# 实时数仓简历素材库

> 从两篇技术文档中提炼的所有可用于简历的素材点，按能力维度分类。每个点标注了简历价值评级（S/A/B/C）和适用场景，所有面试追问均附详细回答。

## 一、架构设计能力

这是最能体现高级别工程师水平的维度，面试官最关注"你为什么这么设计"。

### 1.1 三引擎混合架构设计（S 级）

**素材：** 设计了 Flink 流引擎 + Doris 批引擎 + Blade 状态引擎的混合架构，并制定了基于时效性、数据量级、模型复杂度、状态大小四个维度的引擎选型决策树。

**理由：** 这是整个项目最大的架构亮点。绝大多数团队只用 Flink 或者 Flink+Doris，三引擎混合且有清晰选型标准的非常少见。体现了你对不同计算引擎优劣势的深刻理解，以及面对复杂场景时的架构权衡能力。

**面试追问与回答：**

**Q：三种引擎各自的适用场景？**

A：流引擎（Flink）适合秒级时效、数据量大、状态中等、固定维度去重的场景，典型如业务发展稳定的实时看板。批引擎（Doris2Doris）适合分钟级准实时、数据量中等、模型生产复杂度高、需要灵活维度去重的场景，典型如商业分析的实时宽表——因为 Doris 是纯 SQL 开发，处理 3 个以上流关联、天级时差的复杂 ETL 比 Flink 高效得多，而且去重可以直接 DISTINCT，不需要维护 State。Blade 引擎适合秒级时效、数据量大、状态特别大、业务重要度高能承受较高开发成本的场景，典型如订单相关指标的实时看板——订单的业务生命周期 2-3 天，多流关联的 State 会持续膨胀到千万级，Flink 内部扛不住。

**Q：Blade 引擎解决了什么问题？**

A：解决的是 Flink 有状态计算中 State 过大导致作业不稳定的问题。具体来说，Flink 做多流关联时需要把左右表数据都存在 RocksDB State 里，当业务生命周期长（2-3 天）、关联的流多（5+），State 会膨胀到千万行级别，导致 Checkpoint 制作失败、内存溢出、作业频繁重启。Blade 引擎的思路是把状态外置——Flink 任务变成无状态的，每条流只负责更新 Blade 表中自己对应的字段，关联逻辑由 Blade 表的行级更新来完成。然后通过 Blade 的 Binlog 收集能力，把更新后的数据重新转为流，继续后续的流计算。本质上是把"有状态计算"拆成了"无状态写入 + CDC 重新转流"。

**Q：为什么不用 Cellar/Redis 替代 Blade？**

A：我们对比了四种存储。MySQL 扩展性差，高并发读写性能一般，容易成为瓶颈。Doris 扩展性和读写性能都不错，但不支持 Binlog 收集，没法把存储数据重新转成流——如果用 Doris 存中间结果，下游就变成了批处理而非流计算。Cellar（类似 Redis 的 KV 存储）同样不支持 Binlog 收集。只有 Blade 同时满足三个条件：水平扩展支撑千万级数据、高并发读写、支持 Binlog 收集实现"批转流"。如果是在其他公司没有 Blade，可以用 TiDB + TiCDC 或者 MySQL + Canal 实现类似的效果。

### 1.2 流批切换点设计（A 级）

**素材：** 文档中明确提出了四种"流转批"切换点——从源数据层切换（适合 10 分钟级时效、小数据量、极复杂 ETL）、从事实层切换（5 分钟级、中等数据量）、从轻度汇总层切换（分钟级、较大数据量）、从应用层切换（对外提供 OLAP 服务）。

**理由：** 这个点非常有区分度。大多数人只知道"流批一体"的概念，但很少有人能说清楚在数仓的哪一层做切换、为什么在这一层切。这体现了你对数据流转全链路的掌控力。

**面试追问与回答：**

**Q：怎么判断在哪一层切换？**

A：核心判断依据是两个：时效性要求和模型复杂度。越往上层切换（靠近源数据层），批处理覆盖的链路越长，开发运维越简单，但时效性越差，因为每一层批处理都会叠加延迟。越往下层切换（靠近应用层），流处理覆盖的链路越长，时效性越好，但开发运维成本越高。具体来说：如果需求时效性在 10 分钟以上且 ETL 极其复杂（比如 5+ 流关联、天级时差），直接从 ODS 层就切到批处理，全程 Doris2Doris；如果时效性要求 5 分钟左右，在 Fact 层切换，ODS→Fact 用 Flink 保证基础数据的秒级时效，Fact 之后用 Doris 做聚合；如果时效性要求分钟级但数据量大，在 Topic 层切换；如果需要对外提供灵活的 OLAP 查询，在 App 层切换，前面全部用 Flink，最后写入 Doris 提供查询服务。

**Q：切换后数据一致性怎么保证？**

A：靠两个机制。第一是批流一体化建模——实时和离线用同一套模型定义、同一套字段口径，所以不管在哪一层切换，数据的语义是一致的。第二是实离校验——我们会定期用离线数仓的数据和实时数据做同口径对比，如果发现差异就排查是流处理还是批处理环节出了问题。切换点本身不会引入一致性问题，因为切换的本质只是"同一个模型用不同的引擎来计算"，口径没变。

### 1.3 实时 OLAP 变体架构（A 级）

**素材：** 在服务层采用"实时 OLAP 变体"，将部分聚合、分析、计算下推到 Doris OLAP 引擎承担，使主题层和应用层计算更轻量，从而灵活应对业务分析需求的频繁变更。

**理由：** 这是一个很聪明的架构决策。传统做法是在 Flink 里算好所有指标再写入存储，但业务需求变化快时每次改指标都要改 Flink 任务、重新上线。把部分计算下推到 Doris，利用 OLAP 引擎的灵活查询能力，需求变更时只改 SQL 查询即可，不用动实时任务。

**面试追问与回答：**

**Q：哪些计算适合下推到 OLAP？**

A：适合下推的是"维度灵活、口径经常变"的聚合计算。比如业务方今天想看"按城市+品类"的 GMV，明天想看"按仓库+时段"的 GMV，如果每种组合都在 Flink 里预计算，任务数会爆炸。这类需求适合把明细数据写入 Doris，让 Doris 在查询时做聚合。不适合下推的是"计算逻辑复杂、涉及多流关联"的场景，比如订单和物流状态的关联、跨系统的数据拼接，这些必须在 Flink 里完成，因为 Doris 没有流关联的能力。简单说就是：关联和清洗在 Flink 做，灵活聚合在 Doris 做。

**Q：下推后性能怎么保证？**

A：三个措施。第一，接口层禁止大查询，所有对外查询都有超时限制和扫描行数限制。第二，在业务允许的前提下加缓存，高频查询的结果缓存几十秒到几分钟。第三，Doris 表的模型设计要合理——我们用 Unique 模型保证数据更新的正确性，同时通过合理的分区分桶策略控制单次查询的扫描量。

**Q：和预计算方案的 trade-off？**

A：预计算（在 Flink 里算好所有指标）的优势是查询性能极好（直接读结果），劣势是灵活性差（每加一个维度组合就要新增一个 Flink 任务）和资源消耗大。OLAP 下推的优势是灵活性极好（改 SQL 就行），劣势是查询性能依赖 Doris 集群能力，数据量大时可能慢。我们的策略是：核心指标（访问频率高、时效要求严格的）用预计算，探索性分析（维度组合多变、时效要求宽松的）用 OLAP 下推。大概 70% 的指标走预计算，30% 走 OLAP 下推。

### 1.4 双链路跨机房容灾架构（A 级）

**素材：** 核心任务全链路（Binlog 采集 → Kafka → Yarn → Doris）跨机房主备部署（云谷/兆丰），主备链路保持配置一致、代码版本一致，支持热切换。

**理由：** 实时数仓做到跨机房容灾的团队不多，这体现了对高可用架构的理解。关键亮点是"全链路无单点"——不是只在某一层做备份，而是从数据采集到存储每一层都有独立的备份链路。

**面试追问与回答：**

**Q：主备数据一致性怎么保证？**

A：通过两个手段。第一是"配置一致、代码一致"——主备链路的 Flink 任务代码版本、Kafka topic 配置、Doris 表结构完全相同，由自动化工具同步管理，任何变更必须同时应用到主备两条链路。第二是"主备对比监控"——我们在数据监控体系中有一层"多源校验"，会定期对比主链路和备链路的数据是否一致，如果发现差异会触发告警。正常情况下主备数据是一致的，因为它们消费的是同一份 Binlog 源数据，只是经过不同机房的 Kafka 和 Flink 集群处理。

**Q：切换时有没有数据丢失？**

A：会有短暂的数据不一致窗口，但不会丢数据。因为主备链路都是独立运行的，备链路一直在消费和处理数据，只是正常情况下接口不指向它。切换的时候只需要把接口层的数据源从主 Doris 切到备 Doris，备链路的数据是完整的。可能存在的问题是主备之间有几秒的数据延迟差异（因为两条链路的处理速度不完全同步），但对于我们的业务场景（分钟级刷新的看板）来说，这个差异是可接受的。

**Q：双链路的资源成本怎么控制？**

A：通过任务分级保障来控制。我们把任务分为 S 级和非 S 级，只有 S 级任务才做双链路——这些是直接影响生产调度和资金结算的核心任务，大约占总任务量的 20%。非 S 级任务只跑单链路。这样双链路带来的额外资源开销大约是总资源的 20%，是可控的。同时我们通过任务调优（State 优化、并行度调整）降低单任务的资源消耗，部分抵消了双链路的成本。

## 二、数仓建模能力

数据开发岗的核心硬技能，面试必问。

### 2.1 批流一体化建模体系（S 级）

**素材：** 实时与离线统一主题域划分、统一模型分层（ODS → Fact → Topic → App）、统一命名规范。Fact 层字段定义和口径完全一致，差异只在计算引擎和存储介质。

**理由：** "批流一体"是行业热词，但真正落地的团队很少。你能说清楚怎么做到统一（三个统一），以及这样做的好处（模型复用、实离校验简化），就比大多数候选人强。

**面试追问与回答：**

**Q：实时和离线的模型完全一样吗？有哪些不得不不一样的地方？怎么处理的？**

A：模型定义（字段名、字段含义、口径）是完全一样的，但在三个地方有差异。第一是存储介质不同，离线存 Hive/HDFS，实时存 Kafka（中间层）和 Doris（服务层），这是天然的差异。第二是数据粒度可能不同，离线可以做全量快照，实时一般只处理增量变更（CDC），所以实时的 Fact 层是"变更流"而非"全量表"。第三是部分复杂计算在实时场景下做不了或成本太高，比如全局精确去重（COUNT DISTINCT），离线可以直接算，实时要么用近似算法（HyperLogLog）要么用 State 精确去重但资源消耗大。我们的处理方式是：模型定义层面保持一致，实现层面允许差异但必须在文档中标注清楚，并且通过实离校验来监控差异是否在可接受范围内。

---

### 2.2 四层模型分层设计（A 级）

**素材：** ODS（原始数据接入）→ Fact（清洗过滤关联扩维，星型模型）→ Topic（跨域关联聚合，主题宽表）→ App（面向分析主题，对外服务）+ Dim（离线初始化 + 实时写入，Redis/Tair 存储）。

**理由：** 相比常见的三层（ODS/DWD/ADS），这里多了一个 Topic 主题汇总层，专门做跨域关联和聚合。这个设计决策值得讲——为什么要多加一层？因为物流业务跨多主题分析场景多，如果没有 Topic 层，每个 App 层任务都要自己做跨域关联，重复开发且不一致。

**面试追问与回答：**

**Q：Topic 层和 App 层的边界怎么划分？什么时候该沉淀到 Topic 层？**

A：判断标准是"复用性"。如果一个跨域关联或聚合逻辑被 2 个以上的 App 层任务使用，就应该沉淀到 Topic 层。比如"订单+物流状态"的关联宽表，仓储看板要用、运输看板也要用、管理驾驶舱也要用，这就应该沉淀为 Topic 层的主题宽表，各 App 层任务直接消费它，而不是各自重复做关联。App 层则是面向具体分析场景的最终加工，比如"按城市汇总的日配送完成率"，这个指标只有配送看板用，就放在 App 层。简单说：Topic 层是"公共的跨域数据"，App 层是"专用的业务指标"。

---

### 2.3 维度表实时化方案（B 级）

**素材：** 维度层采用"离线数据初始化 + 实时数据写入"的方式，使用 Redis/Tair 作为维度存储。

**理由：** 实时数仓的维度表处理是个经典难题。离线维度表更新不及时，纯实时维度表又可能丢历史数据。这个"离线打底 + 实时增量"的方案是业界最佳实践，体现你对这个问题的理解。

**面试追问与回答：**

**Q：维度表更新延迟怎么处理？**

A：分两种情况。如果是维度表本身的更新延迟（比如新增了一个仓库，但维度表还没同步），我们通过实时监听维度表的 Binlog 变更来缩短延迟，正常情况下秒级就能同步到 Redis。如果是 Flink 任务关联维度表时维度数据还没到（事实流先到、维度流后到），我们会在 Flink 中做重试——第一次关联不到时先输出一个默认值，同时把这条数据缓存在 State 中，等维度数据到了再更新。对于极少数关联不上的情况，离线数据会在每天凌晨做一次全量刷新兜底。

**Q：缓存一致性怎么保证？**

A：Redis 中的维度数据有两个更新来源：离线全量刷新（每天凌晨）和实时增量更新（Binlog CDC）。一致性靠"实时覆盖离线"的策略——实时更新的数据会直接覆盖 Redis 中的旧值，离线全量刷新时也是全量覆盖。如果出现实时和离线数据不一致的情况，以离线为准（因为离线是全量计算，更可靠），下一次全量刷新会自动修正。

**Q：维度表太大 Redis 放不下怎么办？**

A：我们遇到过这个问题。解决方案是分层存储——高频访问的热维度（如仓库、城市、品类）放 Redis，低频访问的冷维度（如历史供应商信息）放 MySQL，Flink 任务根据维度类型选择不同的数据源做关联。另外对 Redis 中的维度数据设置合理的 TTL，过期的历史维度自动淘汰，需要时从 MySQL 回查。

---

### 2.4 建模思想体系（B 级）

**素材：** 文档总结了七条建模思想——层次精简化、模型宽表化、高内聚低耦合、公共逻辑下沉、单一职责、命名规范清晰、指标统一管理、分离易错点。

**理由：** 这些思想本身不算新，但如果你能结合实际案例讲清楚每一条是怎么落地的（比如"公共逻辑下沉"具体下沉了什么、解决了什么问题），就能体现你不是纸上谈兵。

**面试追问与回答：**

**Q：能举个"公共逻辑下沉"的具体例子吗？**

A：最典型的例子是物流状态的解析逻辑。物流系统有 20+ 个后端系统，每个系统的状态码定义不同，但最终都要映射成统一的物流状态（如"已发货""已到仓""已签收"）。早期每个 App 层任务都自己写一套状态映射逻辑，导致两个问题：一是重复开发，二是不同任务的映射逻辑可能不一致。我们把这个逻辑下沉到 Fact 层，在 Fact 层统一完成状态码到标准物流状态的映射，上层任务直接使用标准状态字段，不再关心原始状态码。这一个改动就消除了十几个 App 层任务的重复逻辑。

## 三、Flink 技术深度

面试官最喜欢追问的技术细节，能体现你是"用过"还是"真懂"。

### 3.1 State 管理与 OOM 治理（S 级）

**素材：** 针对 RocksDB State 堆外内存不受 JVM 管控导致 Container 被 kill 的问题，通过 Flink 1.12 升级（更严格的内存控制）+ JVM Overhead 调大（25%→30%-40%）+ 持久化 Checkpoint 替代 Savepoint（减少磁盘压力）组合解决。

**理由：** 这是 Flink 生产环境中最常见也最棘手的问题之一。能把根因（堆外内存不受 JVM 管控）、排查过程、解决方案讲清楚，直接证明你有大规模 Flink 生产运维经验。

**面试追问与回答：**

**Q：RocksDB 的内存模型是什么？**

A：Flink 使用 RocksDB 作为 State Backend 时，数据存储在本地磁盘上，但 RocksDB 会使用堆外内存（Native Memory）作为 Block Cache（读缓存）和 Write Buffer（写缓冲）。这部分内存不在 JVM 堆内，不受 -Xmx 参数控制，而是直接向操作系统申请。在 Flink 1.9/1.10 版本中，RocksDB 的内存使用几乎是不受限的，当 State 量大时，RocksDB 会申请越来越多的 Native Memory，最终导致整个 Container 的内存超过 Yarn 分配的上限，被 Yarn kill 掉。Flink 1.12 引入了 RocksDB 内存管理机制，通过 state.backend.rocksdb.memory.managed 参数将 RocksDB 的内存纳入 Flink 的 Managed Memory 管理，从而实现了对 RocksDB 内存的上限控制。

**Q：Checkpoint 和 Savepoint 的区别？**

A：两者都是 Flink 状态的快照，但有三个关键区别。第一，触发方式不同：Checkpoint 是 Flink 自动周期性触发的，Savepoint 是手动触发的。第二，存储方式不同：Checkpoint 支持增量模式（只写入自上次 Checkpoint 以来变化的 State 数据），Savepoint 是全量的（每次都写入所有 State 数据）。第三，用途不同：Checkpoint 用于故障恢复（任务挂了自动从最近的 Checkpoint 恢复），Savepoint 用于计划性操作（如升级代码、调整并行度）。在我们的场景中，State 量很大（千万级），全量 Savepoint 会产生巨大的磁盘 IO，所以我们用持久化的增量 Checkpoint 替代定时 Savepoint，磁盘写入量减少了一个数量级。

**Q：增量 Checkpoint 的原理？**

A：增量 Checkpoint 利用了 RocksDB 的 LSM-Tree 特性。RocksDB 的数据存储在多个 SST 文件中，新写入的数据会生成新的 SST 文件，旧的 SST 文件不会被修改（只会在 Compaction 时合并）。增量 Checkpoint 的原理就是：每次 Checkpoint 时，只上传自上次 Checkpoint 以来新生成的 SST 文件，而不是所有 SST 文件。这样每次 Checkpoint 的数据量远小于全量，大幅降低了 Checkpoint 的耗时和磁盘 IO。恢复时，Flink 会从最近的全量 Checkpoint 开始，依次应用后续的增量 Checkpoint，重建完整的 State。

### 3.2 Join 算子深度优化（S 级）

**素材：** 深入到 Flink Join 算子的三种实现——JoinKeyContainsUniqueKey（ValueState，最优）、InputSideHasUniqueKey（MapState，Key 是主键）、InputSideHasNoUniqueKey（MapState，Key 是整行数据，最差）。生产中优先保证关联字段是主键，最大化优化 State 读写效率。

**理由：** 这个深度非常少见。绝大多数 Flink 开发者只知道 Regular Join / Interval Join / Window Join 的区别，但不知道 Regular Join 内部还有三种不同的 State 存储策略。能讲到这个层面，直接拉开和其他候选人的差距。

**面试追问与回答：**

**Q：三种 Join 的 State 大小差异有多大？**

A：差异非常显著。假设左表有 100 万行数据，每行 500 字节。JoinKeyContainsUniqueKey 模式下，State 用 ValueState 存储，每个 Key 对应一行数据，State 大小约 500MB。InputSideHasUniqueKey 模式下，State 用 MapState，Key 是主键（假设 8 字节），Value 是整行数据，额外的 MapState 元数据开销大约 20-30%，State 大小约 600-650MB。InputSideHasNoUniqueKey 模式下，State 用 MapState，Key 是整行数据（500 字节），Value 是计数器（8 字节），Key 本身就很大，State 大小约 1GB+。而且 Key 越大，RocksDB 的查找效率越低（因为比较操作更耗时）。所以从 State 大小和读写效率两个维度看，JoinKeyContainsUniqueKey 都是最优的。

**Q：怎么判断当前任务用的是哪种？**

A：看 Flink 的执行计划（Explain）。在执行计划中，Join 算子会标注 joinType 和 inputUniqueKey 信息。也可以从 SQL 语义来判断：如果 JOIN ON 的字段就是表的主键（PRIMARY KEY），那就是 JoinKeyContainsUniqueKey；如果表有主键但 JOIN ON 的不是主键，就是 InputSideHasUniqueKey；如果表没有声明主键，就是 InputSideHasNoUniqueKey。

**Q：如何改造 SQL 让它走最优路径？**

A：核心是确保 JOIN ON 的字段是表的主键。具体有两个方法。第一，在 DDL 中显式声明 PRIMARY KEY，比如 `CREATE TABLE orders (order_id BIGINT, ..., PRIMARY KEY (order_id) NOT ENFORCED)`，然后 JOIN ON 用 order_id 关联。第二，如果业务上 JOIN 字段确实不是主键（比如用 user_id 关联，但主键是 order_id），可以考虑在上游先做一次按 user_id 的聚合或去重，让输出流的 user_id 具有唯一性，再做 JOIN。实在改不了的情况下，至少要确保表有主键声明，走 InputSideHasUniqueKey 而不是最差的 InputSideHasNoUniqueKey。

### 3.3 Deduplication 替代 ROW_NUMBER 去重（A 级）

**素材：** 文档提到使用 Deduplication 算子替代 ROW_NUMBER 进行去重，性能更优。

**理由：** 这是 Flink SQL 调优的一个实用技巧。ROW_NUMBER 去重在底层会生成 TopN 算子，而 Deduplication 是专门为去重优化的算子，State 更小、性能更好。能说出这个区别，体现你对 Flink SQL 执行计划有研究。

**面试追问与回答：**

**Q：两者在执行计划上的区别？**

A：用 ROW_NUMBER 去重的写法是 `SELECT * FROM (SELECT *, ROW_NUMBER() OVER (PARTITION BY key ORDER BY time) AS rn FROM table) WHERE rn = 1`。Flink 会把它优化为 TopN 算子，TopN 算子需要维护一个排序结构来跟踪每个 Key 的 Top 1 记录，State 中存储的是排序后的数据列表。而 Deduplication 的写法看起来一样，但当 Flink 检测到 ORDER BY 的字段是处理时间（proctime）或事件时间（rowtime），且只取第一条（rn = 1）时，会自动优化为 Deduplication 算子。Deduplication 算子只需要记录每个 Key 是否已经出现过（一个布尔值或第一条记录），State 远小于 TopN。

**Q：Deduplication 的触发条件是什么？**

A：三个条件同时满足：一是 ROW_NUMBER 的 PARTITION BY 字段是去重的 Key；二是 ORDER BY 字段是 proctime 或 rowtime（不能是业务字段）；三是只取 rn = 1（取第一条）或 rn = 1 且是 INSERT ONLY 模式。如果 ORDER BY 的是业务字段（比如 update_time），Flink 无法优化为 Deduplication，只能走 TopN。所以在写去重 SQL 时，如果业务上允许，尽量用处理时间排序。

### 3.4 多流关联方案选型（A 级）

**素材：** 对比了 Window Join、Interval Join、Regular Join 三种方案，结合物流业务特点（关联周期长 2-3 天、不允许丢数），选择了 Regular Join。

**理由：** 多流关联是实时数仓的核心难题。能说清楚三种 Join 的适用场景和 trade-off，体现你的技术选型能力。

**面试追问与回答：**

**Q：三种 Join 各自的优缺点和适用场景？**

A：Window Join 要求两条流的数据在同一个时间窗口内才能关联，窗口外的数据直接丢弃。优点是 State 最小（只保留当前窗口的数据），缺点是会丢数据。适合两条流的数据几乎同时到达的场景，比如同一个事件的不同属性流。

Interval Join 允许两条流的数据在一个时间区间内关联（比如 A 流的数据可以和 B 流前后 10 分钟内的数据关联）。优点是 State 比 Regular Join 小（只保留时间区间内的数据），缺点是超出时间区间的数据关联不上。适合两条流有明确时间先后关系且时间差可预估的场景。

Regular Join 没有时间限制，两条流的数据只要 Key 匹配就能关联，State 中保留所有历史数据（直到 TTL 过期）。优点是不丢数据，缺点是 State 最大。适合关联周期长、不允许丢数据的场景。

我们的物流业务，一个订单从下单到签收的生命周期是 2-3 天，期间会产生多个系统的状态变更事件，这些事件的时间间隔不可预估（可能几秒也可能几小时），而且业务要求不能丢任何一条数据（因为涉及结算）。所以只能选 Regular Join，然后通过 State TTL（设为 3 天）和 Join 算子优化来控制 State 大小。

### 3.5 性能压测方法论（B 级）

**素材：** 上线前做流量回放压测，目标一是确保流量高峰（单表 QPS 4W+，持续 5 分钟的"集单"场景）不出现 Checkpoint 连续失败和长时间反压；目标二是量化任务处理能力（某并行度下单位时间最大处理量），为容量规划提供依据。

**理由：** 大多数团队的实时任务是"上线了再说"，有系统化压测机制的不多。这体现了工程化思维。

**面试追问与回答：**

**Q：压测具体怎么做的？**

A：分三步。第一步，准备压测数据——从线上 Kafka topic 中录制一段包含流量高峰的数据（通常录制"集单"时段的数据），存到一个独立的 Kafka topic 中。第二步，搭建压测环境——在独立的 Yarn 队列上部署待测任务，消费压测 topic 的数据，下游写入独立的 Doris 测试集群，避免影响生产环境。第三步，执行压测——以不同的回放速率（1x、2x、5x）回放数据，观察任务的 Checkpoint 成功率、反压情况、吞吐量、延迟等指标。如果 2x 速率下任务稳定运行，说明有 2 倍的容量余量，可以上线；如果 1x 就出现问题，需要调优或增加并行度。

**Q：压测中关注哪些指标？**

A：四个核心指标。Checkpoint 成功率（必须 100%，连续失败说明任务有严重问题）、反压（Backpressure，高反压说明处理能力不足）、吞吐量（每秒处理的记录数，用来计算容量）、端到端延迟（从数据产生到写入 Doris 的时间）。

---

### 3.6 Blade 引擎状态外置方案（B 级）

**素材：** 将 Flink 的中间计算状态外置到 Blade 数据库，利用 Blade 的水平扩展能力解决千万级大状态问题。多流关联时每条流只更新 Blade 表的部分字段，不再在 Flink 内部保存状态。再通过 Blade Binlog 收集将存储数据重新转为流。

**理由：** 这是一个非常有创意的方案——把"有状态计算"变成"无状态计算 + 外部存储"。虽然 Blade 是内部组件，但这个思路可以迁移到任何支持 CDC 的数据库（如 MySQL + Canal、TiDB + TiCDC），面试时可以抽象来讲。

**面试追问与回答：**

**Q：这个方案的具体实现步骤是什么？**

A：四步。第一步，在 Blade 中预建模型表，表结构就是最终关联后的宽表结构，每个字段对应某条流的数据。第二步，每条 Flink 流独立处理，做完清洗转换后，只更新 Blade 表中自己对应的字段（比如订单流更新订单相关字段，物流流更新物流相关字段），Flink 任务本身不保存任何 State。第三步，Blade 表通过 Binlog 收集能力，将每次行更新事件发送到 Kafka，重新转化为数据流。第四步，下游 Flink 任务消费这个 Binlog 流，做最终的清洗和聚合，输出到 Doris 或 Kafka 供应用层使用。

**Q：这个方案有什么缺点？**

A：两个主要缺点。第一是开发成本高——需要额外维护 Blade 表、Binlog 收集配置、下游解析任务，整体链路比纯 Flink 方案复杂。第二是端到端延迟会增加——数据要经过"Flink → Blade → Binlog → Kafka → Flink"的链路，比纯 Flink 内部处理多了几次网络 IO 和存储 IO，延迟大约增加 1-3 秒。所以这个方案只用于 State 特别大、纯 Flink 方案扛不住的核心场景，不是所有任务都用。

## 四、数据质量治理能力

越来越多的公司把数据质量作为数据团队的核心 KPI，这个维度的素材很有价值。

### 4.1 全生命周期质量保障体系（S 级）

**素材：** 建立了"故障预防 → 故障发现 → 故障恢复 → 故障改进"四环节闭环体系。

**理由：** 这是一个完整的方法论，不是零散的优化点。面试时能画出这个框架，再往每个环节里填具体措施，非常有说服力。体现你不只是"救火"，而是有体系化的治理思维。

**面试追问与回答：**

**Q：四个环节分别做了什么？**

A：故障预防——三个方面：改善集群环境（Yarn 控负载、Kafka 隔离、Doris 治理）、任务性能评估和调优（上线前压测、State 优化、Join 优化）、精细化任务管理（任务分级、开发规范、Code Review）。故障发现——三层监控：集群异常发现（Yarn/Kafka/Doris 的资源和性能指标监控）、任务异常发现（Flink 任务的 Checkpoint 失败、反压、延迟告警）、数据异常发现（多源校验、实离校验、波动校验）。故障恢复——S 级任务 3 分钟发现、6 分钟恢复，依赖双链路热切换和标准化的故障处理 SOP。故障改进——每次故障后做复盘，分析根因，沉淀到自动化检查工具和开发规范中，避免同类问题再次发生。

---

### 4.2 三层数据监控体系（A 级）

**素材：** 多源校验（不同分层间数据对比 + 主备对比）、实离校验（实时与离线同口径对比）、波动校验（同比环比 + 断流监控）。

**理由：** 三层监控覆盖了数据质量的三个核心维度——一致性（多源校验）、准确性（实离校验）、及时性（波动校验）。

**面试追问与回答：**

**Q：实离校验的延迟怎么处理？**

A：实时数据是持续产出的，离线数据是 T+1 产出的，两者的时间点不同。我们的做法是：每天离线数据产出后（通常是凌晨 3-4 点），取前一天的实时数据快照和离线数据做对比。对比的口径是完全一致的（因为批流一体化建模），对比的粒度是核心指标的汇总值（比如总订单数、总 GMV），允许一定的误差范围（通常 0.1% 以内）。如果超出误差范围，触发告警，人工排查是实时链路还是离线链路的问题。

**Q：波动校验的阈值怎么设定？误报怎么处理？**

A：阈值分两种。同比阈值：和上周同一天同一时段对比，波动超过 30% 触发告警（这个值是根据历史数据的正常波动范围统计出来的）。环比阈值：和前一个时段对比，波动超过 50% 触发告警。断流监控：连续 5 分钟无数据更新触发告警。误报的处理：第一，阈值会根据业务特点动态调整，比如大促期间放宽阈值；第二，告警分级，同比波动 30-50% 是预警（大象通知），50% 以上是严重告警（电话通知）；第三，建立了告警确认机制，收到告警后 10 分钟内必须确认是真实问题还是误报，如果是误报就标记并反馈给阈值优化。

---

### 4.3 任务分级保障机制（A 级）

**素材：** 将任务分为 S 级和非 S 级，制定差异化 SLA。

**理由：** 资源永远是有限的，分级保障体现了资源分配和优先级判断能力。

**面试追问与回答：**

**Q：怎么判断一个任务是 S 级还是非 S 级？**

A：两个判断标准。第一是业务影响面——如果任务故障会直接影响一线生产（比如仓库收发货调度依赖的实时数据）或涉及资金结算（比如供应商结算指标），就是 S 级。第二是用户敏感度——如果数据的使用方是外部用户（如供应商、团长）而非内部分析师，对数据中断的容忍度更低，也应该定为 S 级。实际操作中，我们和业务方一起评审，最终 S 级任务约占总任务量的 20%。

**素材：** 从"coding 时长占比过高"到建立完整开发流程：技术方案评审 → 开发 → Code Review → 性能压测 → 数据验证 → 监控配置 → 上线。

**理由：** 体现工程管理意识。

### 4.5 实时 DQC 建设（B 级）

**素材：** 建设了实时数据质量检查（DQC）体系。

**理由：** 实时 DQC 比离线 DQC 难度大很多。

**面试追问与回答：**

**Q：实时 DQC 和离线 DQC 有什么区别？怎么实现的？**

A：离线 DQC 是在数据产出后做检查（比如 Hive 表产出后跑一个校验 SQL），数据是静态的，检查逻辑简单。实时 DQC 的数据是持续流动的，不能等"产出完"再检查，需要在数据流动过程中实时检测。我们的实现方式是：在 Flink 任务中埋入 DQC 算子，对流经的数据做规则检查（比如字段非空、值域范围、主键唯一性），异常数据写入独立的告警 topic，由监控系统消费并触发告警。另外在 Doris 层面，通过定时查询（每 5 分钟）做聚合级别的校验（比如最近 5 分钟的数据量是否在正常范围）。

## 五、集群运维与性能调优

体现你不只是写代码，还能管好线上环境。

### 5.1 Yarn 集群调优（A 级）

**素材：** 三个关键措施——控制集群资源使用率 80% 以内、多磁盘部署分散 State IO 压力、持久化 Checkpoint 替代 Savepoint 降低磁盘写入。

**理由：** 实打实的生产经验。

**面试追问与回答：**

**Q：为什么控制在 80% 而不是 90% 或 70%？**

A：80% 是我们根据实际经验得出的平衡点。低于 70% 资源浪费严重，成本不可接受。高于 85% 时，集群在流量高峰（如"集单"场景）容易出现资源争抢，导致任务 Checkpoint 超时或 TaskManager 启动缓慢。80% 留出了约 20% 的余量来应对流量波动和突发情况（如某个任务需要重启、新任务上线等）。这个值不是固定的，大促期间会提前扩容，把使用率降到 60-70%。

**Q：多磁盘部署具体怎么做的？**

A：Yarn 集群的每台机器配置了多块 SSD 磁盘（通常 4-8 块），Flink 的 State 目录配置为多个磁盘路径（通过 `io.tmp.dirs` 或 `state.backend.rocksdb.localdir` 配置多个路径），Flink 会自动将不同 Subtask 的 State 数据分散到不同磁盘上，避免单块磁盘 IO 成为瓶颈。这个优化对 IO 密集型任务（大量 State 读写）效果非常明显，单磁盘 IO 利用率从 90%+ 降到 40-50%。

---

### 5.2 Kafka 集群治理（B 级）

**素材：** 集群物理隔离（物流业务独立 Kafka 集群）、定时扫描 topic partition 数据量避免倾斜。

**面试追问与回答：**

**Q：为什么要做集群隔离？**

A：物流业务的数据量不大但对可靠性要求极高（涉及结算），如果和流量类业务（如日志、埋点）共用 Kafka 集群，流量类业务的突发流量可能导致 Kafka broker 负载过高，影响物流数据的消费延迟。独立集群可以保证物流数据的 SLA 不受其他业务影响。

**Q：partition 倾斜怎么发现和处理的？**

A：我们写了一个定时任务，每小时扫描所有 topic 的各 partition 数据量，如果某个 partition 的数据量超过平均值的 3 倍，触发告警。处理方式通常是调整上游的分区策略——检查生产者的 Key 选择是否合理，如果某个 Key 的数据量特别大（热点 Key），考虑加盐打散或者调整分区数。

---

### 5.3 Doris 集群治理（A 级）

**素材：** 四个措施——集群隔离、接口层禁止大查询 + 使用缓存、Doris2Doris 任务错峰执行（时间分桶）、限制 Doris 开放。治理后 TP95 从 2500ms 降至 1300ms，磁盘 IO 从 70% 降至 30%。

**面试追问与回答：**

**Q：Doris2Doris 错峰执行具体怎么做的？**

A：我们有 50+ 个 Doris2Doris 定时任务，早期都配置为每 10 分钟执行一次，导致每个整 10 分钟（00、10、20...）会有大量任务同时启动，Doris 集群的事务数瞬间飙升，响应变慢甚至超时。解决方案是时间分桶——把 10 分钟分成 10 个 1 分钟的桶，每个任务随机分配到一个桶中，比如任务 A 在第 0 分钟执行，任务 B 在第 3 分钟执行，任务 C 在第 7 分钟执行。这样同一时刻最多只有 5-6 个任务在跑，集群压力平滑了很多。

**Q：接口层的缓存策略是什么？**

A：根据数据的更新频率和业务容忍度设置不同的缓存时间。实时看板类接口（每分钟刷新）缓存 30 秒，分析报表类接口（每小时刷新）缓存 5 分钟。缓存用 Redis 实现，Key 是查询 SQL 的 hash，Value 是查询结果。缓存失效策略是 TTL 过期自动失效，不做主动失效（因为实时数据本身就在持续更新，短暂的缓存不一致是可接受的）。

---

### 5.4 Doris2Doris 批处理的性能边界（B 级）

**素材：** 百万级数据量 ETL 约 1 分钟，五千万级约 5 分钟。三个瓶颈：分钟级延时、大数据量读写性能瓶颈、集群计算能力短板。

**面试追问与回答：**

**Q：超过五千万级数据量怎么办？**

A：两个方向。第一是优化 Doris 表设计——合理的分区分桶策略可以大幅提升 ETL 性能，比如按天分区、按业务 Key 分桶，让 ETL 只扫描需要的分区。第二是切换引擎——如果数据量确实很大且时效性要求高，就不适合用 Doris2Doris 了，应该切到 Flink 流处理。我们也在探索用 Spark SQL 引擎来做 Doris 上的计算，利用 Spark 的分布式计算能力来缓解 Doris 自身的计算压力。

## 六、开发效率提升

体现你不只是完成需求，还在思考怎么让团队更高效。

### 6.1 FlinkSQL 零 Java 代码开发（A 级）

**素材：** 拥抱 NAU 实时开发平台，实现实时开发零 Java 代码，全部使用 FlinkSQL。

**面试追问与回答：**

**Q：FlinkSQL 有什么场景是表达不了的？怎么处理的？**

A：主要有三类场景。第一是复杂的自定义逻辑——比如需要调用外部 HTTP 接口做数据补全，SQL 表达不了，这种情况用 UDF（User Defined Function）封装，在 SQL 中调用 UDF。第二是精细的 State 控制——比如需要自定义 State 的序列化方式或清理策略，SQL 层面控制不了，但我们通过合理设置 State TTL 和选择合适的 Join 类型来规避。第三是非标准的数据源/目标——比如需要写入一个不支持 Flink Connector 的存储，这种情况开发自定义 Connector。总体来说，95% 以上的需求可以用纯 SQL + UDF 解决，极少数需要写 Java 代码。

---

### 6.2 代码复用与模板化开发（B 级）

**素材：** 发现两条复用规律，建立模板化开发体系。

**面试追问与回答：**

**Q：模板化具体怎么做的？**

A：我们把 Flink SQL 任务拆成三个部分：Source DDL（数据源定义）、Transform SQL（转换逻辑）、Sink DDL（目标定义）。其中 Source DDL 和 Sink DDL 是高度可复用的——同一个 Kafka topic 的 Source DDL 在所有消费它的任务中都一样，同一个 Doris 表的 Sink DDL 也一样。我们把这些 DDL 做成模板，存在配置中心，新建任务时只需要选择 Source 和 Sink 模板，然后写 Transform SQL 就行。Transform SQL 中也有可复用的部分，比如状态码映射、时间格式转换等通用逻辑，封装成 UDF 库。这样一个新需求的开发时间从平均 3 天缩短到 1 天。

---

### 6.3 统一数据服务平台（B 级）

**素材：** 针对服务层口径管理难、取数模块与数仓耦合不清的问题，搭建了统一数据服务平台。

**面试追问与回答：**

**Q：统一数据服务平台解决了什么问题？**

A：解决了三个问题。第一是口径不一致——同一个指标（比如"日配送完成率"），不同的数据产品可能用不同的计算口径，导致数据打架。统一平台把指标的定义、口径、计算逻辑集中管理，所有数据产品从同一个出口取数。第二是取数逻辑和数仓耦合——早期每个数据产品都直接写 SQL 查 Doris，SQL 散落在各个服务代码中，数仓表结构一变，所有下游都要改。统一平台提供 API 接口，下游通过 API 取数，数仓变更只需要改平台内部的映射，对下游透明。第三是权限管理——统一平台可以控制谁能访问哪些指标，避免敏感数据泄露。

## 七、业务理解能力

### 7.1 物流全链路业务理解（A 级）

**面试追问与回答：**

**Q：能用 2-3 句话讲清楚这个业务吗？**

A：这是一个社区电商的物流履约系统。用户在平台下单后，商品从供应商的共享仓出发，经过中心仓（区域分发中心）分拣，运输到网格站（社区级分拣点），最后由团长（社区自提点）交给用户。整个流程是"当天售卖、夜间作业、次日白天履约"，所以物流作业主要发生在夜间，常规的离线数仓只能看到 T-2 的数据，业务对实时数据的需求非常迫切。

**Q："集单"是什么？为什么对实时系统影响大？**

A："集单"是指在一天中的几个固定时间点（通常是晚上），系统把用户当天所有订单汇总计算后，批量生成调拨单下发给仓库。这个动作会在几分钟内对数据库做大量批量写入，产生海量 Binlog 消息，单表 QPS 可达 4W+。对实时系统的影响是：Kafka 瞬间涌入大量消息，Flink 任务如果处理能力不足就会产生反压和延迟，严重时 Checkpoint 失败导致任务重启。所以我们必须在上线前做压测，确保任务能扛住集单高峰。

---

### 7.2 数据直接参与生产决策（A 级）

**面试追问与回答：**

**Q：数据参与生产决策具体是什么意思？能举个例子吗？**

A：举两个例子。第一个是仓库收货调度——中心仓的工作人员需要实时看到"当前有多少车辆在途、预计几点到达、每辆车装了什么货"，根据这些数据安排收货人员和月台。如果实时数据延迟或错误，收货人员可能空等或者来不及安排，直接影响整个仓库的作业效率。第二个是供应商结算——供应商的结算金额是根据实际收货数量和质量实时计算的，数据错误直接导致多付或少付，这是真金白银的资损。所以我们的实时数据不是"看看就好"的 BI 报表，而是直接驱动一线生产动作和资金流转的关键系统。

## 八、未来规划与技术前瞻（加分项）

### 8.1 Flink ETL 语义检测工具（B 级）

**面试追问与回答：**

**Q：这个工具要解决什么问题？**

A：解决的是 FlinkSQL 上线前的语义正确性验证问题。目前 FlinkSQL 的验证主要靠人工 Code Review 和测试环境跑数据，但有些语义错误很隐蔽——比如 Join 条件写错导致数据膨胀、聚合粒度不对导致指标偏大、State TTL 设置不合理导致数据丢失。这些问题在测试环境可能因为数据量小而暴露不出来，上线后才发现。语义检测工具的思路是：解析 FlinkSQL 的执行计划，基于规则引擎自动检测常见的语义问题（如 Join 是否可能产生笛卡尔积、聚合字段是否包含主键、State TTL 是否覆盖业务生命周期等），在上线前给出风险提示。

### 8.2 接口自动切换（B 级）

**面试追问与回答：**

**Q：自动切换和手动切换的区别？为什么要做自动化？**

A：手动切换的流程是：监控告警 → 值班人员收到告警 → 判断是否需要切换 → 登录系统执行切换操作，整个过程最快也要 3-5 分钟。自动切换的思路是：监控系统检测到主链路数据异常（如连续 1 分钟无数据更新或数据量骤降 90%）后，自动将接口层的数据源从主 Doris 切到备 Doris，同时发送通知给值班人员。这样可以把恢复时间从 3-5 分钟缩短到 1 分钟以内。之所以要做自动化，是因为实时数据的故障窗口很短——对于指导现场生产的数据，3 分钟的中断就可能导致仓库作业混乱。
