# 供应链自我介绍
这也是从财务接过来的项目，背景是，我们要统一处理13个子系统的采购、库存、设备等业务数据，用Flink做复杂去重和乱序过滤，依赖Redis存储状态和时间戳，维表也散落在不同的外部系统，导致Flink任务状态很大，Checkpoint 经常膨胀，恢复慢，而且运维成本非常高。

 最核心的思路是把计算和状态下沉到存储层，引入了Apache Paimon作为统一的流式数据湖。首先，我们用Flink CDC把13个子系统的Binlog实时写入Paimon主键表，利用Paimon的`deduplicate`合并引擎，天然实现主键去重，完全替代了Flink里的ROW_NUMBER逻辑；同时配置`sequence.field`，基于数据自带的时间字段来判断事件顺序，解决乱序问题，不再需要Redis做时间戳比较。这样Flink任务就变成了无状态的纯传输，Checkpoint大小一下子下降了90%，任务恢复速度提升了70%，整体吞吐量提升了约40%，而且做到了端到端的Exactly-Once，不再依赖外部组件。

维表我们也全部迁移到Paimon，下游通过Lookup Join实时关联，避免了维护多套存储。基于这套统一的Changelog流，我们做了几件业务价值很高的事：采购执行监控，直接消费Paimon变更流，对交期延迟、到货差异、质检异常等做实时预警，还能利用Time Travel按时间点回溯订单状态，彻底改变了原来T+1报表才发现问题的模式；库存实时水位，我们用Paimon的聚合表，定义SUM、LAST_NON_NULL等聚合函数，让库存增量写入时自动计算实时水位，Flink端彻底无状态，Checkpoint又降了90%，下游消费Changelog就可以触发补货和积压预警。

在上层，我们按指标域在DWS层用聚合表统一计算采购履约率、库存周转率等跨域指标，再同步到Doris做OLAP，Spark SQL每天直接修复Paimon表数据，修数从小时级缩短到了分钟级。同时，我主导建设了写入、加工、产出三层数据质量校验体系，实离对账差异率控制在万分之五以内，并通过分场景的Spark修数及Flink时间戳恢复SOP，让修数资源消耗降低了80%。

最终，整个链路组件从7类收敛到4类，Flink去重和聚合逻辑全部下沉到Paimon，实现了无状态化，故障恢复速度提升70%，核心链路稳定性达到99.9%，并配套了3-10-15分钟的分级故障响应机制。通过这个项目，我们不仅大幅提升了供应链数据的实时性和可靠性，也为公司后续的实时数据湖建设沉淀了可复用的方法论。



# 数据治理
我所在的项目是企业数据智能平台，覆盖 HR、CAP、采购、办公、行政、法务等多条业务线，数据量增长很快。但原有数仓体系存在几个核心问题：数据找数难、指标口径经常打架、T+1 生产经常 delay、OLAP 查询不稳定，直接影响了数据交付质量和下游业务。
23 年团队对标行业标杆，引入了 0987 价值评估体系。我在这个背景下负责数仓治理，核心就做一件事：从元数据、指标口径、SLA、OLAP 稳定性、研发效能五个维度，搭一套体系化的治理闭环，
核心是解决 质量、效率 、体验问题。


五个方向我快速讲一下核心思路。
**元数据体系是底座。** 当时最大的痛点是"找数难"，我设计了业务域、技术域、管理域三域模型，落地了血缘、DQC、SLA、权限这些核心模块，让数据资产从"看不见"变成"可检索、可追溯" 治理效果可量化

**指标一致性治理，核心是统一口径 + 工具化。** 6000 多张表，指标口径对不上，我的做法是先把命名、定义、取数来源、链路加工这四个维度的口径拉齐，然后做成自动化检查工具，不再靠人工巡检。

**SLA 准点率，关键是从单点救火变成全链路治理。** 之前是哪个任务 delay 了就去扑火，我搭了一个"观测—提取—归因—长效机制"的闭环：先让全链路每个节点可观测，再自动提取瓶颈节点做归因，最后把经验沉淀成机制。

**Doris 集群稳定性，思路是全生命周期防控。** 事前卡控高危 SQL 变更，事中做自适应限流和大查询熔断，事后做慢查询归因和红黑榜晾晒，逼着研发改代码。核心是让单点问题不扩散成集群故障。

**研发效能这块**，主要是把常用提效工具基于 NoCode 平台化，同时做了 SQL Copilot 的调研和推广，帮团队用 AI 辅助开发。

### 收尾：效果 + 钩子（约 1 分钟）

整体效果上，几个关键数字：指标一致率从 89.1% 到 97%，SLA 提前 2 小时达成 99%、准点 99%，研发交付时长从 6.88pd 降到 5.5pd，效率提升 20%。Doris 那边在业务量持续增长的情况下，TP95 耗时稳住了，没有因为量涨而劣化。

如果展开聊的话，我觉得最有意思的是两块：一个是 SLA 的全链路归因体系，怎么从几千个任务里自动定位到真正的瓶颈节点，这个方法论后面也复用到了其他场景；另一个是 Doris 的事前卡控机制，怎么在"不误杀正常变更"和"拦住高危操作"之间找平衡，这个踩了不少坑。您对哪个方向比较感兴趣，我可以展开讲。

1、实时链路和离线链路口径怎么保持一致？”
**最开始就是朴素方案——统一指标字典加每天对数。** 实时 Flink SQL 写一套，离线 Hive SQL 写一套，靠指标字典约束口径，靠 T+1 对数任务发现差异。但说白了，**两套代码就注定会漂移**，对数只是事后补救。

**后来引入 Paimon 才真正解决了，核心就是流批同源。** binlog 通过 Flink 写入 Paimon 主键表，去重、乱序、聚合全交给 Paimon 引擎处理。实时流读 changelog，离线批读快照，**读的是同一张表，加工逻辑只有一套**，口径天然一致，不需要对数了。

**一句话说就是：口径不一致的根源是两套代码，解法是流批同源，让两边读同一份存储。*
