# 简历修改说明、底层逻辑与推荐改法

## 一、改写目标

本次输出两份用途不同的简历，而不是把同一份简历简单换标题：

1. **大厂版：** 面向高级数据研发、数仓架构、数据治理及 P7/P8 候选岗位，重点证明复杂度、技术决策、系统性治理和跨团队影响。
2. **小厂版：** 面向高级数据开发、数仓负责人、数据平台核心研发或小型数据团队技术负责人，重点证明能亲自开发、独立交付、解决生产问题和根据成本选择合适方案。

两版均遵循真实性原则：没有可靠基线、口径或业务采用证据的结果，不继续使用“彻底解决、显著提升、极大降低、有效避免”等宣传性词语。

## 二、原简历的核心问题及修改原因

### 1. 缺少职业摘要，筛选者无法在10秒内判断价值

**原问题：** 简历从求职意向直接进入教育经历，最重要的采购、实时、治理成果被埋在后文。

**修改逻辑：** 招聘者首先判断“候选人是谁、能解决什么问题、达到什么规模、结果如何”。因此在头部加入4条职业摘要，使用“年限 + 场景 + 规模 + 能力 + 结果”组合。

**推荐改法：**

- 大厂版突出：复杂业务域、架构、治理指标、技术栈深度。
- 小厂版突出：全链路交付、亲自开发、运维排障、轻量选型。

### 2. 工作职责过于通用，无法体现成长路径

**原问题：** 多家公司都使用“指标体系建设、数据价值、数据治理”等相似话术，早期经历也被包装成高级治理，可信度下降。

**修改逻辑：** 工作经历应呈现明确成长路径：

> 工程开发与任务维护 → 平台和实时链路 → 独立业务域建设 → 架构与治理。

**推荐改法：**

- 谷沃科技：突出采集、离线数仓、任务维护和工程基本功。
- 零零柒科技：突出平台搭建、实时解析、HBase优化和生产排障。
- 纬致芯创：突出复杂业务域、实时架构、指标治理和数据产品。

### 3. “负责、主导、从0到1”过多，但责任边界不足

**原问题：** 高频使用高责任词，却没有交代团队规模、本人决策内容、亲自编码范围及最终结果。

**修改逻辑：** 每个高责任词都应尽量具备四类证据：

> 范围 + 决策 + 动作 + 结果。

**推荐改法：**

不要只写“主导采购数据融合”，应写为：

> 负责12个系统的数据接入方案、公共维度及核心明细模型建设，设计汇率、物料、供应商和组织映射规则，明细覆盖率达到82.3%。

后续若能确认团队规模，可继续补充“协调X个团队 / 指导X名开发 / 评审X个模型”。

### 4. 技术名词很多，技术证据不够

**原问题：** Hive、Spark、Flink、Kafka、Paimon、Doris、HBase、Milvus、Agent 同时出现，但不少组件只证明“用过”，没有说明为什么选、如何设计、解决了什么边界问题。

**修改逻辑：** 高级岗位不以技术词数量定级，而以“问题识别—方案比较—技术决策—工程结果—遗留问题”定级。

**推荐改法：**

- Flink/Paimon：补去重、乱序、状态下沉、Checkpoint、恢复速度和流批同源。
- Doris：补资源隔离、限流、熔断、慢查询归因及 P95 结果。
- HBase：补 RowKey、MOB、Region、GC、写入阻塞根因及优化前后指标。
- Agent：补权限边界、SQL安全、资源限制、成功率、用户量和节省时间。

### 5. 业务结果大量使用宣传性语言

**原问题：** “显著提升”“精准洞察”“有效避免损失”“赋能转型”等表达无法验证。

**修改逻辑：** 结果证据按强弱排序：

1. 收入、成本、风险金额；
2. 时延、吞吐、稳定性、资源、交付周期；
3. 使用人数、调用次数、覆盖率、迁移数量；
4. 制度、发布卡点、统一出口等可验证机制；
5. 主观形容词。

**推荐改法：** 优先使用前四类证据；若没有结果数据，描述可验证的产物和采用范围，不虚构收益。

### 6. 内部术语和汇报式框架影响外部理解

**原问题：** 0987、魔数、Pocket、规范性/完整性/准确性/实时性/产品化等表达更像内部汇报，不适合招聘快速阅读。

**修改逻辑：** 外部简历必须脱离公司上下文仍可理解。

**推荐改法：**

- “魔数看板”改为“BI看板”；
- “Pocket”改为“业务分析系统”；
- 删除“0987”或改为“数据价值评估体系”；
- 项目职责统一使用“问题—方案—结果”，不使用咨询式章节堆叠。

### 7. 文字质量与数据质量岗位形象冲突

**原问题：** 存在“数据数据质量”“工具化落”“处理理”、重复编号、日期空格不统一、项目标题粘连等问题。

**修改逻辑：** 候选人强调 DQC 和规范建设时，简历本身就是最直接的质量样本。

**推荐改法：** 统一日期、标点、编号、公司名称、技术名词和指标写法；删除“学信网可查”和大段自我评价。

## 三、大厂版的改写逻辑

### 1. 核心定位

> 能负责复杂业务域，能做技术方案和关键决策，能通过架构、工具和机制解决系统性问题。

### 2. 信息排序

1. 职业概述：先证明职级潜力；
2. 核心能力：按能力域而非关键词堆砌；
3. 工作经历：证明成长路径；
4. 核心项目：采购 → 企平治理 → 培训 → 反诈 → 保险；
5. 教育经历：放末尾。

### 3. 内容取舍

- 强化复杂度：12系统、6000+表、百亿级数据、300+指标。
- 强化技术决策：Paimon状态下沉、Doris全生命周期治理、SLA关键路径治理。
- 强化组织机制：规范、工具、发布流程、统一出口。
- 弱化“什么都能做”的多面手形象，避免被判断为广而不深。
- 保留未经补全的关键指标时使用克制措辞，不将平台整体成果全部归因到个人。

### 4. 仍需补强的 P7/P8 证据

- 项目总人数、本人带人或评审范围；
- 数据日增、峰值吞吐、端到端延迟；
- Doris P95 优化前后值和故障率；
- HBase 写入吞吐、查询延迟、故障次数；
- Agent 用户数、调用量、成功率和节省人时；
- 至少3个架构方案比较与取舍案例；
- 一个失败、回滚和复盘案例。

## 四、小厂版的改写逻辑

### 1. 核心定位

> 能亲自写代码、独立完成数据链路、理解业务、保障生产，并能在人员和成本有限时做合适选型。

### 2. 信息排序

1. 职业概述先回答“来了能做什么”；
2. 技能覆盖开发、平台、治理、业务和运维；
3. 工作经历强调亲自开发和故障处理；
4. 项目描述使用直观业务语言；
5. 保留采购、培训、反诈三种互补场景。

### 3. 内容取舍

- 强化 SQL、脚本、任务开发、调度、监控和排障；
- 强化从需求到上线的完整交付；
- 强化技术广度，但明确核心方向仍是数据研发与数仓；
- 增加“不照搬大厂重型架构”的成本意识；
- 弱化复杂治理术语、内部平台名和过重流程；
- 用业务方容易理解的语言解释数据建设结果。

### 4. 仍需补强的小厂负责人证据

- 项目周期和分期上线计划；
- 团队规模、任务拆分、Code Review 和新人指导；
- 云资源或机器成本优化；
- 一次凌晨故障定位与恢复案例；
- 只有1—3名开发时如何设计最小可用架构；
- MySQL/PostgreSQL、Doris、Hive、Flink 的选型边界。

## 五、推荐改法 1：稳妥投递版

### 适用目标

- 大厂高级数据开发 / P6+—P7；
- 中型公司高级数据研发；
- 小厂核心数据开发。

### 改法

1. 使用当前两份简历的克制表述，不新增无法验证的数据。
2. 岗位匹配时仅调整求职方向和项目排序。
3. 大厂以采购、治理、反诈为前三重点；小厂以采购、反诈、培训为前三重点。
4. 面试前统一准备指标口径，确保每个百分比都能解释分子、分母和统计周期。

### 优点

- 真实性和稳定性最高；
- 不容易在背景调查或技术追问中出现矛盾；
- 已足以支持大多数高级数据研发岗位。

### 局限

- 对顶级大厂 P8 或小厂数据负责人岗位，组织影响力和成本证据仍偏弱。

## 六、推荐改法 2：结果强化版

### 适用目标

- 大厂 P7；
- 高薪高级研发；
- 供应链、企业服务、实时平台重点岗位。

### 改法

回溯监控、项目文档和周报，为每个重点项目补齐至少一组前后对比：

1. **采购实时链路：** 日处理量、峰值 TPS、端到端 P95/P99 延迟、Checkpoint 大小、恢复时间。
2. **Doris：** 查询 P95、失败率、大查询数量、CPU/内存峰值、扩容或成本情况。
3. **HBase：** 写入 TPS、查询延迟、GC 时间、Region 热点、阻塞次数。
4. **数据质量：** 告警准确率、误报率、平均发现时间、平均恢复时间。
5. **Agent：** 月活用户、月调用量、取数成功率、人工复核率、单次分析节省时间。
6. **业务采用：** 报表访问人数、核心决策场景、迁移报表数量、人工流程减少量。

### 推荐句式

> 针对【问题】，在【约束】下选择【方案】，通过【关键动作】实现【前后对比结果】，并沉淀【机制/复用能力】。

### 优点

- 最能提升面试通过率和定级；
- 技术价值与业务价值都有证据；
- 可将当前“强 P6+/初阶 P7”形象提升为稳定 P7。

### 局限

- 必须从真实记录中恢复数据，不能依靠记忆估算或编造。

## 七、推荐改法 3：岗位定制版

### 适用目标

- 同时投递大厂、小厂、供应链、AI或实时计算岗位；
- 希望提高简历与 JD 的关键词和职责匹配度。

### 改法

为不同岗位维护母版，不是维护十份完全不同的简历。每次投递只调整以下四处：

1. **求职方向：**
   - 大厂数仓：高级数据研发 / 数仓架构；
   - 治理岗位：数据治理 / 数据资产；
   - 小厂：高级数据研发 / 数仓负责人；
   - AI公司：数据工程 / Data Agent 数据基础设施。
2. **职业摘要：** 用 JD 中最重要的3项能力重排摘要。
3. **项目顺序：**
   - 供应链公司：采购 → 治理 → 保险；
   - 实时平台：反诈 → 采购实时 → Doris；
   - AI公司：采购 Agent → 采购底座 → 元数据治理；
   - BI/传统数仓：采购 → 培训 → 保险。
4. **项目子弹：** 每个重点项目保留4—5条，其余压缩；让第一页至少出现 JD 最关心的两项能力。

### 优点

- 对 ATS 和人工筛选都更友好；
- 不需要修改事实，只调整证据顺序；
- 最适合跨类型公司求职。

### 局限

- 需要建立投递记录，避免把错误版本发给不同公司。

## 八、使用建议

### 大厂版建议岗位

- 高级数据研发工程师；
- 数仓架构师；
- 数据治理工程师/专家；
- 实时数据研发；
- 数据平台核心研发。

### 小厂版建议岗位

- 高级数据开发工程师；
- 数仓负责人；
- 数据平台核心研发；
- 数据技术负责人；
- 3—5人数据团队技术骨干。

### 不建议直接使用的岗位名称

- 数据总监；
- 数据部门负责人；
- CTO。

原因是当前材料尚未充分证明预算、招聘、绩效、团队管理和部门级规划能力。如果岗位名称为“数据负责人”但职责实际偏技术，可使用小厂版投递，并在面试中明确管理边界。

## 九、投递前真实性检查表

- 12个系统与13个系统是否属于不同项目或统计口径；
- 7个下线对象是报表、数据集还是系统；
- 82.3%的分母和统计周期；
- 300+指标中一级核心指标数量；
- DQC 100%覆盖的是任务、表、字段还是核心链路；
- 指标一致率89.1%与97%的计算规则；
- SLA两个99%的定义是否存在重叠；
- Checkpoint下降90%、恢复提升70%、吞吐提升40%的测试条件；
- 99.9%稳定性的统计周期；
- 17亿预警话单与百亿历史话单的关系；
- 83起案件是平台整体成果，不能表述为个人直接成果；
- 2023.04—2025.12项目与其他项目并行的原因和投入比例。

所有数据在面试前都要能够回答：

> 分子是什么？分母是什么？基线是什么？统计周期是什么？数据来源是什么？你的个人贡献是什么？
