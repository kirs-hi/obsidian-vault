# 供应链全链路实时数仓建设

## 项目简历版（可直接使用）

### 项目一：供应链全链路实时数仓建设

**项目背景：** 社区电商业务处于高速增长期，覆盖销售、商品、物流三大核心链路，涉及 20+ 后端系统、150+ 业务过程。实时数据直接服务于仓储作业调度、运输排线优化、供应商结算等生产决策场景，对数据时效性（秒级）和准确性（涉及资金结算）要求极高。

**核心职责：** 作为实时数仓核心开发，负责实时数仓架构设计与演进、Flink 实时 ETL 开发、数据质量体系搭建及稳定性治理。

**技术方案与亮点：**

- **设计并落地三引擎混合架构（Flink 流引擎 + Doris 批引擎 + Blade 状态引擎）**，根据时效性、数据量级、状态大小三个维度制定引擎选型决策树，实现流批灵活切换。纯实时链路覆盖率从 20% 提升至 80%，核心指标端到端延迟从 10 分钟级降至秒级。
- **构建批流一体化数仓模型体系**，采用 ODS → Fact → Topic → App 四层架构，实时与离线统一主题域划分和模型设计，公共逻辑下沉至 Fact 层实现跨业务复用，模型复用率提升 60%，新需求开发周期缩短 40%。
- **攻克大状态 Flink 作业稳定性难题**，针对业务生命周期长（2-3 天）、多流关联场景下 State 膨胀导致的 OOM 和 Checkpoint 失败问题，通过 Flink 1.12 版本升级 + JVM Overhead 内存调优（25%→35%）+ Join 算子优化（优先 JoinKeyContainsUniqueKey 模式减少状态读写）+ Deduplication 替代 ROW_NUMBER 去重，将任务故障率降低 70%。
- **建立全链路双链路容灾体系**，核心任务实现 Binlog 采集、Kafka、Yarn、Doris 全链路跨机房主备部署，S 级任务故障发现 3 分钟、恢复 6 分钟，故障频率控制在 1 次/60 天以内。
- **搭建实时数据质量保障体系**，覆盖"预防-发现-恢复-改进"全生命周期：事前建立性能压测机制（支撑 4W+ QPS 流量洪峰）和开发规范检查；事中构建多源校验、实离校验、波动校验三层数据监控；事后建立故障复盘和 SOP 机制。

**项目成果：**

- Doris 集群高峰期 TP95 响应时长从 2500ms 降至 1300ms，Yarn 集群磁盘 IO 从 70% 降至 30%
- 实时任务告警量下降 60%+，有效告警占比提升至 85%+
- 支撑 400+ Flink 实时任务、50+ Doris 批处理任务稳定运行，服务 10+ 核心数据产品
- 实时数据直接参与仓储收发货调度、运输排线优化、供应商结算等生产环节，助力履约配送效率提升

**技术栈：** Flink SQL / Flink DataStream · Kafka · Doris · Blade · Hive · Redis · MySQL · Yarn · FlinkSQL 平台（NAU）

## 面试深挖准备

### Q1：为什么选择三引擎混合架构而不是纯 Flink？

纯 Flink 方案在我们的场景下有三个痛点：一是开发效率低，物流业务有大量复杂 ETL（3+ 流关联、天级时差），用 Flink 实现开发和运维成本极高；二是去重困难，Flink 去重依赖 State 或外部缓存，维度变更成本高；三是大状态不稳定，业务生命周期 2-3 天导致 State 持续膨胀。所以我们引入 Doris 批引擎处理准实时场景（分钟级、复杂 ETL），引入 Blade 引擎解决大状态问题（将状态外置到可水平扩展的存储），三者各取所长。

### Q2：双链路的资源浪费问题怎么解决？

我们通过任务分级保障来控制成本。将任务分为 S 级和非 S 级，只有 S 级任务（直接影响生产和资金结算的）才做双链路，非 S 级任务单链路运行。S 级任务占总任务量约 20%，所以双链路带来的额外资源开销是可控的。同时通过任务调优降低单任务资源消耗，部分抵消了双链路的成本。

### Q3：Flink State OOM 问题的排查和解决过程？

现象是 TaskManager 进程被 Yarn kill，日志显示 Container 内存超用。根因是 RocksDB State 使用堆外内存不受 JVM 管控，在多流 Regular Join 场景下，左右表数据都存在 State 中，业务生命周期长导致 State 持续增长。解决方案分三步：第一步升级 Flink 1.12，该版本对 RocksDB 内存有更严格的控制；第二步调大 JVM Overhead 比例从 25% 到 35%，给 State 更多空间；第三步优化 Join 算子，确保关联字段是主键（JoinKeyContainsUniqueKey），这样 State 用 ValueState 而非 MapState，存储量和读取效率都最优。

### Q4：实时数据质量怎么保障？

分四个环节。预防阶段：上线前做流量回放压测，确保能扛住 4W+ QPS 的集单高峰；建立开发规范和 Code Review 机制。发现阶段：三层监控——多源校验（不同分层间数据对比、主备对比）、实离校验（实时与离线同口径对比）、波动校验（同比环比 + 断流监控）。恢复阶段：S 级任务 3 分钟发现、6 分钟恢复，依赖双链路热切换。改进阶段：每次故障后做复盘，沉淀到 SOP 和自动化检查工具中。

### Q5：批流一体化具体怎么做到模型统一的？

核心是三个统一：统一主题域划分（实时和离线用同一套业务域划分标准）、统一模型分层（ODS/Fact/Topic/App 四层，实时和离线同构）、统一命名规范。在 Fact 层，实时和离线的字段定义、口径完全一致，差异只在计算引擎和存储介质。这样做的好处是：离线模型可以直接复用实时模型的设计，新需求只需要关注增量逻辑；实离校验也变得简单，因为口径天然一致。
