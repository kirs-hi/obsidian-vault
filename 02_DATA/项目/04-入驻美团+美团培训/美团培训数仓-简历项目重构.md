# 版本一：投递版（精简）

**美团培训数仓建设与专题分析　｜　2024.06～至今　｜　美团-商户数字化发展·美团培训数仓建设　｜　数仓开发**

美团培训（美团大学）是生活服务业人才数字化发展平台，累计培训超 7000 万人次、服务 1400 万在线商家、1.5 万+ 课程 / 19 学院。独立负责美团培训数据体系的建设，五层数仓建模、跨端数据打通、埋点全流程治理、核心指标体系定义、专题分析及 20+ 张报表资产沉淀。
- **数仓分层与跨端打通**：建设 ODS→DIM→FACT→TOPIC→APP 五层数仓（新增24个核心模型表 + 6个维表，覆盖课程/讲师/学员/组织），通过 UUID 与用户 ID 多对多映射打通多端散落数据，为下游归因提供统一维度。
- **埋点数据治理：针对功能页面迭代频繁，数据埋点缺少规范流程约束。推行三步治理——控（埋点上线前校验 + 上线后 T+1 离线对比，两阶段验收增量质量）、治（清理存量脏数据，维表/事实表/报表统一收口）、制（拉通产品/前端/后端/测试/运营落地流程规范）。最终实现埋点异常当日发现，取数从手工硬编码转统一出口。
- **核心指标体系**：存在报表繁多、重复计算，多看板同一指标口径不统，为解决这一现状，梳理收敛 20+ 历史报表至 4 个主题数据集，90% 新需求可直接复用；产出时间提升 2.5 小时。
- **专题分析**：
  - **渠道分析**：多渠道投放无效果量化手段、预算分配缺依据。从规模（UV/会话数）、质量（人均停留时长/跳出率）、效率（学习转化率）三层框架评估各渠道效果，产出留存与质量监控报表。实现渠道 ROI 可量化、预算有据可依。
  - **用户分层**：1400 万商家一刀切运营，无法识别高价值用户。基于 AARRR + RFM + 学习时长聚类划分为重度/普通/边缘三层，识别出前 2.53% 用户贡献 52.5% 学习时长，支撑分层精细化运营。
  - **课程效果评估**：1.5 万门课质量参差不齐、无法横向比较。构建学前→学中→学后→业绩四阶段评估体系，结合知识点标签消除课程类型差异，实现同类课程横向对比。量化内容质量，支撑课程迭代与推荐。
  - **学习转化归因**：培训只有完课结果、过程缺失，时长波动无法归因。建"报名→学习→考核→认证"全链路漏斗模型补齐过程数据，同时将学习时长拆解为课程 UV × 设备登录率 × 学习转化率 × 人均学习次数 × 次均学习时长五因子，支持按渠道、新老客、页面来源多维下钻定位波动。

- **报表门户与数据资产**：建 1 总门户 + 7 平台专区 + 12 学院专区两级数据服务门户，按"看规模/看转化/看学习质量/看内容质量/看用户结构"五问组织，沉淀 20+ 张专题报表资产，BA 取数从手工硬编码转自助查表。

**项目收益**：需求响应 5 天→1.5 天，新商家 30 天辅导 100% 触达；埋点异常当日发现、核心指标口径统一；学习时长可归因，波动可下钻到渠道/新老客/页面来源。

# 版本二：讲解版（大白话 + 具体细节，应对面试拷打）

> 用法：每块先看「大白话」秒懂本质，再看「具体干了什么」应对追问，最后看「可能被拷打」预判问题。

## 0. 业务背景与痛点

**大白话**：美团培训是给全国商家上课的平台（7000 万人次、1400 万商家、1.5 万门课、19 个学院）。但它数据又乱又缺——埋点经常埋错埋漏、用户在不同 App 上的行为对不上号、只知道一个人"学完了"却说不清中间过程。我的活就是把这些烂摊子收拾成一套能用的数据底座。

**具体**：接手时三个痛点——
1. **埋点质量差**：漏埋/错埋/同名不同义，BA 取数靠手工硬编码。
2. **跨端打不通**：数据散落多系统，用户跨端行为拼不起来。
3. **效果无法归因**：只有完课结果，学习过程缺失。

任务：从 0 到 1 建培训统一数仓，把这三个问题解决，给商家分层运营和课程推荐提供数据基建。

**可能被拷打**：
- "和采购数仓有什么区别？" → 采购的数据已存在但散、矛盾在主数据口径；培训的数据是埋点采的、矛盾在埋点质量和跨端打通。采购=主数据治理 + DQC + CDC；培训=埋点治理 + 学习路径归因 + UUID 跨端打通 + 原子派生指标 + 三专题分析。

## 1. 数仓分层与跨端打通

**大白话**：同一个商家，在美团 App、微信、网页上的行为各采各的，对不上号。我把所有数据按"原始→清洗→汇总→主题→应用"五层规矩存好，再用一个"身份证号"把同一个人在不同地方的行为串起来，这样就能追溯一个商家的完整学习轨迹。

**具体**：
- 业务问题：数据散落多系统、跨端打不通，没法做归因。
- 方法：多系统埋点日志；UUID 与用户 ID 多对多映射。
- 做了什么：建 ODS→DIM→FACT→TOPIC→APP 五层数仓，24 张核心模型表 + 12 张公共维表，覆盖课程/讲师/学员/组织四类主体。用 UUID 和用户 ID 的多对多映射，把同一用户跨端行为串起来。
- 产出：跨端行为链路打通，下游归因有统一维度。

**可能被拷打**：
- "为什么分五层？" → ODS 存原始日志防丢、DIM 沉淀公共维表复用、FACT 存事实明细、TOPIC 按主题域整合、APP 面向应用直接出报表。分层是为了复用 + 解耦，维表只建一次下游都能用。
- "UUID 多对多映射具体怎么做的？" → 一个设备一个 UUID，一个用户可能多设备（换手机/平板），也可能一个设备多账号。建 UUID↔用户 ID 的映射宽表，关联时按映射关系 join，行为就能归到同一个用户。

## 2. 学习路径归因

**大白话**：以前只知道一个人"学完了/没学完"，中间哪一步流失的不知道。我把"报名→学习→考试→拿证"每一步都记下来做成漏斗，哪一步人跑掉了、跑了多少，一目了然。

**具体**：
- 业务问题：培训只有完课结果，过程缺失，效果无法归因。
- 方法：学习行为日志；漏斗分析。
- 做了什么：建"报名→学习→考核→认证"全链路归因模型，按日/周/月产出转化漏斗报表。
- 产出：补齐过程数据短板，支撑商家分层运营和课程推荐。

**可能被拷打**：
- "漏斗每一层的口径是什么？" → 报名人数=产生报名动作的去重用户；学习人数=进入图文详情页有滑动/进入视频播放器/进入直播间（按课程类型不同定义）；考核人数=参加考试去重；认证人数=通过认证去重。每层都有明确去重规则。

## 3. 埋点治理（控·治·制）

**大白话**：埋点就是在页面上埋代码采集用户行为，但经常漏埋（该采的没采）、错埋（采错了）、同名不同义（两个指标叫一个名但算的不一样）。我搞了套流程：上线前数据 RD 一起验收（控）、存量脏数据清理（治）、定规矩让大家以后都按规范来（制）。

**具体**：
- 业务问题：埋点漏埋/错埋、同名不同义，BA 取数靠手工硬编码。
- 方法：ocean 实时交互 + T+1 双确认机制；跨职能流程规范。
- 做了什么，推"控·治·制"三步：
  - **控**：数据 RD 参与埋点验收，ocean 实时交互确认 + T+1 双确认，把控增量埋点质量。
  - **治**：存量维表/事实表/报表治理收口，解决历史脏数据。
  - **制**：拉通产品/前端/后端/测试/运营，落地埋点流程规范。
- 产出：流量事实表日增 >200w，埋点异常当日发现，取数转统一出口。

**可能被拷打**：
- "T+1 双确认具体怎么确认？" → 埋点上线前数据 RD 用 ocean 实时交互看上报数据对不对，这是第一道；上线后 T+1 跑离线校验对比预期值，这是第二道。两道都过才放行。
- "存量治理治了什么？" → 重复采集的维表去重收口、口径矛盾的报表统一、历史脏字段清洗，让下游只用一套干净的。

## 4. 核心指标体系

**大白话**：之前"学习人数"这个指标，张三算的是进过页面的、李四算的是真学过的，吵不清。我把所有指标按"用户走到哪一步"分类（浏览→学习→供给→讲师→考试认证 5 个阶段），每个指标写死定义和 SQL，谁也别再各算各的。

**具体**：
- 业务问题：指标口径争议频发、"同名不同义"长期未解。
- 方法：按"用户行为阶段"组织指标；原子→派生分级管理。
- 做了什么：建 5 阶段指标体系，4 大设计原则（存量/增量/增速、质量/效率/粘性、来源/结构/去向、转化/留存/流失）。每个指标定义到**含义 + 计算方式 + SQL 逻辑**三层。配套分区数据量波动 + 字段空值率监控。

  | 行为阶段 | 核心指标 |
  |---|---|
  | 浏览 | PV、UV、累计/新增 UV、点击人数、搜索人数、停留时长、人均停留时长 |
  | 学习 | 学习用户数、学习次数、累计/新增学习用户数、学习课程数、学习时长、人均/次均学习时长 |
  | 课程供给 | 在线/新增/累计课程数、课程时长、课时数、课程销售额、付费用户数 |
  | 讲师 | 累计/新增讲师数、课程被关注/收藏/评论人数 |
  | 考试认证 | 累计/新增考试人数、累计/新增认证人数 |

- 产出：统一核心指标定义，消除"同名不同义"。

**可能被拷打**：
- "原子指标和派生指标怎么分？" → 原子指标=基础统计量（如学习时长 sum）；派生指标=原子 + 维度 + 时间修饰（如美酒学院近 7 日人均学习时长）。原子只建一份，派生靠维度组合，改口径只改原子定义。
- "学习时长的口径细节？" → 单次学习时长 < 7.5 小时（round(learn_time/60,6) < 450）才算有效，剔除异常长时长；停留时长剔除单次 > 7200 的，复用旧逻辑。

## 5. 经营分析（周期性看数）

**大白话**：运营每天每周要看数据，之前各写各的 SQL。我做了日报、周报、双周报三套固定报表，大家打开就能看，不用再找人写代码。周报还从 v1 升级到 v2，能按学院×渠道×课程类型多维下钻。

**具体**：
- 业务问题：运营要日/周/月周期性看规模和趋势，但没有统一出口。
- 做了什么，三套周期性报表：
  - **运营周报**：v1 用 `cap_mtdx_report_analysis_info`（流量）+ `cap_mtdx_report_learn_info`（学习）；v2 升级为 `cap_mtdx_aggr_flow_cube_info`（流量）+ `cap_mtdx_aggr_learn_cube_info`（学习）+ `cap_mtdx_stay_analysis_info`（留存），支持学院×渠道×课程类型多维立方体下钻。
  - **大学双周报**：`cap_date_dou_week_info`（周维表）+ 流量/学习表，服务对外合作组。
  - **大学日报推送**：日报指标汇总 + 日报展示表，服务产品 + 运营。
- 另归档方法论知识库（指标体系建设、AARRR、留存、用户分层、生命周期、活动效果评估倍差法、A/B 测试），统一 BA 分析语言。
- 产出：运营有统一周期看数出口。

**可能被拷打**：
- "v1 到 v2 为什么升级？" → v1 是单表平铺，下钻维度有限；v2 拆成流量/学习/留存三张立方体表，支持学院×渠道×课程类型自由组合下钻，灵活性和查询性能都更好。
- "倍差法（DID）是干嘛的？" → 评估平台型大促的增量效果，不是简单看活动前后对比（有自然增长干扰），而是找一个对照组，用活动组前后差值减去对照组前后差值，剔掉自然增长，算出活动真正带来的增量。

## 6. 渠道专题分析

**大白话**：培训课程投放了好几个渠道（抖音、公众号、App 推送、人社局、学习强国…），但哪个渠道效果好、该给哪个多投钱，说不清。我按"来了多少人（规模）→ 人待多久（质量）→ 最后多少人真学了（效率）"三层指标，把每个渠道的投入产出算明白。

**具体**：
- 业务问题：渠道投放 ROI 难量化、预算分配无依据。
- 方法：渠道行为日志；规模/质量/效率三层指标框架。
- 指标分三层：
  - **规模**：UV、PV、用户数、登录用户数、访问次数、会话数。
  - **质量**：人均停留时长、次均会话时长、人均访问次数/页面数/课程数、跳出率。
  - **效率**：用户登录率、学习转化率、课程详情页访问率。
- 产出：渠道留存表 `cap_mtdx_stay_source_analysis_info`、设备类型表 `cap_mtdx_stay_device_info`、渠道类型表 `cap_mtdx_stay_feat_info`、渠道质量监控表 `cap_mtdx_source_quality_info`、账号登录情况表 `cap_mtdx_account_login_info`。
- 解决：支撑渠道预算分配与质量监控。

**可能被拷打**：
- "渠道有哪些？" → 团 APP 商家端（美酒 EB）、三方平台（抖音、头条、学习强国）、机构/协会（人社局、深职院）、微信公众号、其他。
- "ROI 怎么算？" → 渠道没直接花钱投放金额数据，所以用"效率层"指标（学习转化率）代理 ROI——同样拉来 1000 人，哪个渠道学习转化率高，哪个渠道质量好，预算就往哪倾斜。

## 7. 用户专题分析

**大白话**：1400 万商家不可能一视同仁。我用算法把用户按"学习时长"分成重度、普通、边缘三层，发现 2.53% 的重度用户贡献了一半多（52.5%）的学习时长——这帮人就是重点运营对象，得重点培养。

**具体**：
- 业务问题：用户价值无法分层、运营一刀切。
- 方法：AARRR（新增→留存→活跃→传播→盈利）+ RFM + 聚类算法。
- 做了什么：
  - **分层**：按目标路径（新用户→下载→活跃→兴趣→付费）提炼核心节点，优化各节点转化效率。
  - **分群**：按价值贡献分——活跃贡献（长尾/活跃/忠诚）、内容贡献、货币贡献，结合 RFM 定位高价值用户。
  - 用学习时长聚类 + 时长贡献法，分重度/普通/边缘三层：**前 2.53% 用户贡献 52.5% 学习时长**。
- 产出：有效用户事实表 `cap_mtdx_fact_user_active` / `cap_mtdx_fact_user_aggr_active`、各行业有效用户监控表 `cap_mtdx_user_quality_dashboard_info`。
- 解决：支撑 12 学院分类运营、培养高价值学员。

**可能被拷打**：
- "聚类怎么分的？为什么是三层？" → 用学习时长做特征聚类，结合时长累计占比（帕累托式分布）找拐点，前 2.53% 是重度、中间段普通、尾部边缘。三层是业务能落地运营的最简划分，再细运营接不住。
- "分层和分群有什么区别？" → 分层是按目标路径纵向分（用户走到哪一步了），分群是按价值贡献横向分（同一层里谁是高价值）。分层看转化，分群看投入。

## 8. 内容专题分析

**大白话**：1.5 万门课，哪些好哪些差？我从"学之前看不看→学的时候学没学完→学完评不评价→最后有没有带来生意"四个阶段评估每门课，还按知识点给课程打标签，能横向比较同类课程。

**具体**：
- 业务问题：课程学习效果无法量化。
- 方法：四阶段评估框架；知识点第二视角聚合。
- 做了什么，建"学前→学中→学后→业绩"四阶段评估：
  - **学前**：详情页 UV/访问率/退出率。
  - **学中**：参与率/有效学员占比/完课人数占比/近 7、30 日留存。
  - **学后**：好评/收藏/分享/考试通过率/认证通过率。
  - **业绩**：订单量/交易额/评分环比。
  - 课程按知识点（员工培训/品牌营销/会员管理/开店指南/政策解读）打标作第二视角聚合，消除课程类型差异。
- 产出：完课率明细表 `cap_mtdx_crouse_progress_aggr_info`、认证课学习效果闭环表 `cap_mtdx_mj_course_result`。
- 解决：支撑内容迭代与课程推荐。

**可能被拷打**：
- "知识点第二视角是干嘛的？" → 同一门课可能跨多个知识点，不同课程类型（图文/视频/直播）直接比完课率不公平。按知识点聚合后，"品牌营销"类课程之间横向比，消除课程类型差异，才知道哪个知识点的内容做得好。
- "完课率怎么定义？" → 有有效学习行为且学完整门课的人数 / 学习该课的总人数。有效学习行为按课程类型区分（图文=滑动到底、视频=播放完毕、直播=观看时长达标）。

## 9. 学习时长拆解

**大白话**：总学习时长突然跌了，到底哪出了问题？我把它拆成"来的人数 × 登录率 × 学了的人占比 × 每人学几次 × 每次学多久"，哪一环比跌了就查哪一项，能一路下钻到具体渠道、新老用户、哪个页面来的。

**具体**：
- 业务问题：学习时长波动无法归因。
- 方法：按 学习时长 = 课程 UV × 设备登录率 × 学习转化率 × 人均学习次数 × 次均学习时长 拆解。
- 做了什么：支持新老客（复学率/复访率）、页面来源（首页转化 + 渠道 CTR × 学习转化）多角度下钻；配套核心转化、认证课程学习、付费课程支付三套漏斗。
- 解决：时长波动可下钻到渠道/新老客/页面来源。

**可能被拷打**：
- "为什么是这五个因子相乘？" → 这五个因子分别对应"拉新→登录→转化→频次→深度"五个环节，相乘等于总时长，每个因子都可独立监控。总时长跌了，一定是某个因子跌，定位到因子再下钻维度，比直接看总数快。
- "三套漏斗分别是什么？" → 核心转化漏斗（浏览→学习→完课，看整体）、认证课程学习漏斗（学习→考试→认证，看闭环）、付费课程支付漏斗（浏览→下单→支付，看变现），对应三种业务场景。

## 10. 报表门户与数据资产

**大白话**：做了一堆报表，不能到处乱放。我搭了个门户，平台总览放一层、各学院自己的数据放一层，BA 想看什么自己点进去查，不用再找人写 SQL。还按"你想看规模还是看转化还是看质量"五个问题把报表分好类，自助定位。

**具体**：
- 业务问题：BA 取数依赖手工硬编码、响应慢。
- 做了什么：
  - **总门户**：1 个（bi.sankuai.com/portal/21639）。
  - **平台主题专区 7 个**：平台概览、运营周报、大学日报、渠道、学院、用户、内容、页面、营销、课程评论。
  - **学院数据专区 12 个**：培训中心、美酒、闪购商、外卖、配送、房东、优选团长、休娱、美业、优选商家、教育亲子、快驴。
  - **五问框架**：按"看规模变化→看转化效率→看学习质量→看内容质量→看用户结构"组织指标与专区映射，BA 自助定位数据。
  - **数据资产 20+ 张表**：运营周报 v1/v2、双周报、日报推送、页面转化（`cap_mtdx_aggr_transform_page_info`/`cap_mtdx_aggr_transform_model_info`）、渠道专题、内容专题、渠道质量监控、有效用户监控等。
- 解决：BA 取数从手工硬编码转自助查表，需求响应 5 天→1.5 天，新商家 30 天辅导 100% 触达。

**可能被拷打**：
- "五问框架怎么用的？" → BA 想知道"规模变没变"→ 指标 PV/UV/学习用户数 → 去平台概览专区；想看"转化效率"→ 设备登录率/学习转化率/留存 → 去学院/渠道/内容专区。问题导向，不是按表名找数据，是按业务问题找。
- "12 个学院怎么选的？" → 按业务活跃度和数据量选的头部学院，覆盖美酒/外卖/闪购商/配送/房东/优选团长/休娱/美业/优选商家/教育亲子/快驴/培训中心，其他长尾学院走平台总览看。

## 面试 90 秒口述版

起点是美团培训只有完课结果、学习过程数据缺失，埋点漏埋/错埋/同名不同义，BA 取数痛苦、培训效果无法归因。

建设上做了四件：建五层数仓 + 12 公共维表、UUID 多对多映射打通跨端；建"报名→学习→考核→认证"学习路径归因产出漏斗报表；推"控·治·制"治理埋点（控验收、治收口、制规范），取数转统一出口；原子→派生指标分级 + 分区波动/空值率监控，口径可信、埋点异常当日发现。

指标体系按"用户行为阶段"组织（浏览/学习/课程供给/讲师/考试认证 5 阶段），每个指标定义到 SQL 逻辑。在能用的数据上跑三个专题：渠道分析（规模/质量/效率三层算 ROI）、用户分析（AARRR 分层分群 + RFM + 聚类，前 2.53% 用户贡献 52.5% 时长）、内容分析（学前→学中→学后→业绩四阶段评估 + 知识点第二视角）；再加学习时长漏斗拆解。最后产品化成 1 总门户 + 7 平台专区 + 12 学院专区的报表门户，20+ 张专题表，按"看规模/看转化/看学习质量/看内容质量/看用户结构"五问组织。结果：30 天辅导 100% 触达、需求响应 5 天降到 1.5 天。

和采购数仓区分：采购的数据已存在但散、矛盾在主数据口径；培训的数据是埋点采的、矛盾在埋点质量和跨端打通。采购=主数据治理 + DQC + CDC 实时；培训=埋点治理 + 学习路径归因 + UUID 跨端打通 + 原子派生指标 + 渠道/用户/内容三专题分析。
