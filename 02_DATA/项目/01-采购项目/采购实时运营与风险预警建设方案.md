# 采购实时运营与风险预警建设方案

## 一、项目定位

### 1. 背景

现有采购产品已提供采购工作台入口，覆盖申请、项目、合同、订单等模块的基础待办、统计卡片及流程跳转。但采购全流程横跨 PR、项目、招标、合同、PO、验收、付款、财务等多个系统，当前仍有三个典型问题：

1. **待办碎片化**：每个系统只能展示本模块内待办，采购人员需要在多个模块和系统间切换，无法快速识别“当前最应该处理什么”。
2. **风险发现滞后**：订单未验收、合同到期仍在履约、合同额度不足等问题，往往在 T+1 报表或事后复盘中才被发现，错过了最佳处置窗口。
3. **展示与处置割裂**：看板通常只展示待办数量，缺少风险优先级、责任归属、下游影响和一键处理入口，无法形成闭环。

### 2. 建设目标

在不替换现有采购业务系统的前提下，将采购工作台升级为“采购实时运营中枢”：

> 统一工作入口 + 实时事件识别 + 风险分级 + 责任人触达 + 处置闭环

其中，业务系统继续负责流程发起、审批和单据操作；实时数据系统负责跨系统状态关联、风险识别和待办聚合；采购工作台负责展示、触达、跳转原系统及闭环管理。

### 3. 建设原则

- **不为实时而实时**：仅建设能够改变当前动作、避免即时业务损失的实时能力；经营分析、月度成本等仍以小时级或 T+1 为主。
- **不侵入原流程**：不重构 PR、合同、订单、验收等既有系统，采用 CDC 与业务事件实现低侵入接入。
- **一个入口处理问题**：统一聚合待办和风险，但实际操作仍跳转至原业务系统，避免重复建设流程能力。
- **优先闭环、再扩面**：先验证履约—验收—付款场景，再扩展至合同、项目和合规。

## 二、为什么不能仅由单一采购系统解决

采购待办和即时风险的本质是**跨系统、跨阶段、带时间约束的业务状态判断**。

```text
PR系统 → 项目系统 → 招标系统 → 合同系统 → 订单系统 → 验收系统 → 付款/财务系统
```

单个系统只能看到本领域的数据。例如，合同系统知道合同有效期和总额度，订单系统知道在途订单，验收系统知道验收状态；但“合同即将到期且仍有大额订单未验收”需要将三者实时关联后才能识别。

同时，若让多个系统各自建设提醒规则，会带来以下问题：

- 多个团队、多个技术栈分别开发，协调和发版成本高；
- 规则、阈值、风险等级和通知标准不统一；
- 用户仍会收到分散在各个系统中的待办；
- 风险规则每次调整都要多系统改造，无法快速迭代。

因此，最合理的方式是在已有的采购数据基建之上建设统一实时运营层：一次汇聚、多维关联、统一规则、统一入口、回跳处置。

## 三、产品建设方案

## 1. 升级现有采购工作台

现有首页保留按采购流程和业务模块划分的结构，但将“静态统计卡片”升级为“可行动的实时运营卡片”。

### 首页核心区域

```text
实时采购运营中心

我需要处理
- P0：2 条，已影响履约、付款或合规
- P1：8 条，24 小时内需要处理
- P2：16 条，正常待办

重点风险
- 3 笔订单已履约但未验收，涉及金额 128 万
- 2 份合同将在 15 天内到期，仍存在在途订单
- 5 个采购项目关键里程碑已逾期
```

每张卡片应提供：

- 当前待办/风险数量及准实时更新时间；
- P0、P1、P2 风险等级；
- 风险对象、原因、金额、截止时间和下游影响；
- 当前责任人与协同责任人；
- 一键跳转到原系统单据或处理页面。

## 2. 新增“实时待办中心”

建设跨系统待办收件箱，按照风险优先级而非创建时间排序。

| 字段 | 说明 |
| --- | --- |
| 待办类型 | 审批、补充材料、确认、验收、付款、整改等 |
| 风险等级 | P0、P1、P2 |
| 业务对象 | PR、项目、合同、PO、订单行、验收单、付款单 |
| 风险原因 | 例如“已收货 7 天未验收” |
| 当前责任人 | 需要采取动作的用户 |
| 协同对象 | 申请人、项目经理、供应商、财务等 |
| 首次预警时间 | 判断风险持续时间 |
| 截止时间/超时天数 | 支撑优先级排序 |
| 下游影响 | 如付款阻塞、合同到期、项目延期 |
| 处理入口 | 跳转原系统详情或操作页 |
| 处理状态 | 待处理、处理中、已处理、已豁免、已关闭 |

建议以如下维度综合排序：

> 优先级 = 资金影响 + 履约影响 + 时效紧迫度 + 风险持续时间 + 下游阻塞程度

## 四、首批实时场景

实时专项不宜一开始覆盖全部采购流程。第一阶段优先聚焦“履约—验收—付款”主链路，再逐步延伸至合同、项目和合规。

## 场景一：未验收订单行准实时预警闭环（一期 MVP）

### 风险识别规则

订单行满足以下条件时进入风险识别：

```text
订单状态 = 已下单/履约中
且存在收货、服务完成或履约完成信号
且在约定时限内未发起验收，或验收未完成
```

建议按品类配置阈值：

| 场景 | 建议阈值 |
| --- | --- |
| 实物采购 | 收货后 3 天未验收 |
| 服务采购 | 服务期结束后 5 天未验收 |
| 项目型采购 | 里程碑完成后 7 天未验收 |
| 高金额订单 | 收货后 24 小时提示 |

### 闭环流程

```text
收货/服务完成事件
  → 订单行进入待验收状态
  → 达到预警阈值
  → 创建/更新风险待办并触达责任人
  → 用户跳转验收系统处理
  → 验收完成后自动关闭风险
```

### 业务价值

- 缩短未验收订单滞留时长，减少挂账；
- 降低已履约但无法及时结算造成的供应商争议；
- 缩短付款因验收凭证缺失而被阻塞的时间；
- 使管理者获得真实履约进度，而不只看到已下单金额。

## 场景二：合同到期与额度穿透风险（二期）

### 核心规则

1. **合同到期风险**：合同剩余有效期小于等于 15 天，且存在未完成订单、未验收或未付款事项。
2. **额度不足风险**：合同可用额度小于在途订单金额与待下单申请金额之和。
3. **超额度下单风险**：新增 PO 金额与已占用金额之和超过合同额度。

### 产品呈现示例

```text
P0 合同履约风险
合同：HT2026XXXX
剩余有效期：6 天
合同剩余额度：12 万
在途订单金额：86 万
建议动作：续签、补充协议或终止未履约订单
```

## 场景三：采购项目关键里程碑逾期与下游阻塞（二期）

关注采购申请、立项、寻源/招标、定标、合同、下单、验收、付款等关键节点。

| 风险规则 | 业务影响 |
| --- | --- |
| 采购申请提交后 48 小时未受理 | 项目启动延后 |
| 招标截止日临近但有效报价不足 | 可能流标，影响交付 |
| 定标后超时未签合同 | 供应商无法启动履约 |
| 合同完成后超时未下单 | 合同无法转化为实际交付 |
| 关键物料未按计划下单 | 项目排期或上线受影响 |

重点不只提示“项目逾期”，而是识别阻塞链路和下游影响。

## 场景四：采购实时合规审计（三期）

| 风险类型 | 识别逻辑 |
| --- | --- |
| 先下单后补合同 | 订单创建时未匹配有效合同 |
| 超合同额度下单 | 在途订单累计金额超过合同可用额度 |
| 拆单规避审批 | 同申请人、供应商、品类在短时间窗口内多笔小额订单累计超过阈值 |
| 供应商准入失效 | 下单或付款时供应商资质已过期 |
| 非授权采购 | 采购人、组织、品类与授权范围不匹配 |
| 验收付款链路异常 | 已付款但无有效验收，或验收金额与订单/付款金额异常偏离 |

高风险规则可采用强校验或拦截；存在合理例外的规则采用预警、人工处理和豁免审计。

## 六、核心实时模型：订单行累计快照

以 `po_code + po_line_id` 作为履约主粒度，构建“采购全流程信息表（订单行累计快照）”。这是连接离线数仓和实时运营能力的核心模型。

| 领域 | 核心信息 |
| --- | --- |
| 订单 | 订单号、订单行、订单金额、订单状态 |
| 项目 | 项目编号、项目名称、负责人、关键里程碑状态 |
| 合同 | 合同编号、有效期、合同额度、已占用额度 |
| 履约 | 收货状态、服务完成状态、履约完成时间 |
| 验收 | 验收状态、验收金额、验收超期天数 |
| 付款 | 付款申请状态、付款金额、付款阻塞原因 |
| 风险 | 风险等级、风险类型、首次发生时间、处置状态 |
| 责任 | 当前责任人、协同人、组织、采购品类 |

Paimon 主键表设计思路：

```sql
CREATE TABLE paimon_po_lifecycle_snapshot (
    po_code STRING,
    po_line_id STRING,
    contract_code STRING,
    project_code STRING,
    current_status STRING,
    receipt_status STRING,
    acceptance_status STRING,
    payment_status STRING,
    risk_level STRING,
    risk_types ARRAY<STRING>,
    responsible_user STRING,
    event_time TIMESTAMP(3),
    PRIMARY KEY (po_code, po_line_id) NOT ENFORCED
) WITH (
    'merge-engine' = 'deduplicate',
    'sequence.field' = 'event_time',
    'changelog-producer' = 'lookup'
);
```

- `deduplicate`：保证同一订单行保留唯一的当前状态；
- `sequence.field`：解决跨系统事件乱序，避免旧事件覆盖新状态；
- Changelog：将状态变更持续输出给风险规则引擎和待办服务，实现状态变化即识别、风险消失即关闭。

## 七、风险事件与处置闭环

风险不是一次性消息，而应是可管理、可审计的事件。建议以 `risk_type + business_object_id` 作为风险唯一键，避免同一问题被重复创建多条待办。

```text
识别到风险
  → 待处理
  → 处理中
  ├─ 已处理：业务状态恢复正常，系统自动关闭
  ├─ 已豁免：人工填写原因，保留审计记录
  └─ 已升级：超时未处理，升级通知负责人或管理者
```

### 推送分级

| 等级 | 触达方式 | 示例 |
| --- | --- | --- |
| P0 | 站内待办 + IM/短信 + 管理者升级 | 高金额订单已履约但长期未验收 |
| P1 | 站内待办 + IM 提醒 | 合同 15 天内到期且存在在途订单 |
| P2 | 工作台聚合展示/每日摘要 | 普通节点临近超时 |

为避免消息轰炸，同一风险仅在首次触发、关键阈值升级、超时升级和风险解除时触达。

## 八、实时 DQC 与 SLA

实时预警中，错误数据的危害往往大于延迟，因此需同步建设实时数据质量能力。

| 监控项 | 说明 |
| --- | --- |
| CDC 延迟 | 源系统 Binlog 至实时层延迟，核心链路分钟级可观测 |
| 数据断流 | 订单、验收、付款等关键事件异常中断时告警 |
| 主键重复 | 同一业务对象出现不合理重复更新 |
| Schema 漂移 | 多区域或全球化系统字段变动导致消费失败 |
| 状态机校验 | 如未下单即验收、未验收即付款等非法状态 |
| 金额勾稽 | 订单、验收、付款金额的异常偏离 |
| 规则效果 | 命中量、处理率、误报率、豁免率 |

SLA 建议分层：

| 链路 | 建议时效 |
| --- | --- |
| 高风险合规拦截 | 秒级至分钟级 |
| 验收、履约、合同到期风险 | 5 分钟内 |
| 我的待办统一聚合 | 10 分钟内 |
| 管理指标、经营分析 | 小时级或 T+1 |

## 九、分阶段建设路径与价值指标

### 第一阶段：订单履约与验收预警闭环

范围：PO、收货/履约、验收、付款四类核心事件；订单行累计快照；未验收订单行预警；工作台实时待办；原系统跳转和自动关单。

验证指标：

- 未验收订单平均滞留时长；
- 超期验收订单行占比；
- 付款因验收缺失导致的平均阻塞时长；
- 风险待办及时处理率；
- 从风险发生到首次处理的平均时长。

### 第二阶段：合同履约与项目里程碑穿透

范围：合同额度、合同期限、在途订单、项目预算、关键里程碑；识别合同到期履约风险、额度风险和项目阻塞链路。

目标：从处理单据升级为保障采购交付。

### 第三阶段：实时合规审计与风险拦截

范围：先下单后补合同、拆单规避审批、超额度下单、供应商资质失效、验收付款异常等场景。

目标：从事后发现问题升级为事中减少损失。

## 十、面试项目表述

> 我们原来已有采购工作台，能够覆盖申请、项目、合同、订单等模块的基础待办和统计，但待办仍然分散在不同系统中；每个系统只能看到自身流程，无法识别跨合同、订单、履约、验收和付款链路的即时风险。
>
> 因此我推动建设采购实时运营与风险预警能力。底层通过 Flink CDC 接入采购域 22+ 系统的增量事件，在 Flink 中按订单行构建采购全流程累计快照，并利用 Paimon 主键表沉淀当前状态。基于订单、合同、履约、验收、付款状态的实时变更，计算风险事件并回流到现有采购工作台，形成统一待办、风险分级、责任人触达、原系统跳转处置、状态恢复后自动关单的闭环。
>
> 第一阶段优先聚焦履约—验收—付款链路，因为它直接影响供应商结算、资金占用和采购履约效率，且可以通过未验收订单平均滞留时长、付款阻塞时长、风险处理及时率等指标量化效果；后续再扩展到合同穿透、项目节点阻塞和实时合规审计。

> 一句话总结：原采购系统负责把流程走完；采购实时运营中枢负责在跨系统流程即将出问题时及时发现、定位责任人、推动处置，并形成可量化的风险闭环。
