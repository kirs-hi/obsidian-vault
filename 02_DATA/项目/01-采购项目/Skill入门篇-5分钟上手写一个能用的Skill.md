# 入门篇：5 分钟上手，写一个真正能用的 Skill

你有没有遇到过这种情况：给 AI Agent 写了一段很详细的"指令说明"，但 Agent 要么压根不触发，要么触发了却做错了事，要么在它最该报错的地方悄无声息地给你编了一个答案？

这不是 Agent 太蠢，是你写的 Skill 有问题。

这篇文章带你从零开始，5 分钟写出一个结构清晰、真正能跑起来的 Skill。

---

### Skill 是什么？别再叫它"提示词"了

很多人把 Skill 当成"高级提示词"来写——堆一大段文字描述，期待 Agent 看懂之后自动执行。这个思路从根上就错了。

想象你刚招了一位天才实习生。他智商极高，但完全不懂你的业务。你不可能每次布置任务都从头口头交代——那太低效了。你需要给他一本完整的**操作手册（SOP）**：什么情况下做什么，按哪些步骤做，做完怎么交付。

Skill 就是这本手册。Anthropic、OpenAI 都在推动这套标准，本质上是把"每次临时口头交代的 Prompt"，升级成"预先设计好的、可复用的能力模块"。

**Skill 明确告诉 Agent 三件事：**

1. **什么时候该用这个能力**（触发条件）
2. **用什么方式执行**（工具/步骤）
3. **做完什么算完成**（输出标准）

---

### 最简 Skill 长什么样？

一个最精简但能工作的 Skill，只需要三个部分：

```
// 代码块
name: 动作+对象
description: 路由契约（什么时候触发）
正文: 步骤流程
```

#### 字段一：name，用"动作+对象"命名

不要叫 `assistant`、`helper`、`tool_v2`，这些名字对 Agent 毫无意义。

好的命名示例：

- `weekly-report-formatter`（整理周报）
- `sql-query-executor`（执行 SQL 查询）
- `meeting-room-booker`（预订会议室）

名字即功能，一眼就知道这个 Skill 干什么。

---

#### 字段二：description，这是路由契约，不是介绍文案

这是写 Skill 最容易犯错的地方，也是最关键的地方。

**description 不是给人看的功能描述，是给 Agent 看的路由规则。**

Agent 在收到用户请求时，会扫描所有已安装的 Skill 的 description，判断"哪个 Skill 该响应这个请求"。如果你的 description 写得模糊，Agent 要么不触发，要么触发错了。

很多人第一个 Skill 都死在这里。有个真实案例：description 写了"帮助用户处理数据"——结果用户说"帮我分析一下这个文件"和"帮我删除这个文件"，都触发了同一个 Skill。后果自然不用说。

**反例（❌ 介绍文案风格）：**

```
// 代码块
description: 帮用户写周报，支持多种格式，适合各类职场人士使用
```

这段话对 Agent 毫无用处。它不知道什么情况下该触发，也不知道什么情况下不该触发。

**正例（✅ 路由契约风格）：**

```
// 代码块
description: 当用户需要把本周已有的工作事项整理成固定模板格式的周报时触发。
仅适用于已有明确工作内容的整理归纳，不适用于凭空编造工作成果或无材料生成场景。
输入：用户提供的工作事项列表；输出：标准格式周报文本。
```

好的 description 包含三层信息：

- **触发规则**：什么情况下该触发
- **边界规则**：什么情况下不该触发
- **输入输出规则**：需要什么，产出什么

---

#### 字段三：正文，写清楚步骤

```markdown
// 代码块
## 执行步骤

1. 确认用户已提供本周工作事项列表
   - 如果没有提供，提示用户："请先列出本周已完成的工作事项"
   - 不要自行猜测或编造

2. 按照以下模板整理：
   【本周工作总结】
   - 完成事项：...
   - 进行中事项：...
   - 下周计划：...

3. 整理完成后，输出完整的周报文本，并询问用户是否需要调整
```

---

### 写完之后，用三个问题验证

**问题一：会不会触发？**
想几个用户可能会说的话，看 description 能不能覆盖这些场景。

**问题二：触发了结果对不对？**
手动跑一遍主流程，特别关注有没有 Agent 可能"自由发挥"的地方。

**问题三：出错了会不会提醒？**
刻意测试异常情况：给一个不完整的输入，给一个超出范围的请求。

---

### 完整示例：一个可以直接用的 Skill

下面是一个"SQL 查询助手"的完整 Skill，三个字段全部到位，可以直接参考：

**name：**

```
// 代码块
sql-query-assistant
```

**description：**

```
// 代码块
当用户需要查询数据库、执行 SQL 语句、或用自然语言描述查询需求时触发（如"帮我查一下昨天的订单数"、"执行这段 SQL"、"统计过去 7 天的 UV"）。
仅适用于 SELECT 查询操作；不适用于 DDL（CREATE/ALTER/DROP）、DML 写操作（INSERT/UPDATE/DELETE）或任何数据变更场景——这类请求需拒绝并说明原因。
输入：用户的自然语言查询意图或 SQL 语句；输出：查询结果表格 + 简要数据解读。
```

**执行步骤：**

```markdown
// 代码块
## 执行步骤

1. **意图判断**
   - 判断用户是直接提供了 SQL，还是用自然语言描述了查询需求
   - 如果是 DDL 或写操作意图（含 DROP/DELETE/UPDATE/INSERT 等关键词），立即终止并回复：
     "这个 Skill 仅支持 SELECT 查询，数据变更操作请联系数据管理员。"

2. **生成 SQL，确认后再执行**
   - 根据用户意图生成对应的 SELECT SQL
   - 将 SQL 展示给用户，说明查询逻辑，询问："以上 SQL 是否符合你的意图？确认后我将执行。"
   - 等待用户明确确认，不得在未确认前自动执行

3. **执行查询**
   - 调用数据库工具执行 SQL
   - 如果执行超时（>30秒），返回："查询超时，请尝试缩小时间范围或添加过滤条件。"

4. **结果展示**
   - 以表格形式展示查询结果（最多展示前 50 行）
   - 附上 1-2 句数据解读，说明关键发现
   - 如果查询结果为空，明确告知："本次查询无结果，可能原因：条件过严或数据不存在。"

5. **边界处理**
   - 结果超过 1000 行时，提示用户添加 LIMIT 或更精确的过滤条件，不自动截断后返回
```

这个 Skill 体现了三个核心要素：**触发规则清晰**（知道什么时候该接）、**边界规则明确**（知道什么时候该拒）、**执行步骤完整**（每一步做什么、出错怎么办）。

---

### 带走这一条

> **description 是路由契约，不是介绍文案。写 Skill 的核心工作，是把"什么时候触发、什么时候不触发、需要什么输入、产出什么"说清楚。**
