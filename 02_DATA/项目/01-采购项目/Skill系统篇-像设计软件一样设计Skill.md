# 系统篇：像设计软件一样设计 Skill

前两篇，我们学会了写一个能跑的 Skill，也学会了用工程思维让 Skill 好用起来。

但当你的 Skill 开始处理复杂任务——多个步骤、多种分支、多类用户请求——你会发现一个新问题：**Skill 本身变成了一团乱麻。**

这篇文章解决的，是复杂 Skill 的结构化设计问题。核心思路只有一句话：**像设计软件一样设计 Skill。**

---

### 为什么复杂 Skill 会失控？

复杂 Skill 的典型失控路径：

1. 最开始写了一个简单版本，能用
2. 需求增加，往里加步骤
3. 出现异常情况，往里加说明
4. 遇到特殊案例，往里加示例
5. 最终：一个几千行的文档，Agent 每次都要全部加载，慢且不准确

这个问题的根源不是内容太多，而是**没有结构**。软件工程里，我们不会把所有代码塞进一个文件。同样道理，复杂的 Skill 也需要模块化设计。

---

### 七模块结构：给 Skill 搭骨架

一个成熟的 Skill，可以拆成七个功能模块：

**① Overview（概览）**：3-5 句话说清楚这个 Skill 是干什么的，适用什么场景，不适用什么场景。

**② Triage（分诊）**：根据输入特征，把请求分流到不同的处理路径。

```markdown
// 代码块
## Triage
- 用户提供了文件 → 执行 [文件解析流程]
- 用户粘贴了原始数据 → 执行 [直接处理流程]
- 用户只描述了需求但无数据 → 提示用户提供数据
```

**③ Pre-checks（前置检查）**：执行主流程之前，先检查所有必要条件是否满足。早发现，早报错，不让问题带到后面。

**④ Workflow（主流程）**：核心执行逻辑。每步编号清晰，明确"谁来做"（模型还是工具）。

```markdown
// 代码块
## Workflow
1. [工具] 解析用户提供的数据文件
2. [模型] 理解用户想要什么类型的图表
3. [工具] 调用图表生成脚本
4. [模型] 检查图表是否符合要求，输出确认或修改建议
```

**⑤ Output（输出规范）**：明确定义输出格式——长什么样、什么内容、什么格式。模糊的输出规范会导致 Agent 每次输出不一致。

**⑥ Examples（示例）**：提供 2-3 个完整示例。示例的作用不是解释，而是**校准**——帮 Agent 对齐"什么算正确的执行结果"。

**⑦ Failure & Fallback（失败处理）**：明确各种异常情况发生时怎么做。Skill 没有 Fallback，等于没有设计失败路径。

---

### 实战：用七模块拆解一个真实 Skill

用"每日数据报表生成" Skill 走一遍，看每个模块写什么：

**① Overview**

```
// 代码块
自动拉取昨日核心业务指标（订单量、GMV、DAU），按固定模板生成日报并推送到指定大象群。
适用场景：每日定时触发或手动触发生成报表。
不适用：实时数据查询、自定义指标临时分析。
```

**② Triage**

```
// 代码块
- 触发方式为定时任务 → 直接执行，使用默认日期（昨日）
- 用户手动触发且指定了日期 → 使用指定日期
- 用户手动触发但未指定日期 → 询问："请问要生成哪一天的报表？"
```

**③ Pre-checks**

```
// 代码块
- 检查数据源连接是否正常，失败则终止并告警
- 检查目标推送群 ID 是否已配置，未配置则终止
- 检查昨日数据是否已入库（ETL 完成标志位），未完成则等待 30 分钟后重试，最多 3 次
```

**④ Workflow**

```
// 代码块
1. [工具] 从 Hive 拉取指定日期的指标数据（订单量、GMV、DAU）
2. [工具] 计算同比/环比增长率
3. [模型] 根据数据变化生成 1-2 句异动解读文字
4. [工具] 将数据填入报表模板，生成 Markdown 格式日报
5. [工具] 调用大象 API 推送到目标群
```

**⑤ Output**

```
// 代码块
输出格式：Markdown 日报，包含【日期/核心指标表格/同环比/异动解读/数据来源说明】
字数控制在 300 字以内，不附原始数据明细。
```

**⑥ Examples**

```
// 代码块
示例一：正常生成（数据正常，同比+5%，输出标准日报）
示例二：数据异常（GMV 同比下跌 30%，异动解读需标注"显著下跌，建议关注"）
示例三：数据未就绪（入库未完成，触发重试逻辑后成功）
```

**⑦ Failure & Fallback**

```
// 代码块
- 数据源查询失败 → 发送告警消息到群，内容："今日报表生成失败，原因：数据源连接超时，请人工排查。"
- 推送失败（大象 API 报错）→ 保存报表到本地文件，通知负责人："报表已生成但推送失败，文件路径：xxx。"
- 重试 3 次仍失败 → 停止重试，记录日志，等待人工介入
```

七个模块过完一遍，这个 Skill 的完整执行逻辑就一目了然，维护时也能快速定位到出问题的那个模块。

### 渐进式披露：主文件控制在 500 行以内

七模块骨架搭好了，下一个问题是：**怎么控制文件大小？**

OpenAI 在 Cookbook 里把**渐进式披露（Progressive Disclosure）**作为多工具 Agent 的核心设计原则——只在需要的时候加载需要的内容。

这背后有实实在在的技术原因。上下文窗口虽然很大（128k/200k token），但塞满了效果会变差。研究表明，LLM 对处于上下文**中间位置**的内容关注度会显著下降，这被称为"**Lost in the Middle**"效应。把 Skill 主文件控制在 500 行以内，不只是效率考虑，也是质量考虑。

**三层架构：**

```
// 代码块
主文件（≤500行）       ← 路由逻辑 + 主流程入口
  ├── 细节文档         ← 详细步骤说明，按需加载
  └── 脚本/工具        ← 机械操作全封装在这里，不写进主文件
```

**什么该放主文件：** 所有路由和触发逻辑、主流程入口和分支结构

**什么该拆出去：** 超过 50 行的详细步骤、大量示例、所有代码和脚本

---

### 自由度设计：窄桥 vs 开阔平原

复杂 Skill 还有一个容易忽略的设计决策：**该给 Agent 多大的自由度？**

**窄桥加护栏（低自由度）：** 每一步都明确规定 Agent 应该怎么做。

- 适用于：对准确性要求极高的场景，如财务计算、法律文书
- 优点稳定可预测，缺点灵活性差

**开阔平原（高自由度）：** 只规定目标和约束，允许 Agent 自己选路径。

- 适用于：创意类、探索类任务，如写作、分析
- 优点处理多样化输入强，缺点一致性低

**大多数实用 Skill 的最佳姿势：主流程窄、边缘处理宽。** 核心路径明确规定，确保主流程稳定；异常情况给 Agent 判断空间，让它灵活处理未预料到的输入。

### 迭代：测试驱动，小步走

最后聊一下迭代策略。

OpenAI 建议把 Skill 开发看成**测试驱动开发（TDD）**——先写测试用例（评测集），再写 Skill，而不是先写完再测。这和软件开发里的 TDD 思路完全一致：先定义"什么叫做对"，再去实现。

**正确的迭代姿势：**

1. **先定位问题类型：**
  - 没触发 → 问题在 description 触发规则
  - 触发了但结果不对 → 问题在正文步骤或工具调用
  - 触发了不该触发 → 问题在 description 边界规则

1. **每次只改一个地方**，改完跑评测集，确认效果再改下一处

1. **保留修改记录**，需要回退时能找回之前版本

### 一张图总结

```
// 代码块
成熟 Skill 的设计层次
─────────────────────────────────────
主文件（≤500行）
  ├── Overview         ← 高层摘要
  ├── Triage           ← 请求分流
  ├── Pre-checks       ← 前置条件验证
  ├── Workflow         ← 主流程（引用细节文档）
  ├── Output           ← 输出规范
  ├── Examples         ← 2-3个典型示例
  └── Failure&Fallback ← 异常处理

细节文档（按需加载）
  └── 各子流程的详细说明

脚本/工具（机械操作全部在这里）
  └── 数据处理、API调用、格式转换
─────────────────────────────────────
```

### 带走这一条

> **复杂 Skill 失控的根源是没有结构，而不是内容太多。七模块骨架 + 三层架构 + 渐进式披露，让你的 Skill 既能被 Agent 高效执行，又能被人类轻松维护。先写评测集，再写 Skill；小步走，每次只改一处。**
