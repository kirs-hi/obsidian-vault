# Part A · 正式简历（投递用，直接复制）

## 项目名称：采购全链路数据基建

**项目背景：** 公司采购业务分散在境内外 22 个核心系统中（5 个采购自有系统 + 17 个外部关联系统），数据壁垒严重，指标口径不统一，历史数据质量差。为支撑采购数字化、智能化转型，启动采购数仓从 0 到 1 的建设，目标不只是给人看报表，而是建设一套 AI 可消费的数据资产。

**技术栈：** Flink / Flink SQL、Apache Paimon（deduplicate/aggregation merge-engine、sequence.field、Changelog 流读、Lookup Join）、Kafka/Mafka、Hive、Spark SQL、Doris、Python + 大模型 API + Skill 框架

**职责描述：** 负责采购数据平台整体架构设计与核心场景落地，经历离线数仓 0-1 建设、实时湖仓升级、AI 数据资产与 Data Agent 落地三个阶段，覆盖数据接入、分层建模、指标体系、实时监控、数据质量、治理体系和 AI 应用的全链路。

**技术方案：**

**【阶段一 · 离线数仓 0-1 建设】**

- **数仓规范 0-1 制定：** 完成需求对接、数仓建模、指标管理、数据运维四大规范制定。建立约束机制——不登记口径不上看板、不评审不接需求，保障数仓建设流程标准统一。

- **全量数据摸底与入仓：** 对 22 个核心系统 300+ 原始数据表进行整合、清洗和标准化，DWD 明细层覆盖度 82.3%。品类主数据建立三级标准编码体系做跨系统映射，供应商维度建统一编码 + SCD Type2 拉链表处理变更历史，解决境内外编码不一致和层级映射混乱问题。

- **数据质量体系：** 核心链路 DQC 七大维度（完整性/准确性/一致性/唯一性/关联性/及时性/真实性）100% 覆盖，强规则阻断下游、弱规则告警不阻断。建设数据指标可视化门户，业务方自助查看异常和血缘，无需依赖人工对数。

- **指标一致性治理：** 面向 6000+ 表口径不一致问题，执行"盘点→定义→治理→工具化"四步治理，引入"单模型单口径"机制，指标一致率从 89% 提升至 97%。存量应用 100% 切换至标准模型，消除口径差异。

**【阶段二 · 实时湖仓升级】**

- **实时 ODS/DWD 层建设：** 采购核心系统（旭日/Jupiter 等）Binlog 通过 Flink CDC 写入 Paimon 主键表（deduplicate + sequence.field）实现自动去重与乱序过滤，维表统一收口至 Paimon 通过 Lookup Join 关联，替代原 ROW_NUMBER + Redis + 三套维表存储方案，Checkpoint 下降 75%，Kafka Topic 减少 40%。

- **采购执行实时监控与预警：** 基于 Paimon Changelog 流读消费采购订单变更数据，对交期延迟、价格异常、合同执行偏差等关键节点实时规则预警，同时支持 Time Travel 按时间点回溯订单状态，替代原 T+1 报表发现问题的模式。

- **采购金额实时汇总与预算监控：** 采用 Paimon Aggregate Table（SUM / LAST_NON_NULL）替代 Flink Cumulate Window，采购订单增量写入时自动合并计算实时金额汇总，Flink 端无状态写入，Checkpoint 下降 90%，下游消费 Changelog 触发预算超支/异常预警。

- **流批同源与修数体系：** 基于 Paimon 流批同源特性，实时流读 Changelog、离线批读快照，读同一张表加工逻辑只有一套，口径天然一致。配合 Spark SQL 每日批处理修数，修数时间从小时级缩短至分钟级，修数资源消耗降低 80%。

- **实时数据质量保障：** 建设写入层（Paimon 去重 + 乱序过滤）、加工层（实时对账）、产出层（实离对账，差异率 < 0.05%）三层校验体系，设计分场景修数 SOP（Spark SQL 直接修复 Paimon 表 + Flink 指定时间戳恢复），修数资源消耗降低 80%。

- **链路稳定性与运维保障：** 建设全链路系统 + 业务双层监控，制定 3-10-15 分钟故障响应 SOP。引入 Paimon 后链路组件从 7 类收敛至 4 类，Flink 去重/聚合等逻辑下沉存储层实现无状态化，故障恢复速度提升 70%，核心链路稳定性 99.9%。

**【阶段三 · AI 数据资产与 Data Agent】**

- **AI 友好数据资产建设：** 将线上表和字段抽象为结构化元数据描述，AI 直接读取理解，跳过"模拟人翻 DDL"的转换层。数仓开发规范（分层标准、命名规范、DQC 规则、血缘映射）固化为 AI 可理解、可执行的"规范&资产"。

- **Data Agent 设计与落地：** 设计采购取数分析 Skill，覆盖需求澄清、权限预检、多维取数（品类/供应商/城市等）、洞察分析及 HTML 可视化报告生成全链路。采用"大模型理解意图 + 结构化配置 + SQL 模板拼装"的混合方案，内置约束前置（无权数据在 SQL 生成阶段过滤）、分层降级（四级递进兜底）、全链路审计（意图到结果全程可追溯）设计机制，服务于采购经理的日常采购决策分析场景。

**项目成果：**
- 整合 22 个系统 300+ 张表，DWD 明细层覆盖度 82.3%，核心链路 DQC 100% 覆盖
- 指标一致率从 89% 提升至 97%，存量应用 100% 切换至标准模型
- Flink 去重算子 State 降为零，相关任务 Checkpoint 下降 75%（800MB→200MB）
- 聚合类任务 Checkpoint 下降 90%，任务恢复速度提升 70%（五六分钟→一两分钟）
- Kafka Topic 数量减少 40%，中间态数据从不可查变为支持 SQL 点查和时间旅行
- 修数时间从小时级缩短至分钟级，修数资源消耗降低 80%
- 链路组件从 7 类收敛至 4 类，外部组件依赖减少 60%（5 类→2 类）
- 实时与离线数据差异率 < 0.05%，核心链路稳定性 99.9%
- Data Agent 上线，采购经理从"提需求等排期"变为"自助取数分析"

## 话术正文

我介绍一下采购全链路数据基建这个项目。

公司采购业务分散在境内外 22 个核心系统中，5 个采购自有系统加 17 个外部系统，数据壁垒严重，指标口径不统一，历史数据质量差。我负责从 0 到 1 搭建采购数仓，这个项目经历了三个阶段：离线数仓建设、实时湖仓升级、AI 数据资产落地。目标不只是给人看报表，而是建设一套 AI 可消费的数据资产。

**第一阶段是离线数仓 0-1 建设。** 我对 22 个系统 300 多张表做了全面摸底，分优先级入仓，DWD 明细层覆盖度 82.3%。最难的是品类和供应商主数据——各系统编码不统一、层级映射混乱，我建了标准的三级品类体系和统一供应商编码做跨系统映射。同时从 0 到 1 制定了数仓全流程规范，核心是让规范有约束力：不登记口径不上看板，不评审不接需求。数据质量方面，核心链路 DQC 七大维度做到 100% 覆盖，强规则阻断下游、弱规则告警。建了指标可视化门户，业务方自助看异常和血缘。指标一致率从 89% 治理到 97%，存量应用 100% 切到标准模型。

**第二阶段是实时湖仓升级。** 随着业务从看 T+1 报表演进到实时监控采购执行，我们引入 Paimon 构建流式湖仓。核心思路是把状态管理从计算层下沉到存储层。采购核心系统的 Binlog 通过 Flink CDC 写入 Paimon 主键表，用 deduplicate 引擎替代 ROW_NUMBER 去重，sequence.field 替代 Redis 时间戳判断乱序，Flink 端完全无状态，Checkpoint 下降了 75%。维表统一收口到 Paimon 通过 Lookup Join 关联，替代了原来散在各系统的多套维表存储。

基于 Paimon Changelog 流读，我们对采购订单的交期延迟、价格异常、合同执行偏差做实时规则预警，替代原来 T+1 才发现问题的模式。用 Paimon Aggregate Table 做采购金额实时汇总，替代 Flink Cumulate Window，Flink 无状态写入，Checkpoint 又降了 90%。基于 Paimon 流批同源的特性，实时流读 Changelog、离线批读快照，读同一张表加工逻辑只有一套，口径天然一致，修数用 Spark SQL 直接改 Paimon 表，从小时级压到分钟级。整体组件从 7 类收敛到 4 类，实离差异率万分之五以内，核心链路稳定性 99.9%。

**第三阶段是 AI 数据资产与 Data Agent。** 在离线加实时的统一底座之上，我把线上表和字段抽象为结构化元数据，AI 直接读取理解。然后设计了一套采购取数分析 Skill，不是裸 NL2SQL，而是大模型理解意图输出结构化 JSON、再由模板拼装 SQL 的混合方案。内置三个关键机制：约束前置——无权数据在 SQL 生成阶段就过滤掉；分层降级——四级递进兜底；全链路审计——意图到结果全程可追溯。服务采购经理的日常分析，从"提需求等排期"变成"自助取数"。

总结一下：整合了 22 个系统 300 多张表，离线 DWD 覆盖 82.3%，实时层 Checkpoint 下降 75% 和 90%，指标一致率 89% 到 97%，组件从 7 类收敛到 4 类，最终把数据从 T+1 报表升级为 AI 可消费的实时数据资产。最本质的变化是把状态管理从计算层下沉到存储层，Flink 变成无状态编排层，这个思路在不同业务场景都可以复用。如果您对 Paimon 技术细节或 Data Agent 设计感兴趣我可以展开。

## 一、项目背景与业务理解（Q1-Q5）

### Q1. 为什么要从 0 到 1 建采购数仓？之前的痛点是什么？

公司采购业务分散在境内外 22 个核心系统中，5 个采购自有系统（旭日、Jupiter、芥末等）加 17 个外部系统。存在四个层面的痛点：第一，数据壁垒严重，部门间看数需要跨系统导出后手工处理，时效性低、权限申请周期长。第二，全球化采购分析难度高，境外涉及多语言、多币种、多汇率场景，线下手工处理效率低易出错。第三，运营管理粗放，采购涉及几百种品类、近十种流程和十几种单据，基础数据建设不完善，无法还原业务过程细节。第四，数据标准不一，存在同名不同义和同义不同名的问题。所以启动了采购数仓从 0 到 1 的建设。

### Q2. "面向 AI 赋能的数据资产"具体指什么？和传统数仓的区别？

传统数仓面向人和 BI 工具，面向 AI 的数据资产有三个核心差异。第一，数据可消费性：传统数仓用户是人，通过看板看数据；AI 友好数据资产要求数据有规范语义、可被程序化查询。比如把线上表和字段抽象为结构化元数据描述，AI 直接读取理解，跳过"模拟人翻 DDL"的转换层。第二，质量要求更高：传统报表容错率较高，脏数据最多导致某张表不准；但 AI 取数会直接生成分析结论，脏数据会污染整个分析链路。第三，规范驱动 AI 生产：把数仓开发规范固化为 AI 可理解、可执行的"规范&资产"，让 AI 稳定输出符合分层架构和质量要求的数仓资产。我们的 Data Agent 就是建立在这套 AI 友好数据资产之上的。

### Q3. 采购业务的核心业务过程有哪些？

采购业务分三种类型：集中采购（公司统一采购）、生产性采购（直接用于生产经营）、采销（采买后直接或加工后销售）。核心业务过程链路是：需求提报 → 询价比价 → 合同签订 → 下单 → 到货验收 → 入库 → 对账结算。每个环节的数据产出物不同：需求单、报价单、合同、采购订单、验收单、入库单、结算单等十余种单据。间接采购到货后直接形成资产或费用化；直接采购到货后进入库存管理，按生产过程领用。在数仓建模时，按业务过程拆分事实表，品类和供应商作为核心一致性维度贯穿全链路。

### Q4. 22 个系统怎么分类？境内外的数据差异在哪？

22 个系统按数据域分类：5 个采购自有系统（采购域），17 个外部系统（财务域、主数据域、业务域等）。境内系统包括旭日（生产性采购）、芥末（买菜/优选）、驼峰等；境外系统包括 Jupiter、Mars，覆盖海外业务。境内外数据差异是统一口径的难点：币种不同需要统一汇率转换，供应商编码体系不一致需要建映射表统一主键，品类分类体系不同需要层级对齐，数据合规要求不同涉及跨境合规。解决方式是 ODS 层保留原始数据，DWD 层做标准化时统一币种转换、供应商编码映射、品类层级对齐。

### Q5. 这个项目的业务方是谁？不同业务方的诉求差异？

三类业务方，诉求不同。采购经理/品类经理：看采购执行效率、供应商绩效、品类采购成本，需要多维交叉分析能力。财务：看采购成本、对账准确性、预算执行率，需要精确到结算单级别的金额核对。管理层：看品类策略、供应商集中度、合规风险，需要全局视角的汇总指标。数据资产要服务多方，所以指标体系按分析主题设计：采购执行主题、供应商绩效主题、成本分析主题、合规风控主题。不同主题面向不同业务方，但共享同一套一致性维度（品类、供应商、组织、时间）。

## 二、数仓规范与架构（Q6-Q8）

### Q6. 数仓规范 0-1 怎么制定的？怎么让业务和产品达成共识？

规范分四大块：需求对接规范（需求单模板、评审流程）、数仓建模规范（分层架构、表命名、字段命名和词根规范、维度建模方法论）、指标管理规范（原子/派生指标拆解、口径定义和审批流程）、数据运维规范（SLA 目标、变更管理、DQC 配置、权限管理）。让业务和产品达成共识的关键不是写文档，而是建约束机制：不登记口径的指标不上看板，不评审的需求不接，技术方案评审会拉业务和产品一起参加。建指标可视化门户，口径透明化，业务方自己就能看到指标定义和血缘。核心方法论是"模型设计即开发、开发即治理"，规范不是事后约束而是事前驱动。

### Q7. 数仓分层怎么设计？每层职责？

四层架构加 DIM 层。ODS（操作数据层）：业务系统数据直接同步，保留原始细节，binlog 采集业务数据。BI 场景下不开放查询。DWD（明细数据层）：对 ODS 数据清洗、转换、整合，按业务过程组织，基于维度建模设计，事务事实表 + 周期快照事实表 + 累积快照事实表。DWS（汇总数据层）：面向分析对象轻度聚合，按业务主题构建宽表（采购执行、供应商绩效、成本分析等）。ADS（应用数据层）：面向具体应用场景的报表和指标，同步至 Doris 提供 OLAP 查询。DIM（维度层）：品类、供应商、组织、时间等一致性维度统一收口。DWD 和 DWS 的边界：DWD 是明细粒度事实表不做聚合，DWS 是按主题轻度聚合面向分析对象建模。

### Q8. 明细层覆盖度 82.3% 怎么算的？剩下 17.7% 为什么没覆盖？

覆盖度口径 = 已入 DWD 层的业务系统表数 / 应入 DWD 的总表数。22 个核心系统中已覆盖的业务表占比 82.3%。剩下 17.7% 未覆盖的原因：低频使用表（ROI 低，优先级排在后面）、系统待下线（投入产出不划算）、数据质量极差无法入仓（需要源系统先治理）、外部系统无数据访问权限（需走权限申请流程）。82.3% 不是拍脑袋的数字，是做了优先级取舍的结果。先覆盖高价值核心链路，剩余表按迭代节奏逐步纳入。

## 三、数据治理与质量（Q9-Q13）

### Q9. 品类主数据有什么问题？清洗规则怎么定的？

品类主数据的常见问题：品类编码不统一，各系统各编一套；层级混乱，某系统的二级品类对应另一个系统的三级；停用品类未清理，历史品类仍残留；品类重复，同品类不同编码、不同品类同编码。清洗规则四步：统一编码体系（建立品类标准编码表，映射各系统品类编码）、层级映射（统一为三级品类体系，跨系统层级对齐）、停用标记（停用品类标记但保留历史数据）、去重（相同业务含义品类合并为一个标准编码）。品类清洗是持续过程不是一次性，治理机制是新增品类走审批流程，变更走变更评审。

### Q10. 供应商维度模型怎么设计？SCD 怎么处理？

供应商是采购域核心一致性维度。维度表设计：统一供应商编码（主键，映射多系统编码）、供应商名称/注册地/资质信息、合作状态（在合作/暂停/黑名单）、供应商评级、关联品类。SCD Type2 处理变更：加 start_date 和 end_date 拉链表，记录供应商信息变更历史。供应商合并场景（A 收购 B）：建供应商映射表，把 B 的历史数据归到 A，历史事实表回刷供应商编码。境外供应商有额外字段：注册国家、币种、合规信息。境内外供应商通过统一编码体系打通。

### Q11. DQC 100% 覆盖的规则体系怎么设计？

DQC 规则按七大校验维度设计：完整性（非空校验、行数校验）、准确性（枚举值校验、范围校验、格式校验）、一致性（主外键校验、跨表数据对齐）、唯一性（主键去重校验）、关联性（函数关系、主外键关系）、及时性（任务产出时间监控、数据就绪延迟告警）、真实性（数据波动稽查）。100% 覆盖指核心链路所有 DWD 表都配了 DQC 规则。规则分强弱两级：强规则阻断下游（数据不对不产出），电话告警 + 终止任务；弱规则告警不阻断，邮件短信告警，当天处理完再上线。DQC 运营机制：无效 DQC 响应在 1 个工作日内完成，DQC 加白机制（日常观测型 DQC 默认通过）。

### Q12. 指标可视化门户怎么实现无需人工对数？

三层机制实现：第一层产出层实离对账，实时和离线结果比对，差异率低于 0.05%；第二层波动监控，指标值同环比异常自动告警；第三层口径血缘可追溯，下游发现问题不用找人对数，直接在门户查指标定义和血缘链路定位。传统模式是"业务发现数据不对 → 找数据团队 → 人工核对 → 排查问题"，门户模式是"业务在门户看到异常告警 → 点开血缘定位 → 自助解决"。问题响应效率从天级提升到分钟级。

### Q13. 指标一致性治理具体做了什么？89% 到 97% 怎么来的？

面对 6000+ 表口径不一致的问题，四步治理。第一步盘点：梳理所有表的全部指标，建立指标台账。第二步定义：统一口径（命名、定义、取数来源、计算逻辑、加工链路一致性）。第三步治理：找到不一致的指标，推动修改，通过一致性监控工具对比两个模型/应用中相同指标、相同维度值下的数据是否一致。第四步工具化：指标门户登记，不登记口径的指标不上线，一致性监控自动化。一致率口径 = 检测一致的指标次数 / 所有指标检测次数。89% 到 97% 靠的是存量指标口径统一 + 增量管控（新指标必须登记口径才上线）。剩下 3% 不一致的原因：历史遗留口径业务方暂不统一、跨域指标归属不清需协调。

## 四、实时湖仓 · 选型与架构（Q14-Q15）

### Q14. 为什么选 Paimon 而不是 Iceberg 或 Hudi？

三者都是数据湖格式，但定位不同。Iceberg 偏离线分析，流式更新能力较弱，不支持主键表的高效 Upsert。Hudi 支持主键更新，但 Changelog 生成不如 Paimon 原生，MOR 模式在高频更新下 Compaction 压力大。Paimon 的核心优势有三点：一是基于 LSM Tree 的高效流式更新，主键表天然支持 deduplicate、aggregation、partial-update 等多种 merge-engine；二是原生 Changelog 生成能力，下游直接流读变更数据，不需要额外 CDC 组件；三是和 Flink 深度集成，Paimon 本身就是从 Flink Table Store 演进来的，Flink SQL 原生支持，开发体验最好。我们的场景刚好是高频流式更新加 Changelog 流读，所以选了 Paimon。

### Q15. 引入 Paimon 后整体架构是什么样的？原来的组件还用吗？

不是全部推翻重建，而是渐进式迁移。离线部分 Hive + Spark 保留，负责 T+1 批处理和历史数据回刷。实时部分是新增的：ODS 层的 Binlog 接入仍然走 Kafka，因为采集组件原生对接 Kafka，替换成本高且收益小。但 DWD/DWS 层间的数据流转从 Kafka Topic 迁移到 Paimon 表的 Changelog 流读；去重和乱序处理从 Flink State 加 Redis 迁移到 Paimon 主键表；天粒度聚合从 Flink Window 迁移到 Paimon Aggregate Table；维表从 Redis/Cellar/MySQL 收口到 Paimon；ADS 层查询服务仍然用 Doris。整体是"Flink 做计算编排、Paimon 做存储和状态管理、Doris 做查询服务"的三层架构。Kafka 从层间流转主力退化为 ODS 接入通道。

## 五、实时湖仓 · 全链路数据贯通（Q16-Q19）

### Q16. Paimon 主键表去重和原来 ROW_NUMBER 去重的本质区别是什么？

ROW_NUMBER 去重是在计算层做的——Flink 需要维护一个按主键分区、按时间排序的窗口状态，每条新数据都要和状态里的历史数据比较排序，这个状态随数据量线性增长。Paimon 主键表去重是在存储层做的——数据写入 LSM Tree 时，相同主键的多条记录在 Compaction 阶段自动合并，只保留最新一条。Flink 端完全不需要维护去重状态，只负责把数据写入 Paimon 表。本质上是把"有状态计算"变成了"无状态写入加有状态存储"。

### Q17. sequence.field 是怎么解决乱序的？

sequence.field 的原理是：Paimon 在合并相同主键的多条记录时，不是简单取最后写入的那条，而是比较 sequence.field 指定字段的值，比如 event_time，只保留值最大的记录。新写入的数据如果 event_time 比已有记录小，直接丢弃。原来的链路中间层乱序靠 Redis 缓存时间戳判断，每条数据一次网络 IO，高并发下是瓶颈。现在 Paimon 在写入时就保证了顺序正确性，到 Doris 时数据已经有序了。整条链路的乱序防御从"末端兜底"变成了"逐层保证"。

### Q18. Lookup Join 和原来用 Redis 做维表关联有什么区别？

Redis 做维表关联是"外部点查"模式——每条事实数据来了，Flink 算子发一次 Redis 请求查维度数据，网络 IO 是瓶颈。Paimon Lookup Join 是"本地缓存"模式——Flink 算子启动时将 Paimon 维表数据加载到本地，关联时直接本地查找，没有网络开销。我们用的是 Local PrimaryKey Partial Lookup 模式，不需要全量缓存，只在数据触发时按需加载对应 Bucket 的文件，基于 LSM Tree 有序性做定位查找。性能上 Lookup 延迟从 Redis 的毫秒级降到本地微秒级，代价是本地磁盘占用增加，维表更新有一定延迟，依赖 Paimon 快照刷新周期。但我们的维表更新频率不高，天级更新，这个延迟完全可以接受。

### Q19. Checkpoint 下降 75% 具体是什么概念？绝对值大概多少？

原来用 ROW_NUMBER 去重的任务，State 里要存全量的去重窗口数据，Checkpoint 大小大概在 800MB 左右，制作时间 30-40 秒。切到 Paimon 主键表后，Flink 端去重 State 降为零，Checkpoint 只剩下 Source 端的 Offset 和 Sink 端的少量事务状态，大小降到 200MB 以内，制作时间 5-10 秒。所以说下降 75% 不只是存储空间的优化，更重要的是 Checkpoint 制作更快、更稳定，任务从 Checkpoint 恢复的时间也大幅缩短，整体稳定性有质的提升。

## 六、实时湖仓 · 采购监控与预警（Q20-Q21）

### Q20. Paimon 流读和 Kafka 消费在机制上有什么区别？

Kafka 是纯日志流——消费者通过 Offset 顺序读取，读过就没了，数据本身不支持查询。Paimon 流读是基于快照的变更流——每次 Checkpoint 生成新快照，流读消费者读取两个快照之间的差异，本质上是增量快照读。好处有三个：一是数据可查，任意时间点的快照都可以 SQL 直接查询，也就是 Time Travel；二是数据可回溯，指定快照 ID 或时间戳就能读取历史数据，不需要重置 Offset；三是天然去重，因为 Changelog 是基于合并后的结果生成的。代价是时效性受 Checkpoint 间隔影响，通常秒级到分钟级，不如 Kafka 的毫秒级。但在采购场景里秒级完全够用，采购订单的状态变更不需要毫秒级响应。

### Q21. 采购监控的实时规则预警具体怎么实现的？

采购订单有几个关键业务节点：下单、供应商确认、发货、到货、验收、结算。我们监控的规则主要有三类：交期延迟预警，比如供应商确认后超过约定交期还没发货的；价格异常预警，比如同一品类采购价格偏离历史均价超过阈值的；合同执行偏差预警，比如实际到货数量与合同约定偏差超过阈值的。实现方式是在 Flink 消费 Paimon 的 Changelog 流之后，用 CEP（复杂事件处理）或简单的条件判断做规则匹配，命中的数据写入告警 Topic，由下游推送到采购运营看板和大象通知。相比原来 T+1 报表的模式，问题发现时间从一天缩短到分钟级，采购经理能在当天就介入处理。

## 七、实时湖仓 · 采购金额实时汇总（Q22-Q23）

### Q22. Aggregate Table 和 Flink Cumulate Window 在效果上有什么区别？

最终结果是一样的，都是按主键做天粒度聚合。区别在于状态管理方式。Cumulate Window 的状态在 Flink 内存里，窗口越大状态越大，一天的窗口意味着一天的数据都在内存里，Checkpoint 要把这些状态全部持久化，所以大且慢。Aggregate Table 的状态在 Paimon 的 LSM Tree 里，每条新数据写入时和已有记录合并，Flink 端是无状态的，Checkpoint 几乎为空。另一个关键区别是乱序处理：Cumulate Window 依赖 Watermark，迟到数据要么丢弃要么进侧输出流，Aggregate Table 不依赖 Watermark，任何时候来的数据都会正确合并到结果中。这就是 Checkpoint 能下降 90% 的原因——从 Flink 承载的几百 MB 窗口状态直接降到几乎为零。

### Q23. 预算监控的阈值是怎么设定的？

预算监控分两种：超支预警和执行进度预警。超支预警是采购金额汇总超过预算额度的一定比例时触发，比如已下单金额超过年度预算的 90% 就告警。执行进度预警是根据时间进度和采购执行进度的匹配度，比如时间过半但采购执行只到 30%，说明可能有延迟风险。具体阈值不是我们数据平台定的，是业务方根据预算管理和采购策略设定的，我们提供的是实时的金额汇总数据和触发预警的机制。采购金额涉及多个环节——已审批预算、已下单金额、已结算金额，这些汇总就是在 Paimon Aggregate Table 里完成的。

## 八、实时湖仓 · 流批同源与修数（Q24-Q25）

### Q24. 修数流程具体是怎么操作的？停任务不会影响实时性吗？

修数分四步：第一步停掉需要修复的 DWD 层及后续的 Flink 任务；第二步用 Spark SQL 读取 Paimon 表，执行修复逻辑，比如 UPDATE 某个字段或 DELETE 错误数据，修复结果直接写回 Paimon 主键表，利用主键合并覆盖；第三步重启 Flink 任务，指定从某个时间戳开始消费增量数据；第四步验证修复结果。关于实时性影响，确实会有短暂中断，但修数通常安排在业务低峰期，而且 Paimon 修数只需十分钟，比原来的 ODS 重放全链路快得多。对业务的影响从"半天不可用"变成"几分钟不可用"。这也是 Paimon 相对 Kafka 的一个核心优势——中间态数据是可修改的表，而不是只能追加的日志流。

### Q25. 流批同源具体是什么意思？怎么解决实时离线口径不一致的？

最开始是朴素方案——统一指标字典加每天对数。实时 Flink SQL 写一套，离线 Hive SQL 写一套，靠指标字典约束口径，靠 T+1 对数任务发现差异。但两套代码注定会漂移，对数只是事后补救。引入 Paimon 后真正解决了，核心就是流批同源。Binlog 通过 Flink 写入 Paimon 主键表，去重、乱序、聚合全交给 Paimon 引擎处理。实时流读 Changelog，离线批读快照，读的是同一张表，加工逻辑只有一套，口径天然一致，不需要对数了。一句话说就是：口径不一致的根源是两套代码，解法是流批同源，让两边读同一份存储。

## 九、实时湖仓 · 数据质量保障（Q26-Q28）

### Q26. 三层校验体系具体是怎么做的？

写入层靠 Paimon 本身的能力——主键表的 deduplicate merge-engine 保证去重，sequence.field 保证乱序过滤，这是第一道防线。加工层做实时对账——在 Flink ETL 过程中，对关键指标埋入计数器和求和器，和上游源数据做实时比对，比如 ODS 层进来了 1000 条采购单，DWD 层出去也应该是 1000 条，如果对不上就触发告警。产出层做实离对账——每天离线数据产出后，和前一天的实时数据快照做同口径对比，核心指标的差异率必须在万分之五以内。超出阈值的会触发告警并生成修数工单。这套实时三层校验和离线层的 DQC 七大维度是互补关系——DQC 管离线链路的数据质量，三层校验管实时链路的数据质量，两者在产出层通过实离对账交汇。

### Q27. 实离对账差异率万分之五是怎么做到的？差异的常见原因是什么？

万分之五是最终的稳态结果，刚上线时差异率大概在千分之三左右，花了两三个月逐步收敛的。常见的差异原因有三类：第一是时间窗口差异——实时数据截止到某个时间点，离线数据可能多包含或少包含几秒的数据，这种通过对齐快照时间点来解决。第二是乱序数据——极端迟到的数据实时链路可能已经过了窗口，但离线全量计算能包含进去，这种通过延长 Paimon 的快照保留时间和调优 sequence.field 来缓解。第三是口径差异——某些边界 case 在实时和离线的处理逻辑上不完全一致，这种需要逐个排查修复。收敛到万分之五以后，剩下的差异基本都是第一类时间窗口差异，属于可接受范围。引入 Paimon 流批同源后，第三类口径差异基本消除，因为实时和离线读的是同一张表。

### Q28. 修数 SOP 具体是什么流程？修数资源消耗降低 80% 怎么算的？

修数 SOP 分三种场景。第一种是"单表修数"——发现某张 DWD 表的数据有问题，停掉这张表的下游 Flink 任务，用 Spark SQL 直接修复 Paimon 表，然后重启下游任务从指定时间戳恢复。第二种是"链路修数"——问题出在 ODS 层，需要从源头修复，这种先修 ODS 的 Paimon 表，然后依次重启 DWD、DWS 的 Flink 任务，每层都从指定时间戳消费增量。第三种是"紧急修数"——线上数据错误已经影响到业务，这种直接在 Doris ADS 层做临时修复，保证前端展示正确，同时后台按第一种或第二种流程做根因修复。每种场景都有标准化的操作文档、检查清单和回滚方案。关于修数资源消耗降低 80%，原来修数走 ODS 全链路重放——需要启动 ODS 到 DWD 到 DWS 的完整 Flink 任务链来重新消费和计算，多个任务并行跑，通常要占用整条链路的全部计算资源跑数小时。现在直接用 Spark SQL 对 Paimon 目标表做修复，只需要一个 Spark 任务跑几分钟。从"启动全链路 N 个 Flink 任务跑数小时"变成"一个 Spark 任务跑几分钟"，资源消耗大概降了 80%。

## 十、实时湖仓 · 链路稳定性与运维（Q29-Q31）

### Q29. 3-10-15 分钟故障响应 SOP 具体是什么？

3 分钟是发现时间——从数据异常发生到监控系统触发告警，要求在 3 分钟以内。这靠两层监控：系统层监控 Flink 任务的 Checkpoint 状态、反压指标、消费延迟；业务层监控核心指标的波动和断流。10 分钟是定位时间——从收到告警到确认故障根因和影响范围，要求在 10 分钟以内。这靠标准化的排查路径和 Paimon 的可查能力，直接 SQL 查 Paimon 快照就能定位问题出在哪一层哪个时间点。15 分钟是恢复时间——从确认根因到业务恢复正常，要求在 15 分钟以内。恢复手段包括任务重启、从 Checkpoint 恢复、切换备链路等。整个 SOP 有值班机制，S 级任务的告警直接电话通知。

### Q30. 组件从 7 类收敛到 4 类，具体是哪些？外部组件依赖减少 60% 怎么算的？

原来 7 类是：Flink 做计算、Kafka 做层间流转、Doris 做查询、Redis 做维表缓存和乱序状态、Cellar 做维表缓存、MySQL 做维表存储、HBase 做部分维表。收敛后 4 类是：Flink 做计算编排、Paimon 做存储和状态管理、Doris 做查询服务、Spark 做批处理修数。能减少的原因是 Paimon 一个组件替代了原来多个组件的职责——Paimon 主键表替代了 Redis 的乱序状态缓存，Paimon 维表替代了 Redis/Cellar/MySQL 的维表存储，Paimon Changelog 流读替代了 Kafka 的层间流转。Kafka 没有完全去掉，ODS 层接入还在用，但不再算核心链路组件了。外部组件依赖减少 60% 是这样算的：原来 Flink 任务需要依赖的外部组件有 Kafka、Redis、Cellar、MySQL、HBase，一共 5 类外部依赖。引入 Paimon 后，Redis、Cellar、MySQL 的维表职责收口到 Paimon，Kafka 的层间流转也被替代，Flink 任务的外部依赖变成 Paimon 和 Doris，减掉了 3 类，从 5 类降到 2 类，大概就是 60%。

### Q31. 故障恢复速度提升 70% 是怎么算的？

这个主要体现在 Flink 任务从 Checkpoint 恢复的时间上。原来去重和聚合类任务的 State 很大，Checkpoint 动辄几百 MB 到 GB 级，恢复时需要从 HDFS 下载 State 文件、反序列化、重建 RocksDB，整个过程可能要五六分钟。切到 Paimon 后，这些任务的 Flink State 降为几乎为零，恢复时只需要加载 Source Offset 和 Sink 事务状态，几十秒就能完成。从平均五六分钟降到一两分钟，大概就是 70% 的提升。另外一个维度是修数恢复——原来修数需要 ODS 全链路重放，小时级；现在 Spark SQL 直接改 Paimon 表，分钟级。这两个维度加在一起，故障恢复速度整体提升 70% 是有依据的。

## 十一、Paimon 技术深度追问（Q32-Q36）

### Q32. Paimon 的 LSM Tree 结构具体是怎么工作的？

写入时数据先进入内存的 MemTable，积累到一定大小后 Flush 为磁盘上的有序文件，这是 L0 层。随着 L0 文件增多，后台 Compaction 线程将多个 Sorted Run 合并为更大的有序文件并推到更高层级，合并过程中按主键去重并执行 merge-engine 定义的合并逻辑。读取时需要多层合并读，但配合 Deletion Vectors 可以跳过已被覆盖的记录，加速读取。这个结构的好处是写入极快（顺序追加）、天然支持高频更新，代价是读取需要多层合并，所以 Paimon 更适合写多读少的实时数仓场景。

### Q33. Paimon 有什么局限性？哪些场景不适合用？

三个主要局限。第一是时效性上限：Paimon 的流读时效性受 Checkpoint 间隔制约，通常 1-10 秒级，做不到 Kafka 的毫秒级，所以对延迟极高的场景不适合，比如风控拦截。第二是高频更新下的 Compaction 压力：如果单个 Bucket 写入 QPS 非常高，LSM Tree 的 Compaction 可能跟不上，导致读放大严重。我们通过合理设置 Bucket 数量分散压力和调优 Compaction 策略来缓解。第三是生态成熟度：Paimon 相比 Hive 和 Iceberg 生态还年轻，部分 BI 工具支持不够完善，所以我们 ADS 层查询仍然用 Doris。

### Q34. 你提到 Paimon 做了 Bucket 调优，具体怎么做的？有没有踩过坑？

踩过一个比较典型的坑。刚上线时某个热点表的 Bucket 数设得太少，只有 4 个，但有一个业务维度的数据量特别集中，导致某个 Bucket 的写入量是其他的好几倍，Compaction 速度跟不上写入速度，L0 文件堆积，流读延迟从秒级退化到分钟级。排查定位后的解决方案是：第一，增加 Bucket 数到 16，按主键哈希分散写入压力；第二，调大 sort-spill-threshold 和开启 async-file-write，提升单 Bucket 的写入吞吐；第三，调整 Compaction 的触发阈值，让合并更及时。调优后流读延迟稳定在 3-5 秒。这个经验告诉我，Paimon 的 Bucket 数量不能拍脑袋定，要根据数据分布和写入 QPS 做压测。

### Q35. Flink 反压怎么处理？

反压本质上是数据生产速率大于下游消费速率时系统自动触发的反馈机制。我处理反压的核心原则是先定位，再分析，最后对症下药。第一步是精准定位瓶颈节点——打开 Flink Web UI 看算子的 Backpressure 面板，结合 Metrics 指标看 outPoolUsage 和 inPoolUsage 判断是下游拖累还是自身计算能力不足。第二步分三种情况应对：如果是链路过长或资源不足，调整并行度，检查目标表分桶数是否太少导致写入阻塞（之前把分桶数从 4 调到 16 后反压就缓解了）；如果是 Sink 写入端瓶颈，增大 batch.size 和 flush.interval 减少小事务压力；如果是业务逻辑瓶颈，检查是否存在数据倾斜，对 Key 打散或两阶段聚合。第三步注意反压和 Checkpoint 的联动——严重反压会导致 Checkpoint 超时失败，需要同步调整 Checkpoint 间隔或超时时间。

### Q36. Paimon 的 Changelog 是怎么生成的？有几种模式？

Paimon 生成 Changelog 有三种模式。第一种是 Input 模式，直接把写入的原始数据作为 Changelog 输出，适合上游已经是 CDC 数据流的场景，不做额外处理，开销最小。第二种是 Lookup 模式，写入数据时 Lookup 已有的旧值，生成完整的 before/after 变更记录，开销稍大但下游能拿到完整的变更信息。第三种是 Full Compaction 模式，在 Compaction 的时候对比前后两次 Compaction 的结果，生成差异 Changelog，开销最大但最准确。我们大部分场景用的是 Lookup 模式，因为下游需要完整的变更信息做增量计算。对于性能敏感的场景会用 Input 模式，前提是上游的 CDC 数据本身就携带了 before/after 信息。

## 十二、Data Agent（Q37-Q41）

### Q37. AI 友好数据资产具体做了什么？

三个层面。第一，结构化元数据：把线上表和字段抽象为结构化元数据描述，包含业务含义、数据类型、口径定义、关联维度等，AI 直接读取理解，跳过"模拟人翻 DDL"的转换层。第二，规范固化：把数仓开发规范（分层标准、命名规范、DQC 规则、血缘映射）固化为 AI 可理解、可执行的配置，让 AI 稳定输出符合分层架构和质量要求的数仓资产。第三，质量保障升级：传统报表容错率较高，脏数据最多导致某张表不准；但 AI 取数会直接生成分析结论，脏数据会污染整个分析链路，所以对数据质量要求远高于传统场景。我们的 DQC 七大维度 100% 覆盖和实离对账 0.05% 的差异率，就是 AI 友好数据资产的质量底座。

### Q38. Data Agent 整体架构是什么？为什么不用裸 NL2SQL？

架构五层：需求澄清层（理解用户自然语言意图，解析为结构化取数需求）、权限预检层（校验用户是否有权查看目标数据，行级 + 列级）、取数执行层（基于结构化配置生成标准 SQL，不是裸 NL2SQL）、分析层（多维交叉、同环比、趋势分析，宏观-中观-微观三层递进）、报告层（HTML 渲染可视化报告）。其中 analysis_blueprint 是贯穿流程的核心中间对象：需求阶段生成分析蓝图，取数阶段据此生成 SQL，分析阶段按蓝图执行，最后将实际查询结果和 SQL 一起传给报告模块，保证需求、取数、分析和展示口径一致。

不用裸 NL2SQL 的原因：Spider2.0 基准测试中，GPT-4 在 Spider1.0 上 91.2% 准确率，但 Spider2.0（企业级复杂场景）仅 21.3%。裸 NL2SQL 在企业级场景不可靠。我们的方案是"大模型理解意图 + 结构化配置 + SQL 模板拼装"——大模型负责理解用户意图，输出结构化 JSON；取数层基于 JSON 拼装标准 SQL 模板。好处是不会生成危险操作（DROP/DELETE 被模板禁止）、不会有语法错误（模板预校验）、可审计（JSON 意图记录 + SQL 模板可追溯）。SQL 准确性双保险：SQL 模板预校验（EXPLAIN 跑通才执行）+ 结果层合理性校验（空值率、波动率异常拦截）。

### Q39. 约束前置是什么设计？

约束前置 = 把规则在执行前注入 SQL，不是事后过滤。举例：用户要查"全国采购额"，约束前置会在生成的 SQL 中自动注入品类权限范围 WHERE category_id IN (用户有权查看的品类列表)，而不是查出全部数据后再按权限裁剪。好处是安全性（无权数据不进入结果集，不存在泄露风险）、性能（减少扫描数据量）、可审计（约束条件记录在 SQL 中可追溯）。除了权限约束，还有口径约束——已取消/未提交/已关闭的废单子每次都先剔掉再算，保证口径一致。整体安全红线包含取数通道只准走正门、配置只读不许用户改、blueprint 防篡改；数据可信包含不截断、不编数、统一口径过滤；流程约束包含按蓝图执行、自动出报告；可追溯包含 SQL 留痕、提示准确。

### Q40. 分层降级策略是什么？

多级降级，每级有明确触发条件和用户提示。一级全量精确取数（正常路径，实时 + 全维度）；二级降粒度（查不到城市级 → 降到品类级）；三级降时效（实时查不到 → 查 T+1 快照）；四级人工兜底（Agent 无法处理 → 转数据团队）。降级和错误的区别：降级仍然返回可用结果但降了精度（透明告知用户当前结果级别），错误是返回不了结果。除了分析降级链，还有技术执行降级：取数队列降级（首选队列资源不足时自动切备用队列重试）、认证降级（SSO 换票被拦时降级走 CIBA 认证，最终降级到提示用户手动登录）、截图降级（报告截图上传失败时降级为本地展示）。核心原则是每一步失败都有下一个兜底，绝不因为一个环节挂了就直接把整个流程报错终止。

### Q41. 幻觉问题怎么处理？

幻觉控制核心：不让模型自由生成 SQL。大模型只输出结构化意图 JSON，SQL 由模板拼装。幻觉风险主要在意图理解环节，通过反问确认 + 历史模式匹配兜底。具体来说，需求澄清阶段先做中文业务描述到字段映射，只对真正影响结果的歧义进行澄清（例如时间范围、品类范围）；能够从用户表达中推断分析目标时就不重复提问，无法判断时才进行一次引导。比如用户说"帮我看看最近 IT 采购有没有问题"，会拆成：识别"最近"的时间歧义，识别"IT"对应的品类层级，确认"有没有问题"对应综合健康度分析，然后构建宏观 KPI、中观问题定位和异常下钻计划。模型选择用的是内部模型（Friday 平台），成本控制通过结构化输出减少 token 消耗、缓存常用意图模式。

## 十三、综合与开放题（Q42-Q45）

### Q42. 这个项目的技术栈选型是什么？

离线：Hive + Spark（批处理、数仓建设）。实时：Flink + Paimon（实时 ODS/DWD、Changelog 流读、Aggregate Table）。OLAP：Doris（多维查询、看板查询）。修数：Spark SQL（直接修复 Paimon 表）。调度：公司内部调度。Data Agent：Python + 大模型 API + Skill 框架。数据治理：DQC 平台 + 指标门户 + 一致性监控工具。选型理由：和团队已有技术栈对齐；Doris 适合采购多维分析场景（事前优化成本远低于事后）；Paimon 和 Flink 深度集成，开发体验最好。

### Q43. 你个人的核心贡献是什么？

角色：数仓架构设计主导者 + 实时湖仓设计者 + Data Agent 设计者 + 数据治理推进者。核心贡献：规范制定 0-1（需求对接/建模/指标/运维全套规范）；数仓分层架构设计（ODS-DWD-DWS-ADS + DIM）；实时湖仓架构设计与落地（Paimon merge-engine 选型、Changelog 流读、Aggregate Table、流批同源）；DQC 规则体系搭建（七大维度校验 + 强弱分级）；实时三层校验体系（写入层/加工层/产出层）；Data Agent 全链路设计（五层架构 + 约束前置 + 分层降级 + 审计）；指标一致性治理方案设计（四步治理 + 一致性监控工具）。团队协作部分如实说：DQC 规则是团队共同维护，但规则体系设计是我做的；Doris 稳定性治理和 DBA 协作，治理框架和 SOP 是我主导的。

### Q44. 遇到的最大困难是什么？怎么解决的？

准备两个故事。技术困难——22 个系统的数据口径统一，最难的是境内外供应商编码体系不一致。解决方式：建供应商映射表统一主键，品类三级层级映射，币种统一汇率转换。这是持续过程不是一次性。协作困难——推动业务方遵守数仓规范。解决方式：不靠文档靠约束机制。建指标门户让口径透明化，不登记口径不上看板、不评审的需求不接。让规范有约束力而不只是纸面文档。另外实时升级阶段也有挑战——引入 Paimon 时踩过 Bucket 数设太少导致热点、L0 堆积流读退化的坑，通过增加 Bucket 到 16、调大 sort-spill-threshold、开启 async-file-write 解决。

### Q45. 如果重新做这个项目，你会有什么不同的做法？

两个方面。第一是 Paimon 引入的时机可以更早。我们是先搭了离线数仓体系，稳定运行后才引入 Paimon 做实时升级，这意味着有一轮迁移成本。如果一开始就清楚业务会从 T+1 报表演进到实时监控，可以在初始架构中就把 Paimon 作为统一存储层，减少后续的迁移工作。第二是 Data Agent 意图理解可以引入用户历史查询模式做个性化推荐，减少澄清轮次。另外 DQC 自动根因分析也值得做——告警后自动定位异常源头，减少人工排查时间。这些都是具体可落地的改进，不是大方向上的技术栈更换。
