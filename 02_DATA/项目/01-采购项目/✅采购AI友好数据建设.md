### Q1. 为什么要从0到1建采购数仓？之前的具体痛点是什么？

**参考回答**：

### Q2. "面向AI赋能的数据资产"具体指什么？和传统数仓的区别？

**参考回答**：

传统数仓面向人和BI工具，面向AI的数据资产有三个核心差异：

1. 数据可消费性：传统数仓用户是人，通过看板看数据；AI友好数据资产要求数据有规范语义、可被程序化查询。比如把线上表和字段抽象为结构化元数据描述，AI直接读取理解，跳过"模拟人翻DDL"的转换层。
2. 质量要求更高：传统报表容错率较高，脏数据最多导致某张表不准；但AI取数会直接生成分析结论，脏数据会污染整个分析链路，对数据质量要求远高于传统场景。
3. 规范驱动AI生产：把数仓开发规范（分层标准、命名规范、DQC规则、血缘映射）固化为AI可理解、可执行的"规范&资产"，让AI稳定输出符合分层架构和质量要求的数仓资产。

我们的Data Agent就是建立在这套AI友好数据资产之上的——它能理解指标定义、品类层级、供应商维度的语义，从而做准确的多维取数和分析。

### Q3. 采购业务的核心业务过程有哪些？品类/供应商/订单流转链路？

**参考回答**：

采购业务分三种类型：集中采购（公司统一采购）、生产性采购（直接用于生产经营）、采销（采买后直接或加工后销售）。

核心业务过程链路：需求提报 -> 询价比价 -> 合同签订 -> 下单 -> 到货验收 -> 入库 -> 对账结算。每个环节的数据产出物不同：需求单、报价单、合同、采购订单、验收单、入库单、结算单等十余种单据。

间接采购（采购商城&工作系统）到货后直接形成资产或费用化；直接采购（旭日等系统）到货后进入库存管理，按生产过程领用。这两类采购的数据模型设计有差异，间接采购是线性流转，直接采购涉及全生命周期循环管理。

在数仓建模时，按业务过程拆分事实表，品类和供应商作为核心一致性维度贯穿全链路。

### Q4. 22个系统怎么分类？境内外的数据差异在哪？

**参考回答**：

22个系统按数据域分类：5个采购自有系统（采购域），17个外部系统（财务域、主数据域、业务域等）。

境内系统包括旭日（生产性采购）、芥末（买菜/优选）、驼峰、海纳百川等；境外系统包括Jupiter、Mars，覆盖keeta（沙特、香港）、餐饮SaaS、无人机等业务。

- 币种不同：境内人民币，境外涉及沙特里亚尔、港币等，需要统一汇率转换

解决方式是在ODS层保留原始数据，DWD层做标准化时统一币种转换、供应商编码映射、品类层级对齐，建立映射关系表持续维护。

## 二、数仓规范与架构

### Q6. 数仓规范0-1怎么制定的？包含哪些内容？怎么让业务和产品达成共识？

**参考回答**：

规范分四大块：
1. 需求对接规范：需求单模板（明确业务背景、指标定义、维度、粒度、数据源）、需求评审流程
2. 数仓建模规范：分层架构（ODS-DWD-DWS-ADS）、表命名规范（分层+主题+业务过程+粒度）、字段命名和词根规范、维度建模方法论
3. 指标管理规范：指标单登记（原子指标/派生指标拆解）、口径定义和审批流程、指标上下线管理
4. 数据运维规范：SLA目标设定、变更管理流程、DQC配置要求、权限管理

让业务和产品达成共识的关键不是写文档，而是建约束机制：
- 不登记口径的指标不上看板
- 不评审的需求不接
- 技术方案评审会拉业务和产品一起参加，让规范有约束力
- 建指标可视化门户，口径透明化，业务方自己就能看到指标定义和血缘

美团内部方法论是"**模型设计即开发、开发即治理**"，规范不是事后约束而是事前驱动。

### Q7. 数仓分层怎么设计？每层职责？

**参考回答**：

四层架构：
- ODS（操作数据层）：业务系统和流量数据直接同步，保留原始细节，binlog采集业务数据，日志系统采集流量数据。BI场景下不开放查询。
- DWD（明细数据层）：对ODS数据清洗、转换、整合，按业务过程组织，基于维度建模设计，在语义层达到统一和标准。事务事实表+周期快照事实表+累积快照事实表。历史可追溯。
- DWS（汇总数据层）：面向分析对象轻度聚合，按业务主题构建宽表（采购执行、供应商绩效、成本分析等）
- ADS（应用数据层）：面向具体应用场景的报表和指标，同步至Doris提供OLAP查询

DWD和DWS的边界：DWD是明细粒度事实表，不做聚合；DWS是按主题轻度聚合，面向分析对象建模。

> 来源：学城文档《01 数仓建模规范》(km.sankuai.com/collabpage/2707392456)

### Q8. 维度建模怎么做的？总线架构？一致性维度怎么保证？

**参考回答**：

基于阿里巴巴OneData体系演化，维度建模核心是总线矩阵设计——业务过程（行）x 维度（列）。

采购域核心维度：品类（一级/二级/三级）、供应商、组织（部门/BGBU）、时间。这些作为一致性维度在多个事实表中统一编码、统一属性。

一致性维度保证方式：
- 维度统一收口到DIM层，所有事实表关联同一套维度表
- 供应商编码统一映射（多系统编码 -> 统一主键）
- 品类层级统一（三级品类体系，跨系统映射）
- SCD处理：供应商信息变更（改名、合并）用SCD Type2拉链表，加start_date/end_date

供应商合并场景（A收购B）：建供应商映射表 + 历史数据回刷，把B的历史数据归到A。

> 来源：学城文档《美团买菜（履约主题）数仓建模体系》(km.sankuai.com/collabpage/1545857587)

### Q9. 统一建模方法具体指什么？之前为什么不统一？

**参考回答**：

之前的问题：各系统各建各的，没有统一分层和命名规范，导致同指标多口径、重复建设严重。具体表现是"烟囱式开发"——需求直接进开发，缺乏模型设计评审和约束。

统一建模方法 = 统一分层 + 统一命名 + 统一维度建模 + 统一指标口径。核心是体系化建模：以维度建模为理论基础、事前治理为理念，元数据贯穿全流程。上承指标维度定义，下接数据生产，实现"数据需求与模型设计统一"和"模型设计与物理实现统一"。

落地方式：模型设计环节沉淀信息架构元数据，驱动和约束物理模型DDL，防止"烟囱式"开发。上线前自动完成业务定义与物理实现一致性验证。

> 来源：学城文档《数据治理一体化实践之体系化建模》

### Q10. 元数据管理怎么做？血缘怎么管理？

**参考回答**：

元数据分三类管理：
- 技术元数据：表结构、字段类型、分区信息（自动采集）
- 业务元数据：指标定义、业务口径、维度属性、业务术语字典（人工登记+审批）
- 管理元数据：责任人、SLA、权限、密级（流程绑定）

血缘管理：调度系统自动采集DAG依赖关系 + SQL解析补充字段级血缘。通过血缘可以追溯指标从应用层到ODS层的完整链路，支持影响分析和根因定位。

在AI友好资产建设中，新增了"AI友好元数据"——把线上表/字段抽象为结构化元数据描述，AI直接读取理解，跳过模拟人翻DDL的转换层。

## 三、数据治理与质量

### Q11. 22个系统300+表的摸底怎么做的？

**参考回答**：

三步走方法论：
1. 系统调研：系统负责人访谈、ER关系收集、数据流梳理
2. 数据探查：抽样数据、字段分布、空值率、枚举值、数据量级评估
3. 评估登记：表级价值评估（业务使用频率）、数据质量评分、是否纳入数仓

输出物是一张数据资产盘点表，记录每个系统的表清单、业务域归属、数据质量现状、优先级。优先级按业务价值和数据质量双维度排序，高价值高质量先入仓。

### Q12. 明细层覆盖度82.3%怎么算的？剩下17.7%为什么没覆盖？

**参考回答**：

覆盖度口径 = 已入DWD层的业务系统表数 / 应入DWD的总表数。22个核心系统中已覆盖的业务表占比82.3%。

剩下17.7%未覆盖的原因：
- 低频使用表（ROI低，优先级排在后面）
- 系统待下线（投入产出不划算）
- 数据质量极差无法入仓（需要源系统先治理）
- 外部系统无数据访问权限（需走权限申请流程）

82.3%不是拍脑袋的数字，是做了优先级取舍的结果。先覆盖高价值核心链路，剩余表按迭代节奏逐步纳入。面试官追问口径时，要能说清楚分子分母怎么定义。

### Q13. 品类主数据有什么问题？清洗规则怎么定的？

**参考回答**：

品类主数据的常见问题：
- 品类编码不统一：各系统各编一套，同一品类在不同系统名称不同
- 层级混乱：某系统的二级品类对应另一个系统的三级
- 停用品类未清理：历史品类仍残留在数据中
- 品类重复：同品类不同编码、不同品类同编码

清洗规则：
1. 统一编码体系：建立品类标准编码表，映射各系统品类编码
2. 层级映射：统一为三级品类体系（一级/二级/三级），跨系统层级对齐
3. 停用标记：停用品类标记但保留历史数据
4. 去重：相同业务含义品类合并为一个标准编码

品类清洗是持续过程不是一次性，治理机制是新增品类走审批流程，变更走变更评审。

### Q14. 供应商维度模型怎么设计？SCD怎么处理？

**参考回答**：

供应商是采购域核心一致性维度。维度表设计：
- 统一供应商编码（主键，映射多系统编码）
- 供应商名称、注册地、资质信息
- 合作状态（在合作/暂停/黑名单）
- 供应商评级
- 关联品类

SCD Type2处理变更：加start_date和end_date拉链表，记录供应商信息变更历史。供应商合并（A收购B）场景：建供应商映射表，把B的历史数据归到A，历史事实表回刷供应商编码。

境外供应商有额外字段：注册国家、币种、合规信息。境内外供应商通过统一编码体系打通。
### Q15. DQC 100%覆盖的规则体系怎么设计？

**参考回答**：

DQC规则按七大校验维度设计：
- 完整性：非空校验、行数校验（ODS和业务库数据量对比）
- 准确性：枚举值校验（枚举值与注释比对）、范围校验、格式校验
- 一致性：主外键校验、跨表数据对齐
- 唯一性：主键去重校验（count(1) vs count(distinct pk)）
- 关联性：函数关系、主外键关系、索引关系
- 及时性：任务产出时间监控、数据就绪延迟告警
- 真实性：数据波动稽查（同环比异常告警）

100%覆盖指核心链路所有DWD表都配了DQC规则。规则分强弱两级：
- 强规则：阻断下游（数据不对不产出），电话告警+终止任务
- 弱规则：告警不阻断，邮件短信告警，当天处理完再上线

DQC运营机制：无效DQC响应在1个工作日内完成，DQC加白机制（日常观测型DQC默认通过）。

> 来源：学城文档《数据质量管理（DQC）》(km.sankuai.com/collabpage/1419086026)

### Q16. 指标一致性治理具体做了什么？89%到97%怎么来的？

**参考回答**：

面对6000+表、口径频繁不一致的问题，四步治理：
1. 盘点：梳理所有表的全部指标，建立指标台账
2. 定义：统一口径（命名、定义、取数来源、计算逻辑、加工链路一致性）
3. 治理：找到不一致的指标，推动修改。通过一致性监控工具（对比两个模型/应用中相同指标、相同维度值下的数据是否一致）
4. 工具化：指标门户登记，不登记口径的指标不上线。一致性监控自动化，系统自动计算一致率

一致率口径 = 检测一致的指标次数 / 所有指标检测次数。89%到97%靠的是存量指标口径统一 + 增量管控（新指标必须登记口径才上线）。

剩下3%不一致的原因：历史遗留口径业务方暂不统一、跨域指标归属不清需协调。

### Q17. 数据指标可视化门户怎么实现无需人工对数？

**参考回答**：

三层机制实现：
1. 产出层实离对账：实时和离线结果比对，差异率低于0.05%
2. 波动监控：指标值同环比异常自动告警（超过阈值触发）
3. 口径血缘可追溯：下游发现问题不用找人对数，直接在门户查指标定义和血缘链路定位

传统模式是"业务发现数据不对 -> 找数据团队 -> 人工核对 -> 排查问题"，门户模式是"业务在门户看到异常告警 -> 点开血缘定位 -> 自助解决"。问题响应效率从天级提升到分钟级。

### Q18. 存量应用100%数据源切换怎么做的？迁移策略？

**参考回答**：

迁移策略四步走，核心是灰度+回滚：
1. 双跑：新旧数据源并行产出，新旧结果比对验证
2. 验证：差异率达标后进入切换
3. 灰度切换：按应用逐个切，不是一次性全切
4. 回滚预案：每个应用切换后保留旧源N天，发现异常一键切回

100%切换不是一蹴而就，分批次推进。先切低风险应用验证流程，再切核心应用。存量治理的关键是消除口径差异——旧数据源和新标准模型的指标口径不一致，切换后统一以标准模型为唯一数据源。

## 四、Data Agent（简历最大亮点也是最大风险点）

### Q19. Data Agent整体架构是什么？为什么用Skill方案？

**参考回答**：

架构五层：
1. 需求澄清层：理解用户自然语言意图，解析为结构化取数需求
2. 权限预检层：校验用户是否有权查看目标数据（行级+列级）
3. 取数执行层：基于结构化配置生成标准SQL，不是裸NL2SQL
4. 分析层：多维交叉、同环比、趋势分析
5. 报告层：HTML渲染可视化报告

整体分为环境初始化、权限预检、需求澄清、按权限取数、多维分析、洞察生成、HTML 可视化交付几个阶段。  
其中 `analysis_blueprint` 是贯穿流程的核心中间对象：需求阶段生成分析蓝图，取数阶段据此生成 SQL，分析阶段按宏观—中观—微观三层执行，最后将实际查询结果和 SQL 一起传给报告模块，保证需求、取数、分析和展示口径一致。

### Q20. 需求澄清怎么做？歧义怎么消除？

你实际设计得比较有特点：先做中文业务描述到字段映射，只对真正影响结果的歧义进行澄清，例如时间范围、品类范围；能够从用户表达中推断分析目标时就不重复提问，无法判断时才进行一次引导。

面试官可能给你现场案例：

> 用户说：“帮我看看最近 IT 采购有没有问题。”你会怎么澄清？

你可以拆成：识别“最近”的时间歧义，识别“IT”对应的品类层级，确认“有没有问题”对应综合健康度分析，然后构建宏观 KPI、中观问题定位和异常下钻计划。
### Q21. 权限预检校验什么？数据权限粒度？

**参考回答**：

两层权限校验：
- 行级权限：采购经理只能看自己负责的品类/供应商范围
- 列级权限：成本价等敏感字段按角色脱敏

预检在取数前完成，无权字段直接不出现在SQL中（约束前置），不是取完再过滤。这样既保证安全又避免性能浪费。

权限配置和企平权限系统打通，权限变更自动同步，不靠人工配置。

### Q22. 多维取数怎么把用户意图转成SQL？是NL2SQL吗？

**参考回答**：

不是裸NL2SQL，是"大模型理解意图 + 结构化配置 + SQL模板拼装"的混合方案：

1. 大模型负责理解用户意图：要什么指标、什么维度、什么过滤条件，输出结构化JSON
2. 取数层基于JSON拼装标准SQL模板，SQL经过模板约束

好处：
- 不会生成危险操作（DROP/DELETE被模板禁止）
- 不会有语法错误（模板预校验）
- 可审计（JSON意图记录 + SQL模板可追溯）

SQL准确性双保险：
- SQL模板预校验：EXPLAIN跑通才执行
- 结果层合理性校验：空值率、波动率异常拦截

行业参考：Spider2.0基准测试中，GPT-4在Spider1.0上91.2%准确率，但Spider2.0（企业级复杂场景）仅21.3%。所以裸NL2SQL在企业级场景不可靠，结构化配置+模板是更可控的方案。

> 来源：学城文档《基于Spider2.0数据集的NL2SQL分享》
### Q23. 约束前置是什么设计？举个例子？

**参考回答**：

约束前置 = 把规则在执行前注入SQL，不是事后过滤。

举例：用户要查"全国采购额"，约束前置会在生成的SQL中自动注入品类权限范围：

WHERE category_id IN (用户有权查看的品类列表)

而不是查出全部数据后再按权限裁剪。好处：
- 安全性：无权数据不进入结果集，不存在泄露风险
- 性能：减少扫描数据量
- 可审计：约束条件记录在SQL中可追溯

对应的是事后过滤（查出全部再裁剪），有数据泄露风险和性能浪费。

### Q24. 分层降级策略是什么？降级到什么？

**参考回答**：

多级降级，每级有明确触发条件和用户提示：

1. 一级：全量精确取数（正常路径，实时+全维度）
2. 二级：降粒度（查不到城市级 -> 降到品类级）
3. 三级：降时效（实时查不到 -> 查T+1快照）
4. 四级：人工兜底（Agent无法处理 -> 转数据团队）

降级和错误的区别：降级仍然返回可用结果但降了精度（透明告知用户当前结果级别），错误是返回不了结果。

### Q25. 审计机制记录什么？怎么追溯？

**参考回答**：

审计日志记录：
- 用户意图（自然语言原文）
- 解析结果（结构化JSON）
- 执行SQL（模板+参数）
- 返回数据摘要（行数、关键字段范围，不记录具体值）
- 降级路径（走了哪级降级）
- 耗时

审计日志存表，支持按用户/时间/指标维度检索。审计数据本身的合规：不记录具体数据值，只记录摘要。

Agent服务于采购经理的日常采购决策分析场景。采购经理90%的分析需求是临时的（"看看这个月某品类的采购趋势"、"对比一下两个供应商的交期"），不是标准看板能覆盖的。

### Q27. 为什么不直接用BI看板？Agent和BI的区别？

**参考回答**：

BI和Agent是互补关系：

### Q28. 用的大模型是什么？成本？幻觉问题怎么处理？

### Q23. 约束前置是什么设计？举个例子？

**安全红线**——防越权、防注入、防用户绕过权限

- 🔒1 取数通道（只准走那道会查权限的正门取数，不准走后门——走后门就等于没人拦着，想看啥数据看啥数据）
- 🔒4 配置只读（"你能看哪些品类"这份名单只能系统自己填，不许用户开口说"把我改成能看全部"）
- 🔒5 blueprint 防篡改（这次要分析啥、看哪些范围是系统定好的，不许用户自己塞张纸条进来说"照我这个来"，那样他就能偷偷把范围撑大、绕过限制）

**数据可信**——保证数是真的、全的、口径一致

- 🔒2 不截断（不许为了图快只取一部分数据，你算总金额、算占比，少取一行就全错了）
- 🔒3 不编数（数必须是真从数据库查出来的，不许 AI 自己"我估摸着差不多这个数"——这条最要命，就是防 AI 张嘴瞎编）
- 🔒10 统一口径过滤（那些已取消、没提交、已关闭的废单子，每次都先剔掉再算，不然今天算的和明天算的对不上）

**流程约束**——保证执行不跑偏、交付闭环

- 🔒7 按蓝图执行（该看哪几个角度开头就定死，AI 不许干到一半自己临时加戏，不然同一个问题问两遍给两个答案）
- 🔒9 自动出报告（分析完文字必须接着把图表报告出出来、打开给人看，不许写完字就停那儿等人催）

**可追溯/可解释**——出问题能查、给用户的话不误导

- 🔒6 SQL 留痕（每份报告都留着这次真正用的查询语句，将来有人说"这数不对"，能翻出来一行行对）
- 🔒8 提示准确（没权限的时候，得告诉用户去申请对的那张表，别让人家申请错了，折腾半天还是用不了

## 二、降级：技术执行失败时的兜底链（这是"出错了怎么办"）

降级是指某一步技术环节挂了，不直接报错终止，而是自动退到下一个可用方案。你 skill 里有好几条降级链：

**取数队列降级**——首选队列资源不足（QUEUE_FULL / NO_RESOURCE）时，`run_query_with_mtdata_bi()` 自动切到备用队列 `FALLBACK_QUEUE` 重试一次。降级到:换个队列继续跑。

**认证降级**——SSO 换票被 IP 校验拦住时，降级走 CIBA 认证；如果 SSO 页面白屏彻底修不好，最终降级到"提示用户手动登录"。降级到:换认证方式，最后退到人工。

**截图/图片降级**——报告截图上传 S3Plus 失败时，降级为本地 `image_read` 直接展示。降级到:本地展示，不依赖云存储。

**意图识别降级**——正常情况下场景标签由 `analysis_blueprint.scene_tags` 直接给；万一蓝图没给，用 `detect_intent_tag()` 关键词匹配兜底打标签。降级到:关键词兜底。

核心原则一句话：**每一步失败都有下一个兜底，绝不因为一个环节挂了就直接把整个流程报错终止**。

### Q25. 审计机制记录什么？怎么追溯？

**参考回答**：

审计日志记录：
- 用户意图（自然语言原文）
- 解析结果（结构化JSON）
- 执行SQL（模板+参数）
- 返回数据摘要（行数、关键字段范围，不记录具体值）
- 降级路径（走了哪级降级）
- 耗时

审计日志存表，支持按用户/时间/指标维度检索。审计数据本身的合规：不记录具体数据值，只记录摘要。

### Q26. 这个Agent上线后的效果？采购经理怎么用的？

**参考回答**：

Agent服务于采购经理的日常采购决策分析场景。采购经理90%的分析需求是临时的（"看看这个月某品类的采购趋势"、"对比一下两个供应商的交期"），不是标准看板能覆盖的。

上线效果（如果数据不好看，说当前是POC到生产过渡阶段，重点验证了可行性和准确性）：
- 日均取数次数
- 平均耗时（对比之前提需求等排期）
- SQL结果准确率
- 用户覆盖（多少采购经理在用）

### Q27. 为什么不直接用BI看板？Agent和BI的区别？

**参考回答**：

BI和Agent是互补关系：
- BI看板：固定维度组合、标准化监控、适合日常经营检视
- Data Agent：交互式取数、自然语言描述需求、Agent理解意图后动态取数，适合长尾临时分析需求

采购经理90%的分析需求是临时的，不是标准看板能覆盖的。BI管标准化、Agent管长尾临时需求。不替代关系，是互补。

### Q28. 用的大模型是什么？成本？幻觉问题怎么处理？

**参考回答**：

模型选择：如果是内部模型就说内部模型（Friday平台），如果用外部模型说具体名称。

成本控制：结构化输出减少token消耗、缓存常用意图模式。

幻觉控制核心：不让模型自由生成SQL。大模型只输出结构化意图JSON，SQL由模板拼装。幻觉风险主要在意图理解环节，通过反问确认 + 历史模式匹配兜底。

行业参考：Spider2.0测试中GPT-4在复杂场景仅21.3%准确率，说明裸NL2SQL不可靠。所以我们的方案是"大模型理解 + 结构化约束"，把幻觉风险控制在意图层而非SQL层。

### Q30. 300+表的ETL怎么调度的？依赖管理怎么做？

**参考回答**：

### Q31. SLA怎么保障的？

**参考回答**：

- 事前：DQC规则拦截脏数据，不合规数据不产出
- 事中：任务监控+异常告警，核心链路重点关注
- 事后：修数SOP + 复盘机制

从局部单点优化升级为全链路结构性治理，搭建"观测-提取-归因-长效机制"闭环体系。T+1生产SLA准点率99%。

### Q34. 遇到的最大困难是什么？怎么解决的？

**参考回答**：

### Q35. 采购数据敏感，数据安全/合规怎么保障的？

**参考回答**：

三层保障：
- 存储层：敏感字段加密存储
- 访问层：行级+列级权限控制（Data Agent权限预检是合规落地关键）
- 审计层：所有取数行为可追溯（审计日志记录意图+SQL+摘要）

Data Agent的约束前置设计就是合规落地的核心——无权数据不进入SQL结果集，从源头杜绝数据泄露。

## 六、延伸对比与开放题

### Q36. 这个项目和供应链实时数仓项目的区别？

**参考回答**：

两个项目互补：

一个解决"数据有多快"，一个解决"数据有多好用"。

### Q37. 你之前也做过从0到1的数仓，这个项目的从0到1差异化在哪？

**参考回答**：

多了两个维度：
1. 面向AI的数据资产建设：不只是给人看，还要给Agent用。数据要有规范语义、可被程序化查询、有质量保障。
2. Data Agent产品化：数仓不只是数据仓库，还是AI的数据底座。Agent直接消费数据资产做取数分析。

之前的从0到1是标准数仓建设，这个是数仓+AI的融合建设。

### Q38. 如果重新做这个项目，会怎么改进？

**参考回答**：

别说没改进空间，也别说大方向改（换技术栈），要说具体可落地的改进。
