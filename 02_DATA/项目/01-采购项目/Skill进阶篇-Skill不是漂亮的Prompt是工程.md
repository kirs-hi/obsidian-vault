# 进阶篇：Skill 不是漂亮的 Prompt，是工程

上一篇，我们用 5 分钟写出了一个能跑的 Skill。但"能跑"和"好用"之间，差的不只是一些细节——差的是一整套工程思维。

这篇文章要解决的问题是：**为什么你的 Skill 越写越乱、越改越脆、效果越来越差？**

答案是：你在润色 Prompt，不在设计能力。

---

### Prompt 思维 vs 工程思维

很多人写 Skill 的方式是这样的：

1. 写一个初版
2. 效果不好，在上面加更多描述
3. 还是不好，把语气改得更强调
4. 继续加，加到文档越来越长、越来越难维护

这是 Prompt 润色的思路——以为文字堆得越多、描述越详细，Agent 就会越听话。

**但 Skill 不是作文，是系统组件。** 你的目标不是"让 Agent 读懂你的意思"，而是"设计一个在边界内可靠执行的能力模块"。

---

### 四件事，从 Prompt 思维切换到工程思维

#### 第一件事：把任务拆原子化

一个 Skill 只做一件事。

**原子化的判断标准：如果这件事可以独立触发、独立完成、独立交付，它就应该是一个独立的 Skill。**

把复杂任务拆成原子 Skill，每个 Skill 的 description 更精确，出问题了容易定位，还可以像积木一样组合成更复杂的工作流。

---

#### 第二件事：专练路由描述

一个好的 description，需要同时包含三层规则：

**① 触发规则**：什么用户请求应该触发这个 Skill

**② 边界规则**：什么情况下不该触发（同样重要，甚至更重要）

**③ 输入输出规则**：需要什么输入，产出什么格式

为什么边界规则这么重要？见一个血泪教训：有人做了一个"帮助写作"的 Skill，没有边界规则。用户说"帮我删掉这篇文章"，Agent 触发了这个 Skill——然后真的把文件删了。

---

**一个场景，两种写法**

场景：你写了一个"帮我整理会议记录"的 Skill。

❌ **烂的写法：**

```
// 代码块
description: 整理会议记录，提取关键信息，生成结构化摘要，方便用户快速了解会议内容。
```

问题显而易见：只写了功能，没有触发边界，没有说需要什么输入，也没有定义输出格式。Agent 看到"帮我写会议记录"（根本没有原始内容）或者"帮我把这段文字翻译成英文"（完全不相关），都可能误触发。

✅ **好的写法：**

```
// 代码块
description: 当用户提供了会议录音文本、聊天记录截图或手写笔记，需要整理成结构化会议纪要时触发（触发词：整理会议记录、生成会议纪要、会议内容总结）。
仅适用于有原始会议素材的整理场景；不适用于凭空生成虚假会议记录、翻译文档、或与会议无关的文字处理需求。
输入：会议原始文本或笔记内容；输出：包含【议题/决议/待办/负责人/截止日期】的标准会议纪要格式。
```

**区别一目了然：烂的 description 只告诉 Agent "我能做什么"，好的 description 同时告诉 Agent "什么时候触发、什么时候拒绝、拿什么进来、交什么出去"。** 三层规则少一层，Skill 就会在某个角落出问题。

---

#### 第三件事：把机械活剥给工具

这是工程思维最核心的一点：**模型做语义理解，脚本做重复操作。**

这不只是效率问题，也是准确性问题。LLM 的"注意力"在长文本中会漂移——越到后面越容易漏掉内容。如果你让模型从 100 条记录里筛选数据，前 20 条可能没问题，后 80 条精度就开始下降。但脚本不会"发挥"，也不会漂移。

**分工原则：**

- 语义理解、意图识别、自然语言生成 → 交给模型
- 数据处理、重复操作、格式转换、API 调用 → 交给脚本/工具

如果你的 Skill 里有大量"让模型做计算机本来该做的事"，这是一个需要重构的信号。

---

#### 第四件事：建评测集

这是最容易被跳过、但最能提升 Skill 质量的一步。

Anthropic 在官方 Blog 里专门强调：**"先定义成功标准，再做评测"**，是 Agent 开发的核心流程——*Good evaluations help teams ship AI agents more confidently.*（好的评测，让团队能更有把握地交付 Agent）

**评测集的最小规模：20 条样本，覆盖 4 类情况：**

| 类型 | 说明 |
| --- | --- |
| 正常触发 | 应该触发且结果正确的标准用法 |
| 边界触发 | 表述不标准但意图明确的模糊请求 |
| 不该触发 | 相关但不对口的请求 |
| 异常输入 | 缺少必要信息的不完整输入 |

有了评测集，每次修改 Skill 之后都可以跑一遍，确保你"修了这里没坏那里"。没有评测集，Skill 的迭代基本靠感觉，很容易越改越不稳定。

---

### 六种工程思维，快速自查

| 思维 | 核心问题 |
| --- | --- |
| 路由思维 | 触发条件，Agent 能准确识别吗？ |
| 边界思维 | 明确说了自己不做什么吗？ |
| 接口思维 | 输入输出有没有明确定义？ |
| 状态机思维 | 执行过程中各状态的成功/失败路径写清楚了吗？ |
| 最小权限思维 | 有没有申请超出需要的权限？ |
| 失败优先思维 | 出错时行为是可预测的吗？ |

---

### 一个诊断工具

当一个 Skill 表现不好，先对号入座：

```
// 代码块
没有触发         → 检查 description 触发规则是否太窄
触发了但结果错   → 检查正文步骤逻辑、工具调用是否正确
触发了不该触发   → 检查 description 边界规则是否缺失
出错了没有提示   → 检查异常处理和 Fallback 逻辑
越改越乱         → 可能需要拆成多个原子 Skill
```

---

### 带走这一条

> **不要把 Skill 当成 Prompt 去润色，要把 Skill 当成能力去设计。拆原子、写边界、剥机械活给工具、建评测集——这四件事做完，你的 Skill 才算从"能用"走向"好用"。**
