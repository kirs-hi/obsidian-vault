# AI Skill 书写与设计技巧

> 核心心智模型：**Skill 不是文档，而是写给"另一个 AI"看的上岗手册。** 它唯一的读者是大模型，唯一的目标是让模型在正确的时机、用正确的方式、稳定地完成任务。本文每一部分都配有可直接套用的枚举清单与正反示例。

---

> 📑 **文档目录**

### O、底层第一性原理

先理解这三条，其余技巧都是它们的推论。

#### 1. 上下文是公共资源

`name + description` 常驻上下文，正文触发后才加载，资源文件按需加载（渐进式披露）。能省的 token 必须省，细节要下沉到 references。

**枚举：哪些内容应该下沉，而不是留在正文**

- 超过 100 行的 API 字段说明
- 完整的数据库表结构（schema）
- 各种变体/框架的差异化细节（AWS vs GCP vs Azure）
- 长篇示例集合
- 出错码对照表
- 历史变更说明

#### 2. 模型本身已经很聪明

不要教模型常识，只补它不知道的——你的私有 schema、内部 API、业务规则、团队约定。每写一句话都要自问："它真的需要这句吗？"

**枚举：典型的"废话"（应删除）**

- "SQL 是一种结构化查询语言……"
- "JSON 是一种数据交换格式……"
- "你需要仔细阅读用户的需求……"
- "请确保输出准确无误……"
- "Markdown 是一种轻量级标记语言……"

**枚举：典型的"干货"（应保留）**

- "本系统的日期字段统一用 `biz_date`，格式 `yyyyMMdd`，不是 `yyyy-MM-dd`"
- "查询前必须先调用 `checkPermission` 做权限预检"

"返回码 4001 表示配额耗尽，需退避 60 秒"

##### 快速判断

每写一句话，都问：

> 删除这句话后，Agent会不会更容易做错？

- 不会：删除；
- 会影响执行路径、指标口径或工具调用：保留；
- 只在特定场景使用且内容较长：下沉到`references/`。

一句话总结：

> 不要教模型已经知道的知识，只补充它不知道、且遗漏后会真正做错的规则。

##### **会议skill为例：**

| 类型 | 不推荐写法 | 问题 | 推荐写法 | 判断原则 |
| --- | --- | --- | --- | --- |
| 通用常识与泛泛要求 | "会议时长是员工参加会议所花费的时间""分析前请仔细理解用户需求""请确保结果准确清晰" | 模型本身已经知道，且没有提供可执行规则 | "仅对命中异常条件的指标下钻；分别从子部门和职级×序列维度计算正向贡献Top3" | 删除不影响执行结果的常识和空话，保留能改变分析路径的规则 |
| 内部业务口径 | "会议次数是员工参加会议的数量" | 只有概念解释，没有说明企业内部如何统计 | "日均会议时长为团队内正式员工日均会议时长的中位数" | 模型无法自行知道的私有指标口径必须保留 |
| 正文内容过载 | 在`SKILL.md`中写入所有指标公式、字段定义、边界情况和大量案例 | 每次触发都加载大量无关内容，核心流程容易被淹没 | "日均会议时长异常时读取`references/日均会议时长贡献规则.md`；长会异常时读取`references/长会分析规则.md`" | 正文保留核心流程和导航，复杂细节按需下沉到`references/` |

##### 最终判断标准

决定一句内容是否值得留在 Skill 中，可以依次问五个问题：

1. **模型本身是否已经知道？**
已经知道的通用常识，删除。
2. **这条规则是否是公司或业务特有的？**
是，则保留。
3. **遗漏后是否容易产生严重错误？**
是，则放在 `SKILL.md`。
4. **是否只在特定指标或少数场景中使用？**
是，则放进 `references/`。
5. **是否属于稳定计算或固定格式？**
计算放 `scripts/`，格式放 `assets/`。

#### 3. 自由度要匹配任务脆性

| 自由度 | 适用场景 | 给模型的东西 | 枚举例子 |
| --- | --- | --- | --- |
| 高（文字启发） | 多种方案都可行、依赖上下文判断 | 原则、启发式 | "根据数据量大小选择合适的聚合粒度" |
| 中（伪代码/带参脚本） | 有偏好模式、允许一定变化 | 模板、可配置脚本 | "用 `query.py --granularity day/week` 取数" |
| 低（固定脚本） | 操作脆弱、要求强一致 | 固定脚本、极少参数 | "PDF 旋转固定调用 `rotate_pdf.py`，不要自己写" |

### 一、基础结构设计（名称 & 描述）

> description 是 Skill 的命门——它是模型唯一用来判断"该不该触发我"的依据。90% 的失败都来自 description 写得烂。

#### 1.description黄金法则

- **description 必须同时写"做什么"+"何时用"（触发场景）**，"何时用"绝不能写进正文（正文触发后才加载，写在正文里模型根本看不到）。
- **触发词要穷举真实用户的口语**，包括同义词、缩写、中英文混说。
- **name 用 kebab-case、动宾或领域名词**，见名知意，不堆形容词。
- **description 要划清边界**——既写"适用"，也写"NOT for（不适用）"，防止误触发抢了别的 skill。
- **frontmatter 只放 **`**name**`** 和 **`**description**`（外加可选 license/compatibility），不要塞自创字段。

##### description 的五要素结构

| 要素 | 回答的问题 | 会议 Skill 示例 |
| --- | --- | --- |
| 核心能力 | 它能做什么 | 分析部门会议指标和变化来源 |
| 分析对象 | 分析什么数据 | 会议时长、频次、长会、大会、会议负担 |
| 使用场景 | 用户何时需要它 | 指标异动、同比环比、贡献来源定位 |
| 用户表达 | 用户可能怎么说 | "会议时长为什么涨了""哪个部门贡献最大" |
| 排除边界 | 什么任务不属于它 | 会议纪要、排期、订室、内容总结 |

##### Description 案例与黄金法则、五要素对应关系

| 案例 | 对应的黄金法则 | Description五要素表现 | 核心问题 |
| --- | --- | --- | --- |
| **Bad Case 1：范围太泛**`name: meeting-analysisdescription: 帮助用户分析会议相关问题。` | 违反法则1：没有同时写"做什么＋何时用"；违反法则2：没有真实用户表达或触发词；违反法则4：没有写适用与不适用边界；法则3格式基本符合，但名称粒度太宽。 | 核心能力：模糊；分析对象：缺失；使用场景：缺失；用户表达：缺失；排除边界：缺失。 | 模型只知道它与"会议"有关，不知道是分析会议数据、整理纪要，还是安排会议。 |
| **Bad Case 2：只写功能，不写何时使用**`name: meeting-metrics-analysisdescription: 计算日均会议时长、长会占比、大会占比和员工会议负担。` | 违反法则1：只写"做什么"，没有写"何时用"；违反法则2：没有覆盖用户真实口语；违反法则4：没有排除相邻任务；法则3符合，名称清晰具体。 | 核心能力：有；分析对象：有；使用场景：缺失；用户表达：缺失；排除边界：缺失。 | 模型知道它会算哪些指标，但不知道用户提出什么问题时应该调用，也不知道是否支持异动分析和来源下钻。 |
| **Bad Case 3：把执行流程塞进Description**`description: 首先读取部门数据，然后计算日均会议时长……排序并输出Top3。` | 法则1中的"做什么"写得过细，但"何时用"仍不突出；违反法则2：写的是系统流程，不是用户怎么说；违反法则4：没有清晰边界。 | 核心能力：被流程细节淹没；分析对象：部分体现；使用场景：不清晰；用户表达：缺失；排除边界：缺失。 | Description 被写成简化版 `SKILL.md`。它应该负责判断是否进入，而不是描述进入后每一步怎么执行。 |
| **Good Case：**帮助管理者和HRBP分析部门会议时长、会议频次、高会议负担员工、长会和大会等指标，判断同比或环比异动，并从子部门和职级×序列维度定位主要正向贡献来源。当用户需要了解部门会议水位、分析会议指标变化原因、定位贡献Top3，或提到"日均会议时长""长会占比""大会次数""大于4小时人数""子部门贡献""职级序列"等内容时使用。不要用于会议纪要、会议内容总结、会议排期、订会议室、邀请函撰写，或仅凭会议时长直接评价会议效率。 | 符合法则1：同时写清做什么和何时用；符合法则2：覆盖用户真实表达和业务术语；符合法则3：`meeting-metrics-analysis` 使用 kebab-case，见名知意；符合法则4：同时写适用场景和"不适用场景"；符合法则5：Frontmatter只保留规范字段。 \| 五要素 \| Good Case 中的对应内容 \| \| --- \| --- \| \| 核心能力 \| 分析会议指标、判断同比环比异动、定位主要正向贡献来源 \| \| 分析对象 \| 会议时长、会议频次、高会议负担员工、长会和大会 \| \| 使用场景 \| 了解会议水位、分析指标变化原因、定位贡献Top3 \| \| 用户表达 \| "日均会议时长""长会占比""大会次数""大于4小时人数""哪个部门贡献最大" \| \| 排除边界 \| 会议纪要、内容总结、会议排期、订会议室、邀请函和无依据的效率评价 \| | 核心能力：分析指标、判断异动、定位来源；分析对象：会议时长、频次、长会、大会和会议负担；使用场景：会议水位、同比环比、贡献Top3；用户表达：日均会议时长、长会占比、哪个部门贡献最大等；排除边界：纪要、排期、订室、内容总结和无依据效率评价。 | Description形成完整路由合同，模型能够判断什么时候调用，也能避免与其他会议类Skill冲突。 |

> Bad Case 1是什么都没说清；Bad Case 2只说了能做什么；Bad Case 3说了太多进入后的流程；Good Case则刚好回答了"能做什么、分析什么、何时使用、用户怎么说、什么不属于它"。

#### 2.Name 命名规则

> **ℹ️ 信息：**
>
> Name 的作用是让人和模型快速判断：这个 Skill 处理什么对象、提供什么稳定能力。
>
> 命名时建议遵循以下规则：
>
> 1. **使用 kebab-case**
> 全部使用小写字母，多个单词之间用连字符连接。
> 2. **使用"领域对象＋核心动作"或"业务对象＋能力类型"**
> 3. **名称应表达稳定能力，不使用临时项目名或版本号**
> 版本信息应放在版本记录中，不应写进 Name。
> 4. **名称粒度应与 Skill 的能力边界一致**
> 5. **避免使用无业务含义的通用词**
> 6. **不要把所有功能都塞进名称**

##### Name 命名规则举例：

| 类型 | 不推荐示例 | 问题 | 推荐示例 | 核心原则 |
| --- | --- | --- | --- | --- |
| 名称缺少能力信息 | `helper`、`tool1`、`processor`、`smart-agent` | 看不出 Skill 处理什么对象、提供什么能力。 | `meeting-metrics-analysis`、`meeting-load-diagnosis` | 使用"领域对象＋核心动作"或"业务对象＋能力类型"，采用 kebab-case。 |
| 名称粒度过宽 | `meeting` | 无法判断它处理会议排期、纪要、订会议室、内容总结还是指标分析。 | `meeting-metrics-analysis` | Name 的粒度要与 Skill 的实际能力边界一致。 |
| 名称过长或混入临时信息 | `department-meeting-duration-frequency-long-meeting-large-meeting-analysis`、`meeting-v2-final` | 前者过长、难维护；后者混入版本和临时状态，无法表达稳定能力。 | `meeting-metrics-analysis` | Name 只表达稳定能力，具体指标和版本信息放进 Description 或正文。 |

##### name + Description 案例对照

| 类型 | 示例 | 主要问题或优点 | 改进原则 |
| --- | --- | --- | --- |
| Bad Case：Name与Description都过于空泛 | `name: helperdescription: 一个很有用的工具，可以帮助处理各种文档相关的事情。` | Name没有信息量；Description没有说明具体能力、使用场景、用户表达和排除边界，模型无法判断何时调用。 | Name负责说明"它是谁"，Description负责说明"什么时候让它上场"。 |

### 二、参数定义规范（类型 & 必填项）

> 适用于带工具调用/脚本入参的 Skill。参数定义是"模型与你的代码之间的契约"。

#### 黄金法则

- **每个参数都写清：类型 + 是否必填 + 默认值 + 一句话用途 + 格式约束**，缺一不可。
- **必填项最小化**——能从上下文推断或给默认值的，就别设为必填。
- **用枚举(enum)收敛开放参数**——只要取值有限，就列成枚举。
- **参数命名自解释**，并在描述里标注格式约束。
- **给"反例约束"**——明确写不接受什么，比只写正例更能防错。

##### 参数定义规范：五条黄金法则与示例

| 黄金法则 | 不推荐示例 | 推荐示例 | 说明 |
| --- | --- | --- | --- |
| **1. 每个参数都写清完整定义** | `参数：分析日期` | `分析开始日期`类型：日期字符串是否必填：是默认值：无用途：指定会议数据分析周期的开始日期格式：`YYYY-MM-DD`，如`2026-05-01` | 参数至少应说明：类型、是否必填、默认值、用途和格式约束，避免模型自行猜测。 |
| **2. 必填项最小化** | 要求用户必须提供：部门、起止日期、对比方式、下钻维度、Top数量、小样本阈值 | 必填：目标部门、分析周期默认：对比方式为环比、下钻维度为子部门和职级×序列、返回Top3、小样本阈值为10人 | 能从上下文推断或有合理默认值的参数，不要设为必填，减少无意义追问。 |
| **3. 有限取值使用枚举** | `对比方式：文本，用户自由填写` | `对比方式：枚举`允许值：环比、同比、上一周期对比 | 用户可能说"跟上个月比""看环比"，但传给工具时应统一收敛为固定值。 |
| **4. 参数命名自解释，并写清格式或范围** | `date`、`type`、`num` | `分析开始日期：YYYY-MM-DD对比方式：环比/同比返回排名数量：整数，范围1—10，默认3` | 参数名本身要体现含义；日期、数量、集合等参数还要注明格式、范围或分隔方式。 |
| **5. 明确反例约束** | 只写：`分析开始日期示例：2026-05-01` | 正确：`2026-05-01`不可直接传入：`昨天`、`上个月`、`2026年5月1日`正确：`返回排名数量=3`不可传入：`前三`、`Top3`、`0`、`-1` | 正例告诉模型应该怎么传，反例明确哪些自然语言或非法值不能直接传入。 |

一句话总结：

> 参数定义要让模型明确知道"传什么、怎么传、哪些值合法，以及缺少参数时应该使用默认值、从上下文推断还是向用户追问"。

#### 枚举：一个完整参数定义应包含的字段

| 字段 | 必要性 | 示例 |
| --- | --- | --- |
| 参数名 | 必须 | `start_date` |
| 类型 | 必须 | `string` / `int` / `enum` / `bool` / `array` |
| 是否必填 | 必须 | 必填 / 可选 |
| 默认值 | 可选项必须 | 默认 `"day"` |
| 取值范围 | 有限制时必须 | `1-100` / `day/week/month` |
| 格式约束 | 有格式时必须 | ISO-8601，如 `2026-06-16` |
| 用途说明 | 必须 | "返回的数据条数上限" |
| 反例约束 | 易错时建议 | "不接受相对日期如'昨天'" |

#### 枚举：常见参数类型与收敛方式

| 类型 | 收敛建议 | 例子 |
| --- | --- | --- |
| 日期 | 强制绝对格式 + 标注格式 | `start_date (string, ISO-8601)` |
| 选项 | 用 enum 列全 | `type (enum: line/bar/pie)` |
| 数量 | 给范围 + 默认值 | `limit (int, 1-100, 默认 10)` |
| 开关 | 用 bool + 默认值 | `dry_run (bool, 默认 false)` |
| ID 集合 | 标注上限 + 分隔符 | `ids (最多 5 个，逗号分隔)` |

#### Bad Case ❌

```
// 代码块
参数：date, type, num
执行查询。
```

> 类型不明、是否必填不明、格式不明、type 取值范围不明，模型只能瞎猜。

#### Good Case ✅

```
// 代码块
参数：
- start_date (string, 必填)：起始日期，ISO-8601 格式，如 "2026-06-16"
- granularity (enum, 可选, 默认 "day")：聚合粒度，取值 day | week | month
- limit (int, 可选, 默认 10, 范围 1-100)：返回条数
注意：日期不接受相对值（如"昨天"），需先解析为绝对日期再传入。
```

### 三、意图识别优化（如何让模型更精准命中）

> 模型选 skill 的逻辑：把用户输入与所有 skill 的 description 做语义相似度匹配，选最像的那个。

#### 黄金法则

- **触发词覆盖"用户怎么说"而非"功能叫什么"**。
- **多 skill 共存时，靠"具体度"竞争**——越具体越容易胜出；通用 skill 要主动让路。
- **正向锚点 + 负向排除双管齐下**。
- **场景编号化**——把触发场景写成 (1)(2)(3)。
- **中英/缩写/别名全列**——一个都不能少。

##### 意图识别优化：五条黄金法则与会议 Skill 示例

| 黄金法则 | 不推荐或常见问题 | 会议 Skill 示例 | 作用 |
| --- | --- | --- | --- |
| **1. 触发词覆盖"用户怎么说"，而不只写"功能叫什么"** | 只写：会议指标分析、会议负担诊断。用户没有使用专业名称时，可能无法命中。 | 用户可能会说："最近团队是不是会太多了？""为什么这个月会议时长涨了？""哪个部门把会议时间拉高了？""帮我看下长会是不是变多了。" | 覆盖用户真实表达，使模型即使没看到专业术语，也能识别用户意图。 |
| **2. 多 Skill 共存时，用具体度区分；通用 Skill 主动让路** | 通用数据分析 Skill："分析业务数据、指标变化和异常问题。"描述过宽，容易与专业 Skill 抢任务。 | 会议指标分析 Skill："分析部门会议时长、会议频次、长会、大会及高会议负担员工指标，并定位子部门和职级×序列贡献 Top3。"用户问"哪个子部门对日均会议时长上涨贡献最大"时，应由会议分析 Skill 处理。 | 越具体的 Skill 越能准确承接专业任务，避免通用 Skill 错误抢占。 |
| **3. 正向锚点和负向排除同时写** | 只写适用场景，不写不适用场景，容易和会议纪要、会议排期等 Skill 冲突。 | `description: 当用户需要分析会议时长、会议频次、长会、大会、员工会议负担、同比环比变化或贡献Top3时使用。不要用于会议纪要、会议排期、会议室预订、单场会议内容总结或会议邀请撰写。` | 正向锚点告诉模型何时触发；负向排除告诉模型何时不要触发。 |
| **4. 将多个触发场景编号化** | 把所有场景挤在一个长句中，模型和阅读者不容易识别边界。 | `description: 当用户需要：（1）了解部门整体会议水位；（2）判断会议指标同比或环比是否异常；（3）从子部门和职级×序列维度定位正向贡献Top3；（4）分析长会、大会或高会议负担员工情况时使用。` | 将不同使用场景拆开，提升可读性，并使触发边界更加清晰。 |
| **5. 覆盖中文、英文、缩写和常见别名** | 只写一种正式中文名称，用户使用英文、缩写或口语时可能漏触发。 | 中文正式名：日均会议时长、长会占比、高会议负担员工。中文口语：会是不是太多了、天天都在开会、长会变多了吗。英文或缩写：meeting load、meeting duration、MoM、YoY、Top3。常见别名：大会、多人会议、15人以上会议。 | 覆盖真实使用中的语言变体。例如用户说"帮我看 meeting load 的 MoM 变化"，仍能识别为会议分析任务。 |

一句话总结

> 意图识别优化的核心，是让 Description 同时覆盖用户真实表达、专业场景和排除边界，而不是简单堆砌功能名称或关键词。

#### 枚举：触发词要覆盖的语言变体

| 变体类型 | 例子（学城文档 skill） |
| --- | --- |
| 中文正式名 | 学城、知识库 |
| 中文口语 | 文档、写个文档、记到文档里 |
| 英文/拼音 | km、wiki、citadel |
| 链接特征 | 内部文档平台URL特征（collabpage、page路径） |
| 相关概念 | contentId、parentId、流程图、划词评论、密级 |

#### 枚举："用户怎么说" vs "功能叫什么"对照

| 功能术语（不该只写这个） | 用户口语（必须补上） |
| --- | --- |
| 执行 BI 数据集查询 | 拉个数、查下 GMV、看下昨天订单量 |
| 文档内容提取 | 看下这个文档讲了啥、总结一下 |
| 节点截图 | 截个图、snapshot、看下屏幕现在啥样 |
| 创建日程 | 安排个会、约一下、定个时间 |

#### 枚举：负向排除（NOT for）的典型场景

- 易混淆的相邻能力（SQL skill 排除 DDL、Kafka 实时流）
- 通用 skill 主动让位给专用 skill
- 明确不支持的数据源/文件类型
- 高危操作（写生产、删除）需走另一条路径

#### Bad Case ❌

```YAML

```

> 用户说"昨天哪个接口请求最多"时，语义里没有"SQL/Hive"，命不中。

#### Good Case ✅

```YAML
// 代码块
description: 执行 SQL 或用自然语言描述查询需求（如"昨天哪个 appkey 请求最多"
  "统计 xxx 趋势"），即使未显式提到 SQL 或 Hive 也触发；搜索表/查字段结构
  （"看下 xxx 表""有哪些字段"）；检查 SQL 质量。
  NOT for：Kafka 实时流、DDL(CREATE/ALTER/DROP)、数据写入、非公司数据源。
```

### 四、边界与异常处理

> 模型最容易在"出错时该怎么办"上裸奔。好 Skill 会把失败路径也写进剧本。

#### 黄金法则

1. **预设失败分支**——明确每类错误对应的处理动作。
2. **脆弱步骤用脚本而非文字**——一错就崩的操作，提供脚本兜底。
3. **写明前置校验**——把校验左移。
4. **空结果/弱结果要有退路**。
5. **划定红线（安全边界）**——明确哪些操作必须停下来问人。

##### 边界与异常处理：五条黄金法则与会议 Skill 示例

| 黄金法则 | 会议 Skill 示例 | 正确处理 | 核心作用 |
| --- | --- | --- | --- |
| **1. 预设失败分支：每类异常都对应处理动作** | 用户要求分析产品部5月会议数据，但接口返回空结果。 | ①检查部门名称和分析周期；②检查数据是否已更新；③使用相同口径重查一次；④仍为空时，明确告知"当前周期暂无可用数据"，不生成分析结论。 | 避免工具出错或数据为空时，Agent直接停止、无限重试或编造结果。 |
| **2. 脆弱步骤使用脚本处理** | 贡献值计算、正向贡献筛选和Top3排序容易在大批量数据下出错。 | 调用脚本完成：①计算贡献值；②过滤贡献值≤0的对象；③按贡献值降序排序；④提取正向贡献Top3；⑤不足3项时按实际数量返回。Agent只负责解释结果。 | 让脚本负责稳定计算，让Agent负责业务判断和结果解释。 |
| **3. 写明前置校验：把错误拦在分析前** | 在计算会议指标和贡献Top3之前，需要确认数据是否具备分析条件。 | 先检查：目标部门、分析周期、日期顺序、对比周期数据、分项与整体数据是否一致，以及占比指标是否同时具备分子和分母。缺少对比周期时，不进入同比、环比和贡献Top3分析。 | 避免分析做到一半或输出结论后，才发现输入数据不可用。 |
| **4. 空结果或弱结果要有退路** | 情况一：A部门贡献30，B部门贡献10，C部门贡献-5。情况二：某职级×序列贡献排名第一，但只有4人。 | 情况一：只输出A、B，不使用负贡献补足Top3。情况二：可以展示，但必须标记"小样本"，不得作为强结论。 | 数据不足或证据较弱时，仍输出可信的部分结果，而不是强行凑齐或过度解读。 |
| **5. 划定红线：高风险操作必须先问用户** | 发送员工会议负担名单、写入正式数据库、覆盖正式报告、批量修改员工标签或权限、自动向员工发送提醒。 | 默认只读取数据、计算并生成草稿。执行发送、写入、覆盖或批量修改前，必须明确询问用户确认。例如："已生成高会议负担员工名单，是否确认发送给部门管理者？" | 防止Agent在未经授权的情况下执行外发、写入或不可撤销操作。 |

#### 枚举：必须预设处理动作的异常类型

| 异常类型 | 预设处理动作 |
| --- | --- |
| 命令不存在（command not found） | 重读 SKILL.md → 跑安装步骤 → 重试 |
| 认证失败 / token 过期 | 清缓存重新登录，仍失败则提示用户 |
| 限流 429 | 按 Retry-After 退避重试，最多 N 次 |
| 查询结果为空 | 放宽条件/换同义词重查一次，仍空才告知"无数据" |
| 参数校验失败 | 指出缺哪个字段，只追问必要字段 |
| 版本冲突（如文档被他人改动） | 重新拉取最新内容后重试，禁止用旧内容覆盖 |
| 权限不足 | 立即停止，告知用户无权限，不伪造结果 |

#### 枚举：必须列入红线的高危操作（执行前停下问人）

- 删除类：DELETE / DROP / 删文件 / 删评论
- 写生产类：数据写入、配置变更、发布上线
- 外发类：发邮件、发公开帖、对外传文件
- 不可撤销类：覆盖还原、批量改权限
- 资金/合同类：任何涉及钱或对外承诺的操作

#### 枚举：前置校验左移清单

- 调用写接口前 → 先做权限预检
- 处理日期前 → 先把相对日期解析为绝对日期
- 批量操作前 → 先确认数量与范围，避免误伤
- 编辑文档前 → 先拉取最新内容（先读后改）

#### Bad Case ❌

```
// 代码块
运行 `mytool query` 获取数据，然后总结结果。
```

> 工具不存在怎么办？查询超时怎么办？返回空怎么办？全是空白。

#### Good Case ✅

```
// 代码块
## 执行
运行 `mytool query`。

## 异常处理
- 报错含 "command not found" → 重新读本 SKILL.md，执行「安装」章节，再重试
- 返回 429 → 按 Retry-After 退避后重试，最多 3 次
- 结果为空 → 放宽筛选条件或换同义词重查一次；仍为空才告知用户"无数据"

## 红线（必须停下问人）
- 任何 DELETE / DROP / 数据写入操作，先向用户确认再执行
```

### 五、内容组织与渐进式披露（结构心法）

> 决定一个 Skill 是"轻盈好用"还是"上下文黑洞"的关键。

#### 黄金法则

1. **三级加载，各司其职**。
2. **正文只留"核心流程 + 导航"**，细节下沉 references 并写明"何时去读哪个文件"。
3. **按领域/变体拆分 references**。
4. **引用只下沉一层**——不要套娃式嵌套。
5. **不放冗余文件**——README/CHANGELOG/安装指南统统不进 Skill。

##### 黄金法则与会议 Skill 示例

| 黄金法则 | 不推荐做法 | 会议 Skill 示例 | 作用 |
| --- | --- | --- | --- |
| **1. 三级加载，各司其职** | 把触发条件、分析流程、所有指标公式和输出模板全部写进一个 `SKILL.md`。 | **L1：**`name + description`，判断是否调用会议分析 Skill。**L2：**`SKILL.md`，规定异常判断、下钻流程和通用 Top3 规则。**L3：**`references/长会分析.md`、`scripts/贡献计算.py`、`assets/管理层汇报模板.md`，按需读取或使用。 | 让不同层级只承担自己的职责，减少无关内容占用上下文。 |
| **2. 正文只留"核心流程＋导航"** | 在正文中写入每个会议指标的完整公式、边界情况和大量案例。 | `SKILL.md`只写：①判断哪些会议指标命中异常；②仅对命中指标下钻；③从子部门和职级×序列维度计算贡献 Top3。当长会占比异常时，读取 `references/长会分析.md`，用于分析长会次数和总会议次数变化。 | 保持正文简洁，同时让 Agent 知道什么情况下读取哪份详细规则。 |
| **3. 按领域或变体拆分 References** | 所有详细规则都放在 `references/所有会议规则.md`。 | 按完整分析问题拆分：`日均会议时长分析.md长会分析.md大会分析.md高会议负担员工分析.md` | Agent只读取当前问题对应的规则，避免加载整套会议分析知识。 |
| **4. 引用只下沉一层，避免套娃嵌套** | `SKILL.md` → `长会分析.md` → `占比分析.md` → `分母变化规则.md`。 | `SKILL.md`直接指向 `references/长会分析.md`，该文件中包含完成长会分析所需的完整规则，不再继续引用其他 Reference。 | 减少多层追踪导致的漏读、错读和规则优先级混乱。 |
| **5. 不放冗余文件** | 将 `README.md`、`CHANGELOG.md`、安装指南、创建过程记录和测试报告放入 Skill，并要求 Agent读取。 | Skill执行目录只保留：`SKILL.mdreferences/scripts/assets/`版本记录、测试报告和开发说明保留在 Skill 执行上下文之外。 | 避免与实际任务无关的开发资料消耗上下文、干扰执行。 |

一句话总结

> 主文件负责指路，Reference负责讲细节，Script负责稳定计算，Asset负责最终交付；当前任务用不到的内容，不要加载。

#### 枚举：三级加载系统

| 层级 | 内容 | 加载时机 | 体量约束 |
| --- | --- | --- | --- |
| L1 | name + description | 常驻上下文 | ~100 词 |
| L2 | SKILL.md 正文 | 触发后加载 | < 500 行 |
| L3 | references / scripts / assets | 按需加载（脚本可不读直接执行） | 无限 |

#### 枚举：三种资源目录的用途

| 目录 | 放什么 | 是否读入上下文 | 例子 |
| --- | --- | --- | --- |
| `scripts/` | 可执行代码 | 否（可直接执行） | `rotate_pdf.py` |
| `references/` | 供模型按需查阅的文档 | 是（按需） | `schema.md`、`api_docs.md` |
| `assets/` | 进入最终产物的文件 | 否 | `logo.png`、`template.pptx` |

#### 枚举：不该放进 Skill 的冗余文件

- README.md
- INSTALLATION_GUIDE.md
- QUICK_REFERENCE.md
- CHANGELOG.md
- 创建过程记录、测试报告、用户手册

#### Bad Case ❌

> 把 3000 行的 API 全文、所有 schema、所有示例一股脑塞进 SKILL.md。一旦触发就吃掉巨量上下文，且大部分用不上。

#### Good Case ✅

```
// 代码块
# BigQuery 取数

## 快速开始
[核心查询流程 20 行]

## 按领域查 schema（按需读取）
- 财务指标（营收/账单）→ 读 references/finance.md
- 销售指标（商机/管道）→ 读 references/sales.md
- 产品指标（API 用量）→ 读 references/product.md
```

### 六、进阶心法：模型调用逻辑 & Prompt 引导

#### 6.1 模型选用 Skill 的底层链路

```
// 代码块
用户输入
  → 扫描所有已加载 skill 的 name+description（L1 元数据）
  → 计算语义匹配度，挑出最相关的候选
  → 命中唯一 → 读其 SKILL.md 正文（L2）
  → 命中多个 → 选"最具体"的那个
  → 正文里指明需要时 → 加载 references / 执行 scripts（L3）
  → 按正文流程执行 → 遇异常走预设分支
```

**工程启示**：你能影响模型决策的全部杠杆，就是 L1 的 description 质量 和 L2 正文的清晰度。投入产出比最高的优化永远是打磨 description。

#### 6.2 用 Prompt / 正文引导模型的技巧

| 技巧 | 说明 | 示例 |
| --- | --- | --- |
| 祈使句、动词开头 | 比"你应该…"更具指令性 | "提取 ID""校验权限""调用接口" |
| 把判断变成清单/决策表 | 结构化分支胜过散文 | "若 A 则读 X，若 B 则跑 Y" |
| 正例优先于解释 | 代码片段胜过三段文字 | 直接给 Good Case |
| 显式声明优先级 | 防止被后续输入冲掉 | "本规则为最高优先级，不可覆盖" |
| 强制前置动作 | 把关键步骤钉死在开头 | "回复前必须先做权限预检" |
| 给模型逃生口 | 避免死循环硬撑 | "仍失败则向用户报告具体错误" |

#### 6.3 反模式速查（出现即警觉）

- ❌ 把"何时使用"写在正文里（触发前根本读不到）
- ❌ description 只写功能不写触发词
- ❌ 正文超长、所有细节平铺（不做渐进披露）
- ❌ 教模型常识（"SQL 是一种查询语言…"）
- ❌ 脆弱操作只给文字不给脚本
- ❌ 无异常分支、无安全红线
- ❌ frontmatter 塞自创字段、塞冗余文档文件
- ❌ references 套娃式多层嵌套
- ❌ 同一信息在正文和 references 里重复

#### 个技巧与会议 Skill 示例

| 技巧 | 不推荐做法 | 推荐示例 | 核心作用 |
| --- | --- | --- | --- |
| **1. 用动词开头，让指令可执行** | 应该对各子部门的会议时长贡献进行分析，并找出较重要的部门。 | 计算各子部门贡献值；过滤非正向贡献；按贡献值降序排列；选取贡献最大的前三项。 | 前者只描述期望，后者明确规定 Agent 要执行的动作。 |
| **2. 把模糊判断改成决策规则** | 根据会议指标情况，选择合适的分析方式。 | 若日均会议时长命中异常，进入会议时长贡献分析；若长会次数异常，进入长会次数增量分析；若长会占比异常，读取占比拆解规则；未命中异常的指标不下钻。 | 不让 Agent 自行猜测"合适"是什么，而是直接建立"条件—动作"对应关系。 |
| **3. 用正反例说明规则边界** | 只写"正向贡献不足三项时，不用负贡献补足"，但不给示例。 | 研发一部贡献30、产品一部贡献12、销售一部贡献−4。正确结果：只输出研发一部和产品一部；错误结果：为了凑Top3加入销售一部。 | 具体案例可以固定模型的处理模式，减少对抽象规则的误解。 |
| **4. 明确规则冲突时的优先级** | 用户要求输出Top3时，Agent为了满足格式，使用负贡献结果补足。 | ①数据真实性优先于输出数量；②正向贡献规则优先于"必须输出Top3"；③正向贡献不足三项时按实际数量输出。 | 防止不同规则冲突时，Agent为了满足次要要求而破坏数据准确性。 |
| **5. 把关键校验放在分析前面** | 先计算贡献Top3，最后才发现缺少上月数据或下钻明细。 | 计算前先检查当前周期、对比周期、指标口径、下钻维度明细及分项与整体数据是否一致；校验通过后再计算Top3。 | 将错误拦截在流程前面，避免无效计算和错误输出，即"校验左移"。 |
| **6. 给无法完整完成的任务预设退路** | 数据不全时直接停止，或强行编造完整分析。 | 缺少对比周期时，只输出当前会议水位，不计算变化贡献Top3；只有整体数据时，完成整体判断，并说明当前无法进行子部门下钻。 | 无法完整执行时，仍提供可信的部分结果，同时明确数据限制。 |

#### 完整前后对比

| Bad Case | 存在的问题 | Good Case |
| --- | --- | --- |
| 分析会议指标，找出贡献最大的三个部门并输出结论。 | 没有说明分析哪些指标、何时下钻、贡献如何计算、负贡献如何处理、数据不足怎么办、小样本怎么办。 | ①校验当前周期、对比周期和下钻数据；②仅对命中异常的指标下钻；③按照指标规定口径计算贡献值；④仅保留正向贡献并降序排列；⑤选取前三项，不足三项时按实际数量保留；⑥职级×序列人数小于10人时标记小样本；⑦数据不足时输出可完成部分并说明限制；⑧贡献来源不直接解释为会议低效。 |

一句话总结

> 好的正文不是告诉 Agent "做一份分析"，而是把任务写成可执行、可判断、可校验、可降级的操作路径。

### 七、一页速查（贴墙版）

| 维度 | 一句话铁律 |
| --- | --- |
| 名称 | kebab-case，见名知意 |
| 描述 | 做什么 + 何时用 + 触发词 + NOT for，四要素齐全 |
| 参数 | 类型/必填/默认/格式/反例约束，必填项最小化 |
| 意图 | 写"用户怎么说"，不写"功能叫什么"；正负锚点并用 |
| 异常 | 预设失败分支 + 脆弱步骤上脚本 + 安全红线 |
| 结构 | 三级渐进披露，正文 <500 行，细节下沉 references |
| 文风 | 祈使句、清单化、正例优先、只补非常识 |
| 心法 | 把 Skill 当成写给另一个 AI 的上岗手册，不是写给人看的文档 |

### 九：分析步骤和输出步骤不要混在一起

#### 1、定义区别

| 类型 | 回答的问题 | 主要包含 |
| --- | --- | --- |
| **分析步骤** | 结论是怎么得出来的？ | 数据校验、指标计算、口径比较、条件判断、排序筛选、异常处理 |
| **输出步骤** | 得到结论后，怎么展示？ | 展示字段、表格或文字、输出顺序、标题层级、措辞和样本标记 |

#### 2、快速判断方法

| 判断问题 | 结果 | 类型 |
| --- | --- | --- |
| 删除这一步后，最终Top3或结论会改变吗？ | 会改变 | 分析步骤 |
| 删除这一步后，结论不变，只是展示方式变化吗？ | 不会改变 | 输出步骤 |
| 这一步在决定"选谁"吗？ | 是 | 分析步骤 |
| 这一步在决定"怎么展示已经选中的对象"吗？ | 是 | 输出步骤 |

---

#### 3、会议 Skill 示例

| 分析场景 | 分析步骤 | 输出步骤 |
| --- | --- | --- |
| **日均会议时长贡献Top3** | ①计算各子部门日均会议时长变化贡献；②过滤非正向贡献；③按贡献值降序排列；④选取正向贡献最大的前三项；⑤不足三项时按实际数量保留。 | ①用表格展示Top3；②展示当期值、对比期值、变化值和贡献值；③先输出整体结论，再展示Top3明细。 |
| **高会议负担员工分析** | ①计算各职级×序列中日均会议时长大于4小时的员工人数增量；②过滤非正向增量；③按人数增量降序排列；④选取前三项；⑤检查样本人数是否小于10人。 | ①展示职级×序列、当期人数、对比期人数和增量；②小于10人时标记"小样本"；③将结果放入"重点关注人群"板块。 |
| **长会次数分析** | ①计算各子部门单次会议时长大于1.5小时的会议次数增量；②仅保留正向增量；③按增量降序排列；④选取增长贡献最大的前三项。 | ①展示子部门名称、当期值、对比值和增量；②将第一名描述为"首要增长来源"；③不将长会增加直接描述为会议低效。 |
| **指标异常判断** | ①计算日均会议时长环比变化；②与异常阈值比较；③命中异常时进入子部门和职级×序列下钻；④未命中时不计算贡献Top3。 | ①命中异常时放入"核心发现"；②未命中时只放入整体指标表；③统一使用"明显上涨""基本稳定"等表述。 |

---

#### 4、容易混淆的动作

| 动作 | 类型 | 原因 |
| --- | --- | --- |
| 计算贡献值 | 分析步骤 | 决定排名依据 |
| 过滤负贡献 | 分析步骤 | 决定哪些对象能进入结果 |
| 按贡献值排序 | 分析步骤 | 决定对象排名 |
| 选取Top3 | 分析结果筛选 | 如果任务目标就是找Top3，它属于分析的一部分 |
| 检查样本量 | 分析步骤 | 会影响结论可信度 |
| 标记"小样本" | 输出步骤 | 规定风险如何展示 |
| 用表格展示 | 输出步骤 | 只改变呈现方式 |
| 展示当前值和对比值 | 输出步骤 | 补充证据字段，不改变排名 |
| 先输出整体结论再输出明细 | 输出步骤 | 只规定信息顺序 |
| 两类结果一致时合并成一句话 | 输出步骤 | 只改变表达方式 |

#### 5、Bad Case 与 Good Case

| 类型 | 示例 | 问题或优点 |
| --- | --- | --- |
| **Bad Case** | Step 1：输出贡献最大的三个子部门。Step 2：用表格展示。Step 3：写入核心结论。 | 只规定了结果怎么展示，没有说明贡献如何计算、如何排序、负贡献怎么办、正向贡献不足三项怎么办。 |
| **Good Case：分析步骤** | ①按指标规定口径计算各子部门贡献值；②过滤非正向贡献；③按贡献值降序排列；④选取正向贡献最大的前三项；⑤不足三项时按实际数量保留。 | 明确规定了结论如何产生。 |
| **Good Case：输出步骤** | ①用表格展示结果；②展示当期值、对比期值、变化值和贡献值；③在核心结论中按贡献值顺序描述Top3；④小样本结果必须明确标记。 | 明确规定了分析结果如何交付。 |

一句话总结

> 分析步骤决定"答案是什么"，输出步骤决定"答案怎么讲"。
