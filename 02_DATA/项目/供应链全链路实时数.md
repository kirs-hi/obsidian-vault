
**项目名称：mt供应链全链路实时数仓平台**

**核心技术：** Flink, Doris, Blade (外置State), HBase, Kafka, Hive, Flink CDC, Grafana
**项目背景：**
面向mt供应链全链路流转（覆盖计划、采销、仓配、制造等数十个环节），构建千万级包裹量、数十亿规模的大型实时数据仓库。针对原传统Lambda架构多链路开发冗余、大促多流Join状态过载及数据回溯困难等痛点进行底层架构重构，为企平、财务等业务提供高可用（99.99%）的实时/离线数据分析服务。

可以合并 **STAR法则**：每个项目确保有**Situation→Task→Action→Result**的完整叙事与技术突破
*   **【架构重构与Kappa演进】**
    落地优化版Kappa流批一体架构，核心计算链路完全基于Flink+Doris构建。通过将宽表产出的Kafka消息定时备份至Hive，并逐级构建冷热分离的分钟/小时/天级分层体系，彻底消除了传统双链路带来的口径不一致与回溯难题。使数据研发效率整体提升30%+，存储成本显著降低40%+。

*   **【多流Join与大状态存储攻坚】(底层硬核攻坚)**
    针对全链路数十个Kafka流Join引发的Flink RocksDB大状态存储瓶颈，**创造性引入 Blade 作为 Flink 外置 State 存储**。配合Watermark与侧输出流（Side Output）机制妥善处理跨网传输带来的严重数据乱序与迟到问题。完美突破单机内存限制，成功产出500+字段供应链超宽表。

*   **【端到端0误差保障机制】(实时质量与对账)**
    主导构建全链路数据质量保障体系。利用 Kafka 幂等性与 Doris 两阶段提交（2PC）机制保障端到端 Exactly-Once 语义。设计分钟级时间窗口对齐机制，比对 Flink 实时流与 Hive 离线 T+1 结果，通过公式计算并监控数据准确率 $$ P = \left( 1 - \frac{\text{差异记录数}}{\text{总记录数}} \right) \times 100\% $$，做到全量千万级包裹实时状态流转 0 误差。

*   **【大促极限性能调优】(性能调优)**
    面对大促峰值 **150万+读QPS、100万+写TPS** 的数据洪峰，实施多维极限调优策略：
    1. 采用 MurmurHash 散列 HBase Rowkey 彻底消除底层写热点；
    2. 自定义Flink窗口聚合机制对上游触发流进行预去重，配合合理设置外置存储的 TTL 和 LRU 缓存大小，大幅削减外部系统读放大问题；
    3. 针对背压（Backpressure）问题，开启非对齐检查点（Unaligned Checkpoint）优化快照耗时，保障大促期间系统 0 延迟、0 宕机。

*   **【实时维表同步与统一数据服务】(动态数据与业务赋能)**
    基于 Flink CDC 实时捕获网点/司机等业务线 MySQL 变更。针对海量流与动态维表 Join 耗时高的问题，采用异步I/O（Async I/O）配合旁路缓存大幅压降处理延迟。在计算层之上，封装统一数据服务（Data Service）接口，提供在线SQL解析、限流降级及黑白名单功能，屏蔽了下游100+报表对底层数据源的直接依赖，支撑上千个指标的亚秒级查询，常年线上0事故。

*   **【全生命周期监控治理】(数据治理)**
    拒绝盲盒式黑盒运行，基于 Grafana 深度定制 Flink 任务与底层 Doris 集群的全生命周期监控。不仅覆盖基础的Failover、吞吐量（RPS）监控，更针对业务属性定制了 Event Time 与 Processing Time 的全链路端到端延迟监控告警，确保核心业务指标SLA稳固在 2 秒以内。

## 项目二：实时数据质量监控与治理平台

### 项目背景
随着实时数仓规模扩大至200+实时任务，数据质量问题频发（如数据延迟、丢失、重复），缺乏统一的实时数据质量保障体系，严重影响下游业务决策的可靠性。

### 项目职责
作为核心开发者，负责实时数据质量监控平台的架构设计与核心模块开发。

### 技术方案

**1. 多维度质量检测引擎**
- 基于Flink CEP（Complex Event Processing）构建实时异常检测引擎
- 设计五维质量检测模型：
  - **完整性**：基于Watermark机制检测数据丢失，对比上游Kafka Offset与下游Sink条数
  - **及时性**：监控Event Time与Processing Time偏移量，动态计算端到端延迟分位数（P99/P95）
  - **准确性**：流批对比校验，将Flink实时结果与Hive T+1离线结果做diff
  - **一致性**：跨层数据血缘追踪，DWD→DWS聚合结果反向校验
  - **唯一性**：基于Flink State + BloomFilter实现高效去重检测

**2. 智能告警与根因定位**
- 设计三级告警策略：
  ```
  L1（秒级）：数据断流 → PagerDuty电话告警
  L2（分钟级）：数据延迟超阈值 → IM消息推送
  L3（小时级）：流批diff超1% → 邮件通知
  ```
- 构建实时任务依赖DAG图谱，异常发生时自动向上游回溯，定位根因节点
- 基于历史异常模式训练时序异常检测模型（Isolation Forest），误报率降低60%

**3. 自愈机制**
- 实现Flink任务健康度评分系统，综合Checkpoint成功率、背压指标、GC频率
- 开发自动化Failover策略：Checkpoint恢复 → Savepoint回滚 → 数据重放三级兜底
- 基于Kafka Consumer Group Offset实现精准数据重放，保证Exactly-Once语义

### 核心成果
| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 异常发现时效 | 平均30min | < 30s |
| 数据问题人工排查耗时 | 2h/次 | 10min/次 |
| 月均数据事故 | 12次 | 2次 |
| 告警准确率 | 45% | 92% |

## 项目三：实时用户行为分析与实时标签引擎

### 项目背景
运营团队需要基于用户实时行为（浏览、下单、评价等）进行精准营销触达，传统T+1标签体系无法满足实时场景需求，需构建秒级更新的实时用户标签系统。

### 项目职责
负责实时标签计算引擎的整体架构设计与开发，以及与下游推荐/营销系统的对接。

### 技术架构

**1. 实时数据接入层**
```
客户端埋点 → Kafka(行为日志Topic)
                 ↓
服务端事件 → Kafka(业务事件Topic)  → Flink多流Join → 标签计算
                 ↓
数据库Binlog → Kafka(CDC Topic)
```
- 基于Flink Interval Join实现行为流与订单流的时间窗口关联，窗口大小动态可配
- 采用Flink Broadcast State处理维度数据变更的实时广播，避免频繁查询外部存储

**2. 标签计算引擎**
- **实时统计型标签**：基于Flink Window（Tumbling/Sliding/Session）计算
  - 近30分钟浏览品类Top3
  - 当日累计下单金额
  - 实时会话深度（Session Window, gap=30min）
- **实时规则型标签**：基于Flink CEP模式匹配
  - 连续3次浏览未下单 → 犹豫用户
  - 下单后5分钟内取消 → 冲动消费用户
  - 浏览→加购→下单 完整转化链路识别
- **实时模型型标签**：Flink集成PMML/ONNX模型在线推理
  - 实时流失概率预测
  - 实时价格敏感度评分

**3. 标签存储与服务层**
- 采用Redis Cluster作为热标签存储（TTL动态管理），QPS支撑50W+
- HBase作为全量标签宽表存储，支持标签历史版本回溯
- 设计Lambda查询层：实时标签（Redis）+ 离线标签（HBase）合并输出
- 封装统一标签查询API，支持单点查询（<5ms）和批量圈人（<500ms/万人）

**4. 状态管理优化（技术深度）**
- 针对大状态场景（用户级State，亿级Key），采用RocksDB State Backend
- 实现增量Checkpoint + 本地恢复，Checkpoint耗时从120s降至15s
- 设计State TTL分级策略：
  ```
  热标签State：TTL = 24h，滑动更新
  温标签State：TTL = 7d，惰性清理
  冷标签State：异步转储至HBase后清除
  ```
- 自定义State序列化器（替换Kryo），序列化性能提升3倍，State体积减少40%

### 核心成果
- 支撑**500+实时标签**的秒级更新，覆盖用户画像、行为偏好、营销响应等维度
- 实时标签驱动的营销触达，点击率较T+1标签提升**25%**
- 标签计算引擎日均处理**50亿+**事件，端到端延迟P99 < **3秒**

## 项目四：基于Flink CDC的实时数据湖构建

### 项目背景
业务数据库（MySQL集群200+实例）的数据同步至数仓依赖传统Sqoop T+1批量抽取，无法满足业务对数据时效性的要求，且全量同步对源库造成较大压力。

### 项目职责
主导设计并实现基于Flink CDC的实时数据入湖方案，替代传统ETL链路。

### 技术方案

**1. CDC采集架构**
- 基于Flink CDC 2.x实现**增量快照读取**，核心优势：
  - 无锁快照：基于Chunk分片并行读取，不加全局锁，对源库零影响
  - 断点续传：Chunk粒度的Checkpoint，故障恢复无需全量重跑
  - 精确一致性：结合Binlog Position实现Exactly-Once保障
- 单任务支撑**1000+表**并行采集，通过正则匹配动态感知新增表

**2. Schema Evolution处理**
- 设计DDL变更感知模块，实时捕获`ALTER TABLE`事件
- 实现Schema Registry服务，维护表结构版本链：
  ```
  V1: (id, name, amount) 
  → V2: (id, name, amount, discount)  -- ADD COLUMN
  → V3: (id, name, total_amount, discount)  -- RENAME COLUMN
  ```
- 下游Hudi/Iceberg表自动Schema合并，兼容历史数据读取

**3. 实时入湖（Hudi）**
- 采用Hudi MOR（Merge On Read）表类型，平衡写入性能与查询效率
- 关键参数调优：
  - Compaction策略：`num_commits=5`，避免读放大
  - Index类型：Bucket Index（适合大表），Bloom Index（适合小表）
  - 文件大小：`target_file_size=256MB`，平衡小文件与并行度
- 实现**分区级别增量查询**，下游Presto/Spark可基于Commit Timeline读取增量数据

**4. 数据一致性保障**
- 设计End-to-End校验框架：
  - 源端：MySQL Binlog GTID追踪
  - 传输层：Flink Checkpoint Barrier对齐
  - 目标端：Hudi Commit Metadata行数校验
- 实现**分钟级对账任务**，自动发现并修复数据不一致

### 核心成果
| 维度 | Sqoop方案 | Flink CDC方案 |
|------|-----------|---------------|
| 数据延迟 | T+1 | < 5分钟 |
| 源库影响 | 高（全量查询） | 极低（Binlog读取） |
| 表变更适配 | 人工修改 | 自动感知 |
| 故障恢复 | 全量重跑（小时级） | 增量恢复（秒级） |
| 日均同步数据量 | 10TB | 10TB+ |
