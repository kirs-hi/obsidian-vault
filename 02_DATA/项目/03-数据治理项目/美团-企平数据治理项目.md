# Part A · 正式简历（投递用，直接复制）

## 项目名称：企业数据智能平台数据治理

**项目背景：** 企业数据智能平台覆盖 HR、CAP、采购、办公、行政等多条业务线，随着业务数据快速增长，原有数仓体系面临"找数难、价值难量化、研发成本高"等严峻挑战，严重影响数据生产效率、数据服务质量、管理层决策效率及下游业务系统稳定性。2023 年对标行业及公司优秀团队，引入字节 & 外卖 0987 价值评估体系，确保团队质量交付追上内外优秀团队。

**技术栈：** Hive / Spark SQL、Doris（OLAP 查询）、元数据体系（血缘/DQC/SLA/权限）、NoCode 平台、SQL Copilot、大象告警/群助手

**职责描述：** 负责数仓治理的整体体系建设，从元数据、指标口径、SLA、OLAP 稳定性、研发效能五个维度，搭一套体系化的治理闭环，核心解决质量、效率、体验问题。

**技术方案：**

- **元数据体系建设：** 搭建元数据仓库与数据治理指标体系，设计业务/技术/管理三域模型及分层架构，落地血缘、DQC、SLA、权限等核心模块，实现数据资产可视化与治理数字化，显著提升找数效率与问题定位效率。

- **指标一致性治理：** 面对数据表 6000+、数据产品指标口径频繁不一致的问题，统一命名、定义、取数来源、链路加工四个维度的口径，推动工具化落地，建立"单模型单口径"机制与标准化查询 API，指标一致率由 89.1% 提升到 97%。

- **SLA 准点率治理：** 从局部单点优化升级为全链路结构性治理，提出核心链路概念并做三阶段切割（数据采集/数仓计算/数据下发），搭建"观测—提取—归因—长效机制"闭环体系，实现 T+1 生产的 SLA 服务提前 2 小时达成 99%、SLA 服务准点产出占比 99%。

- **Doris 集群稳定性：** 建设面向多租户、高并发场景的 Doris 治理看板，通过精细化资源计量与管控体系，落地全生命周期治理——事前 SQL 变更风险卡控、事中自适应限流与大查询熔断、事后慢查询归因与红黑榜晾晒，实现业务高速增长下的稳态运行与降本增效。

- **需求承接率提升：** 基于研发流程诊断与价值场景挖掘，通过 NoCode 将常用提效工具与文档平台化，完成 SQL Copilot 能力调研与团队推广，需求承接平均交付时长由 6.88pd 缩短至 5.5pd，开发效率提升 20%。

**项目成果：**
- 指标一致率从 89.1% 提升到 97%，不一致问题从周均 3 次降到月均不到 1 次
- SLA 服务提前 2 小时达成 99%，SLA 服务准点产出占比 99%
- 研发交付时长从 6.88pd 降到 5.5pd，开发效率提升 20%
- Doris TP95 耗时在业务量持续增长下稳住，未因量涨而劣化
- 数据资产从"看不见"变为"可检索、可追溯"，找数效率与问题定位效率显著提升

## 话术正文

### 开场：背景与定位（约 1 分钟）

我所在的项目是企业数据智能平台，覆盖 HR、CAP、采购、办公、行政、法务等多条业务线，数据量增长很快。但原有数仓体系存在几个核心问题：数据找数难、指标口径经常打架、T+1 生产经常 delay、OLAP 查询不稳定，直接影响了数据交付质量和下游业务。

23 年团队对标行业标杆，引入了 0987 价值评估体系。我在这个背景下负责数仓治理，核心就做一件事：从元数据、指标口径、SLA、OLAP 稳定性、研发效能五个维度，搭一套体系化的治理闭环，核心是解决质量、效率、体验问题。

### 中间：怎么做的（约 2 分钟）

五个方向我快速讲一下核心思路。

**元数据体系是底座。** 当时最大的痛点是"找数难"，我设计了业务域、技术域、管理域三域模型，落地了血缘、DQC、SLA、权限这些核心模块，让数据资产从"看不见"变成"可检索、可追溯"。

**指标一致性治理，核心是统一口径 + 工具化。** 6000 多张表，指标口径对不上，我的做法是先把命名、定义、取数来源、链路加工这四个维度的口径拉齐，然后做成自动化检查工具，不再靠人工巡检。

**SLA 准点率，关键是从单点救火变成全链路治理。** 之前是哪个任务 delay 了就去扑火，我搭了一个"观测—提取—归因—长效机制"的闭环：先让全链路每个节点可观测，再自动提取瓶颈节点做归因，最后把经验沉淀成机制。

**Doris 集群稳定性，思路是全生命周期防控。** 事前卡控高危 SQL 变更，事中做自适应限流和大查询熔断，事后做慢查询归因和红黑榜晾晒，逼着研发改代码。核心是让单点问题不扩散成集群故障。

**研发效能这块**，主要是把常用提效工具基于 NoCode 平台化，同时做了 SQL Copilot 的调研和推广，帮团队用 AI 辅助开发。

### 收尾：效果 + 钩子（约 1 分钟）

整体效果上，几个关键数字：指标一致率从 89.1% 到 97%，SLA 提前 2 小时达成 99%、准点 99%，研发交付时长从 6.88pd 降到 5.5pd，效率提升 20%。Doris 那边在业务量持续增长的情况下，TP95 耗时稳住了，没有因为量涨而劣化。

如果展开聊的话，我觉得最有意思的是两块：一个是 SLA 的全链路归因体系，怎么从几千个任务里自动定位到真正的瓶颈节点，这个方法论后面也复用到了其他场景；另一个是 Doris 的事前卡控机制，怎么在"不误杀正常变更"和"拦住高危操作"之间找平衡，这个踩了不少坑。您对哪个方向比较感兴趣，我可以展开讲。

## 二、指标一致性治理（Q3-Q10）

### 主线讲述

背景上，我们平台累计建了 6000 多张表、多个数据产品和 API 服务，不同产品之间指标口径频繁打架，平均每周有一些指标冲突反馈，严重影响数据公信力和决策效率。

我先把问题做了系统拆解，发现指标不一致有三种表现：同名不同义、同义不同名、同名同义但值不同（一个指标绑定多个模型）。根因集中在四类——语义理解不统一、需求变更没同步、生产逻辑分散在各 ETL 里、查询口径各写各的。归根结底就是指标定义散在"人脑"和"各个脚本"里，缺统一标准和集中控制。

基于这个分析，我设计了一套"三位一体"的治理体系：

第一是**统一语义**，建立统一指标标准，把业务定义、技术口径、查询规范全部沉淀到元数据体系里。

第二是**统一生产**，引入"单模型单口径"机制——一个指标在一个模型里只有唯一计算逻辑，所有下游 ETL 必须引用统一口径而不是各自实现。口径变更时通过集中入口修改，自动联动更新下游，避免逻辑漂移。

第三是**统一查询**，通过标准化数据 API 做统一出口，减少业务侧直接写 SQL 导致的口径分裂。同时配套了命名规范、口径认证机制、立项到上线的 SOP 流程，形成"标准 + 工具 + 流程"的闭环。

最终效果是指标一致率从 89.1% 提升到 97%，不一致问题从周均 3 次降到月均不到 1 次，核心数据链路口径统一，跨团队沟通成本显著下降。

### Q3：你怎么发现哪些指标口径不一致的？全量扫描还是有重点？

我们不是全量铺开，而是按优先级推进。首先从问题反馈入手——每周 3 次冲突反馈就是线索库，先治高频问题。其次按指标使用频次和下游消费方数量排序，优先治理"用得多、影响面大"的指标。检测方法上，三种表现各有手段：同名不同义靠语义比对，同义不同名靠指标字典交叉匹配，同名同义但值不同靠取数链路比对——看两个指标背后的 SQL 逻辑是否一致。

### Q4：三种不一致表现，能各举一个例子吗？

同名不同义：比如"活跃用户数"，A 产品定义为当日登录用户，B 产品定义为当日有操作行为的用户，名字一样但含义不同。

同义不同名：A 产品叫"GMV"，B 产品叫"交易额"，指的是同一个东西但命名不统一。

同名同义但值不同：都叫"订单量"、定义也一样，但 A 从订单表 count，B 从支付表 count，取数来源不同导致数值有差异。

### Q5："单模型单口径"具体怎么落地的？下游 ETL 怎么保证引用统一口径而不是自己写？

核心是把指标计算逻辑从"散落在各 ETL 里"变成"集中在一个模型里"。具体来说，每个指标有唯一的口径定义和计算逻辑，沉淀在统一的汇总层。下游 ETL 只能引用这个汇总表的字段，不能自己重新算。落地保障上，一是通过代码审查卡控，二是通过元数据工具自动检测——如果一个指标在多个地方有独立计算逻辑，系统会标红告警。

### Q6：口径变更时"自动联动更新下游"，具体怎么实现的？

这依赖元数据体系的血缘能力。每个指标关联了它的定义、计算逻辑和所有下游引用关系。当口径变更时，在集中入口修改定义，系统通过血缘关系自动识别所有受影响的下游 ETL 和数据产品，生成变更通知和影响范围报告。需要重新跑的 ETL 会自动加入调度队列，需要人工确认的变更会推送通知给对应负责人。

### Q7：统一查询用标准化 API，这个 API 层是你们自研的还是基于现有工具？业务方接受度怎么样？

我们是在现有数据服务能力上做的标准化封装，核心是定义了一套统一的指标查询接口规范，业务方调 API 时传指标名和维度，底层自动路由到对应的统一口径表。接受度方面，初期有阻力——业务方习惯了自己写 SQL 灵活取数，被限制后会不适应。推动策略是先提供 API + SQL 双通道，让业务方对比结果一致性，同时 API 附带口径说明，让业务方逐步建立信任，最后再收紧直接写 SQL 的权限。

### Q8：指标一致率从 89.1% 到 97%，剩下 3% 为什么没治理掉？

剩下一部分是历史包袱——一些老旧链路改动成本高、风险大，暂时做了标注隔离没有强行改。另一部分是合理的业务差异——比如不同业务线对"活跃用户"的定义确实存在合理差异，这不是口径不一致而是业务本身不同。关键是区分"该治但还没治"和"不该治的合理差异"，对前者有治理排期，对后者做了明确标注防止误判。

### Q9：SOP 流程具体是什么样的？怎么保证研发真的按流程走？

SOP 覆盖了从立项到上线全流程：立项时要提交指标定义（业务含义、计算口径、数据来源）；评审时由数据架构师审核口径是否与已有指标重复或冲突；开发时必须引用统一口径模型，不允许自建计算逻辑；上线前做口径一致性自动校验，不通过就拦截。保证执行靠两个手段：一是工具卡控——上线校验不通过就发不出去，二是流程卡控——没有口径认证的指标不能接入数据产品。

### Q10：你在这个项目里最大的挑战是什么？

最大的挑战不是技术，而是推动力。指标治理本质上是在"改别人的习惯"——研发习惯了自己写 SQL，业务方习惯了按自己理解定义指标。我的做法是"先治痛、再建规"：先用高频冲突案例展示治理价值，让大家看到"治了之后确实少了麻烦"，再推规范和流程。另外就是"工具替代人工"——如果只靠制度约束，执行会衰减；把约束嵌到工具里（上线校验、自动告警），才能持续生效。

## 三、SLA 准点率治理（Q11-Q13）

### Q11：SLA 链路的核心难点是什么？怎么甄别瓶颈任务？

**难点一：** 每个 SLA 应用上游计算层级最多达 26 层，涉及任务上百个，呈网状，如何有效甄别瓶颈任务，确保优化的每个任务或链路是有效的，优化链路 ROI 最高。

**举措：**

- **举措一：** 基于链路复杂的背景，我们提出核心链路的概念。核心链路：SLA 末端任务到数据采集层迭代寻找同层级内最晚执行完成的任务，这个任务的集合组成的链路为核心链路。通过准确找出核心链路，我们能判定哪些任务优化后应用产出肯定会提前，哪些任务优化后对应用执行就绪是无效的。

- **举措二：** 外部任务、数仓任务、数据下发任务治理方式大相径庭，所以把核心链路做了切割，把数仓生产抽象成三个阶段：
    - **数据采集阶段**（所有 O 层任务和其他数仓部门任务）：优化策略——跨部门沟通，协调提供更早产出的上游依赖，或推动上游优化依赖。
    - **数仓计算阶段**（部门内数仓层、集市层、应用层）：优化策略——Spark 参数优化、逻辑优化、链路精简，尽可能缩短该层生产时间。
    - **数据下发阶段**（数仓同步 OLAP 数据库的任务）：优化策略——Doris 模型调整、Load 策略调整、高级参数优化。

### Q12：长期治理怎样有效落地？抓手是什么？

**难点二：长期治理怎样有效落地，抓手是什么？**

建立长线运营机制：

- **事前—预警：** 通过分析历史应用执行情况，存在风险 SLA 大象告警（每周同步一次）。
- **事中—提供优化分析框架和治理建议：** 开发 SLA 治理大盘——当应用有延时风险时，用户可以快速定位优化方向、优化任务及优化策略，从复杂的网状链路找问题的核心所在。包括：应用分析、链路分析、任务分析、人员分析。
- **事后—追踪：** 提供处理进展追踪、任务影响权重、权责划分。

### Q13：不同根因的监测指标和解决方案是什么？

**难点三：** 针对不同问题根因，梳理监测指标以及解决方案。

## 四、Doris 集群稳定性（Q14-Q17）

### Q14：Doris 分桶和索引怎么设置？

查看分区、分桶情况：

```sql
show partitions from epdw.aggr_personflow_quit_dept_analysis
```

**分桶和索引设置要点：**

- rollup 索引、前缀索引、位图索引
- 前缀索引顺序：int 类型、varchar 类型，分区表的分区字段放在前缀索引的后面，分区字段可以进行分区裁剪，全量表的时间字段需要放在第一个
- 分区放在前缀索引的后面，用 date 类型
- 指标自助查询绑定部门字段必须要使用 varchar 类型，char 会被当成数值；分区字段必须使用 date 类型
- rollup 索引：聚合模型和唯一模型用于聚合数据，冗余模型用于调整前缀索引；rollup 包括查询的所有字段，查询会命中 rollup

### Q15：分桶列和分桶数量怎么设置？

**分桶列选择的关键因素：**

1. **基数（Cardinality）：**
    - 高基数列：选择基数（唯一值）较高的列作为分桶字段，可有效避免数据倾斜。具有大量唯一值的列（如用户 ID、订单 ID）是良好候选列。
    - 低基数列：避免选择基数很低的列（如布尔值、二进制列），会导致数据不均匀分布，部分桶压力过大。

2. **查询模式（Query Patterns）：**
    - 选择在查询过滤条件中经常使用的列作为分桶字段。
    - 考虑使用多列分桶（Composite Bucketing），在不同查询条件下达到更好的均匀分布。

3. **数据分布：** 理想情况下分桶字段能将数据均匀分布到各桶中，避免数据倾斜。

4. **数据量：** 确保桶的数量合理，过多或过少都会影响查询和加载性能。

HR 场景高基数列一般选择**员工工号**，将员工工号设置为分桶列。

**分桶数量设置：**

- 分区表分桶数和集群数量保持一致，HR 集群数为 50，CAP 集群数为 15
- 如果分桶数超过了集群数量（如集群 10 台、分桶设 20），看数据量——数据大设 20 没问题，分桶多可提升单查询执行并发，但会降低吞吐量
- 增加分桶数，聚合模型和唯一模型会增加表存储空间，冗余模型不会（因为聚合度问题）
- Doris 集群 epdw 机器数量现为 50 台，大表可考虑分桶数同机器数，超过机器数也没问题

分桶数量参考：
- 小数据集（< 1GB）：5-10 个分桶
- 中等数据集（1GB-10GB）：10-30 个分桶
- 大数据集（> 10GB）：50 个分桶

### Q16：Doris 索引怎么设置？各有什么区别？

- **rollup 索引：** 聚合模型和唯一模型用于聚合数据，冗余模型用于调整前缀索引；rollup 包括查询的所有字段，查询会命中 rollup。
- **前缀索引：** 数据的前 36 个字节作为这行数据的前缀索引。遇到 VARCHAR 类型时前缀索引会直接截断。平台建议前缀索引顺序为 int 类型、varchar 类型，分区字段放在前缀索引后面，可进行分区裁剪。指标自助查询绑定部门字段必须使用 varchar 类型，char 会被当成数值；分区字段必须使用 date 类型。
- **Bitmap 索引：** 适用于高基数的列（如用户 ID）或低基数的列（如性别、状态等）。
- **BloomFilter 索引：** 本质上是一种位图结构，用于快速判断一个给定的值是否在一个集合中。

### Q17：常见 Doris 查询性能问题和优化策略？

| 问题 | 策略 | 优化优先级 |
| --- | --- | --- |
| SQL 查询分区表扫描全表 | 需要分区裁剪。将分区查询的函数替换为 Doris 支持分区裁剪的常量函数；SQL 中存在 IF 函数查询，替换为 case when 查询。参考 [06 Doris查询性能调优 & 治理](https://km.sankuai.com/collabpage/727356486#b-4bf7857452b942fb92774bd33b7dba98) | 1、SQL 优化；2、模型优化可批量通过 SQL 修改分桶数。如果 1、2 解决不了再考虑前缀索引优化 |
| 表分桶数为 3，集群机器数为 15，需修改分桶数 | 平台建议分桶数和集群机器数保持一致，可修改为 15。Doris 表可直接通过 SQL 修改分桶数，但 HASH 列需和 DDL 中保持一致 | — |
| 前缀索引设置不合理 | 参考 [06 Doris查询性能调优 & 治理](https://km.sankuai.com/collabpage/727356486#b-4bf7857452b942fb92774bd33b7dba98)。索引列顺序基于建表时指定的排序列（Sort Key）生成，无法单独修改前缀索引列。优化方式：重新建表优化索引；或使用 ROLLUP 表——在冗余模型中可通过创建 Rollup 表指定新的排序键顺序生成不同前缀索引。冗余模型是唯一支持通过 Rollup 调整前缀索引顺序的模型；聚合模型和唯一主键模型的 Rollup 主要用于预计算和投影优化，无法改变前缀索引结构 | — |
