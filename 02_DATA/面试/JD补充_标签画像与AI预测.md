## 二、标签体系建设（5题）

### Q41. 你是如何理解标签体系的？标签和维度有什么区别？

**参考回答**：

标签体系是从业务运营视角出发，对实体（人、货、场等）进行多维度刻画的结构化标签集合。它和维度有关联但不完全相同：

**维度**是数仓建模概念，强调从业务过程中抽象出分析视角，比如供应商维度表包含供应商编码、名称、类型等基础属性，服务于OLAP分析。

**标签**是运营视角的概念，强调对实体的"画像"刻画和分层分群，比如"战略供应商"、"高风险供应商"、"快速响应供应商"等。标签通常基于维度数据 + 业务规则 + 统计模型计算得出，是对维度数据的二次加工和价值提炼。

在我的采购数仓项目中，供应商维度模型设计实际上就包含了标签体系建设的思路：
- 基础属性层：供应商编码、名称、注册地、主营品类等（对应维度表）
- 统计标签层：累计采购金额、订单数量、平均交货周期、质量合格率等（基于事实表聚合计算）
- 业务标签层：战略供应商/核心供应商/一般供应商/长尾供应商（基于规则引擎分层）
- 预测标签层：风险等级、合作趋势（上升/稳定/下降）、推荐评级（基于历史数据预测）

标签体系的核心价值在于**把分散的数据转化为可操作的运营策略**，支撑分层分群精细化运营。

> 来源：学城文档《10 标签建设规范》(km.sankuai.com/collabpage/2710782951)——标签分类体系、标签处理流程、标签注册管理

### Q42. 如果让你从0搭建采购域的标签体系，你会怎么设计？

**参考回答**：

我会按照"实体识别 → 标签分类 → 标签生产 → 标签管理 → 标签应用"五步走：

**第一步：实体识别**。采购域核心实体有四类——供应商（人）、品类（货）、采购组织/城市（场）、采购订单（事件）。每个实体都需要独立的标签体系。

**第二步：标签分类设计**。以供应商为例，按"基础属性 → 统计事实 → 业务规则 → 预测模型"四层设计：
- 基础属性：注册地、企业性质、主营品类、资质认证等级
- 统计事实：累计采购额、年度订单量、平均交货天数、质量合格率、价格竞争力指数
- 业务规则：战略/核心/一般/长尾分层（基于帕累托分析）、合作状态（活跃/休眠/流失）
- 预测模型：风险等级、推荐评级、合作趋势预测

**第三步：标签生产链路**。基础属性直接从ODS同步；统计事实从DWS聚合计算；业务规则通过规则引擎配置化产出；预测模型接入算法团队产出的ML结果。每类标签都有独立的产出任务和调度依赖。

**第四步：标签管理平台**。包括标签注册（名称、业务口径、计算逻辑、更新频率、负责人）、标签审批、标签血缘、标签生命周期管理（上线/下线/废弃）、标签质量监控（覆盖率、准确率、更新及时率）。

**第五步：标签应用场景**。供应商分层运营、风险预警、推荐寻源、品类策略制定。标签通过API或宽表形式对外服务。

> 来源：学城文档《10 标签建设规范》——标签分类（基础/统计/规则/预测四层）、标签处理流程、标签注册管理

### Q43. 标签的质量怎么保证？标签覆盖率怎么衡量？

**参考回答**：

标签质量我从三个维度衡量和保障：

**覆盖率**：有标签值的实体数量 / 实体总数。比如3000个供应商中，有"合作趋势"标签的只有2000个，覆盖率就是66.7%。覆盖率低说明标签生产链路有断点或数据源缺失。我会设定SLA——核心标签覆盖率不低于90%，一般标签不低于70%。

**准确率**：抽样校验标签值是否正确。比如"战略供应商"标签，人工抽样50个，看是否符合战略供应商的定义标准。准确率低于95%的标签需要排查计算逻辑或数据源质量。

**时效性**：标签数据的更新是否及时。统计类标签一般T+1更新，预测类标签可能周更或月更。通过调度SLA监控，超时告警。

保障手段：
1. 标签血缘追溯——每个标签可追溯到源表和计算逻辑，出问题能快速定位
2. DQC覆盖——标签产出任务配置空值率、波动率、一致性校验
3. 人工抽检机制——新增标签上线前必须经过业务方确认抽检
4. 标签反馈闭环——业务方使用过程中发现标签不准，通过标签管理平台反馈，触发修正流程

### Q44. 标签体系和用户画像是什么关系？B端画像和C端画像有什么区别？

**参考回答**：

标签体系是**基础设施**，用户画像是**应用场景**。画像是在标签体系之上，针对特定运营目标对实体做群体性刻画和分层。标签是点，画像是由点组成的面。

B端画像和C端画像有本质区别：

**C端画像**面向消费者个体，强调个性化推荐和精准营销。标签维度多——人口属性、行为轨迹、消费偏好、兴趣标签等，标签数量可能上千。核心目标是"千人千面"，颗粒度到个体。

**B端画像**面向企业客户或内部业务角色，强调分层分群和资源倾斜。标签维度少但业务含义重——比如供应商画像的核心标签可能只有20-30个（采购规模、合作年限、风险等级、品类覆盖等），但每个标签都直接关联运营策略。核心目标是"分层运营"，颗粒度到群组。

我在采购域做的供应商画像就是典型的B端画像：
- 画像对象：供应商
- 核心标签集：规模分层（帕累托）、风险等级、合作趋势、品类集中度
- 应用场景：采购经理对不同层级的供应商采取不同管理策略——战略供应商深度绑定，长尾供应商优化整合，高风险供应商启动备选寻源
- 画像消费方式：通过数据看板 + Data Agent查询，而非C端的实时推荐流

### Q45. 你提到供应商分层用了帕累托分析，具体怎么做？

**参考回答**：

帕累托分析就是我们说的"二八法则"，在供应商分层中就是把供应商按累计采购额降序排列，累计贡献前80%的供应商划为"头部/战略供应商"，中间15%划为"核心供应商"，最后5%划为"长尾供应商"。

具体SQL实现思路：

```sql
WITH supplier_amount AS (
  SELECT supplier_id, SUM(amount) AS total_amount
  FROM dwd_proc_order_di
  WHERE dt >= date_sub(current_date, 365)
  GROUP BY supplier_id
),
supplier_rank AS (
  SELECT supplier_id, total_amount,
    SUM(total_amount) OVER (ORDER BY total_amount DESC) AS cum_amount,
    SUM(total_amount) OVER () AS grand_total,
    ROW_NUMBER() OVER (ORDER BY total_amount DESC) AS rn
  FROM supplier_amount
)
SELECT supplier_id, total_amount,
  cum_amount / grand_total AS cum_pct,
  CASE
    WHEN cum_amount / grand_total <= 0.80 THEN '战略供应商'
    WHEN cum_amount / grand_total <= 0.95 THEN '核心供应商'
    ELSE '长尾供应商'
  END AS supplier_tier
FROM supplier_rank
```

实际落地中还会结合其他维度做交叉校验——比如有的供应商采购额不大但供应的是关键品类（单一供应商），也需要划为战略级。所以帕累托是基础分层，再叠加业务规则做调整。

## 三、用户画像与B端运营（5题）

### Q46. 采购域的"用户"是谁？画像标签怎么设计？

**参考回答**：

采购域的用户不是C端消费者，而是**B端业务角色**，主要包括三类：

1. **采购经理（CoE）**：负责采购策略制定、供应商管理、品类规划。关注供应商绩效、品类集中度、价格趋势、风险预警。
2. **品类经理（Category Manager）**：负责特定品类的寻源、招标、谈判。关注品类供应商池、价格竞争力、交货质量。
3. **管理层**：关注采购大盘、成本节约、合规风险。

针对采购经理的画像标签设计：
- 角色属性：所属部门、管理品类、采购权限范围
- 行为标签：常用查询维度（品类/供应商/城市）、查询频率、报表偏好
- 决策标签：审批金额区间、平均决策周期、历史决策记录
- 偏好标签：关注的风险指标、偏好的数据可视化方式

这些标签在Data Agent中直接使用——Agent根据用户画像预加载常用查询维度、默认时间范围和可视化模板，减少交互轮次，提升取数效率。这就是"用户画像驱动的个性化数据服务"。

> 来源：学城文档《采购数仓建设规划》——业务角色与数据需求矩阵

### Q47. 画像数据存在哪？宽表还是图数据库？

**参考回答**：

我的选择是**分层存储，按场景选择**：

**宽表存储（Hive/Doris）**：标签数据最终产出为供应商画像宽表，每行一个供应商，列是各维度标签值。适合批量分析、BI看板查询、Data Agent的多维取数场景。宽表的优势是查询效率高、SQL友好、和数仓体系无缝衔接。

**标签服务层（MySQL/Redis）**：高频查询的标签（如风险等级、合作状态）同步到MySQL或Redis，通过API对外提供实时查询。适合Data Agent在需求澄清阶段做权限预检和用户画像加载。

**图数据库（可选）**：如果要做供应商关系网络分析——比如供应商的关联关系（同一实控人、同一地址的潜在围标风险），图数据库（Neo4j/JanusGraph）更合适。但这个属于高阶应用，优先级低于基础画像建设。

我的实践是先做宽表 + API服务层，覆盖80%的日常运营场景。图关系分析属于Phase 2规划。

### Q48. 画像标签怎么更新？实时还是离线？

**参考回答**：

按标签类型分策略更新：

**基础属性标签**（注册地、企业性质等）：变更驱动更新。源系统变更后通过Flink CDC或T+1批量同步到数仓，变更频率低，T+1即可。

**统计事实标签**（累计采购额、订单量等）：T+1批量更新。依赖DWS层聚合任务，每天凌晨产出后写入画像宽表。

**业务规则标签**（战略/核心/长尾分层）：月度重算。分层不需要每天变，月度评估足够。

**预测模型标签**（风险等级、合作趋势）：按模型训练频率更新，一般周更或月更。依赖算法团队的模型产出。

**实时标签**（如供应商当日异常行为告警）：通过Flink实时计算，写入Redis，由风控系统消费。这类标签不是画像宽表的一部分，而是实时事件流。

整体策略是**离线为主、实时为辅**，避免过度追求实时性增加系统复杂度。大部分B端运营决策对时效的要求是T+1级别，只有风控场景需要实时。

### Q49. 如何衡量画像体系对业务的价值？

**参考回答**：

从三个层面衡量：

**数据层**：标签覆盖率（有标签的供应商占比）、标签使用率（被下游查询/引用的标签占比）、标签准确率（抽样校验通过率）。

**业务层**：
- 供应商分层运营覆盖率——有多少供应商纳入分层管理体系
- 风险预警命中率——标签预警的供应商中有多少确实出现了风险事件
- 寻源推荐采纳率——Data Agent推荐的供应商有多少被采购经理实际使用

**决策层**：
- 采购集中度优化——通过分层运营，头部供应商采购占比是否优化
- 供应商风险事件响应时长——从标签预警到业务介入的时间缩短
- 采购成本节约——通过供应商画像驱动的谈判和寻源，带来的成本优化

在实际项目中，数据层指标容易量化，业务层和决策层指标需要和业务方共同定义和持续追踪。

### Q50. 如果面试官说"你没有做过C端用户画像，怎么胜任"，你怎么回应？

**参考回答**：

我的回应策略是**承认差异，强调方法论复用**：

"确实，我的画像实践集中在B端——供应商画像和采购角色画像，没有做过C端消费者画像。但两者在方法论层面是相通的：

第一，建设方法论相同——都是'实体识别 → 标签分类设计 → 标签生产链路 → 标签管理平台 → 应用场景'的五步框架。B端画像我完整走过了这五步。

第二，数据基础设施相同——都依赖数仓分层架构、维度建模、DWS聚合、DQC质量保障。这些是我的核心能力。

第三，核心差异在标签规模和应用场景——C端标签数量可能上千，强调千人千面的实时推荐；B端标签几十个，强调分层运营。如果做C端画像，标签生产链路的调度和数据量会更大，但工程架构不变。

而且我认为，B端画像的难点不在于技术，而在于**业务理解深度**——要把采购业务逻辑转化为标签规则，这需要对业务的深入理解。这种业务到数据的转化能力，无论B端C端都是核心能力。"

## 四、AI预测与数据驱动决策（5题）

### Q51. 采购域可以做哪些AI预测？你的思路是什么？

**参考回答**：

采购域的AI预测场景我从"需求侧"和"供给侧"两个维度梳理：

**需求侧预测**：
- 采购需求预测：基于历史采购数据 + 业务计划，预测未来品类/物料的采购量。帮助采购提前寻源、锁价、备货。
- 价格趋势预测：基于大宗商品市场价格 + 历史采购价 + 汇率波动，预测采购成本的变动趋势，辅助锁价时机决策。

**供给侧预测**：
- 供应商风险预警：基于供应商的交货延迟率、质量异常率、订单取消率等指标，预测供应商的违约/断供风险。
- 供应商绩效预测：基于历史合作数据，预测供应商未来N个月的绩效表现，辅助供应商池优化。
- 寻源推荐：基于品类需求 + 供应商画像 + 历史合作记录，推荐最优供应商组合。

**我的落地思路是"数据先行、规则先行、模型渐进"**：
1. Phase 1——先把数据底座建好（已完成，300+表整合、DWS聚合、指标体系）
2. Phase 2——用规则引擎做基础预警（如交货延迟连续3次即触发风险标签），这是统计规则，不需要ML
3. Phase 3——接入算法团队做ML模型，用历史数据训练预测模型
4. Phase 4——预测结果回流到Data Agent，形成"预测 → 决策建议 → 执行 → 反馈"的闭环

我在项目中已完成Phase 1和Phase 2，Phase 3正在规划中。

> 来源：学城文档《采购数字化方向2026年工作量评估》——预测模型建设规划

### Q52. NL2SQL准确率不高怎么解决？你的Data Agent怎么保证取数准确性？

**参考回答**：

这是我在Data Agent设计中最核心的挑战。根据学术研究，Spider2.0基准测试中GPT-4在企业场景下NL2SQL准确率只有21.3%，直接让大模型生成SQL不可靠。

我的解决方案是**"结构化配置 + SQL模板 + 约束前置"的三层架构**，而非纯NL2SQL：

**第一层：意图分类与意图澄清**。Agent不直接生成SQL，而是先做意图分类——用户是想看汇总数据、明细数据还是趋势分析？然后通过多轮对话澄清参数（品类范围、时间范围、供应商范围、聚合维度）。意图分类用大模型做，准确率远高于直接NL2SQL。

**第二层：SQL模板引擎**。预先定义好高频分析场景的SQL模板——品类采购额趋势、供应商绩效排名、跨城市采购对比等。Agent根据意图分类结果选择模板，填充参数生成SQL。模板经过人工验证，SQL准确性100%保证。

**第三层：约束前置 + 审计回溯**。SQL执行前做权限校验（用户能看哪些品类的数据）、成本预估（查询数据量是否超限）、约束注入（强制加分区条件、强制加时间范围）。执行后做结果校验（行数异常、数值波动）和审计日志（谁查了什么数据）。

**降级策略**：如果意图分类置信度低或参数澄清失败，Agent降级为"推荐已有看板"模式——不生成新SQL，而是从已有看板库中匹配最接近的看板给用户。这是fail-safe机制。

核心思路是：**把大模型擅长的事情（意图理解、参数抽取、结果解读）交给大模型，把不擅长的事情（SQL生成、权限管控）用工程化手段保证。**

### Q53. 你提到Spider2.0准确率只有21.3%，能不能展开讲讲这个数据？

**参考回答**：

Spider2.0是2024年发布的NL2SQL基准测试，由滑铁卢大学等机构推出，专门评估大模型在企业级复杂数据库场景下的Text-to-SQL能力。

核心发现：
- GPT-4在Spider2.0上的执行准确率只有21.3%
- 失败原因主要集中在：复杂表关联（多表JOIN）、嵌套子查询、窗口函数、业务语义理解（同名字段不同含义）、大规模schema理解（企业级数百张表）

对比来看，Spider 1.0（2018年发布，单库200张表以内）上GPT-4准确率可达85%+，但在Spider2.0（企业级，千张表+复杂业务语义）骤降至21.3%。

这个数据给我的启示是：**企业级NL2SQL不能依赖端到端生成**。原因有三：
1. 企业schema复杂——300+张表，大模型context window放不下全部schema
2. 业务语义不透明——"采购金额"在不同口径下定义不同，大模型无法从表结构推断
3. SQL正确性要求极高——BI取数错了直接影响决策，容错率低

所以我的Data Agent没有走纯NL2SQL路线，而是用"结构化配置 + SQL模板"的方式。这个设计决策就是受Spider2.0研究启发。

### Q54. Data Agent的约束前置具体怎么做？能举个例子吗？

**参考回答**：

约束前置是我在Data Agent中设计的核心安全机制，在SQL执行之前完成所有校验和限制注入。举几个具体例子：

**权限约束**：采购经理A只负责"间接采购"品类，Agent在生成SQL前先查权限表，自动注入 `WHERE category_type = 'indirect' AND city IN ('北京','上海')` 条件。用户甚至感知不到这个约束被加进去了，但数据安全得到保障。

**成本约束**：Agent预估SQL扫描数据量，如果超过100GB，自动加分区限制 `WHERE dt >= '2025-01-01'` 或强制走DWS聚合层而非DWD明细层。避免一个查询把集群打挂。

**语义约束**：业务方定义了"采购金额"的统一口径——必须用"含税金额"且"排除退货单"。Agent在模板SQL中预设这个逻辑，大模型生成的SQL不能覆盖这个约束。

**时效约束**：强制注入分区字段条件，禁止全表扫描。`WHERE dt IS NOT NULL` 作为兜底。

这些约束不是在SQL执行后拦截，而是在SQL生成阶段就注入到模板中，确保**生成的SQL天然安全**。这是"左移"思路——把风险控制在最早环节。

### Q55. AI预测结果不准确怎么办？业务方不信任怎么办？

**参考回答**：

这是AI落地最常见的问题。我的应对策略是"置信度透明 + 人机协同 + 反馈闭环"：

**置信度透明**：预测结果不能只给一个值，要给置信区间。比如"供应商A下月违约概率70%（置信度中等）"，而不是简单说"高风险"。让业务方知道模型有多大把握。

**人机协同**：Agent不做最终决策，只做决策辅助。预测结果以"建议"形式呈现——"建议关注供应商A，近期交货延迟率上升30%，建议启动备选寻源评估"。最终决策由采购经理判断。

**反馈闭环**：业务方的采纳/拒绝决策回流到模型训练数据中。如果采购经理连续忽略某个预测，说明该维度需要优化。通过持续反馈迭代模型准确率。

**渐进式上线**：先在低风险场景试跑——比如先做"采购量预测"（预测错了影响不大），跑通后再做"供应商风险预警"（影响较大）。建立信任后逐步扩展到高价值场景。

**可解释性**：预测不能是黑盒。Agent在给出预测时附带关键因子——"风险预警依据：近30天交货延迟率从5%升至15%，质量异常率从0.3%升至0.8%"。让业务方理解预测逻辑，而不是盲信。

核心原则是：**AI是辅助决策工具，不是决策者。建立信任靠透明和持续准确，不靠强制。**

## 五、团队管理与项目推进（5题）

### Q56. 你的团队管理经验是什么？带过几个人？

**参考回答**：

在采购数仓项目中，我承担的是**技术负责人（Tech Lead）**角色，不是正式的管理岗，但承担了团队管理和项目推进的职责。具体包括：

**团队规模**：2-3名开发同学（含外包），我负责需求拆解、方案设计、任务分配、代码评审和上线验收。

**项目管理**：
- 需求对接：与业务方、产品经理对齐需求和优先级，制定迭代计划
- 方案评审：所有数仓模型设计需要过我review，确保符合分层规范和建模方法论
- 进度跟踪：每日站会同步进展，识别风险和阻塞，协调资源
- 质量把控：代码上线前必须过DQC校验和代码review，我签字才能发布

**跨团队协作**：项目中需要和算法团队（Data Agent的NL2SQL部分）、产品团队（看板需求）、业务团队（口径对齐）协作。我作为技术方代表，负责拉齐各方认知、推进决策落地。

**规范推广**：数仓规范制定后，我负责向团队宣贯和落地执行。制定checklist，新需求接入时逐项检查——是否有需求文档、是否有模型设计评审、是否有DQC配置、是否有上线验收。

我的管理风格是**以身作则 + 规范驱动**——最难的模型设计和攻坚我来做，日常开发分配给团队，通过规范和review保证质量。这种方式在2-3人小团队中效果很好。

### Q57. 如果团队中有人不认同你的数仓规范，怎么处理？

**参考回答**：

这个场景我实际遇到过。处理思路是"对事不对人，用数据和结果说话"：

**第一步：理解反对理由**。不认同规范通常有几种原因——规范太重增加工作量、规范不适配当前业务场景、规范限制了技术选型自由度。先听对方说，不要急着反驳。

**第二步：区分是规范问题还是执行问题**。如果对方说的确实有道理——比如某个规范过于理想化不适配当前业务，那就调整规范。规范不是一成不变的，要迭代。如果对方只是嫌麻烦不想遵守，那需要坚持。

**第三步：用案例说服**。拿实际案例——"上次没按规范走，模型上线后下游口径不一致，排查了3天。按规范走，DQC提前拦截了。"用结果说话比讲道理有效。

**第四步：降低遵守成本**。规范的执行成本要尽量低——比如我做了checklist和代码模板，按规范走其实比不按规范走更快。把规范工具化、模板化。

**第五步：升级到流程**。如果反复沟通仍不遵守，把规范遵守嵌入到上线流程中——CI/CD流水线自动检查模型命名规范、分层规范，不合规直接拦截。把人的问题变成系统的问题。

### Q58. 你怎么平衡需求排期和技术债？

**参考回答**：

这是数据团队最常面临的矛盾。我的原则是"保交付、控技术债、预留重构窗口"：

**保交付**：业务需求优先级永远最高，因为数据团队的价值就是支撑业务。但保交付不等于无脑接需求——每个需求接入前先做评估，明确数据源是否ready、模型设计是否合理、工期是否现实。不合理的需求宁可砍需求也要保证质量。

**控技术债**：技术债不可怕，可怕的是不受控。我建了一个"技术债看板"——所有临时方案、不优雅的实现、缺失的DQC都记录在案。每周review，按影响严重程度排序。高风险技术债（影响SLA、影响数据准确性）优先还，低风险的可以缓。

**预留重构窗口**：每个迭代周期预留15%-20%的时间做技术债偿还。比如这周排了5个需求，预留1天做DQC补齐、模型优化、文档更新。这个时间窗口是制度化的，不能被业务需求挤占。

**前置评审**：需求接入时做方案评审，识别潜在技术债——如果这个需求需要临时绕过数仓规范，评估是否可接受，如果不能接受就调整方案。前期投入1小时评审，省下后期10小时重构。

实际项目中，我通过这套机制把技术债控制在了可管理范围，没有出现"需求越做越多，系统越来越烂"的恶性循环。

### Q59. 如何向非技术背景的管理层汇报数仓建设成果？

**参考回答**：

向管理层汇报的核心原则是**业务价值先行、技术细节后置**：

**讲价值不讲技术**：不说"我建了ODS-DWD-DWS-DIM四层架构，300张表标准化"，而是说"以前采购经理看数要找3个系统导数据手工处理，现在统一看板直接看，决策效率提升60%"。

**讲数字不讲过程**：不说"我做了DQC全覆盖、SLA治理、指标一致性治理"，而是说"数据质量问题从每月30+次降到5次以内，数据可用准点率99%，指标口径统一后业务方质疑数据准确性的工单下降了80%"。

**讲ROI不讲投入**：不说"投入了2个人力6个月"，而是说"采购数仓上线后，支撑了3个核心业务场景，间接节省采购成本X万，数据建设投入产出比1:N"。

**讲下一步不讲过去**：管理层更关心未来。汇报完成果后，重点讲"下一步AI预测供应商风险，预计减少断供损失X万；Data Agent推广到更多业务角色，预计覆盖效率提升Y%"。

**可视化**：一张数据治理前后的对比图胜过千言万语。我做了数据指标可视化门户，管理层可以直接看趋势——指标一致率从89%到97%的曲线、SLA准点率曲线、DQC拦截次数下降曲线，一目了然。

### Q60. 你觉得从IC到Tech Lead最大的转变是什么？

**参考回答**：

最大的转变是**从"自己做好"到"让团队做好"**：

**IC阶段**：关注个人产出——我写的SQL质量高不高、我做的模型设计好不好、我解决的bug快不快。衡量标准是个人交付质量和效率。

**Tech Lead阶段**：关注团队产出——团队的规范执行率、需求交付准点率、模型设计的一致性、团队成长速度。衡量标准是团队整体效能。

具体体现在三个方面：

**1. 从执行者到决策者**：IC是接到需求就做，Tech Lead是先判断该不该做、怎么做最优。每天做的最多的决策是"这个需求用什么方案、分配给谁、工期多长"。

**2. 从技术视角到业务视角**：IC关注技术实现，Tech Lead要理解业务战略——这个季度业务重点是什么、数仓建设怎么支撑这个重点、资源应该投向哪个方向。

**3. 从个人成长到团队成长**：IC关注自己技术提升，Tech Lead要关注团队成员成长——谁的模型设计能力需要提升、谁的业务理解不够、怎么通过review和指导提升团队整体水平。

我还在转型过程中，不是完全转过来了。但采购数仓项目给了我这个机会——从一个人做模型设计，到带领团队按规范推进建设，这个过程让我对"技术管理"有了真实的体感。
