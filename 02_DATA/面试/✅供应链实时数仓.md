## 供应链实时数仓建设

### 项目背景

公司硬件供应链覆盖采购、仓储、物流、维修、交付全链路，底层涉及 PMS/WMS/TMS/OMS/MMS 等13个业务子系统，业务形态包括销售类（线性流转）和资产运营类（全生命周期循环管理）。供应链控制塔（SCCT）定位为一线作战工具，数百名仓管、采购员、物流人员依赖实时数据直接指导操作决策，对数据贯通性、时效性和准确性要求极高。

### 个人职责
**1、全链路数据贯通与统一底座**
13个子系统 Binlog 通过 Flink CDC 写入 Paimon 主键表（deduplicate + sequence.field）实现自动去重与乱序过滤，维表统一收口至 Paimon 通过 Lookup Join 关联，替代原 ROW_NUMBER + Redis 存储的方案，Checkpoint 大小减少 90%，任务恢复速度提升 70%，整体吞吐量提升约 40%。

13个子系统 Binlog 经 Flink CDC 写入 Paimon 主键表：`deduplicate` 引擎替代 ROW_NUMBER() 实现主键去重，`sequence.field` 替代 Redis 时间戳判断实现乱序过滤；维表统一迁移至 Paimon，下游通过 Lookup Join 关联。整体任务吞吐量提升了约 40%，Checkpoint 更加稳定。

 好处：开发简洁易维护  不依赖外部主件，实现了Exactly-Once 语义。 读取时候 Paimon Changelog    LSM Tree 结构支持高效合并与索引查找。

**2、采购执行实时监控与异常预警**
基于 Paimon Changelog 流读消费采购订单变更数据，对交期延迟、到货差异、质检异常等关键节点实时规则预警，同时支持 Time Travel 按时间点回溯订单状态，替代原 T+1 报表发现问题的模式。


原来采购异常依赖 T+1 报表，延期或质检问题通常第二天才能发现。我们通过 Flink CDC 将 PMS 采购订单、WMS 到货入库和 MES 质检数据写入 Paimon 主键表，由 Paimon维护订单最新状态并输出 Changelog。下游 Flink 只消费发生变化的订单，按采购订单行聚合采购量、到货量和质检结果，实时判断交期延迟、到货差异和质检异常。命中后生成幂等告警事件，避免重复通知。对于仅由时间流逝触发的延期，还配合定时器或周期扫描处理。Paimon保留的历史 Snapshot 可以通过 Time Travel 恢复指定时间点的订单状态，用于解释异常是何时、因为什么字段变化产生的。最终把异常发现从次日缩短到分钟级，但离线报表仍保留用于日终对账和经营汇总。
指标
延期交付时间：  当前时间 > 约定交付时间 AND 待入库数量 > 0
到货差异量 = 实际到货数量 - 本次应到数量


**3、库存实时水位监控与预警**
采用 Paimon Aggregate Table（SUM / LAST_NON_NULL）替代 Flink Cumulate Window，库存增量事件写入时自动合并计算实时水位，Flink 端无状态写入，Checkpoint 下降90%，下游消费 Changelog 触发补货/积压预警。
**4、供应链实时指标体系与决策支撑**
DWS 层按指标域建立 Aggregate Table 统一聚合采购履约率、库存周转、设备在线率等跨域指标，ADS 层同步至 Doris 提供 OLAP 查询，配合 Spark SQL 每日批处理修数，修数时间从小时级缩短至分钟级。
**5、实时数据质量保障体系**
建设写入层（Paimon 去重 + 乱序过滤）、加工层（实时对账）、产出层（实离对账，差异率 < 0.05%）三层校验体系，设计分场景修数 SOP（Spark SQL 直接修复 Paimon 表 + Flink 指定时间戳恢复），修数资源消耗降低80%。
**6、链路稳定性与运维保障**
建设全链路系统 + 业务双层监控，制定 3-10-15 分钟故障响应 SOP。引入 Paimon 后链路组件从7类收敛至4类，Flink 去重/聚合等逻辑下沉存储层实现无状态化，故障恢复速度提升70%，核心链路稳定性99.9%。

## 话术正文

我介绍一下供应链控制塔实时数据平台这个项目。

旭日硬件供应链覆盖采购、仓储、物流、维修、交付全链路，底层有 13 个业务子系统。业务分两种形态：销售类，像骑行设备、充电宝，走采购入库出库交付的流程；资产运营类，设备投放之后要运维、回收、报废，是全生命周期管理。供应链控制塔是这些业务的统一数据作战台，不是给管理层看报表的驾驶舱，而是一线仓管、采购员、物流人员每天盯着干活的工具，有几百人在用。我负责整个实时数据平台的架构设计和核心场景落地。

这个项目面临三个核心挑战：一是 13 个子系统数据要贯通到统一视图，二是 用户依赖实时数据做操作决策、时效性要求极高，三是数据必须准确，用户看到矛盾数据就不信任系统。

针对这些挑战，我的整体思路是引入 Paimon 构建流式湖仓，把原来散在 Flink State、Redis、Kafka 各处的状态管理统一下沉到 Paimon 存储层。具体落地了几个关键场景。

第一个是全链路数据贯通，这是整个平台的底座。1系统的 Binlog 通过 Flink CDC 写入 Paimon 主键表，用 deduplicate merge-engine 加 sequence.field，写入时自动按 event_time 去重和过滤乱序，Flink 端完全不需要维护去重状态。原来去重靠 ROW_NUMBER 加 Redis，State 膨胀严重，现在全省了。维表也做了收口，原来散在 Redis、Cellar、MySQL 三套存储里，现在统一到 Paimon，事实表通过 Lookup Join 本地查找，不走网络。这个场景实践下来，Checkpoint 下降了 75%，Kafka Topic 减少了 40%。

第二个是采购执行的实时监控。基于 Paimon Changelog 流读消费采购订单变更，对交期延迟、到货差异、质检异常做实时规则预警，替代原来 T+1 才发现问题的模式。

第三个是库存实时水位监控。用 Paimon Aggregate Table 替代 Flink Cumulate Window，库存增量事件写入时自动合并计算水位，Flink 无状态写入，Checkpoint 下降了 90%，下游消费 Changelog 触发补货和积压预警。

第四个是统一指标体系。DWS 层按指标域建 Aggregate Table，统一聚合采购履约率、库存周转、设备在线率等跨域指标，同步到 Doris 做 OLAP 查询。配合 Spark SQL 批处理修数，修数从小时级压到了分钟级。

最后两个是横向能力。数据质量建了三层校验——写入层靠 Paimon 保证去重和乱序过滤，加工层做实时对账，产出层做实离对账，差异率控制在万分之五以内。稳定性方面，链路组件从 7 类收敛到 4 类，Flink 去重和聚合逻辑下沉存储层实现无状态化，故障恢复速度提升 70%，做了 3-10-15 分钟的故障响应 SOP，核心链路稳定性 99.9%。

总结一下成果：业务上打通了 13 个子系统，支撑控制塔一线作战看板上线。技术上关键数字是 Checkpoint 下降 75% 和 90%，Kafka Topic 减少 40%，修数从小时级到分钟级，组件从 7 类收敛到 4 类，实离差异率万分之五以内。最本质的变化是把状态管理从计算层下沉到了存储层，Flink 变成了无状态的编排层，任务更轻、更稳、恢复更快，中间态数据从 Kafka 黑盒变成了 Paimon 可查表。

这个项目里我觉得最有意思的是 Paimon 不同 merge-engine 的场景化选型，以及 Changelog 流读和 Kafka 消费在时效性和可查性上的取舍，如果您感兴趣我可以展开讲。

## 一、整体架构与选型决策（Q1-Q4）

### Q1：为什么选 Paimon 而不是 Iceberg 或 Hudi？

三者都是数据湖格式，但定位不同。Iceberg 偏离线分析，流式更新能力较弱，不支持主键表的高效 Upsert。Hudi 支持主键更新，但 Changelog 生成不如 Paimon 原生，MOR 模式在高频更新下 Compaction 压力大。Paimon 的核心优势有三点：一是基于 LSM Tree 的高效流式更新，主键表天然支持 deduplicate、aggregation、partial-update 等多种 merge-engine；二是原生 Changelog 生成能力，下游直接流读变更数据，不需要额外 CDC 组件；三是和 Flink 深度集成，Paimon 本身就是从 Flink Table Store 演进来的，Flink SQL 原生支持，开发体验最好。我们的场景刚好是高频流式更新加 Changelog 流读，所以选了 Paimon。

### Q2：引入 Paimon 后整体架构是什么样的？原来的组件还用吗？

不是全部推翻重建，而是渐进式迁移。ODS 层的 Binlog 接入仍然走 Kafka，因为采集组件原生对接 Kafka，替换成本高且收益小。但 DWD/DWS 层间的数据流转从 Kafka Topic 迁移到 Paimon 表的 Changelog 流读；去重和乱序处理从 Flink State 加 Redis 迁移到 Paimon 主键表；天粒度聚合从 Flink Window 迁移到 Paimon Aggregate Table；维表从 Redis/Cellar/MySQL 收口到 Paimon；ADS 层查询服务仍然用 Doris。整体是"Flink 做计算编排、Paimon 做存储和状态管理、Doris 做查询服务"的三层架构。Kafka 从层间流转主力退化为 ODS 接入通道，Redis 从维表存储收缩为极低延迟缓存，Doris Sequence 列从主要乱序防线变为最后兜底。

### Q3：13 个子系统数据怎么贯通的？最大的难点是什么？

13 个子系统涵盖 PMS 采购、WMS 仓储、TMS 物流、OMS 订单、MMS 维修等等，数据模型和更新频率差异很大。贯通的做法是：每个子系统的核心业务表通过 Flink CDC 监听 Binlog，写入 Paimon 主键表做 ODS 层，然后在 DWD 层按业务域做清洗关联扩维，最后在 DWS 层按控制塔的指标域做跨域聚合。最大的难点是维度数据的一致性——13 个子系统的维度定义不统一，比如"仓库"在 WMS 里是作业单元，在 TMS 里是发货节点，在 INV 里是库存位置，同一个物理仓库在不同系统里的 ID 和属性都不一样。我们的解决办法是在 Paimon 里建一套统一的维度主表，以 MDM 主数据系统为准，其他系统的维度通过映射关系关联过来。事实表统一通过 Lookup Join 关联这套主维表，而不是各自关联各自的维表，从根源上解决一致性问题。

### Q4：你说业务分"销售类"和"资产运营类"，数据处理上有什么区别？

区别主要在数据生命周期和指标体系上。销售类业务，比如骑行设备和充电宝的采购销售，数据流是线性的——采购、入库、出库、交付，生命周期比较短，关注的是采购履约率、库存周转、交付时效这些流转效率指标。资产运营类业务，比如骑行设备投放后的运维、充电宝的借还、取餐柜的运营，数据流是循环的——投放、使用、回收、维修、再投放，生命周期很长，关注的是设备在线率、故障率、维修周转天数这些运营效率指标。在数据处理上，销售类更多是事件驱动的实时流处理，资产运营类需要维护设备的全生命周期状态，State 更大，用到的 Paimon partial-update 场景也更多。控制塔要把这两类数据统一呈现在同一个看板上，所以统一指标体系的设计就很关键。

## 二、场景一深挖：全链路数据贯通与统一底座（Q5-Q9）

### Q5：Paimon 主键表去重和原来 ROW_NUMBER 去重的本质区别是什么？

ROW_NUMBER 去重是在计算层做的——Flink 需要维护一个按主键分区、按时间排序的窗口状态，每条新数据都要和状态里的历史数据比较排序，这个状态随数据量线性增长。Paimon 主键表去重是在存储层做的——数据写入 LSM Tree 时，相同主键的多条记录在 Compaction 阶段自动合并，只保留最新一条。Flink 端完全不需要维护去重状态，只负责把数据写入 Paimon 表。本质上是把"有状态计算"变成了"无状态写入加有状态存储"。

### Q6：sequence.field 是怎么解决乱序的？和 Doris Sequence 列什么关系？

sequence.field 的原理是：Paimon 在合并相同主键的多条记录时，不是简单取最后写入的那条，而是比较 sequence.field 指定字段的值，比如 event_time，只保留值最大的记录。新写入的数据如果 event_time 比已有记录小，直接丢弃。这和 Doris Sequence 列原理类似，但作用位置不同：Doris Sequence 列解决的是"数据已经到了 Doris 之后的乱序"，而 Paimon sequence.field 解决的是"数据在中间层流转过程中的乱序"。原来的链路中间层乱序靠 Redis 缓存时间戳判断，每条数据一次网络 IO，高并发下是瓶颈。现在 Paimon 在写入时就保证了顺序正确性，到 Doris 时数据已经有序了。整条链路的乱序防御从"末端兜底"变成了"逐层保证"。

### Q7：Paimon 的 LSM Tree 结构具体是怎么工作的？

写入时数据先进入内存的 MemTable，积累到一定大小后 Flush 为磁盘上的有序文件，这是 L0 层。随着 L0 文件增多，后台 Compaction 线程将多个 Sorted Run 合并为更大的有序文件并推到更高层级，合并过程中按主键去重并执行 merge-engine 定义的合并逻辑。读取时需要多层合并读，但配合 Deletion Vectors 可以跳过已被覆盖的记录，加速读取。这个结构的好处是写入极快（顺序追加）、天然支持高频更新，代价是读取需要多层合并，所以 Paimon 更适合写多读少的实时数仓场景。

### Q8：Lookup Join 和原来用 Redis 做维表关联有什么区别？

Redis 做维表关联是"外部点查"模式——每条事实数据来了，Flink 算子发一次 Redis 请求查维度数据，网络 IO 是瓶颈。Paimon Lookup Join 是"本地缓存"模式——Flink 算子启动时将 Paimon 维表数据加载到本地，关联时直接本地查找，没有网络开销。我们用的是 Local PrimaryKey Partial Lookup 模式，不需要全量缓存，只在数据触发时按需加载对应 Bucket 的文件，基于 LSM Tree 有序性做定位查找。性能上 Lookup 延迟从 Redis 的毫秒级降到本地微秒级，代价是本地磁盘占用增加，维表更新有一定延迟，依赖 Paimon 快照刷新周期。但我们的维表更新频率不高，天级更新，这个延迟完全可以接受。

### Q9：Checkpoint 下降 75% 具体是什么概念？绝对值大概多少？

原来用 ROW_NUMBER 去重的任务，State 里要存全量的去重窗口数据，Checkpoint 大小大概在 800MB 左右，制作时间 30-40 秒。切到 Paimon 主键表后，Flink 端去重 State 降为零，Checkpoint 只剩下 Source 端的 Offset 和 Sink 端的少量事务状态，大小降到 200MB 以内，制作时间 5-10 秒。所以说下降 75% 不只是存储空间的优化，更重要的是 Checkpoint 制作更快、更稳定，任务从 Checkpoint 恢复的时间也大幅缩短，整体稳定性有质的提升。

## 三、场景二深挖：采购执行实时监控（Q10-Q11）

### Q10：Paimon 流读和 Kafka 消费在机制上有什么区别？

Kafka 是纯日志流——消费者通过 Offset 顺序读取，读过就没了，数据本身不支持查询。Paimon 流读是基于快照的变更流——每次 Checkpoint 生成新快照，流读消费者读取两个快照之间的差异，本质上是增量快照读。好处有三个：一是数据可查，任意时间点的快照都可以 SQL 直接查询，也就是 Time Travel；二是数据可回溯，指定快照 ID 或时间戳就能读取历史数据，不需要重置 Offset；三是天然去重，因为 Changelog 是基于合并后的结果生成的。代价是时效性受 Checkpoint 间隔影响，通常秒级到分钟级，不如 Kafka 的毫秒级。但在供应链场景里秒级完全够用，采购订单的状态变更不需要毫秒级响应。

### Q11：采购监控的实时规则预警具体怎么实现的？

采购订单有几个关键业务节点：下单、供应商确认、发货、到货、质检。我们监控的规则主要有三类：交期延迟预警，比如供应商确认后超过约定交期还没发货的；到货差异预警，比如到货数量与采购数量偏差超过阈值的；质检异常预警，比如质检不合格率突然升高的。实现方式是在 Flink 消费 Paimon 的 Changelog 流之后，用 CEP（复杂事件处理）或简单的条件判断做规则匹配，命中的数据写入告警 Topic，由下游推送到控制塔看板和大象通知。相比原来 T+1 报表的模式，问题发现时间从一天缩短到分钟级，采购员能在当天就介入处理。

## 四、场景三深挖：库存实时水位监控（Q12-Q13）

### Q12：Aggregate Table 和 Flink Cumulate Window 在效果上有什么区别？

最终结果是一样的，都是按主键做天粒度聚合。区别在于状态管理方式。Cumulate Window 的状态在 Flink 内存里，窗口越大状态越大，一天的窗口意味着一天的数据都在内存里，Checkpoint 要把这些状态全部持久化，所以大且慢。Aggregate Table 的状态在 Paimon 的 LSM Tree 里，每条新数据写入时和已有记录合并，Flink 端是无状态的，Checkpoint 几乎为空。另一个关键区别是乱序处理：Cumulate Window 依赖 Watermark，迟到数据要么丢弃要么进侧输出流，Aggregate Table 不依赖 Watermark，任何时候来的数据都会正确合并到结果中。这就是 Checkpoint 能下降 90% 的原因——从 Flink 承载的几百 MB 窗口状态直接降到几乎为零。

### Q13：库存预警的阈值是怎么设定的？

库存预警分两种：补货预警和积压预警。补货预警的阈值是动态的，根据历史消耗速度和补货周期计算安全库存水位线，当实时库存低于安全水位时触发。积压预警的阈值也是动态的，根据库存周转天数，超过历史均值一定倍数时触发。具体阈值不是我们数据平台定的，是业务方根据经验和供应链模型设定的，我们提供的是实时的库存水位数据和触发预警的机制。这里有个细节，库存数据涉及多个系统——WMS 管仓内库存，TMS 管在途库存，PMS 管在采库存，三者要汇总才是"可用库存"。这个汇总就是在 Paimon Aggregate Table 里完成的。

## 五、场景四深挖：统一指标体系与批处理修数（Q14-Q15）

### Q14：修数流程具体是怎么操作的？停任务不会影响实时性吗？

修数分四步：第一步停掉需要修复的 DWD 层及后续的 Flink 任务；第二步用 Spark SQL 读取 Paimon 表，执行修复逻辑，比如 UPDATE 某个字段或 DELETE 错误数据，修复结果直接写回 Paimon 主键表，利用主键合并覆盖；第三步重启 Flink 任务，指定从某个时间戳开始消费增量数据；第四步验证修复结果。关于实时性影响，确实会有短暂中断，但修数通常安排在业务低峰期，而且 Paimon 修数只需几分钟，比原来的 ODS 重放全链路快得多。对业务的影响从"半天不可用"变成"几分钟不可用"。这也是 Paimon 相对 Kafka 的一个核心优势——中间态数据是可修改的表，而不是只能追加的日志流。

### Q15：DWS 层的 Aggregate Table 和场景三的 Aggregate Table 有什么区别？

场景三的 Aggregate Table 聚焦在单一指标域——库存水位，聚合函数相对简单，主要是 SUM 和 LAST_NON_NULL。场景四的 DWS 层 Aggregate Table 是跨指标域的，一张表里可能同时聚合采购履约率、库存周转、设备在线率等不同域的指标，不同字段用不同的聚合函数——有的用 SUM，有的用 COUNT，有的用 LAST_NON_NULL。设计上的关键决策是按指标域拆表还是合表。我们选择的是按指标域拆表——采购域一张、库存域一张、设备域一张，在 Doris ADS 层再通过视图或宽表做跨域关联查询。这样做的好处是单表的聚合逻辑清晰，不会因为某个域的数据异常影响其他域。

## 六、场景五深挖：数据质量保障（Q16-Q18）

### Q16：三层校验体系具体是怎么做的？

写入层靠 Paimon 本身的能力——主键表的 deduplicate merge-engine 保证去重，sequence.field 保证乱序过滤，这是第一道防线。加工层做实时对账——在 Flink ETL 过程中，对关键指标埋入计数器和求和器，和上游源数据做实时比对，比如 ODS 层进来了 1000 条采购单，DWD 层出去也应该是 1000 条，如果对不上就触发告警。产出层做实离对账——每天离线数据产出后，和前一天的实时数据快照做同口径对比，核心指标的差异率必须在万分之五以内。超出阈值的会触发告警并生成修数工单。

### Q17：实离对账差异率万分之五是怎么做到的？差异的常见原因是什么？

万分之五是最终的稳态结果，刚上线时差异率大概在千分之三左右，花了两三个月逐步收敛的。常见的差异原因有三类：第一是时间窗口差异——实时数据截止到某个时间点，离线数据可能多包含或少包含几秒的数据，这种通过对齐快照时间点来解决。第二是乱序数据——极端迟到的数据实时链路可能已经过了窗口，但离线全量计算能包含进去，这种通过延长 Paimon 的快照保留时间和调优 sequence.field 来缓解。第三是口径差异——某些边界 case 在实时和离线的处理逻辑上不完全一致，这种需要逐个排查修复。收敛到万分之五以后，剩下的差异基本都是第一类时间窗口差异，属于可接受范围。

### Q18：修数 SOP 具体是什么流程？修数资源消耗降低 80% 怎么算的？

修数 SOP 分三种场景。第一种是"单表修数"——发现某张 DWD 表的数据有问题，停掉这张表的下游 Flink 任务，用 Spark SQL 直接修复 Paimon 表，然后重启下游任务从指定时间戳恢复。第二种是"链路修数"——问题出在 ODS 层，需要从源头修复，这种先修 ODS 的 Paimon 表，然后依次重启 DWD、DWS 的 Flink 任务，每层都从指定时间戳消费增量。第三种是"紧急修数"——线上数据错误已经影响到一线操作，这种直接在 Doris ADS 层做临时修复，保证前端展示正确，同时后台按第一种或第二种流程做根因修复。每种场景都有标准化的操作文档、检查清单和回滚方案。

关于修数资源消耗降低 80%，原来修数走 ODS 全链路重放——需要启动 ODS 到 DWD 到 DWS 的完整 Flink 任务链来重新消费和计算，多个任务并行跑，资源消耗大、时间也长，通常要占用整条链路的全部计算资源跑数小时。现在直接用 Spark SQL 对 Paimon 目标表做修复，只需要一个 Spark 任务跑几分钟，修的是哪张表就操作哪张表，不需要全链路重跑。从"启动全链路 N 个 Flink 任务跑数小时"变成"一个 Spark 任务跑几分钟"，资源消耗大概降了 80%。

## 七、场景六深挖：链路稳定性与运维（Q19-Q21）

### Q19：3-10-15 分钟故障响应 SOP 具体是什么？

3 分钟是发现时间——从数据异常发生到监控系统触发告警，要求在 3 分钟以内。这靠两层监控：系统层监控 Flink 任务的 Checkpoint 状态、反压指标、消费延迟；业务层监控核心指标的波动和断流。10 分钟是定位时间——从收到告警到确认故障根因和影响范围，要求在 10 分钟以内。这靠标准化的排查路径和 Paimon 的可查能力，直接 SQL 查 Paimon 快照就能定位问题出在哪一层哪个时间点。15 分钟是恢复时间——从确认根因到业务恢复正常，要求在 15 分钟以内。恢复手段包括任务重启、从 Checkpoint 恢复、切换备链路等。整个 SOP 有值班机制，S 级任务的告警直接电话通知。

### Q20：组件从 7 类收敛到 4 类，具体是哪些？外部组件依赖减少 60% 怎么算的？

原来 7 类是：Flink 做计算、Kafka 做层间流转、Doris 做查询、Redis 做维表缓存和乱序状态、Cellar 做维表缓存、MySQL 做维表存储、HBase 做部分维表。收敛后 4 类是：Flink 做计算编排、Paimon 做存储和状态管理、Doris 做查询服务、Spark 做批处理修数。能减少的原因是 Paimon 一个组件替代了原来多个组件的职责——Paimon 主键表替代了 Redis 的乱序状态缓存，Paimon 维表替代了 Redis/Cellar/MySQL 的维表存储，Paimon Changelog 流读替代了 Kafka 的层间流转。Kafka 没有完全去掉，ODS 层接入还在用，但不再算核心链路组件了。

外部组件依赖减少 60% 是这样算的：原来 Flink 任务需要依赖的外部组件有 Kafka（读写）、Redis（维表查询+乱序判断）、Cellar（维表查询）、MySQL（维表查询）、HBase（维表查询），一共 5 类外部依赖。引入 Paimon 后，Redis、Cellar、MySQL 的维表职责收口到 Paimon，Kafka 的层间流转也被替代，Flink 任务的外部依赖变成 Paimon（读写）和 Doris（写入），减掉了 3 类，从 5 类降到 2 类，大概就是 60%。组件减少的直接好处是运维复杂度降低——以前排查问题要登录五六个系统看日志，现在三四个就够了。

### Q21：故障恢复速度提升 70% 是怎么算的？

这个主要体现在 Flink 任务从 Checkpoint 恢复的时间上。原来去重和聚合类任务的 State 很大，Checkpoint 动辄几百 MB 到 GB 级，恢复时需要从 HDFS 下载 State 文件、反序列化、重建 RocksDB，整个过程可能要五六分钟。切到 Paimon 后，这些任务的 Flink State 降为几乎为零，恢复时只需要加载 Source Offset 和 Sink 事务状态，几十秒就能完成。从平均五六分钟降到一两分钟，大概就是 70% 的提升。另外一个维度是修数恢复——原来修数需要 ODS 全链路重放，小时级；现在 Spark SQL 直接改 Paimon 表，分钟级。这两个维度加在一起，故障恢复速度整体提升 70% 是有依据的。

## 八、Paimon 技术深度追问（Q22-Q24）

### Q22：Paimon 有什么局限性？哪些场景不适合用？

三个主要局限。第一是时效性上限：Paimon 的流读时效性受 Checkpoint 间隔制约，通常 1-10 秒级，做不到 Kafka 的毫秒级，所以对延迟极高的场景不适合，比如风控拦截。第二是高频更新下的 Compaction 压力：如果单个 Bucket 写入 QPS 非常高，LSM Tree 的 Compaction 可能跟不上，导致读放大严重。我们通过合理设置 Bucket 数量分散压力和调优 Compaction 策略来缓解。第三是生态成熟度：Paimon 相比 Hive 和 Iceberg 生态还年轻，部分 BI 工具支持不够完善，所以我们 ADS 层查询仍然用 Doris。

### Q23：你提到 Paimon 做了 Bucket 调优，具体怎么做的？有没有踩过坑？

踩过一个比较典型的坑。刚上线时某个热点表的 Bucket 数设得太少，只有 4 个，但有一个业务维度的数据量特别集中，导致某个 Bucket 的写入量是其他的好几倍，Compaction 速度跟不上写入速度，L0 文件堆积，流读延迟从秒级退化到分钟级。排查定位后的解决方案是：第一，增加 Bucket 数到 16，按主键哈希分散写入压力；第二，调大 sort-spill-threshold 和开启 async-file-write，提升单 Bucket 的写入吞吐；第三，调整 Compaction 的触发阈值，让合并更及时。调优后流读延迟稳定在 3-5 秒。这个经验告诉我，Paimon 的 Bucket 数量不能拍脑袋定，要根据数据分布和写入 QPS 做压测。

### Q24：Paimon 的 Changelog 是怎么生成的？有几种模式？

Paimon 生成 Changelog 有三种模式。第一种是 Input 模式，直接把写入的原始数据作为 Changelog 输出，适合上游已经是 CDC 数据流的场景，不做额外处理，开销最小。第二种是 Lookup 模式，写入数据时 Lookup 已有的旧值，生成完整的 before/after 变更记录，开销稍大但下游能拿到完整的变更信息。第三种是 Full Compaction 模式，在 Compaction 的时候对比前后两次 Compaction 的结果，生成差异 Changelog，开销最大但最准确。我们大部分场景用的是 Lookup 模式，因为下游需要完整的变更信息做增量计算。对于性能敏感的场景会用 Input 模式，前提是上游的 CDC 数据本身就携带了 before/after 信息。

## 九、业务理解与价值追问（Q25-Q27）

### Q25：控制塔的"一线作战工具"和传统 BI 看板有什么本质区别？

本质区别在于数据的使用方式和容错要求。传统 BI 看板是给管理层"看趋势"的——数据延迟半小时、有点误差都没关系，管理层看的是大方向。一线作战工具是给仓管、采购员、物流人员"指导操作"的——仓管根据实时库存水位决定要不要紧急补货，采购员根据供应商交期预警决定要不要催单，物流人员根据在途状态决定仓库月台的分配。数据延迟或错误直接导致错误的操作决策，影响是即时的、具体的。所以对数据平台的要求完全不同：时效性从分钟级提升到秒级，准确性从"大差不差"变成"分毫不差"，可用性从"偶尔看看"变成"时刻依赖"。这也是为什么我们要在数据质量和链路稳定性上投入这么多精力。

### Q26：这个项目的成果你觉得最有价值的是什么？

业务维度上最有价值的是"把数据从报告变成了工具"。上线前一线人员用的是 T+1 的 Excel 报表，出了问题第二天才知道。上线后控制塔变成了他们的工作入口，采购执行进度、库存水位、物流状态都是实时的。举个例子，以前供应商延迟交货可能两三天后才发现，现在交期预警实时推送，采购员当天就能催单或启动备选方案。技术维度上最有价值的是"状态管理下沉到存储层"这个架构思路。Paimon 主键表让 Flink 从有状态计算变成无状态编排，这不只是性能优化，而是从根本上改变了实时数仓的运维模型——任务更轻、恢复更快、修数更简单。这个思路在其他场景也可以复用，比如未来如果要做实时数据中台，同样可以用 Paimon 做统一的状态管理层。

### Q27：如果重新做这个项目，你会有什么不同的做法？

两个方面。第一是 Paimon 引入的时机可以更早。我们是先搭了 Flink+Kafka+Doris 的体系，稳定运行后才引入 Paimon 做架构升级，这意味着有一轮迁移成本。如果一开始就清楚业务会从管理层看板演进到一线作战工具，可以在初始架构中就把 Paimon 作为中间层，减少后续的迁移工作。第二是数据质量体系可以更前置。我们的三层校验体系是在业务跑起来之后逐步建的，期间有一段"裸奔"期，出过几次数据不一致的问题影响了一线信任度。如果从第一天就把实离对账、实时 DQC 作为基础设施同步建设，业务上线会更平滑。

## 十、压轴问题（Q28-Q29）

### Q28：你在这个项目中遇到的最大挑战是什么？怎么解决的？

最大的挑战不是技术本身，而是"13 个子系统数据口径的统一"。13 个系统由不同团队开发，字段命名、状态码定义、时间精度都不一样。比如同一个设备，PMS 里叫 asset_id，WMS 里叫 device_sn，MMS 里叫 equipment_no。同一个状态"已入库"，不同系统的状态码可能是 3、IN_STOCK、或者 RECEIVED。我的做法是三步：第一步，和每个系统的开发对齐，梳理出一份跨系统的字段映射文档；第二步，在 Paimon DWD 层统一做字段标准化和状态码映射，把这个逻辑下沉到公共层，上层任务只用标准字段；第三步，建了一套自动化的映射校验——新系统接入时必须提供字段映射表，CI 流程自动检查映射完整性。这个工作技术含量不高但极其重要，因为底层数据不统一，上面所有的监控、预警、指标都是空中楼阁。

### Q29：Paimon 在你的场景中有踩过什么坑吗？

踩过一个比较典型的坑。刚上线时有个热点 Bucket 的写入 QPS 特别高，导致 LSM Tree 的 L0 文件堆积——写入速度超过了 Compaction 速度，流读延迟从正常的秒级退化到了分钟级，下游看板数据刷新变慢。排查后发现是 Bucket 数量设少了，热点 Key 集中在同一个 Bucket。解决方案是三步：增加 Bucket 数量按主键哈希分散写入压力，调大 sort-spill-threshold 让 Compaction 更积极，开启 async-file-write 提升写入吞吐。调优后流读延迟恢复到秒级。这个经验让我后来给每个 Paimon 表都做了 Bucket 数量的容量评估，根据预估 QPS 和主键分布来设定，避免类似问题。

## PMS · 采购管理

**📋 业务详细版**  
PMS 是供应链"采购到付款"流程的核心系统，覆盖从需求发起到货款结算的全链路。主要功能包括：采购计划制定（基于 SCP 下发的补货计划或业务手动发起）、供应商询比价与定价、采购订单下达与跟踪、到货验收与入库触发（联动 WMS）、发票核销与付款申请。系统与 SRM 打通，实现供应商资质自动校验；与 CST 联动，完成采购成本核算。在骑行、充电宝、餐饮 SaaS 等多业务线中承担硬件物资的集中采购管理职责。

**📋 业务详细版**

**🎤 面试版**

WIP 覆盖硬件产品的生产制造过程管理，主要功能包括：生产工单创建与下达、工序流转与在制品状态跟踪、物料投料管理（联动 WMS 扣减原材料库存）、生产完工入库（联动 WMS 生成成品入库）、生产成本核算（联动 CST）。在自动车、无人机等涉及生产制造的业务线中，WIP 承担了从原材料到成品的全过程数字化管理，是制造类业务供应链数据的重要来源。

**🎤 面试版**  
WIP 是在制品管理系统，覆盖生产工单、工序流转、物料投料到完工入库，与 WMS 联动管理原材料和成品库存，服务自动车、无人机等制造类业务线。

---

## CST · 成本管理

**📋 业务详细版**  
CST 是供应链财务核算的核心模块，负责归集和核算各环节产生的成本。主要功能包括：采购成本核算（来源 PMS 采购订单）、仓储成本分摊（来源 WMS 仓库运营数据）、运输成本归集（来源 TMS 运费结算）、返修成本核算（来源 MMS 维修工单）、生产制造成本核算（来源 WIP 工单）、成本报表与利润分析。CST 是供应链"财务入账"流程的核心，其数据是业务成本分析和 TCO（总拥有成本）优化的主要依据。CST 内嵌数据中心/报表模块，是数仓接入的重点财务系统。

**🎤 面试版**  
CST 是成本管理系统，汇聚采购、仓储、运输、返修、生产各环节成本数据，支持成本核算和利润分析，是供应链 TCO 优化的核心财务模块。

---

## ARS · 应收管理

**📋 业务详细版**  
ARS 负责销售侧的应收账款全生命周期管理，主要功能包括：应收账款生成（来源 OMS 订单完成事件）、账单核对与对账、收款登记与核销、逾期账款预警与催收、坏账管理。系统与 OMS 紧密联动，实现销售履约到财务收款的自动化衔接；与 CST 联动，支持销售毛利的实时核算。ARS 是"销售到收款"流程的财务收口，其数据质量直接影响公司应收账款报表的准确性。

**🎤 面试版**  
ARS 是应收管理系统，覆盖应收账款生成、对账、收款核销到逾期催收，与 OMS 联动实现销售到收款的财务闭环。

## SRM · 供应商关系管理

**📋 业务详细版**  
SRM 是供应商全生命周期管理平台，覆盖供应商从引入到淘汰的全过程。主要功能包括：供应商准入审核与资质认证、供应商基础信息维护（联动 MDM 主数据）、采购寻源与供应商邀标、绩效评估（交期达成率、质量合格率、服务评分）、供应商分级分类管理、协同对账与结算。SRM 为 PMS 采购流程提供合格供应商名录，是供应商数据的权威来源，也是供应链风险管理（供应商集中度、资质到期预警）的重要抓手。

**🎤 面试版**  
SRM 是供应商关系管理系统，覆盖供应商准入、资质管理、绩效评估到协同对账，为 PMS 提供合格供应商名录，是供应商数据的权威来源。

## MDM · 主数据管理

**📋 业务详细版**  
MDM 是整个旭日系统的数据基础设施，统一管理供应链各类主数据，包括：物料主数据（物料编码、规格参数、计量单位）、商品主数据（商品分类、价格体系）、供应商主数据（供应商编码、银行账户、资质信息）、仓库主数据（仓库编码、库区库位、仓库属性）、渠道主数据（销售渠道、配送区域）。MDM 为 PMS、WMS、OMS、SRM 等各子系统提供统一的基础数据标准，是解决跨系统数据口径不一致问题的核心，也是数仓主数据层建设的直接来源。

**🎤 面试版**  
MDM 是主数据管理系统，统一管理物料、商品、供应商、仓库、渠道等基础主数据，为各子系统提供一致的数据标准，是供应链数据治理的基础底座。
