# 供应链实时数仓面试问答

> [!info] 使用说明
> 这组问答围绕简历中的“美团-供应链实时数仓”项目展开，从业务根因依次追到跨系统建模、库存与运输口径、Paimon 技术方案、数据质量和成果量化。回答采用面试口述视角，但对无法由现有材料直接证明的数字保留了证据边界，避免把方案设想说成已经验证的事实。

## Q1：为什么要建实时控制塔？

**结论**：控制塔不是把离线报表改成实时刷新，而是把分散的采购、库存、运输、维修状态拼成一条可操作的供应链事实链。

**形式化**：`传统报表 = 看结果；控制塔 = 看状态 + 找异常 + 立即行动`

**怎么想到的**：

- PMS、WMS、TMS、OMS、MMS 等系统各自只记录一个业务环节，单个系统无法回答“订单现在卡在哪里”。
- 仓管、采购员和物流人员需要根据当前状态补货、催交、调拨或处理异常，而不是第二天再看汇总结果。
- 因此数据平台既要统一主数据和状态口径，也要把分钟级异常推送到看板和通知链路。

**边界**：采购和物流业务通常不需要毫秒级响应；如果是风控拦截或交易撮合，仅靠秒级快照和 Changelog 可能不够。

## Q2：13 个系统怎么真正贯通？

**结论**：系统贯通不是把表集中存放，而是用统一主数据、业务单据关联键和标准状态，把各环节拼成可追踪的业务链路。

**形式化**：`源系统 ID → 映射表 → 统一 ID；PR → PO → 入库 → 运输 → 签收/维修`

**怎么想到的**：

- 各系统先通过 Flink CDC 接入 ODS，每张核心表保留来源系统、业务主键、操作类型、源端更新时间和接入时间。
- MDM 作为物料、仓库、供应商等核心维度的标准来源，其他系统中的本地编码通过映射表归一到统一维度 ID。
- DWD 按采购、库存、运输、维修等业务域还原单据和事件，DWS 再按指标域聚合履约率、库存水位和设备在线率。
- 贯通后的结果必须支持从一个采购订单继续追到发货、在途、入库或质检状态，而不只是得到几张互不相干的宽表。

**边界**：如果源系统没有稳定的关联键，只能依赖时间、物料和数量等弱条件匹配，链路准确率必须单独评估，不能宣称百分之百贯通。

> [!tip] 面试前补充
> 准备好 13 个系统的完整清单：PMS、WMS、TMS、OMS、MMS、CST、ARS、INV、WIP、RMS、SRM、SCP、MDM，并能用一句话解释每个系统负责什么。

## Q3：库存水位到底怎么算？

**结论**：库存预警必须先区分仓内现有量、仓内可用量和供应可用量，否则同一个“库存”会产生三种答案。

**形式化**：`仓内可用 = 现有 - 占用 - 冻结；供应可用 = 仓内可用 + 采购在途 + 调拨在途`

**怎么想到的**：

- WMS 的现有量表示账面持有数量，但已被订单占用或因质检、盘点等原因冻结的部分不能继续分配。
- PMS 提供尚未完成入库的采购在途，WMS/TMS 提供仓间调拨或运输在途。
- 补货预警通常比较供应可用量和安全库存，积压预警则结合库存周转天数、历史耗用和需求预测。
- 粒度一般落在“组织 + 仓库 + 物料”，再按品类、车型或区域向上汇总。

**边界**：采购在途是否计入“可用”取决于场景；如果到货日期不可靠或订单可取消，应分别展示而不是无条件相加。

## Q4：调拨为什么最容易算错？

**结论**：调拨是一笔业务、两个仓库视角，若只按原始事务行正负汇总，就可能漏算收货仓或把库内移动误算成库存变化。

**形式化**：`一次调拨 = 发货仓 -N + 收货仓 +N；全局库存净变化 = 0`

**怎么想到的**：

- 普通事务可根据数量正负判断入库和出库，但带发货仓、收货仓的调拨必须拆成两个仓库事件。
- 发货仓产生负向库存变化，收货仓产生正向库存变化，全公司总库存不应因此增加。
- 库内移位只改变库区或库位，不改变仓库级库存，需要按单据类型或操作类型排除。
- 进销存对账用“期初 + 入库 - 出库 = 期末”验证，差异再从迟到事务、冲销、补录和快照时点排查。

**边界**：若业务把调拨在途单独作为一个虚拟库存位置，发出和收货发生在不同时间，仓库合计在中间时段不会立刻守恒，需要把在途库存纳入等式。

## Q5：运输宽表怎么避免放大？

**结论**：先固定分析粒度，再分别聚合一对多子表；直接把配送单、运单和派车单平铺 Join，会让计划数量和金额重复累加。

**形式化**：`配送单 1:N 运单 1:N 派车单；先聚合后关联 ≠ 多表直接相乘`

**怎么想到的**：

- 配送单描述运输需求，整车运单组织运输任务，派车单记录具体车辆和执行过程，三者的粒度不同。
- 如果看执行状态，宽表可以采用派车单粒度，但配送单级计划数量只能分摊或以窗口口径保留，不能直接求和。
- 如果看配送需求达成率，应先在派车或签收侧聚合实际数量，再回到配送单粒度关联计划数量。
- 取消重派、部分签收和迟到签收必须用业务主键、状态优先级和版本字段处理。

**边界**：不存在一个宽表能同时完美服务所有指标；配送履约、车辆执行和承运商分析可能需要不同粒度的事实表。

## Q6：为什么去重下沉到 Paimon？

**结论**：下沉去重的价值不是少写一段 `ROW_NUMBER`，而是把高基数业务状态从 Flink Checkpoint 转成可查询、可合并的表状态。

**形式化**：`旧：Flink State + Redis 判版本；新：Flink 写入 + Paimon 主键合并`

**怎么想到的**：

- 流式 `ROW_NUMBER` 或 Top1 需要按业务主键维护最新记录，高基数和长 TTL 会扩大 Flink State。
- Redis 保存最大版本号时，每条数据还要进行外部网络点查，并承担缓存一致性和故障恢复问题。
- Paimon 主键表用 `deduplicate` 合并相同主键，再用 `sequence.field` 比较源端版本，避免迟到旧数据覆盖新状态。
- Flink 端因而主要保留 Source Offset、Sink 事务和少量算子状态，业务去重状态转移到存储层。

**边界**：`sequence.field` 必须可靠单调；只有秒级 `update_time` 时，同一秒多次更新可能无法稳定排序，最好使用源库版本号或 Binlog 位点，并验证删除事件的顺序语义。

## Q7：维表为什么收口到 Paimon？

**结论**：维表收口首先解决的是口径一致性，其次才是减少 Redis、Cellar、MySQL 多种外部点查和运维成本。

**形式化**：`三套维表 × 三套口径 → 一套主维 + 来源映射`

**怎么想到的**：

- 同一个仓库在 WMS 可能是作业单元，在 TMS 是运输节点，在库存系统又是库存位置，ID 和属性并不一致。
- 以 MDM 为标准建立统一物料、仓库和供应商维度，再维护源系统编码到统一 ID 的映射。
- 事实流通过 Paimon Lookup Join 获取统一属性，避免每个任务各自关联自己的维表。
- Lookup 能结合本地缓存或局部加载减少外部网络点查，但具体性能取决于缓存命中、Bucket 和快照刷新配置。

**边界**：高频变化且要求毫秒级生效的维度不一定适合完全依赖 Paimon 快照刷新，可保留 Redis 等低延迟缓存作为补充。

## Q8：实时预警如何避免误报？

**结论**：预警必须由“业务状态 + 时间条件 + 数量差异 + 幂等抑制”共同决定，单独看到一条变更事件不能直接报警。

**形式化**：`有效告警 = 命中规则 + 状态未关闭 + 超过阈值 + 未重复通知`

**怎么想到的**：

- 交期延迟应判断“约定交付时间已过且待入库数量大于 0”，不能只看订单创建至今多少天。
- 到货差异要比较采购、发货、收货或入库数量，并明确允许的业务容差。
- 质检异常要区分抽样不合格、整批拒收、返修和让步接收，不能把所有 NG 都视为同一种风险。
- Flink 消费 Paimon Changelog 后执行规则判断，将告警事件写入消息通道；告警键、冷却时间和恢复通知用于抑制重复告警。

**边界**：阈值本身通常由采购、库存或质量业务方定义；数据团队负责口径、计算和触发机制，不应代替业务方决定风险标准。

## Q9：数据质量怎么守住？

**结论**：只在最终看板对数无法定位问题，必须把质量检查拆成接入、加工和产出三层，并让每层都有可追踪的差异证据。

**形式化**：`写入正确 → 加工守恒 → 产出同口径；任一层失败 = 结果不可交付`

**怎么想到的**：

- 写入层验证 CDC 完整性、主键去重、版本乱序、删除语义和最大更新时间。
- 加工层不能只比较 `COUNT(*)`，还要比较主键数、数量 `SUM`、金额 `SUM`、状态分布、空值率和孤儿记录数。
- 产出层按同一快照时点、同一过滤范围和同一指标定义进行实时与离线对账。
- 出现差异时按单表修复、链路重算或紧急 ADS 兜底分类处理，并保留修数前后审计记录。

**边界**：使用 `DISTINCT` 消除多表 Join 后的重复，可能掩盖粒度设计错误；时间窗口未对齐时，差异率再低也不能证明口径一致。

## Q10：优化成果怎么证明？

**结论**：百分比只有绑定绝对值、统计范围和对照条件才可信，否则“下降 75%”只是一句无法复核的宣传。

**形式化**：`可信成果 = 前值 + 后值 + 计算式 + 同条件 + 数据来源`

**怎么想到的**：

- Checkpoint 下降 75% 的口径应是相同任务、并行度、Checkpoint 间隔和相近流量下，例如 800 MB 降到 200 MB，计算为 `(800-200)/800`。
- 除大小外，还应对比制作时长、失败率和恢复时长；当前材料中的参考值是 30～40 秒降至 5～10 秒。
- Kafka Topic 减少 40% 必须说明统计边界和绝对数量，例如核心链路从 50 个降到 30 个，主要下线去重、扩维和层间中转 Topic。
- 稳定性 99.9%、恢复提升 70% 和修数资源下降 80% 也应分别绑定监控周期、平均恢复时间、CPU/内存时长或任务资源账单。

**边界**：现有需求文档能证明业务背景和口径，但不能独立证明上述技术数字；如果拿不出监控记录，应改成可确认的绝对值或弱化表述，不能编造证据。

> [!warning] 量化数字证据清单
> 面试前补齐改造前后任务名、并行度、Checkpoint 配置、Checkpoint Size/Duration、Topic 绝对数量和统计范围、平均恢复时长、修数资源账单或监控截图。

## 一句话收口

这套项目真正的价值不是“用了 Paimon”，而是把多系统供应链状态压成一条可追踪、可校验、可行动的事实链，再用合适的存储和计算边界保证它持续可信。
