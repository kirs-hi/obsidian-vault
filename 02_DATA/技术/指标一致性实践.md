## 前言

指标平常的表现是数值，但要表达出它的意义还需要对数值进行解释，本文中提到的指标的语义就是指对数值的解释。例如2020是一个数值，它可以是年份，可以是价格，年份及价格就是2020的语义。

本文的指标一致性是指多个数据产品间的指标语义一致且数值一致。

## 一、背景：指标的一致性随着业务发展越来越重要！

### 为满足业务的快速发展，某业务建设了大量的数据表及数据产品，不同产品的指标不一致问题频发

某酒店业务自成立后持续的快速增长，2014年Q2间夜量超过某竞品，2015Q4间夜量超过某竞品，2017年间夜破两亿，2019年Q2超过某竞品系成为国内TOP1。某酒店四大法宝之一是科技驱动，对系统产品建设很重视，同时对数据以及数据产品也很重视。为了满足业务的发展需要，住宿数据组建设了大量的数据以及数据产品。据2019年5月统计，住宿在线数据表6000+张，160+工具报表，14个定制化专题产品，以230+数据API服务对接了18个业务系统。经常性（周均3次）发生不同产品的指标不一致问题。

### 指标不一致阻塞业务的发展

不一致的指标会导致数据的失真及失信，进而导致决策风险，直接影响业务发展。错误的数据往往会造成错误的决策，策略的错误导致目标错误或运营失败；失信的数据也很难快速实现决策（需先确认数据的准确性），尤其住宿战役期间，一步慢（决策慢）步步慢（运营慢），直接影响业绩达标。

不一致的指标需要额外的沟通成本及开发成本才能对齐，从而导致业务发展的滞后。数据的开发者之间、开发者与需求者之间、数据的不同使用者之间都需要额外的沟通成本才能最终达成一致的结果。当多个项目以及多方的额外成本叠加后，业务的发展即滞后。

### 指标一致性的必要性及重要性逐步提高

住宿场景囊括了探索期（海外）、进攻期（高星）、发展期（简约）、变革期（品质）不同的业务阶段，不同阶段不同的业务发展需要越来越多的数据产品支持，并且产品的迭代也会越来越频繁。住宿的数据服务为了服务业务的发展，指标一致性的必要性及重要性也就越来越高。

### 住宿在解决指标不一致问题上的历程

**探索**：业务驱动指标的一致性服务。2018年是某公司的服务年，酒店也大搞服务，对内对外都要求提高服务质量。据2018年住宿数据问卷调研显示指标不一致是住宿数据服务不满意的最重要指标之一。因此指标一致性作为我们提高数据服务一个重要目标，开始探索指标一致性的解决方案。

**立项**：技术自驱指标一致性建设。2019年某公司强调苦练基本功，数据开发的基本功之一就是做好业务的数据服务。住宿数据组所在的数据BP中心成立了大禹项目，数据一致性项目是其的一个重要子项，数据一致的重要内容就是指标一致性，正式立项制定解决方案。

**持续**：业务的发展以及数据的规划都需要长期的解决方案。2020年酒店业务3个核心关键词之一是"数据化"，数据的准确性及一致性提高了更高的要求。2019年住宿数据组在核心及重要的产品上实现了指标一致，如何应用到住宿全场景以及复用到数据BP其他业务甚至更广，需要持续的完善和推广。

## 二、目标：指标全局一致性

### 2.1 指标不一致有三种表现形式

指标由语义和值两部分组成，语义由名称和含义组成，即指标可分解为名称、含义、值三个基本元素。指标不一致可分解为三个元素中存在不一致的情况。如下所示：

![[attachments/704328266/img_00.png]]

在上述指标不一致的元素组合中，✅为不一致的正常现象，❌为数据不一致的异常情况，即如下三种表现形式：

**同名不同义**：相同的名称但含义不同。

**同义不同名**：相同的含义但名称不同。

**同名同义不同值**：相同的含义且相同的名称，但数据值不同。

以指标"订单转化率"举例分别说明三种表现形式：不同的人员对相同的指标名"订单转化率"理解含义不同，不同的人员对相同含义的指标名称呼不同，不同的产品的相同含义相同名称的指标的数值不同。如下所示：

### 2.2 指标不一致的四个主要原因

指标不一致问题的产生过程是使用者（人）从产品获得数据后进行对比，因此从"人"、"产品"、"数据"三个模块拆解分析指标不一致的原因，拆解内容如下：

![[attachments/704328266/img_02.png]]

如上图拆解后，可拆解为八个原因，再对该八个原因进行分析：

| 模块 | 原因 | 解释 | 概率 | 数据相关 | 同名不同义 | 同义不同名 | 同名同义不同值 | 指数 |
|------|------|------|------|----------|-----------|-----------|--------------|------|
| 用户 | 操作问题 | 用户在产品上操作错误或失误。 | 低 | 弱 | | | ✔️ | ★ |
| 用户 | 理解问题 | 产品上线后，不同用户对产品上数据的语义理解不一致，各说各话。 | 高 | 强 | ✔️ | ✔️ | ✔️ | ★★★★ |
| 产品 | 需求问题 | 产品需求设计与业务要求不一致，产品开发与产品需求不一致 | 高 | 强 | ✔️ | ✔️ | ✔️ | ★★★ |
| 产品 | 功能问题 | 产品功能BUG。 | 低 | 弱 | | | ✔️ | ★ |
| 数据 | 生产逻辑问题 | 多个数据表的生产逻辑不同（开发RD不同或者迭代有遗漏）。 | 高 | 强 | ✔️ | ✔️ | ✔️ | ★★★★ |
| 数据 | 调度问题 | 调度不同步或者调度内容不一致。 | 低 | 中 | | | ✔️ | ★ |
| 数据 | 引擎问题 | 引擎配置不一致（如kylin配置），数据库引擎故障。 | 低 | 中 | | | ✔️ | ★ |
| 数据 | 查询问题 | 查询的数据表不一致，相同数据表查询的逻辑不一致。 | 高 | 强 | ✔️ | ✔️ | ✔️ | ★★★ |

> 说明：指数 依据 概率、解决难易度、与数据开发相关程度等因素综合制定。

综上，问题1和6发生率较低并且可通过规范和培训进一步降低概率，问题4和7也属于低概率且与数据开发关系较小，另外4个问题发生概率高且与数据开发强相关，因此导致指标不一致的**四个主要原因**为：

1. 理解问题
2. 需求问题
3. 生产逻辑问题
4. 查询问题

### 2.3 指标不一致的解决方式：指标标准化

前面确定了四个主要原因，其中理解问题和需求问题可以通过对齐数据使用者、数据开发者、需求管理者之间的数据语义，形成一致化的语义解决；生产逻辑问题可通过保障不同的ETL中使用相同的生产逻辑，且当逻辑变化时所有的ETL联动更新，保障一致化的生产解决；查询问题可通过查询相同的数据表及相同的查询语法，保障一致性的查询解决。综上可总结三个解决办法：

- 一致化语义
- 一致化生产
- 一致化查询

上述三个问题可进一步总结为 **统一数据标准**。

建设好指标标准后仍需保证持续有效的运营，因此还需要技术工具以及流程制度，进一步总结为**指标标准化**。

因此，**指标标准化**作为解决指标不一致的解决方式。

## 三、实践：用技术工具和流程制度保障指标的持续标准化

### 3.1 整体方案设计

根据指标标准化的要求，需要有统一的指标标准，有技术工具支撑，有流程制度做保障。其中：

- 指标标准包含标准的指标语义、标准的生产逻辑、标准的查询逻辑；
- 技术工具包括标准的管理工具、保障生产一致的生产工具、保障查询一致的查询工具；
- 流程制度是要制定标准化SOP，保障指标标准、技术工具的规范性和可持续运营。

- 与数据应用平台合作，优化起源系统，作为指标标准语义与标准查询的管理工具
- 与数据生产平台合作，新建达芬奇系统，作为指标标准生产的管理工具

### 3.2 建立指标标准

指标标准化的第一步也是最重要的一步就是建立指标标准。建立标准的过程为：

首先要收集现有的指标信息，包括指标池、ETL生产语法、产品查询语法等；

然后与数据的使用人员、管理人员、开发人员对齐信息，形成一致性的共识，包括指标的业务语义、技术规范及技术加工逻辑等；

最后将一致性的信息总结归纳为指标标准，包括业务元数据和技术元数据。

### 3.3 使用技术工具

#### 3.3.1 标准管理及查询管理工具-起源

起源是住宿数据组2017年内部孵化的产品，初衷就是要实现业务标准的统一管理和数据查询的统一管理，后来数据应用平台对起源的完善更加贴合了住宿的场景，因此起源作为标准管理及查询管理工具。

标准管理包含指标及维度的业务信息（包括业务定义、业务名称等）、技术信息（包括英文名称、对应数据表字段、计算公式等），同一个指标的业务及技术信息打通。

查询管理主要有3个模块：数据应用模块、外部应用模块、数据地图模块。其中数据应用模块包括元数据应用和数据应用，分别对应元数据API服务和数据API服务。

数据应用模块主要面向数据产品服务。其中元数据应用提供数据表、指标、维度的业务及技术的信息查询服务，例如模型支持的指标及维度列表、某个场景的指标列表等；数据应用是提供数据表数据查询服务，通过数据API以及指定的查询参数（指标ID、维度、过滤条件等）提供可直接查询数据的SQL。

外部应用模块是后台RD负责，面向各个数据产品，一个外部应用对应一个appkey。每个外部应用内部可以细分子模块，例如某报表平台是一个外部应用，平台内的报表则是子模块。外部应用调用数据应用API需要数据RD将数据应用授权子模块。

数据地图模块支持从数据表、模型、指标、数据应用、外部应用五个维度查找数据的上下游信息，展示数据流概览图。用户可以从单个节点为切入口，全局浏览整个数据服务链路，对数据问题定位及信息查询非常有帮助。

#### 3.3.2 生产管理工具-达芬奇

为系统化的解决生产不一致的问题，住宿数据组在2019年4月正式成立ETL配置化项目（大禹子项目之一），即通过工具化实现数据生产。后与数据开发平台在2020年2月实现了生产管理工具上线，即达芬奇。

达芬奇系统分为三个模块：模型设计、指标定义、智能构建。其中：

- 模型设计从数据模型系统同步数据模型，然后在达芬奇配置维度及模型的关联条件及过滤条件；
- 指标定义是从起源调取指标信息，在达芬奇配置指标在某个模型的计算公式；
- 智能构建是依据已配置的模型及指标，选择最终产出的指标和维度以及数据源模型，智能化的生成ETL，称之为画板。

在达芬奇系统中，数据模型、指标、维度等生产必须的信息即作为数据的生产标准，画板作为数据生产的管理模块。

> 提示：当前处于在住宿业务落地实验阶段，尚未推广使用，感兴趣可以查看参考文档中达芬奇手册。

数据模型可以配置主表、关联表、表级的过滤条件。

维度支持字段级的原子维度、衍生维度配置 以及 复杂的维度枚举值独立配置。原子维度对应模型的单个字段，无需额外的加工。衍生维度是需要1个或多个字段加工的维度。

指标口径采用自定义SQL方式，同时支持单个指标的过滤条件配置。

画板配置包括构建定义、模型结构、参数配置、调度配置。其中构建定义是选择需要生产的指标、维度、数据模型，模型结构是调整目标表的数据结构（例如表名、字段顺序、分区等），参数配置是可自定义资源参数，增量生产的条件（$now.delta）等，调度配置是同ETL平台配置生产计划及依赖配置等。

指标生产一致性实现方式：

每个模型的每个指标只有唯一的计算口径，即所有同一个模型的下游ETL中的指标共用唯一的计算口径，因此ETL生成时可保证指标的生产逻辑是一致的。

当指标的计算口径发生变更时，只有唯一的变更入口，口径变更发布后，达芬奇系统会自动修改下游全部的ETL中计算逻辑，自动完成数据测试并同步上线全部的下游ETL，可保障指标生产逻辑变更后也是一致的。

因此达芬奇可保障同一个模型的生产的指标是一致的。

## 四、总结：指标一致性对业务和人员都有积极影响

通过指标标准化解决了住宿产品指标不一致的问题：

- 业务多指标不一致问题的日常反馈由原先的3次/周减少到<1次/月。
- 2019年及2020年住宿数据服务调研问卷对产品数据不一致问题0反馈（存在手写SQL不一致问题，在其他方案解决）。

指标一致性对业务和人员都有积极的作用。

**业务方面**：

- 制定及管理一致的数据标准，可保证指标语义的一致性，促进形成业务一致的数据视图，促进信息的沟通与共享。
- 通过制定标准的过程及评估已有的标准建设，可发现现有的指标问题，提高数据质量。
- 指标标准可作为新数据产品的依据，便于统筹业务整体的规划设计，也保障新数据产品的指标标准性。

**人员方面**：

- 对业务人员而言，指标标准可提升业务、需求、产品的规范性，保障人员对指标的业务含义认知一致，支撑数据的有效沟通及共享。
- 对技术人员而言，可参考指标标准，提升开发效率，降低出错概率及返工概率，保障开发的规范性，并且提升数据质量。
- 对管理人员而言，指标标准的建设可提供更加可信、更加完整的数据，可以更好的支撑业务决策。

## 五、规划：短期提升系统化，长期智能化

**推动工具功能性及易用性建设，提升系统化程度。**

现在虽然建设了工具，但工具的功能性及易用性依然还有很大的拓展空间，需要结合业务场景持续的优化，不断的抽象工作流程，弱化人工，提高工作效率和准确性，尽可能实现工作链路的系统化闭环。

**指标标准化工作的自动化及智能化服务。**

前面介绍了指标标准化的三个工具系统，但实际工作中涉及的工作远远不止，比如ETL平台、数据质量平台、报表平台等，长期的规划是各个工具间系统化协同，实现标准化工作的自动化，更长期是实现智能化。
