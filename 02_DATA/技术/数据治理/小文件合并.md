# 一、背景与治理目标

## 1.1 小文件问题定义

小文件通常指显著小于 HDFS Block（128M 或 256M）的文件。

常见特征：

- 单文件 < 32M
- 单目录文件数 > 1000
- 动态分区目录呈指数级增长

---

## 1.2 小文件带来的影响

### 对 HDFS

- NameNode 元数据膨胀
- RPC 请求激增
- GC 压力上升

### 对计算引擎

- getSplits 变慢
- Task 数暴涨
- Shuffle 开销增加
- 作业启动时间变长

---

## 1.3 治理目标

| 指标 | 推荐标准 |
|------|----------|
| 单文件大小 | 256M ~ 1G |
| 单目录文件数 | < 2000 |
| 动态分区目录数 | 可控且可预估 |

---

# 二、小文件产生原因分类

| 类型 | 原因 |
|------|------|
| 输入侧 | 上游已产生大量小文件 |
| Shuffle侧 | reducer 过多 |
| 分区设计 | 动态分区过细 |
| 写后阶段 | 单task输出过小 |

---

# 三、治理总体原则

治理顺序：

```
1. 控制 Shuffle
2. 优化分区分布
3. Map端输入合并
4. 写后合并（兜底）
```

原则：

- ✅ 优先减少“生成”
- ❌ 尽量避免“事后合并”

---

# 四、核心治理方案

---

# 4.1 Shuffle阶段控制（最重要）

## ✅ 开启 AQE

```text
# 开启自适应执行能力
spark.sql.adaptive.enabled = true
# 允许 Spark 自动合并 Shuffle 分区
spark.sql.adaptive.coalescePartitions.enabled = true
# Shuffle 后合并分区的目标大小由 `spark.sql.adaptive.shuffle.targetPostShuffleInputSize` 决定，默认 128MB，推荐 256MB 左右。
- 结果：减少 reducer 数 → 减少输出文件数 → 减少小文件
```

---

## ✅ 控制 reducer 目标数据量

```text
`spark.sql.adaptive.shuffle.targetPostShuffleInputSize` 决定，默认 128MB，推荐 256MB 左右。
```

推荐范围：

- 中等规模：256M
- 大规模：512M
- 超大规模：1G

---

## ✅ 控制最大分区数

```text
spark.sql.shuffle.partitions = 300~800
```

不要使用默认 2000。

---

## ✅ 文件数估算公式

```
文件数 ≈ 总数据量 / targetPostShuffleInputSize
```

---

# 4.2 动态分区优化（防止文件爆炸）

动态分区文件数公式：

```
文件数 = 分区组合数 × reducer数
```

---

## ✅ 必加 distribute by

```sql
INSERT OVERWRITE TABLE t PARTITION (dt, region)
SELECT ...
DISTRIBUTE BY dt, region;
```

作用：

- 相同分区数据进入同一个 reducer
- 减少重复文件

---

## ✅ 分区设计原则

- 分区字段 ≤ 2 层
- 不使用高基数字段做分区
- 日分区优于小时分区（除非强需求）

---

# 4.3 Map端输入合并

适用于上游已产生大量小文件。

---

## ✅ 推荐统一设置 256M

```text
spark.hadoopRDD.targetBytesInPartition = 256M
spark.hadoop.mapreduce.input.fileinputformat.split.maxsize = 256M
spark.hadoop.mapreduce.input.fileinputformat.split.minsize = 256M
```

---

## ✅ ORC 特殊设置

```text
spark.hadoop.hive.exec.orc.split.strategy = ETL
```

否则 ORC 可能不会按大小合并。

---

# 4.4 写后自动合并（兜底）

慎用，仅用于无法从源头治理场景。

---

## ✅ 参数设置

```text
> 当某个目录下的平均文件大小 < 20MB 时，Spark 会触发“写后小文件合并”。 写出之后的兜底合并
spark.sql.mergeSmallFileSize = 20M

spark.sql.targetBytesInPartitionWhenMerge = 256M
```

触发条件：

```
平均文件大小 < spark.sql.mergeSmallFileSize
```

---

## ⚠ 成本

- 额外启动一个 job
- IO 增加
- NN RPC 增加
- 作业时间增加

---

# 五、标准生产配置模板

推荐作为默认参数模板。

```text

# Map端输入合并

1 控制 **HadoopRDD 读取阶段 每个 partition 期望的数据大小。
Spark 会根据输入数据总量来决定生成多少个 partition。
- 会生成大约 400 个 partition
100GB / 256MB ≈ 400 个 partition 
它只影响 **读阶段**（Input Stage）， 不直接决定最终写表时的文件数。
spark.hadoopRDD.targetBytesInPartition = 256M

spark.hadoop.mapreduce.input.fileinputformat.split.maxsize = 256M
spark.hadoop.mapreduce.input.fileinputformat.split.minsize = 256M


#shuffle端  开启 AQE 
spark.sql.adaptive.enabled = true
spark.sql.adaptive.coalescePartitions.enabled = true

#控制 **Shuffle 之后，每个 reducer 期望处理的数据量**
- Shuffle Read = 1500GB
- 目标 = 1024MB（1GB）
  1500GB / 1GB ≈ 1500 个 reducer
✅ 它影响 reducer 数  
✅ reducer 数影响最终 task 数  
✅ task 数影响文件数
但：
❌ 它不是合并文件  
❌ 它不会在写完之后再 merge 文件
spark.sql.adaptive.shuffle.targetPostShuffleInputSize = 256M

# 控制并行度
spark.sql.shuffle.partitions = 500


# ORC
spark.hadoop.hive.exec.orc.split.strategy = ETL

# 写后合并兜底
spark.sql.mergeSmallFileSize = 128M
spark.sql.targetBytesInPartitionWhenMerge = 256M
```

---

# 六、排查 Checklist

出现小文件时，按以下顺序排查：

1. ✅ 是否开启 AQE？
2. ✅ shuffle 后 partition 数多少？
3. ✅ 是否存在动态分区爆炸？
4. ✅ 单个 reducer 处理数据量多少？
5. ✅ 是否存在 distribute by？
6. ✅ 输入目录是否大量小文件？
7. ✅ 是否开启写后自动合并？

---

# 七、不同数据规模推荐策略

| 数据量 | 推荐文件大小 | targetPostShuffleInputSize |
|--------|--------------|----------------------------|
| < 100G | 128M | 128M |
| 100G ~ 5T | 256M | 256M |
| > 5T | 512M ~ 1G | 512M |

---

# 八、最佳实践总结

✅ 小文件问题本质是：  
> 并行度与数据规模不匹配。

✅ 核心控制点：  
> reducer 数量。

✅ 动态分区是指数级放大器。

✅ 写后合并只是补救手段。

✅ 治理目标不是“文件越少越好”，而是：
> 单文件大小合理 + 并行度合理 + NN 压力可控。
