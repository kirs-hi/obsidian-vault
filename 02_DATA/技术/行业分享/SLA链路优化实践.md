## 一、背景及问题

### 1.1 业务背景

某生鲜即时零售业务，定位为社区居民的"30分钟快送超市"，采用"手机App+线下服务站"的模式，通过销量预测、订货、采购、质检、仓储、运输、拣货、履约等核心业务环节为社区居民提供最快30分钟送达的即时零售服务。业务复杂度较高。目前已在北京、上海、深圳、广州、武汉、苏州等城市上线，服务范围为站点附近2-3公里，基本覆盖各城区核心区域。

**业务特点：** 2021年开始业务进入快速发展期，2年内月均订单量翻了近4倍。同时，生鲜即时零售的业务特点是链路长、损耗大、利润低，生鲜自营赛道竞争的核心是精细化运营能力。所以，在业务快速发展且对精细化运营能力要求较高的大背景下，业务对数据时效性的要求越来越高。

**成本考量：** 成本治理的大背景下，预算及单均大数据成本需要严格把控，成本治理的核心是资源的高效利用，SLA治理是资源高效利用的主要手段之一。

**业务痛点：** 业务X3节点每天早上晨会对数据有强诉求，而数据组的大部分SLA实际完成时间在8点后，实际就绪时间逼近X3晨会8点半的用数基线要求，链路任务执行时长或集群稳定性稍有波动，都会大面积延时，影响业务用数诉求。

### 1.2 问题

**用户视角：**

- **SLA应用上线前：** 开发初期，客户从实际业务场景出发给出初步SLA，但在开发中后期，任务的实际就绪时间往往和客户期望的有较大落差。由于任务上游链路长且复杂，RD只能优化任务本身，无法从链路层面做深度优化，收效甚微。在这种情况下，客户往往会做出妥协，重新根据任务的实际执行情况制定SLA就绪时间。
- **SLA应用上线后：** 数据产品使用方频繁收到SLA延时通告，影响早上用数诉求。

**研发视角：**

夜间生产集群0-6点多时段资源空闲，但是SLA却在持续延时，资源利用率明显不足，大任务优化后收效甚微。

## 三、治理方案

SLA上游任务错综复杂，要想准确高效地实现常态化治理，实现预期目标，需要做到以下几点：

- **看得见问题：** 建立完善的观测指标体系，让大家清楚问题的所在。
- **理得清链路：** SLA卡点链路自动化识别，让大家明确问题出在哪里，优化哪些链路ROI最高。
- **改得准任务：** 实现卡点任务自动化归因。SLA卡点链路的优化最终还是会落在任务上，需要找出任务的问题在哪，如何优化，并实现策略线上化。
- **保得好成果：** 建立常态化治理机制，确保措施长期执行，促进问题根治。

### 3.2 数仓链路优化策略

**局部视角：** 业务SLA应用上游计算层级最多可达26层，涉及任务上百个，呈网状，并不是优化任何一条链路、任何一个任务对SLA就绪都是有效的，所以在单个SLA的优化上，我们要准确地提取到卡点链路，确保做功任务的有效性。

**全局视角：** 在全局视角，哪些任务是大量SLA应用共同的卡点任务，这些任务需要具备识别能力，确保ROI最高，能快速拿收益。

#### 3.2.1 卡点链路的提取

**卡点链路提取逻辑：** SLA末端任务到数据采集层迭代寻找同层级内最晚执行完成的任务，这些任务的集合组成的链路就是卡点链路。通过准确找出卡点链路，我们能够确定哪些任务优化后，应用产出肯定会提前，哪些任务优化后对SLA应用的执行就绪是无效的。这能极大地促进SLA应用的优化和改进。

如上图所示，"SLA应用A"配置了三个任务，其中任务C最晚完成。因此，任务C是影响该SLA应用就绪的卡点任务。任务C的直接上游有两个任务，其中"task17"是"任务C"的卡点任务。同理，链路 `任务C -> task17 -> task18 -> task20` 是该SLA应用的卡点链路。通过优化该链路上的任何一个任务，都可以有效地优化"SLA应用A"的就绪时间，反之，通过优化卡点链路外的任何任务，都是无效的。

![[attachments/img_4.png]]
*卡点链路提取示意图*

**大节点任务识别：** 如果某个任务出现在多个SLA应用的卡点链路上，则该任务就是大节点任务。如上图所示，"task18"是"SLA应用A"与"SLA应用B"的共同卡点任务。通过优化该任务的执行时间，可以提前全部SLA应用的就绪时间。通过识别并优化少量大节点任务，实现SLA应用完成时间的大面积提前，具有非常高的ROI。

![[attachments/img_5.png]]
*部分大节点任务影响40+SLA应用*

**总结：** 通过以上方式，我们可以提取到每个SLA应用的卡点链路和全局的大节点任务。如上图，最大的大节点任务直接影响62个SLA应用的就绪，所以我们的做功重点围绕大节点任务优化展开。大节点任务按数仓的生产流程分三类（不同类型优化策略不同），即数据采集类任务、数仓计算类任务、数据下发类任务。

#### 3.2.2 卡点链路的拆解

卡点链路提取以后，需要把卡点链路执行总时长通过数仓生产流程切割分段，直观分析卡点原因：

**链路执行总时长 = 数据采集时长 + 数仓计算时长 + 数据下发时长 + 调度延时时长（上游完成下游未调度的时间段）**

- **数据采集时长：** 如占链路总时长比例较大，则存在新任务数据采集调度配置不合理或其余BU上游就绪时间过晚的问题。
- **数仓计算时长：** 如占链路总时长比例较大，则可能是数仓生产链路过长或任务本身存在问题。
- **数据下发时长：** 如占链路总时长比例较大，则可能是Doris性能出现瓶颈或数据下发任务存在问题。
- **调度延时时长：** 如占链路总时长比例较大，则可能存在资源瓶颈或彻底没有可用资源的情况，需要适度延后B级任务的调度时间。

![[attachments/img_6.png]]
*链路拆解举例——经营大盘*

**总结：** 通过工具化，能快速完成卡点链路提取、大节点任务识别、卡点链路拆解，问题定位时长大幅缩减且定位准确性100%；但是找到卡点任务后如何优化，又成了比较棘手的问题，下文主要围绕任务级别如何归因、如何优化展开。

### 3.3 卡点任务归因

找到卡点任务后，需要通过元数据进一步识别卡点任务存在的问题，并通过规则路由自动化地识别卡点任务存在的问题。

一个优质的SLA链路具备"跑的早"、"跑的快"、"链路短"、"不阻塞"等特点，针对以上四点，从数据生产和资源两个角度进行原因分析：

| 分类 | 常见问题名称 | 说明 |
|------|-------------|------|
| 生产视角 | 调度就绪时间不合理 | 部分任务调度配置时间大于0点，导致下游链路执行开始时间整体后延 |
| 生产视角 | O层表类型不合理 | 数据采集全量表与快照表有30-40分钟延时，增量表可大幅缩短采集时长，使O层就绪时间提前 |
| 生产视角 | 高耗时任务 | 常见大任务，可通过参数优化、数据裁剪等方式进行优化 |
| 生产视角 | 大文件任务 | 存在部分任务过度合并小文件，文件切割参数未配置，拉取数据时单个task处理数据量过大，不能充分发挥分布式性能 |
| 生产视角 | 小文件任务 | 当划分分片（getSplits）的时候，都要从NN获取文件信息。这个过程的耗时与文件数成正比，同时受NN压力的影响。在NN压力大、上游小文件多的情况下，下游的getSplits操作就会比较慢 |
| 生产视角 | 维表瓶颈任务 | 由于维表具有公用性强、下游多等特点，可能会存在一些产出较晚的维表，从而影响到大批下游任务 |
| 生产视角 | 数据倾斜任务 | 由于存在热点数据、上游文件大小不均或空值等问题，可能会导致任务执行缓慢，从而直接影响到SLA的就绪状态 |
| 生产视角 | 文本存储任务 | 文本存储会导致IO开销增大，从而浪费集群资源，同时也会增加执行时间 |
| 生产视角 | Doris load选型不合理 | Sparkload、Streamload选型不合理，会导致导入时间延长，如冗余模型使用Sparkload |
| 生产视角 | Doris全bitmap字段较多 | 大量字段用bitmap去重字段，构建字典会产生大量性能消耗，影响导入效率 |
| 生产视角 | 全量下发任务 | 部分Hive2Doris全量导入，每次导入历史全量数据势必会影响导入效率 |
| 资源视角 | 调度延时任务 | 上游任务已经就绪，任务迟迟未启动 |
| 资源视角 | 弹性队列配置不合理 | 夜间是否存在生产队列资源不够、其他队列资源过剩的情况 |

![[attachments/img_7.png]]
*卡点任务问题归因线上化示意*

**总结：** 通过对存量卡点任务专项分析，将卡点任务常见问题实现自动化识别并给出优化建议，实现常见问题任务优化由高线行为向基线行为转变。通过对10个左右的大节点任务优化，夜间资源利用率提高15%左右，SLA达标率提高6pp。

### 3.4 长期治理策略落地

长期治理体系落地，需要建立以下机制：

- **事前：** 建立风险感知机制，对延时和风险应用进行提前介入。
- **事中：** 追踪治理进度，建立晾晒机制。
- **事后：** 明确问题的影响范围和影响程度，确定优化的优先级。

#### 3.4.1 风险感知机制

随着业务发展，任务增多，靠人工去发现可能存在延时风险的应用显然不可靠，同时目前SLA预警具有滞后性，所以需要建立风险应用感知机制。

建立完善的晾晒机制，并通过人员级管理推动治理进度，有助于确保治理成果落地。

#### 3.4.3 影响评估机制

当SLA延时发生时，仅凭SLA延时时长并不能直接评估对用户造成的影响面和影响程度。由于每个产品在SLA延时时间段访问的用户规模不同，而且用户层级也有所不同（用户层级越高，对决策影响程度越大），因此需要建立影响评估机制来评估影响的权重。

具体的影响权重评估核算机制如下：

$$
AppInfStage = \sum_{i=0}^{n} (PV_0 \cdot k_{grade_0} + PV_1 \cdot k_{grade_1} + \ldots + PV_n \cdot k_{grade_n}) \cdot delayhour
$$

- `delayhour`：应用延时小时数（应用实际产出时间 - 应用承诺时间）
- `PV_i`：用户在上周延时时段使用该应用的PV值
- `k_grade_i`：用户层级系数（BU负责人: 1000；X3节点: 100；X2节点: 10；其他: 1）

![[attachments/img_10.png]]
*影响评估机制示意*

**总结：** 通过事前风险感知机制、事中优化进度晾晒机制、事后影响评估机制的建立，实现SLA治理的常态化，流程上实现闭环。

## 四、项目成果

**服务视角：**
- SLA承诺时间7点前 >= 90%
- SLA平均完成时间提前1小时

**资源视角：**
- 0-6点ETL队列资源使用率 >= 95%
- Hadoop计算资源节约1200 CU·天（约占当时配额的10%）

**研发视角：**
- **问题定位效率：** SLA定位问题需要数小时到半天，现在可实现分钟级定位
- **SLA优化范式：** 之前SLA链路是单点优化，优化价值小，很快就遇到优化瓶颈。而新的优化范式是结构性优化，站在全局视角，对数据生产链路结构进行优化
- **从事后到事前：** 之前SLA只有频繁出现延时，才会启动优化，短期见效，频繁复发；新的治理策略是风险识别，提前预防

## 个人笔记

> 这篇文章的核心方法论：**卡点链路提取 + 大节点任务识别 + 自动化归因 + 长期治理机制**，是一套完整的 SLA 治理闭环。
>
> 对数仓开发的启示：
> 1. SLA 优化不能只看单个任务，要从链路全局视角找"大节点任务"，ROI 最高
> 2. 资源利用率低 + SLA 延时并存，说明问题不在资源量，而在调度策略
> 3. 治理要形成机制（事前感知 + 事中晾晒 + 事后评估），否则容易反弹
