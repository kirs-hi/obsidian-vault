## 一、适用范围

本规范适用于基础研发平台/数据平台部下所有团队。

## 二、概念解释

通过明确建模涉及的概念，理解基础含义，保障各方的基础认知一致，对相关概念定义如下：


| 名词 | 含义 | 举例 |
| --- | --- | --- |
| **数据域** | 将业务过程或者维度进行抽象的集合，是站在业务视角对业务流程的直接抽象和刻画，对应到明细数据层。 | 交易数据域、营销数据域 |
| **分析主题** | 独立的分析集合，站在分析视角对业务流程的再组合及刻画，对应到汇总数据层及应用数据层。 | 用户分析、财务分析、物流分析 |
| **业务过程** | 业务过程指企业的业务活动事件，业务过程是一个不可拆分的行为事件，通俗地讲，业务过程就是企业活动中的事件。 | 下单、支付、退款 |
| **业务活动** | 业务过程中的行为事件，有两个或多个对象参与。 | 下单业务过程中的下单事件 |
| **业务对象** | 参与活动的主体或者具体事务的抽象，业务对象一般可以生成维度表。 | 用户、商家、商品 |
| **实体** | 实体是指现实业务中可独立存在且具有唯一标识的对象或概念，是数据建模的核心单元，实体分为业务对象和业务活动。 | - |
| **维度** | 维度是度量的环境，用来反映业务的一类属性，这类属性的集合构成一个维度，也可以称为实体对象。 | 商家维度、商品维度 |
| **维度属性** | 维度属性隶属于一个维度，是维度表中用于描述维度特征的列。 | 商家维度中的商家名称、商家类目 |
| **指标** | 指标分为原子指标、复合指标和计算指标。原子指标和度量含义相同，基于某一业务事件行为下的度量，是业务定义中不可再拆分的指标，具有明确业务含义的名词。 | 支付金额 |

## 四、分层规范

### 4.1 操作数据层


|  |  |
| --- | --- |
| 层级定义 | **操作数据层（Operational Data Store）**</br>ODS层是数据仓库的最底层，用于存储最近一段时间内的详细业务数据。这些数据通常是从源系统（如业务系统、日志等）直接抽取过来的，保留了业务系统的原始细节。 |
| 建模方式 | </br>1. 分类：业务系统数据和流量数据，业务系统数据binlog采集，流量数据通过日志系统采集；</br>2. 同步：结构化数据增量或全量同步到数据仓库；</br>3. 结构化：非结构化（日志）结构化处理并存储；</br>4. 累积历史：根据数据业务需求及稽核和审计要求保存历史数据。</br></br> |
| 上游依赖 | 业务系统、日志系统 |
| 指标关系 | 不直接产出指标 |
| 开放情况 | BI场景下不开放查询 |

### 4.2 明细数据层


|  |  |  |  |
| --- | --- | --- | --- |
| 层级定义 |  | **明细数据层（Data Warehouse Detail）**</br>DWD层是对ODS层数据的进一步清洗、转换和整合，形成统一的明细数据层。DWD层的数据通常按照业务过程进行组织，保留了业务细节，但比ODS层更加规范化和标准化。 |  |
| 特点 |  | </br>1. 明细数据，按照业务数据域分类，以业务为驱动设计表结构和表间关系；</br>2. 数据集成，整体上基于维度建模方法进行设计，并在语义层达到统一和标准；</br>3. 历史可追溯，包括生命周期中的所有变化状态。</br></br> |  |
| 分类 |  | 公共数据 | 业务数据 |
| 职责 |  | 面向公共业务系统完成统一的数据接入、清洗和分发，屏蔽底层公共业务系统的变化。 | 面向各个业务建设统一的明细数据表，可从公共数据中抽取业务数据，或直接从业务特有业务系统中接入。 |
| 建模方式 | **表类型** | 事务事实表、周期快照事实表、累积快照事实表。</br>其中事务事实表又分为单事务事实表（针对每个业务过程设计一个事实表）和多事务事实表（用一个事实表包含不同的业务过程）。 |  |
| 建模方式 | **数据范围** | </br>1. 面向公共数据域，如商家、商品、营销、交易等，这些数据域通常包括多个业务的数据；</br>2. 在行上通常是全量的，如公共订单模型需要包含全部业务的订单；</br>3. 字段需要包含公共字段，业务特有字段可通过扩展字段存储；</br>4. 单表内不跨数据域，可以适当建设融合多业务过程的多事务事实表。</br></br> | </br>1. 面向各个业务线的数据域，如某医疗业务、某到家业务、某即时零售业务交易等，通常只包括单业务或少量业务的数据；</br>2. 在行上仅包括某一业务，如某即时零售业务交易数据域只包括某即时零售业务业务的订单；</br>3. 字段通常更丰富，可包括业务特有分析字段；</br>4. 单表内不跨数据域，可以适当建设融合多业务过程的多事务事实表。</br></br> |
| 建模方式 | **数据粒度** | 选择最细级别的原子粒度。 |  |
| 建模方式 | **维度** | 事实表需适量冗余维度，如较稳定的维度city_id、city_name等。 |  |
| 建模方式 | **其他约束** | </br>1. 出口表需要对关键实体进行统一清洗和映射，如用户、商家、商品、订单、城市等；</br>2. 出口表需要增加业务线标识字段biz_code、bu_code、bg_code。</br></br> | - |
| 上游依赖 |  | 只依赖操作数据层（ODS）、明细数据层（DWD）、维度层（DIM） |  |
| 指标关系 |  | 产出原子指标，如订单量，并且口径唯一；因为数据量较大，流量类原子指标可以不从这层产出，各业务可根据数据量灵活判断。</br>口径由属性（字段） + 公式组成，必须从唯一的一张表的属性生成。</br>举例：原子指标实付交易额（GTV），只能绑定在唯一的表dwd_ord_xxx上，使用sum(total)计算得到。 |  |
| 开放情况 |  | 对外开放查询，对接指标管理平台数据集/全量资产池 |  |

### 4.3 汇总数据层


|  |  |
| --- | --- |
| 层级定义 | **汇总数据层（Data Warehouse Summary）**</br>DWS层是对DWD层数据的进一步加工和汇总，形成公共粒度的汇总指标事实表。该层次主要功能是加工多维度冗余的宽表（解决复杂的查询）、多角度分析的汇总表。 |
| 特点 | </br>1. 数据模型相对稳定，在明确的分析主题下建设，有复合指标和计算指标；</br>2. 产出的模型通常是跨多个数据域的宽表模型，对数据分析更友好、更通用；</br>3. 多角度汇总，从多个角度分析的汇总模型。</br></br> |
| 建模方式 | </br>1. 在分析主题下，明确汇总维度进行聚合；</br>2. 可按照相同粒度将多事实表进行关联或冗余维度形成宽表；</br>3. 不要在同一个表中存储不同层次的数据，比如不能有些行存放按天汇总、有些行存放按月汇总。</br></br> |
| 上游依赖 | 不可依赖操作数据层（ODS）、数据应用层（ADS） |
| 指标关系 | 产出原子指标、复合指标和计算指标，其中复合指标和计算指标可以使用指标管理平台配置生成，也可以直接在表中计算出结果并使用单独字段存储。需要保障指标一致性，对相同的指标必须提供与查询DWD数据一致的查询结果。 |
| 开放情况 | 对外开放查询，对接指标管理平台数据集/全量资产池 |

### 4.4 应用数据层


|  |  |  |
| --- | --- | --- |
| 层级定义 | **应用数据层（Application Data Store）**</br>ADS层是数据仓库的最顶层，按照项目和应用组织数据，规划多样化、个性化的复合指标体系、分析模型和数据应用。该层级的主要功能是提供差异化的数据服务、满足业务方的需求，支持快速查询和分析。 |  |
| 特点 | </br>1. 数据模型不稳定，随着分析算法和应用的变更随时变化或下线；</br>2. 数据高度汇总，可做交叉分析、钻取、切片、切块、旋转等多维分析操作；</br>3. 更高级的数据分析或挖掘应用，衍生出信息类、知识类数据；</br>4. ADS层的数据粒度最粗，通常是按具体业务需求进行高度聚合后的结果，直接服务于特定的应用场景。</br></br> |  |
| 建模方式 | 满足性能和客户化展示需求，非必要不建设，以下几种场景可以建设数据应用层：</br>1. 满足应用的个性化分析需求，如行列转换、特殊排序要求、特殊授权范围、强业务规则和过滤条件等；</br>2. 满足查询性能优化，如建设cube模型等，物化工具生产的过程表可以放到这一层；</br>3. 满足各类标签的建设，如用户、商家标签等；</br>4. 直接对接业务系统的数据表，通常包括业务系统定制的数据口径。</br></br> |  |
| 上游依赖 | 不可依赖操作数据层（ODS） |  |
| 指标关系 | 产出原子指标、复合指标和计算指标，其中复合指标和计算指标可以使用配置生成，也可以直接在表中计算出结果。需要保障指标一致性，对相同的指标必须提供与查询DWD/DWS数据一致的查询结果。 |  |
| 开放情况 | BI场景下非必要不开放查询，可以对接应用系统和指标管理平台数据集 |  |

### 4.5 维度层


|  |  |
| --- | --- |
| 层级定义 | **维度层（Dimension）**</br>主要负责存储关于业务实体的详细描述信息，为事实数据提供上下文，帮助解释和分析事实数据。 |
| 建模方式 | </br>1. 从业务源系统抽取转化的维表，或根据业务分析需求构建的维表；</br>2. 通常使用快照维表来记录每天的全量快照数据；</br>3. 必须有主键，通常为业务实体标识。</br></br> |
| 上游依赖 | 只依赖操作数据层（ODS）、明细数据层（DWD）、维度层（DIM） |
| 指标关系 | 不产出指标 |
| 开放情况 | 对外开放查询，对接指标管理平台数据集/全量资产池 |

## 六、表命名规范


通用规范：

（1）表命名使用英文小写字母，单词之间用下划线分开，总长度不能超过40个字符（除项目组名）；

（2）上传表统一添加后缀_upload；

（3）视图统一添加后缀_view；

（4）表中文名建议从英文名直译过来，并说明刷新周期和存储策略。
