savepoint 和 checkpoint 都是 Apache [[Flink SQL 完整语法教程|Flink]] 中用于容错和状态恢复的机制，但它们在使用场景、触发方式和功能上有一些关键区别。

**1. Checkpoint（检查点）**

*   **是什么**: Checkpoint 是 Flink 作业**自动**、**周期性**创建的快照。它记录了作业在某个特定时刻的完整状态（包括算子状态、键控状态、状态后端数据等）和消费的数据流的偏移量（offset）。
*   **目的**: 主要用于**容错和故障恢复**。当作业因为节点故障、程序错误等原因崩溃时，Flink 可以从最近一个成功的 checkpoint 自动恢复，将作业状态和消费位置回滚到那个时刻，然后继续处理数据。
*   **触发方式**: 由 Flink 框架**自动触发**，基于用户配置的间隔时间（如每30秒一次）。
*   **存储**: 通常存储在分布式文件系统（如 HDFS、S3）或 JobManager 的内存中（不推荐生产使用）。
*   **特点**:
    *   **自动管理**: Flink 负责创建、维护和清理（基于保留策略）。
    *   **轻量级**: 设计目标是快速、高效，尽量减少对作业性能的影响。
    *   **内部使用**: 主要用于 Flink 自身的故障恢复，用户一般不直接操作。

**2. Savepoint（保存点）**

*   **是什么**: Savepoint 是 Flink 作业在**特定时刻**的**手动**或**程序化**创建的状态快照。它本质上也是一个 checkpoint，但创建方式和用途不同。
*   **目的**: 主要用于**有计划的操作**，比如：
    *   **升级 Flink 版本**
    *   **修改作业代码**（如修复 bug、添加新功能）
    *   **调整作业的并行度**（增加或减少 TaskManager）
    *   **迁移作业到不同的集群**
    *   **暂停作业并稍后恢复**
*   **触发方式**: 由用户**手动触发**（通过命令行工具 `flink savepoint <jobID>` 或 REST API）或**程序化触发**。
*   **存储**: 存储在用户指定的路径（通常是分布式文件系统）。
*   **特点**:
    *   **手动触发**: 用户主动控制创建时机。
    *   **持久化**: 通常存储在可靠的外部存储中，以便长期保留。
    *   **可移植性**: 可以在不同的 Flink 集群、不同的作业版本之间使用。
    *   **用于变更**: 是进行有状态作业变更（如升级、扩容）的**唯一安全方式**。

**关键区别总结:**

| 特性 | Checkpoint | Savepoint |
| :--- | :--- | :--- |
| **触发方式** | 自动、周期性 | 手动或程序化 |
| **主要目的** | 容错、故障恢复 | 有计划的操作（升级、修改、迁移等） |
| **管理方式** | Flink 自动管理 | 用户手动管理 |
| **存储位置** | 通常自动管理（如 HDFS） | 用户指定路径 |
| **使用场景** | 故障恢复 | 作业变更、暂停/恢复 |
| **可移植性** | 通常限于同一作业 | 可在不同集群/版本间使用 |
| **创建开销** | 轻量级，追求性能 | 相对较重，但更可靠 |

**类比理解:**

*   **Checkpoint** 就像你的手机自动每5分钟自动保存一次游戏进度。如果手机突然没电了，你重启后可以从最近的自动存档点继续玩。
*   **Savepoint** 就像你在打 Boss 前，手动按下一个存档按钮。这样，即使你打 Boss 失败了，或者你想尝试不同的打法，你都可以明确地回到这个手动存档点重新开始。

**简单来说:**  
*   **Checkpoint 是“自动存档”**，用于意外崩溃后的恢复。  
*   **Savepoint 是“手动存档”**，用于有计划地暂停、修改或迁移你的作业。  

为了进行任何有状态作业的变更（比如升级 Flink 版本、修改代码逻辑、调整并行度），**必须使用 Savepoint**。仅仅依赖 Checkpoint 进行恢复，在作业变更后可能会导致状态不兼容或数据丢失。
