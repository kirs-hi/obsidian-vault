## 一、背景与目标

### 1.1 业务痛点

在企业采购场景中，供应商围串标、关联交易、虚假投标等风险行为严重损害采购公平性与企业利益。现有基于关系型数据库的风险检测系统面临以下瓶颈：

传统方案依赖规则表与多表 JOIN 查询，当供应商通过"人→公司→银行账号→IP"等多跳链路进行隐蔽关联时，SQL 查询复杂度呈指数级增长，且难以发现超过 2 度的隐性关联。当前系统虽已实现了投标 IP 相同、基本信息雷同、关联关系等风险大类的检测，但在深层关联挖掘、实时风险感知、围串标团伙识别等方面存在明显不足。

### 1.2 核心目标

本方案旨在引入图数据库技术，构建供应商风险知识图谱，实现三个层次的能力升级：第一层是将现有的规则型风险检测（投标 IP 相同、银行账号相同、联系人相同、注册地址雷同等）从多表 JOIN 改造为图查询，提升查询效率 10 倍以上；第二层是基于图拓扑结构发现隐性关联网络，识别通过中间人、壳公司、共用账户等手段隐藏的围串标团伙；第三层是引入图神经网络（GNN），实现基于图结构的智能风险评分，从"规则驱动"演进到"数据驱动"。

## 二、业务场景分析

### 2.1 现有风险检测体系（基于供应商风险行为 PRD）

当前系统的风险检测分为两大类：**关联关系风险**和**疑似围串标风险**。

#### 2.1.1 关联关系风险（客商接口数据）

关联关系风险通过随单实时调用客商接口扫描识别，根据关系类型划分为三类：

**股权型关系**：投资关系，通过股权链路识别实际控制人

**契约型关系**：分支机构关系

**其他型关系**：包括法定代表人、任职、股权质押、动产抵押、土地抵押、股权出质、竞争合作（供应商+客户+对外担保+招投标+同商标）、法律诉讼（开庭公告+裁判文书+失信人+司法拍卖+法院公告）

#### 2.1.2 疑似围串标风险（多维数据比对）

疑似围串标风险通过库内数据+外部数据+随单数据综合判定：

| 风险类型 | 判定依据 | 数据来源 |
|---------|---------|---------|
| 投标 IP 相同 | 同一项目多家供应商报价 IP 一致 | 采购项目随单数据 |
| 银行账号相同 | 供应商收款账户一致 | 供应商主数据-实名认证记录 |
| 注册 IP 相同 | 供应商注册阶段 IP 一致 | 供应商主数据-实名认证记录 |
| 联系人信息相同 | 联系人姓名/手机号/邮箱一致 | 供应商主数据-实名认证记录 |
| 详细地址雷同 | 供应商办公地址高度相似 | 供应商主数据-实名认证记录 |
| 工商注册电话/地址相同 | 企业联系电话、通讯地址一致 | 客商接口工商快照 |
| 投标文件雷同 | 投标文件编辑人、内容相似 | 采购项目投标文件 |
| 投标保证金账户雷同 | 保证金缴纳账户一致 | 采购项目保证金记录 |
| 历史疑似围串标记录 | 存在历史违规记录 | 供应商风险信息库 |

#### 2.1.3 风险记录唯一性规则

风险记录的唯一性由以下字段组合确定，确保同一风险不会被重复记录：

`供应商 + 风险大类 + 风险类型 + 风险/违规性质 + 关系类别 + 违规详情 + 单据号`

### 2.2 数据来源现状

目前系统的数据主要来自以下渠道：客商接口（已替代原金镜接口，提供基础的供应商风险数据）、工商快照（企业注册信息、股东结构、主要人员）、企业联系方式（电话、邮箱、地址等联系信息），以及主要人员信息（法人、董监高、实际控制人）。这些数据源天然具备图结构特征——实体（供应商、人员、账户）之间存在丰富的关联关系，非常适合用图数据库来建模和查询。

### 2.3 图数据库可解决的新场景

在现有检测能力之上，图数据库可以进一步支撑以下场景：

**多跳隐性关联发现**：供应商 A 的法人张三，同时是供应商 B 的监事，而供应商 B 的实控人李四又是供应商 C 的股东——这种跨 3 跳甚至更多跳的关联在关系型数据库中几乎无法高效查询，但在图数据库中只需一条 Cypher/nGQL 语句即可遍历。

**围串标团伙识别**：通过社区发现算法（如 Louvain）自动将密切关联的供应商聚类成"团伙"，结合投标行为数据（同一项目、相近报价、相同 IP）判定围串标概率。

**资金流向追踪**：从某个可疑银行账号出发，沿转账路径追溯资金流向，发现"投标保证金→中间账户→实控人个人账户"的回流模式。

**风险传播评估**：当某供应商被标记为高风险时，通过图遍历评估其关联网络中其他供应商的受影响程度，实现"一点触发、全网预警"。

## 三、行业案例研究

### 3.1 震坤行 × NebulaGraph：工业品供应链图谱

震坤行是国内领先的工业品 MRO（维护、维修、运营）供应链平台，其供应链管理面临 SKU 数量庞大（百万级）、供应商层级深、物料关系复杂等挑战。震坤行采用 NebulaGraph 构建供应链知识图谱，将供应商、品牌、品类、物料、仓库等实体建模为图中的节点，以供货关系、品牌授权、品类归属、仓储配送等作为边。核心应用包括：供应商寻源推荐（根据物料需求自动匹配最优供应商路径）、供应链风险传导分析（某供应商断供时快速评估影响范围并推荐替代方案）、以及物料关联分析（发现可替代物料降低采购成本）。实施效果方面，供应商寻源效率提升 60%，供应链风险评估从天级缩短到分钟级。

### 3.2 携程金融：反欺诈关联网络

携程在金融风控场景中使用图数据库构建用户-设备-账户关联图谱，核心目标是识别"团伙欺诈"。图谱中的节点包括用户、手机号、设备指纹、银行卡、IP 地址、Wi-Fi SSID 等，边表示"使用""绑定""登录"等行为关系。通过社区发现算法识别出紧密关联的用户群体，再结合行为特征（短时间内大量申请、相似填写模式）判定为欺诈团伙。该方案将欺诈识别率从 70% 提升到 92%，误报率下降 40%。携程案例与供应商围串标检测高度相似——都是通过共享属性（设备/IP/银行账号）发现隐性团伙关联。

### 3.3 中国移动：电信反诈知识图谱

中国移动利用图数据库构建电信诈骗知识图谱，将电话号码、IMEI、基站、银行账号、社交账号等建模为节点，以通话、转账、社交联系等建模为边。关键技术包括：异常子图检测（发现呈星型拓扑的诈骗呼叫网络）、时序图分析（追踪诈骗电话的时间序列模式）、以及基于 GNN 的风险传播模型。该案例验证了图数据库在大规模关系网络中进行异常模式检测的有效性。

### 3.4 围串标检测专利方案

相关专利公开了一种基于知识图谱的围串标检测方法：首先从招投标数据中提取实体（投标人、标段、项目）和关系（投标、中标、联合体），构建招投标知识图谱；然后定义围串标模式模板（如"环形投标""轮流中标""陪标模式"），通过子图匹配算法在图谱中检索符合模式的子图；最后结合报价分析（价格梯度异常、报价区间聚集）综合判定围串标概率。该方法将围串标检测准确率提升至 85% 以上，且可解释性强——每个判定结果都可追溯到具体的图模式和数据证据。

### 3.5 GNN + 知识图谱：学术前沿

学术界在图神经网络（GNN）与知识图谱结合的风控方向有大量研究。GraphSAGE 通过采样和聚合邻居节点特征进行归纳学习，适合供应商图谱中新节点（新注册供应商）的风险评估。GAT（Graph Attention Network）引入注意力机制动态分配邻居权重，能够区分不同类型关联关系（如"共用银行账号"比"同一城市"的风险信号更强）的重要性。异构图神经网络（如 HAN）专门处理包含多种节点和边类型的图谱，天然适配供应商知识图谱的异构特性。

## 四、数据建模：供应商风险知识图谱

### 4.1 节点（Vertex）设计

基于现有业务数据和风险检测需求，设计以下核心节点类型：

**Supplier（供应商）**：属性包括 supplier_id（供应商 ID）、supplier_name（供应商名称）、unified_social_credit_code（统一社会信用代码）、register_date（注册日期）、register_capital（注册资本）、business_status（经营状态）、risk_level（当前风险等级）、register_address（注册地址）、business_scope（经营范围）。

**Person（自然人）**：属性包括 person_id、person_name、id_card_hash（身份证哈希，脱敏处理）、phone_hash（电话哈希）、is_legal_representative（是否法人）、is_actual_controller（是否实控人）。

**BankAccount（银行账户）**：属性包括 account_id、bank_name（开户行）、account_type（账户类型，区分对公/对私/保证金账户）。

**IPAddress（IP 地址）**：属性包括 ip_address、ip_location（IP 归属地）、ip_type（区分注册 IP / 投标 IP / 登录 IP）。

**BidProject（招标项目）**：属性包括 project_id、project_name、bid_open_date（开标日期）、budget_amount（预算金额）、category（采购品类）。

**BidDocument（投标文件）**：属性包括 doc_id、doc_hash（文件指纹，用于相似度比对）、submit_time（提交时间）、bid_amount（投标报价）。

**Address（地址）**：属性包括 address_id、full_address（完整地址）、normalized_address（标准化地址）、geo_location（经纬度坐标）。

**ContactInfo（联系方式）**：属性包括 contact_id、phone_number_hash、email_hash、contact_person_name。

### 4.2 边（Edge）设计

边的设计直接映射现有风险检测维度：

| 边类型 | 起点 → 终点 | 核心属性 | 业务含义 |
|---|---|---|---|
| `IS_LEGAL_REP` | Person → Supplier | start_date, end_date | 法人代表关系 |
| `IS_SHAREHOLDER` | Person → Supplier | share_ratio, invest_amount | 股东关系 |
| `IS_DIRECTOR` | Person → Supplier | position（董事/监事/高管） | 董监高关系 |
| `IS_ACTUAL_CTRL` | Person → Supplier | ctrl_path（控制链路描述） | 实际控制关系 |
| `OWNS_ACCOUNT` | Supplier → BankAccount | purpose（收款/保证金/一般） | 开户关系 |
| `PERSON_ACCOUNT` | Person → BankAccount | relationship | 个人账户关系 |
| `USED_IP` | Supplier → IPAddress | action（注册/投标/登录）, timestamp | IP 使用记录 |
| `SUBMITTED_BID` | Supplier → BidProject | bid_amount, bid_time, result | 投标行为 |
| `BID_DOC_OF` | BidDocument → BidProject | — | 投标文件归属 |
| `SUBMITTED_DOC` | Supplier → BidDocument | — | 文件提交关系 |
| `LOCATED_AT` | Supplier → Address | address_type（注册/办公） | 地址关系 |
| `HAS_CONTACT` | Supplier → ContactInfo | — | 联系方式关系 |
| `TRANSFER_TO` | BankAccount → BankAccount | amount, timestamp, purpose | 资金流向 |

### 4.3 图 Schema 设计

#### 4.3.1 NebulaGraph nGQL Schema

```ngql
-- 创建图空间
CREATE SPACE IF NOT EXISTS supplier_risk (
  partition_num = 100,
  replica_factor = 3,
  vid_type = FIXED_STRING(64)

USE supplier_risk;

-- 节点定义
CREATE TAG IF NOT EXISTS supplier (
  supplier_name string,
  unified_social_credit_code string,
  register_date date,
  register_capital double,
  business_status string,
  risk_level int DEFAULT 0
);

CREATE TAG IF NOT EXISTS person (
  person_name string,
  id_card_hash string,
  phone_hash string
);

CREATE TAG IF NOT EXISTS bank_account (
  bank_name string,
  account_type string
);

CREATE TAG IF NOT EXISTS ip_address (
  ip string,
  ip_location string,
  ip_type string
);

CREATE TAG IF NOT EXISTS bid_project (
  project_name string,
  bid_open_date datetime,
  budget_amount double,
  category string
);

-- 边定义
CREATE EDGE IF NOT EXISTS is_legal_rep (start_date date, end_date date);
CREATE EDGE IF NOT EXISTS is_shareholder (share_ratio double, invest_amount double);
CREATE EDGE IF NOT EXISTS is_director (position string);
CREATE EDGE IF NOT EXISTS is_actual_ctrl (ctrl_path string);
```

## 五、系统架构设计

### 5.1 整体架构

系统采用分层架构，从下到上分为数据层、图存储层、计算层和应用层。

**数据层**：对接现有数据源——客商接口提供供应商基础风险数据（替代原金镜接口），工商快照提供企业注册与股权数据，招投标系统提供投标行为数据（IP、报价、文件），银行系统提供账户与资金流水数据。数据通过 ETL 管道清洗、标准化后注入图数据库。

**图存储层**：选用 NebulaGraph 作为核心图数据库。NebulaGraph 是国产开源分布式图数据库，支持千亿级节点和边，查询延迟毫秒级，且在国内有成熟的企业级落地案例（震坤行、美团内部已有使用经验）。同时部署 NebulaGraph 的图计算引擎 NebulaGraph Analytics 用于离线图算法计算。

**计算层**：包含三个子模块——规则引擎（将现有风险规则改写为图查询语句，实现等价功能的同时提升性能）、图算法引擎（运行社区发现、PageRank、环路检测等图算法）、GNN 模型引擎（基于 PyTorch Geometric 训练 GraphSAGE/GAT 模型进行风险评分）。

**应用层**：提供风险查询 API（供现有供应商风险行为查询系统调用）、风险大屏（可视化展示供应商关联网络和风险态势）、预警推送（实时风险事件通知）。

### 5.2 技术选型

| 组件 | 技术选型 | 选型理由 |
|---|---|---|
| 图数据库 | NebulaGraph 3.x | 国产开源，千亿级节点支持，毫秒级查询，社区活跃 |
| 图可视化 | NebulaGraph Explorer / D3.js | 原生集成，支持子图探索与交互式分析 |
| 图算法 | NebulaGraph Analytics + NetworkX | 分布式大规模图计算 + Python 原型验证 |
| GNN 框架 | PyTorch Geometric (PyG) | 生态丰富，支持异构图，与 PyTorch 无缝集成 |
| ETL | [[Flink SQL 完整语法教程|Flink]] + [[spark|Spark]] | 实时流处理 + 离线批处理 |
| 数据源对接 | 客商接口 + 工商快照 API | 已有对接，数据覆盖全面 |
| 消息队列 | Kafka | 实时投标行为事件流 |
| API 网关 | Spring Cloud Gateway | 与现有微服务体系一致 |

### 5.3 数据流设计

数据流分为离线和实时两条链路。

**离线链路（T+1）**：每天从客商接口和工商快照拉取供应商基础数据和工商变更数据，经 Spark ETL 清洗转换后全量/增量写入 NebulaGraph。随后触发离线图算法任务——运行 Louvain 社区发现更新供应商团伙标签，运行 PageRank 更新节点影响力评分，运行 GNN 模型推理更新风险评分。

**实时链路**：投标行为（提交投标文件、投标报价、保证金缴纳等）通过 Kafka 实时推送，Flink 消费事件后实时写入 NebulaGraph，并触发实时规则检测——检查是否存在同 IP 投标、同保证金账户等即时风险信号，命中则立即推送预警。

## 六、图查询与算法设计

### 6.1 现有规则的图查询改写

以下将现有各风险检测维度改写为图查询语句，同时提供 NebulaGraph nGQL 和 Neo4j Cypher 两种语法。

#### 6.1.1 投标 IP 相同检测

**NebulaGraph nGQL**：
```ngql
-- 查找同一项目中共用 IP 的供应商组
MATCH (s1:supplier)-[:submitted_bid]->(p:bid_project)<-[:submitted_bid]-(s2:supplier),
      (s1)-[:used_ip{action:"投标"}]->(ip:ip_address)<-[:used_ip{action:"投标"}]-(s2)
WHERE id(s1) < id(s2)
RETURN p.bid_project.project_name AS project,
       s1.supplier.supplier_name AS supplier_1,
       s2.supplier.supplier_name AS supplier_2,
       ip.ip_address.ip AS shared_ip;
```

**Neo4j Cypher**：
```cypher
// 查找同一项目中共用 IP 的供应商组
MATCH (s1:Supplier)-[:SUBMITTED_BID]->(p:BidProject)<-[:SUBMITTED_BID]-(s2:Supplier)
MATCH (s1)-[u1:USED_IP]->(ip:IPAddress)<-[u2:USED_IP]-(s2)
WHERE u1.action = '投标' AND u2.action = '投标' AND s1.supplier_id < s2.supplier_id

       s1.supplier_name AS supplier_1,
```

### 6.3 图算法应用

**Louvain 社区发现（围串标团伙识别）**：Louvain 算法通过最大化模块度将图中紧密连接的节点划分为社区。在供应商网络中，一个社区即对应一个"疑似围串标团伙"。实施步骤为：首先构建供应商关联图（以共用 IP、共用账户、共同人员等作为边权），然后运行 Louvain 算法识别社区，最后将社区内供应商数量超过阈值（如 ≥ 3）且社区内密度超过阈值的标记为"疑似围串标团伙"。

**PageRank（关键节点识别）**：PageRank 评估节点在网络中的重要性。在供应商网络中，PageRank 得分高的节点可能是"核心中间人"或"关键壳公司"——它们连接了多个可疑供应商，是风险网络的枢纽。将 PageRank 得分作为风险评分的输入特征之一。

**最短路径（关联紧密度评估）**：两个供应商之间的最短路径长度反映了它们的关联紧密度。路径越短、经过的节点类型越敏感（如共用银行账号比同城市更敏感），关联风险越高。定义"加权最短路径"，对不同边类型赋予不同权重：共用银行账号（权重 0.1，极高风险信号）、共用 IP（0.2）、共同法人（0.3）、共同股东（0.4）、同地址（0.6）。

**环路检测（资金回流识别）**：检测图中的有向环路，特别是沿 `transfer_to` 边（资金转账关系）形成的环路。存在资金环路意味着可能存在"保证金回流"或"围标资金倒手"行为。

### 6.4 GNN 风险评分模型

**模型架构**：采用两层 GraphSAGE + 注意力聚合的异构图神经网络。输入为节点特征（供应商的注册资本、经营年限、历史中标率等数值特征，以及经营范围的文本 [[01基础_08从文本到向量之理解Embedding|Embedding]]）和图结构特征（度数、聚类系数、PageRank 值、所属社区 ID）。输出为每个供应商节点的风险概率（0-1 连续值）。

**训练数据构造**：以历史已确认的围串标案例和关联交易案例作为正样本，以正常供应商作为负样本，构造有标签的训练集。考虑到围串标样本稀少（正负样本不平衡），采用 Focal Loss 或 SMOTE 过采样策略。

**模型推理流程**：离线每天对全量供应商运行一次 GNN 推理，更新风险评分。对于新注册供应商，GraphSAGE 的归纳学习能力可以在无需重新训练的情况下，基于其邻居信息生成风险评分。

**可解释性**：引入 GNNExplainer 对高风险预测进行解释，输出"哪些邻居节点和边对该供应商的高风险评分贡献最大"，例如"该供应商风险评分为 0.87，主要贡献因素为：与已确认围串标供应商 B 共用银行账号（贡献 0.35）、法人张三同时担任 3 家投标供应商的法人（贡献 0.28）、近 6 个月内 5 次同 IP 投标（贡献 0.24）"。

**GNNExplainer 实现示例**：
```python
from torch_geometric.explain import GNNExplainer

class ExplainableRiskGNN(SupplierRiskGNN):
    def explain_prediction(self, x_dict, edge_index_dict, target_node):
        """生成风险预测的可解释性报告"""
        explainer = GNNExplainer(
            model=self,
            algorithm='GNNExplainer',
            epochs=200,
            return_type='log_prob'
        )
        
        # 获取目标节点的预测解释
        node_feat_mask, edge_mask = explainer.explain_node(
            target_node,
            x_dict=x_dict,
            edge_index_dict=edge_index_dict
        )
        
        # 提取贡献度最高的邻居
        top_contributors = []
        for edge_type, mask in edge_mask.items():
            edge_importance = mask.cpu().detach().numpy()
            top_edges = np.argsort(edge_importance)[-5:]
            
            for idx in top_edges:
                src, dst = edge_index_dict[edge_type][:, idx]
                contribution = edge_importance[idx]
                
                top_contributors.append({
                    "relation_type": edge_type,
                    "source": src.item(),
                    "target": dst.item(),
                    "contribution": float(contribution)
                })
        
        # 按贡献度排序
        top_contributors.sort(key=lambda x: x["contribution"], reverse=True)
        
        return {
            "target_node": target_node,
            "feature_importance": node_feat_mask.cpu().detach().numpy(),
            "top_contributors": top_contributors[:10]
        }

# 生成风险报告示例
def generate_risk_report(supplier_id: str, explainer: ExplainableRiskGNN) -> str:
    """生成自然语言风险报告"""
    explanation = explainer.explain_prediction(
        x_dict, edge_index_dict, target_node=supplier_id
    )
    
    report = f"""
    供应商 {supplier_id} 风险分析报告
    ================================
    风险评分：{prediction_score:.2f}
    风险等级：{risk_level}
    
    主要风险因素：
    """
    
    for i, contrib in enumerate(explanation["top_contributors"][:5], 1):
        factor_desc = describe_relation(contrib["relation_type"])
        report += f"{i}. {factor_desc} (贡献度: {contrib['contribution']:.2%})\n"
    
    return report

def describe_relation(relation_type: str) -> str:
    """将关系类型转换为中文描述"""
    descriptions = {
        "(supplier, used_ip, ip_address)": "共用投标 IP 地址",
        "(supplier, owns_account, bank_account)": "共用银行账户",
        "(person, is_legal_rep, supplier)": "同一人担任法人",
        "(person, is_shareholder, supplier)": "同一人持股",
        "(supplier, located_at, address)": "同一注册地址"
    }
    return descriptions.get(relation_type, "未知关联")
```

## 七、风险评分体系

### 7.1 综合风险评分模型

采用规则评分与 GNN 评分加权融合的方式，兼顾可解释性和智能化。

**规则评分（Rule Score）**按照现有风险维度计算：投标 IP 相同（+30 分/次）、银行账号相同（+40 分/次）、联系人信息相同（+25 分/次）、注册 IP 相同（+35 分/次）、详细地址雷同（+20 分/次）、投标文件雷同（+45 分/次）、投标保证金账户雷同（+40 分/次）、历史围串标记录（+50 分/次）。规则评分归一化到 0-100。

**图结构评分（Graph Score）**基于图算法输出计算：PageRank 得分（反映网络中心度）、社区规模与密度（所在社区越大越密，风险越高）、最短路径（与已知高风险节点的最短加权路径越短，风险越高）、环路参与度（参与的资金环路数量）。图结构评分归一化到 0-100。

**GNN 评分（GNN Score）**直接取模型输出的风险概率 × 100。

**综合风险评分** = α × Rule Score + β × Graph Score + γ × GNN Score，其中初始阶段 α=0.5, β=0.3, γ=0.2，随着 GNN 模型成熟逐步调整为 α=0.3, β=0.3, γ=0.4。

**实时风险评分更新流程**：
```python
class RiskScorer:
    def __init__(self, alpha=0.5, beta=0.3, gamma=0.2):
        self.alpha = alpha
        self.beta = beta
        self.gamma = gamma
        self.graph = Neo4jGraph()  # 图数据库连接
        
    def calculate_rule_score(self, supplier_id: str) -> float:
        """计算规则评分"""
        rules = [
            ("same_ip_bid", 30),       # 投标IP相同
            ("same_bank_account", 40), # 银行账号相同
            ("same_contact", 25),      # 联系人相同
            ("same_reg_ip", 35),       # 注册IP相同
            ("similar_address", 20),   # 地址雷同
            ("similar_file", 45),      # 投标文件雷同
            ("same_deposit_account", 40), # 保证金账户相同
            ("history_collusion", 50), # 历史围串标记录
        ]
        
        score = 0
        for rule_name, points in rules:
            violations = self.check_rule_violation(supplier_id, rule_name)
            score += violations * points
            
        return min(score, 100)  # 上限100分
    
    def calculate_graph_score(self, supplier_id: str) -> float:
        """计算图结构评分"""
        # 从图数据库获取算法结果
        pagerank = self.graph.run(
            "MATCH (s:Supplier {supplier_id: $id}) "
            "RETURN s.pagerank_score AS score",
            id=supplier_id
        ).data()[0]["score"]
        
        community_density = self.graph.run(
            "MATCH (s:Supplier {supplier_id: $id})-[:COMMUNITY]->(c:Community) "
            "RETURN c.density AS density",
            id=supplier_id
        ).data()[0]["density"]
        
        # 综合计算图评分
        graph_score = (pagerank * 40 + 
                      community_density * 30 +
                      self.get_shortest_path_risk(supplier_id) * 30)
        return min(graph_score, 100)
    
    def get_comprehensive_score(self, supplier_id: str) -> dict:
        """获取综合风险评分"""
        rule_score = self.calculate_rule_score(supplier_id)
        graph_score = self.calculate_graph_score(supplier_id)
        gnn_score = self.gnn_model.predict(supplier_id) * 100
        
        total = (self.alpha * rule_score + 
                self.beta * graph_score + 
                self.gamma * gnn_score)
        
        return {
            "supplier_id": supplier_id,
            "rule_score": rule_score,
            "graph_score": graph_score,
            "gnn_score": gnn_score,
            "comprehensive_score": total,
            "risk_level": self.get_risk_level(total),
            "factors": self.get_risk_factors(supplier_id)
        }
```

## 九、总结

本方案以现有供应商风险行为检测体系为基础，通过引入图数据库和知识图谱技术实现三层能力升级：第一层将规则检测从关系型数据库迁移到图数据库，获得数量级的性能提升；第二层利用图算法（社区发现、PageRank、环路检测）发现传统方法无法触及的深层隐性关联；第三层通过 GNN 模型实现从"规则驱动"到"数据驱动"的范式转换。

方案的设计紧密贴合现有业务需求——围串标检测的七大维度（银行账号相同、注册 IP 相同、联系人信息相同、详细地址雷同、投标文件雷同、投标保证金账户雷同、历史围串标记录）均有对应的图查询实现，同时引入了行业验证过的最佳实践（震坤行供应链图谱、携程反欺诈网络、围串标专利方案）。分阶段实施路线确保了技术风险可控、业务价值逐步兑现。
