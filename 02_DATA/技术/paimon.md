# 0. 关于paimon

Apache Paimon 是一个高吞吐、低延迟的流式[[数据湖项目|数据湖]]，专注于实时数据更新与高效查询。它由 Apache [[Flink SQL 完整语法教程|Flink]] 社区孵化，旨在解决传统数据湖在实时数据处理中的痛点，如数据更新延迟、流批分离架构复杂等问题，Paimon最大优势是，可以有效解决现代大数据系统面临的三大核心挑战：平衡低延迟与成本、简化复杂的数据生产架构，以及无缝集成 AI 场景。Apache Paimon可部署在S3、HDFS等系统上，支持通过Flink SQL实时处理数据，并支持SparkSQL任务进行批处理，并且支持通过Doris、Presto等OLAP引擎来直接查询。

| 概念             | 含义                                                                                                                                                                                                                                                                                                |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 分桶             | 每个分区内的数据分布在不同的桶内，用于更高效的查询。分桶是paimon读写操作的最小存储单元，因此存储桶的数量限制了最大处理并行度。每条数据的分桶取决于分桶键的hash值。                                                                                                                                                                                                            |
| LSMT           | Paimon采用LSM树（Log-Structured Merge-Tree）作为文件存储的数据结构。LSM树的核心思想<br><br>- **分层存储**  <br>    数据按层级（Level）组织，新数据写入内存（MemTable），后台异步合并到磁盘（SSTable），形成多层结构（如Level 0 → Level 1 → ...）<br>    <br>- **顺序写入**  <br>    所有写操作以追加（Append）方式完成，避免随机写，显著提升吞吐量。                                                   |
| 分区（Partition）  | Paimon采用与Apache Hive相同的分区概念来分割数据。<br><br>分区是一种基于特定列（如日期、城市、部门）的值将表划分为关联部分的可选方法。每张表可以有一个或多个分区键，用于标识特定分区。<br><br>通过分区，用户可以高效地操作表中的部分数据。                                                                                                                                                             |
| 表结构（Schema）    | 记录了字段，主键、分区、分桶，版本号等信息，是所有数据与元数据关联的基础，每次更新调整Schema，都会生产一个新版本的Schema文件，借助此项能力来实现SchemaEvolution。                                                                                                                                                                                                    |
| 合并（Compaciton） | Paimon使用类似RocksDB的Universal Compaction策略，好处是Compaction更轻量，每轮Compaction不需要将同层数据全部参与Compaction，写放大控制的较好，但是问题是因为每轮Compaciton都不会合并全部数据，所以读放大会相对增加，所以相当于在写放大和读放大之间做了动态权衡，如果写多读少，比如没有cdc的需求，可以适当的增加一次参与compaciton的文件数。当Paimon将记录追加到LSM树时，它也会根据需要执行Compaction。用户还可以选择在“专用Compaction作业”中独立执行所有Compaction。 |
doris对比paimon

|对比项|Paimon|Doris|
|---|---|---|
|存储结构|LSM 树|列式存储|
|写入方式|追加写 + 多层合并|批量写入列文件|
|主要场景|实时流写 + Upsert|高性能 OLAP 查询|
|是否典型 LSM|✅ 是|❌ 不是|
