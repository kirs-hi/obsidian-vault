## Doris 的核心架构包括哪些组件？各自的作用是什么？
**一句话记住：FE 管调度，BE 管存算，Meta 管元数据，Broker 管导入导出。**
## 回答重点

Doris 是一款现代化的大数据实时分析数据库，它的核心架构主要包括以下几个组件：

1）Frontend（FE）：负责用户接口和查询调度。
2）Backend（BE）：负责数据存储和查询执行。 
3）MetaServer：存储和管理元数据。 
4）Broker：数据导入和导出辅助工具。

各自的作用如下：

1）Frontend（FE）：它充当用户与系统之间的接口，处理用户查询请求和调度查询任务。FE 还负责管理用户权限，以及规划和优化查询。

2）Backend（BE）：它承担数据存储和实际查询执行的任务。BE 处理分布式数据存储的魔法，确保数据的可靠性和可用性，同时执行 FE 分配的查询任务。

3）MetaServer：主要用来存储数据库的元数据，包括表结构、用户信息、元数据备份等。这些信息对确保系统运行稳定性和数据一致性至关重要。

4）Broker：它的主要任务是帮助完成数据的导入和导出操作，特别是在与外部存储系统（如 HDFS、S3 等）交互时非常有用。

## 扩展知识

除了核心组件之外，还可以对 Doris 的一些特性和优势展开了解：

1）高性能查询：Doris 支持 MPP（大规模并行处理）架构，能高效地处理复杂的 SQL 查询，适合大数据实时分析场景。

2）亚秒级延迟：它通过列式存储、索引加速、内存计算等技术手段，大幅提升查询速度，实现了亚秒级延迟的查询响应。

3）强大的数据导入功能：利用 Broker 组件，Doris 可以非常方便地从各种大数据存储系统（如 HDFS、S3、HBase）导入数据，并支持多种导入方式，如批量导入和实时导入。

4）灵活的数据建模：Doris 支持星型、雪花型等多种数据建模方法，并且提供灵活的分区、分桶策略来提升查询效率和数据管理的便捷性。

5）高可用性和可扩展性：通过多副本机制、故障节点自动恢复、高效的资源调度等特性，Doris 保证了系统的高可用性和可扩展性，能适应不同规模的业务需求

### ⚔️ Doris 性能优化与治理“独孤九剑”

#### 第一式：心法总纲（治理核心）
> **“Latency 是表象，Resources 是真相。”**

*   **误区**：只盯着 `Latency`（查询耗时）看。
*   **正解**：`Latency` 只是结果，甚至可能是被别人堵出来的。要治理集群，必须看**破坏力指标**：
    1.  **Scan CPU Time**：它是导致 BE 节点负载飙升、Scanner 线程池打满的元凶。
    2.  **Shuffle Bytes**：它是导致网络拥堵、心跳丢失的元凶。

#### 第二式：望闻问切（诊断口诀）
> **“先看排队判拥堵，再看扫描抓元凶。”**

当集群卡顿或查询变慢时，按此顺序排查：
1.  **看 `ScanWorkerWaitTime`**：如果飙升，说明 Scanner 线程池爆满了（大家都在排队等吃饭）。
2.  **查 `ScanTime` & `Scan CPU Time`**：去抓那些正在“暴食”的 SQL（真正占用线程不释放的任务）。
3.  **看 `ShuffleBytes`**：如果 CPU 不高但网络 IO 高，查大表 Join 或 Group By 导致的全网数据重传。

#### 第三式：IO 裁剪（优化之王）
> **“物理跳过胜过一切索引。”**

*   **最高优先级**：**前缀索引（ShortKey）** + **分区裁剪**。
    *   *原理*：直接不读磁盘上的数据块（Data Block）。
*   **次级优先级**：BloomFilter / Bitmap 索引。
    *   *陷阱*：如果过滤度不高，读索引本身的开销反而比直接扫描数据还大（起到负作用）。

#### 第四式：Join 调优（借力打力）
> **“不要轻信优化器，手动干预保平安。”**

*   **RuntimeFilter**：利用这个机制，把大表 Join 变成“单表前缀索引扫描”（面试必杀技）。
*   **Hint 强制**：Doris CBO 偶尔会犯蠢，把大表广播（Broadcast）导致 OOM。手动加 `[shuffle]` 让它乖乖重分布。
*   **Colocate Join**：高频 Join 表（如订单+订单详情），建表时划入同一组，实现**零网络传输** Join。

#### 第五式：大宽表迷思（辩证思维）
> **“大宽表不是万能药，巧用 Join 更高效。”**

*   **反常识**：在 Doris 中，合理利用 **ShortKey + RuntimeFilter** 的 Join 查询，往往比没有特定排序的“大宽表”全表扫描要快得多。不要为了宽表而宽表。

#### 第六式：去重秘籍（Bitmap）
> **“越密集，越高效。”**

*   **原理**：`count(distinct)` 改用 `Bitmap` 类型。
*   **技巧**：Bitmap 存的数据越**密集**（例如连续整数），内存占用越小，计算越快。尽量将稀疏 ID 映射为稠密 ID。

#### 第七式：大文件处理（化整为零）
> **“切得越细，跑得越快。”**

*   **场景**：H2D（Hadoop to Doris）导数或查询。
*   **手段**：调整 Split Size，避免单个 Task 处理几个 G 的数据。强制增加并发度，利用多核 CPU 优势。

#### 第八式：熔断机制（止损防崩）
> **“杀掉一个，救活全家。”**

*   **策略**：建立基于 **Scan CPU Time 突刺** 的熔断机制。
*   **逻辑**：检测到某个 SQL 在 1 秒内 CPU 消耗斜率异常（比如瞬间唤醒 100 个线程），直接 Kill，防止它把整个 BE 节点拖垮。

#### 第九式：面试话术（高分金句）

如果面试官问：“你对 Doris 原理有什么深入理解？” —— **请直接背诵这段话：**

> “很多人只关注 SQL 跑得快不快，但我更关注 **BE 节点的执行模型**。
> 我发现 Doris 的 `Fragment Instance` 只是逻辑调度，真正干重活的是底层的 `OlapScanNode`。
> 它依赖一个公共的 **Scanner 线程池**。一旦某个烂 SQL 进行了大规模全表扫描，就会瞬间耗尽线程池资源，导致其他几十个简单查询全部卡死在队列里。
> 所以，我治理的核心不是看 Latency，而是监控 **Scan CPU Time** 的突刺，并优先通过 **ShortKey** 和 **RuntimeFilter** 来做 IO 裁剪，从源头降低对线程池的破坏力。”

### **Doris 查询性能调优与治理结构化分析**

这份文档是关于 Apache Doris 数据仓库查询性能优化的深度指南，主要面向数据开发和运维人员。它从问题现象出发，深入到底层原理，并提供了全方位的调优手段。

以下是基于文档内容的结构化分析：

#### **1. 核心问题与治理原则**
*   **常见性能痛点**：
    *   **查询超时**：默认 20 秒超时，常见于数据重刷高峰或大查询。
    *   **耗时抖动**：平时快但在高峰期变慢，通常源于资源竞争（大查询抢占资源）。
    *   **资源瓶颈**：CPU 或 IO 负载过高。
*   **治理原则（Fast-Fail）**：
    *   Doris 设计遵循快速失败原则，不具备复杂的中间状态存储或重试机制。
    *   **策略**：不建议随意调大超时时间。如果必须运行大查询（Ad-hoc），建议更换引擎或申请独立集群资源。
    *   **扩容观**：扩容仅在硬件资源（CPU/内存/磁盘）完全饱和时有效，优先进行查询调优。

#### **2. 核心调优手段（三大支柱）**
文档强调 **“预聚合”** 和 **“前缀索引”** 是 Doris 性能的核心，未能利用这两点的查询通常不适合 Doris。

**A. 数据划分（Partition & Bucket）**
*   **分区（Partition）**：
    *   **不可变性**：建表后分区分桶列不可更改，需提前规划。
    *   **最佳实践**：推荐按天（YYYYMMDD）分区。
    *   **查询铁律**：**必须**显式过滤分区，避免全表扫描。需注意谓词类型匹配（如字符串 vs 数字）以防分区裁剪失效。
*   **分桶（Bucket）**：
    *   **选列原则**：高基数（如 User ID）、分布均匀、常用于查询过滤。避免使用低基数（如城市）或时间列导致数据倾斜。
    *   **数量规划**：单 Tablet 建议大小 **1GB - 10GB**。
        *   *点查为主*：减少分桶数以提升吞吐。
        *   *范围/扫描为主*：增加分桶数以利用多核并行计算。

**B. 索引与模型优化**
*   **前缀索引（Prefix Index）**：
    *   **机制**：利用前 36 字节快速定位数据。
    *   **命中条件**：使用 `=` 或 `IN`，且位于查询条件的最左前缀。`OR` 和 `!=` 无法命中。
    *   **注意事项**：避免 VARCHAR 类型在前缀索引靠前位置，因为它会截断索引。
*   **Bitmap 索引**：适用于低基数列（性别、省份），配合前缀索引使用。
*   **Rollup / 物化视图**：
    *   **核心价值**：**预聚合**。通过空间换时间，减少现场计算量。
    *   **命中逻辑**：查询涉及的所有列（维度、指标、过滤条件）必须包含在 Rollup 中。

**C. Join 策略优化**
Doris 适合中小数据量（GB 级）Join，超大数据量建议预处理。
*   **Shuffle Join**：默认方式，需注意网络开销和数据倾斜。
*   **Broadcast Join**：适合“大表 Join 小表”，将小表广播到所有节点。
*   **Bucket Shuffle Join**：Join Key 包含全部分桶列时触发，网络开销较低。
*   **Colocate Join**：性能最高（零网络开销），但限制严格（分桶数、副本数、分桶列类型必须一致）。

#### **3. 查询原理与诊断**
*   **执行架构**：
    *   FE 生成分布式查询计划（PlanFragment）。
    *   BE 执行具体计算（Instance），涉及 Scan、Aggregation、Exchange 等节点。
*   **Profile 分析**：
    *   每个查询实例都有独立的 Profile，包含 CPU 耗时、数据扫描量等关键指标，是深度调优的依据。
*   **监控体系**：
    *   **关键指标**：CPU Top N（定位计算密集型大查询）、Scan Rows Top N（定位扫描过多的查询）、Fragment Instance 数量（定位高并发复杂查询）。

#### **4. 治理与排查流程**
*   **排查三板斧**：
    1.  **确认现象**：是执行变慢、请求压力变大，还是资源瓶颈？
    2.  **瓶颈定位**：
        *   *计算瓶颈*：CPU 高，复杂计算或大查询。
        *   *扫描瓶颈*：IO 高，分区/分桶裁剪失效，未命中索引。
        *   *网络瓶颈*：Shuffle 数据量过大。
    3.  **限流措施**：对单 DB 或特定 SQL 模式进行限流，保护集群稳定性。

### **第一部分：导入性能优化 SOP（根源治理）**

**背景**：查询慢，往往是因为数据“进”得不好（碎片多、版本多、基数没处理好）。买菜业务实战证明，选对导入方式能带来 **3倍** 以上的性能提升。

#### **Step 1.1：导入方式选型决策树**
*执行导入任务前，请根据以下逻辑选择 `SparkLoad` 还是 `StreamLoad`：*

1.  **是聚合模型 (Aggregate Key) 吗？**
    *   **YES** → **数据量 > 1亿行 (或 >10GB)？**
        *   **YES** → ✅ **必须选 SparkLoad**。
            *   *实战数据*：买菜场景下，1.3亿行聚合表，SparkLoad 耗时 **641秒**，StreamLoad 耗时 **1975秒**。
            *   *原理*：SparkLoad 利用 Spark 集群的 Shuffle 能力完成预聚合，减轻 BE 节点的 Compaction 压力。
        *   **NO** → ✅ **选 StreamLoad**（省去 Spark 资源申请开销）。
    *   **NO (是明细/主键/冗余模型)** → ✅ **首选 StreamLoad**。
        *   *实战数据*：买菜场景下，1.3亿行冗余表，StreamLoad 耗时 **1329秒**，SparkLoad 耗时 **14218秒**（反向优化）。
        *   *原理*：冗余模型无需预聚合，SparkLoad 的 Shuffle 纯属多余，且会导致数据膨胀。

#### **Step 1.2：导入并发设置（防雪崩）**
*不要盲目增加并发，买菜业务的 90BE 节点集群压测结论如下（供参考）：*

*   **聚合模型**：最佳并发 **40-50**。超过 80 会导致 BE 内存耗尽 (OOM) 或 Write 耗时飙升。
*   **冗余模型**：最佳并发 **70-80**。冗余模型写入开销小，可适当调大。
*   **SOP 动作**：
    1.  从 **3-5 个并发**起步。
    2.  观察监控：BE CPU < 70%，内存 < 80%。
    3.  逐步上调，一旦发现 `Commit` 阶段耗时显著增加，立即回调。

#### **Step 1.3：解决导入倾斜（SparkLoad 专项）**
*如果在 SparkLoad 阶段这就慢，Doris 也没法快。*

*   **现象**：Spark 任务中某几个 Task 运行时间极长，其他 Task 很快结束。
*   **SOP 动作**：
    1.  **源文件倾斜**：如果 Hive 源表文件大小不均，添加参数强制切分：
        ```java
        set spark.hadoop.hive.exec.orc.split.strategy=BI; // 或 ETI
        ```
    2.  **Bitmap 构建倾斜**：如果是构建 Bitmap 字典导致倾斜：
        *   开启 **全局字典复用** (`dict_reuse`)。
        *   添加 **跳过 NULL 值** 参数 (`skip_null_value_columns`)，避免大量 NULL 值挤在一个 Reduce 中。

### **第二部分：模型设计 SOP（Bitmap 专项）**

**背景**：去重（Count Distinct）和留存分析（Retention）是 Doris 最耗资源的场景。最佳实践是**“用空间换时间”**。

#### **Step 2.1：高基数去重建模（User ID）**
*   **场景**：`count(distinct user_id)`，UV 计算。
*   **SOP 动作**：
    *   **低/中基数**（千万级以内）：
        *   直接建表时将列类型设为 `BITMAP`，聚合函数用 `BITMAP_UNION`。
    *   **高基数**（亿级，且 ID 为 String）：
        *   **方法 A (推荐)**：在 Hive/Spark 层先生成全局字典，将 `String` 映射为 `Int/BigInt`，再导入 Doris 的 `BITMAP` 列。
        *   **方法 B (省事)**：使用 Doris 的 `bitmap_hash()` 函数，但要注意 Hash 冲突风险（虽极低）。
    *   **查询改写**：
        *   ❌ `SELECT count(distinct user_id) ...`
        *   ✅ `SELECT BITMAP_UNION_COUNT(user_id) ...`

#### **Step 2.2：留存/复购分析建模**
*   **场景**：计算昨天来了，今天又来的用户。
*   **SOP 动作**：
    *   利用 `intersect_count` 或 `orthogonal_bitmap`（正交位图）。
    *   **建模**：建立一张以 `user_id` 为 Bitmap 聚合列，`tag`（如日期、渠道）为维度的聚合表。
    *   **查询**：
        ```sql
        -- 计算同时满足 filter_1 和 filter_2 的用户数
        select intersect_count(user_id_bitmap, tag, 'filter_1', 'filter_2') ...
        ```
    *   *收益*：买菜业务中，利用 Bitmap 优化复购率计算，性能从 **几十秒** 提升至 **亚秒级**。

### **第三部分：查询执行优化 SOP（Join 与 资源）**

**背景**：Explain 只是计划，Profile 才是真相。

#### **Step 3.1：Join 策略强制干预**
*Doris 优化器有时会选错 Join 方式，需人工 Hint。*

1.  **Bucket Shuffle Join (BSJ)**：
    *   **条件**：左表的分桶列 = Join Key，且数据类型一致。
    *   **收益**：减少数据在网络间的 Shuffle，只传右表。
    *   **SOP 动作**：确保 Join Key 是分桶列。如果不是，考虑新建 Rollup 将 Join Key 设为第一列（并设为分桶列？注意分桶列不能改，这里通常指新建表或调整逻辑）。*修正：分桶列建表后不可改，只能重建表。*
2.  **Colocate Join (最强)**：
    *   **条件**：两张表分桶列、分桶数、副本数完全一致，且归属于同一个 Group。
    *   **SOP 动作**：
        ```sql
        -- 建表时指定
        "colocate_with" = "group_x"
        ```
    *   *验收*：Explain 中显示 `Colocate: true`，且没有 `EXCHANGE` 节点。
3.  **Broadcast Join (广播)**：
    *   **场景**：右表极小 (<1G)，左表极大。
    *   **风险**：如果右表统计信息失效（实际很大），会导致 BE OOM。
    *   **SOP 动作**：显式 Hint `... JOIN [broadcast] table_b ...` 仅在确信右表很小时使用。

#### **Step 3.2：并行度 (Parallelism) 调优**
*   **参数**：`parallel_fragment_exec_instance_num` (默认 1)。
*   **场景**：
    *   **QPS 高的小查询**：保持默认 **1**，避免线程争抢。
    *   **QPS 低的大查询 (OLAP 分析)**：调大该值（如 8 或 16）。让一个查询利用单台 BE 的多个核。
*   **SOP 动作**：
    *   在 Session 级别设置：`set parallel_fragment_exec_instance_num = 8;`
    *   *注意*：不要超过机器 CPU 核数的一半，否则上下文切换开销过大。

#### **Step 3.3：Runtime Filter 检查**
*   **原理**：右表生成的过滤器推给左表。
*   **SOP 动作**：
    *   检查 Profile，搜索 `RuntimeFilter`。
    *   如果 **WaitTime** 很久：说明右表太慢，拖累了左表。→ 优化右表。
    *   如果 **Effect** (过滤行数) 为 0：说明过滤无效。→ 考虑关闭 RF 减少开销 (`set runtime_filter_mode = OFF`)。

### **第四部分：SOP 执行核对单 (Checklist)**

**在提交 SQL 或上线新表前，请逐一勾选：**

| 阶段 | 检查项 | 标准/阈值 |
| :--- | :--- | :--- |
| **导入** | 导入方式选择 | 聚合大表用 SparkLoad，其他用 StreamLoad |
| **导入** | 分区时间跨度 | 必须对齐（如按天分区），避免跨分区导入 |
| **模型** | 分桶数设置 | `数据量 / 300MB`，单 Tablet 控制在 1-5GB |

| **查询** | 分区裁剪 | Explain 中 `partitions` 必须大幅 < 总分区数 |

| **查询** | 结果集限制 | 生产环境禁止无 Limit 的 `SELECT *` |
| **查询** | Join 顺序 | 大表在左，小表在右；Join Key 类型严格一致 |

### **SparkLoad 与 StreamLoad 的核心区别**

**Simple Answer:**

**SparkLoad** 利用外部 Spark 集群资源对数据进行预处理（主要是排序、分区和预聚合），适合**超大数据量（亿级）的初次导入**或**聚合模型**表，能显著降低 Doris 后端（BE）的 Compaction 压力；**StreamLoad** 是通过 HTTP 协议直接将数据流式传输给 Doris BE 节点进行处理，适合**微批高频**导入、**实时**同步以及**非聚合模型**的中小批量数据，胜在轻量级和低延迟。

---

### **详细对比分析**

#### **1. 核心原理与架构**
| 特性 | **StreamLoad** | **SparkLoad** |
| :--- | :--- | :--- |
| **执行主体** | **Doris BE 节点** | **外部 Spark 集群** |
| **数据处理** | BE 节点负责接收数据、解析、分发、排序和写入。 | Spark 集群负责读取数据、排序、**预聚合**、划分 Bucket，生成中间文件；Doris BE 仅负责拉取文件和建立索引。 |
| **资源消耗** | 消耗 **Doris 集群** 的 CPU 和内存。 | 消耗 **Spark 集群** 资源，**节省 Doris 资源**。 |
| **协议/方式** | HTTP / HTTPS 协议，流式传输。 | 提交 Spark 任务（YARN/K8s）。 |

#### **2. 性能表现 (基于美团买菜实战数据)**

根据之前的**《买菜 Doris 离线场景的性能优化与实践》**案例，不同模型下两者性能差异巨大：

*   **场景 A：聚合模型 (Aggregate Key)**
    *   **结论**：**SparkLoad 完胜**（快 3 倍）。
    *   **数据**：1.3 亿行数据导入。
        *   StreamLoad：**1975 秒**。
        *   SparkLoad：**641 秒**。
    *   **原因**：聚合模型需要合并相同 Key 的数据。StreamLoad 写入后，BE 需要大量 CPU 资源进行 Compaction（合并）；而 SparkLoad 在外部就利用 Shuffle 完成了预聚合，进入 Doris 的数据已经是合并好的，BE 压力极小。

*   **场景 B：冗余/明细模型 (Duplicate Key)**
    *   **结论**：**StreamLoad 完胜**（快 10 倍）。
    *   **数据**：1.3 亿行数据导入。
        *   StreamLoad：**1329 秒**。
        *   SparkLoad：**14218 秒**（反向优化，极慢）。
    *   **原因**：冗余模型不需要合并数据。SparkLoad 强制进行的 Shuffle 和排序变成了纯粹的额外开销（浪费时间），不如 StreamLoad 直接并发写入来得快。

#### **3. 功能特长**
*   **StreamLoad**：
    *   **实时性**：支持秒级延迟，适合 Flink/CDC 实时写入。
    *   **简单**：无需配置 Spark 环境，Client 一个 curl 命令即可触发。
    *   **并发**：并发能力强，适合小文件高频写入。

*   **SparkLoad**：
    *   **全局字典 (Bitmap)**：**独家优势**。对于高基数 String 类型的去重（如 `user_id` 是字符串），SparkLoad 可以在构建阶段生成全局字典并转换为 Bitmap，这是 StreamLoad 难以做到的（通常需要借助外部 KV 存储）。
    *   **抗倾斜**：对于源数据严重倾斜的场景，Spark 可以更好地处理分区倾斜问题。

### **选型决策树**

在实际生产中，请参照以下逻辑进行选择：

1.  **是否是实时/流式写入？**
    *   👉 是：**StreamLoad** (配合 Flink Doris Connector)

2.  **是否需要构建全局字典 (Bitmap 去重 String 字段)？**
    *   👉 是：**SparkLoad** (利用 Spark 计算全局字典)

3.  **表模型是聚合模型 (Aggregate) 且数据量 > 1 亿行？**
    *   👉 是：**SparkLoad** (利用 Spark 预聚合能力)
    *   👉 否 (明细表/小数据量)：**StreamLoad**

**总结建议**：
默认优先使用 **StreamLoad**，它更轻量、运维成本更低。只有在 **"聚合模型大数据量导入慢"** 或者 **"需要构建全局字典"** 这两个特定痛点出现时，才升级为 **SparkLoad**。
