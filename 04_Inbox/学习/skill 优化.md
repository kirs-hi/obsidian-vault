你问的是两个层面的事情：一是"这种 Skill 的设计方案文档本身该怎么写"，二是"这个 Skill 有哪些可以落地的优化方案"。我结合刚才完整读过的 2003 行 SKILL.md 和所有辅助文件，给你从数仓工程师的视角讲清楚。

## 一、方案怎么写（Skill 设计方案的写法方法论）

核心问题是：SKILL.md 既是"给人看的设计文档"，也是"给 LLM 执行的 prompt"，这两个目标是冲突的。设计文档追求"人能快速理解全貌"，prompt 追求"LLM 在执行时能精确遵守规则"。当前这份 SKILL.md 试图用一个文件同时满足两者，结果是 2003 行的巨型文件，人和 LLM 都吃力。

好的 Skill 方案应该分三层来写：

**第一层：总控文件（SKILL.md，控制在 300-500 行）**

只放三样东西：全局强制约束（当前的🔒1-10）、阶段流水线概览（一句话说清楚每个阶段做什么、输入输出是什么）、以及各阶段的路由入口（指向具体的子文件）。类比数仓的调度 DAG——DAG 定义只管节点依赖关系，不管每个节点内部的 SQL 怎么写。

**第二层：阶段子文件（每个阶段独立一个 .md）**

比如 `phases/phase-0-permission.md`、`phases/phase-1-clarify.md`、`phases/phase-2-query.md`。每个文件只描述本阶段的详细逻辑。LLM 执行到哪个阶段就读哪个文件，不需要一次性加载全部 2003 行。这和你熟悉的数仓分层是同一个道理——ODS/DW/ADS 各管各的，不会把所有 SQL 写在一个文件里。

**第三层：引用资料（references/ 目录，当前已有）**

SQL 模板、HTML 骨架、洞察 prompt、字段映射表等。这层当前做得已经不错了。

写方案时还有一个关键原则：**约束要"代码化"，不要"[[提示词]]化"**。当前的🔒1"严禁 mtdata query"、🔒2"严禁 LIMIT"，都是写在自然语言里让 LLM 自律的。但 `procurement_utils.py` 里的 `run_query_with_mtdata_bi()` 其实已经做了 LIMIT 检测和自动注入行权限。那方案文档里就不需要再花大量篇幅"教"AI 怎么做，只需要写"调用 `run_query_with_mtdata_bi(sql)` 执行取数"，把规则执行的责任交给代码。方案文档的角色是"调度编排"，不是"手把手教 LLM 写代码"。

## 二、可落地的优化方案

下面按优先级排列，每条都说清楚"为什么要做"和"怎么做"。

### P0：SKILL.md 模块化拆分

**为什么**：2003 行单文件是所有问题的根源。LLM 上下文窗口越长，后段对前段约束的遵守率越低。你自己在 memory 里也记了"否定性约束最易丢失"——而当前 Skill 恰恰有大量否定性约束（10 条"严禁"）。

**怎么做**：
```
SKILL.md（300行以内）
  ├── 强制约束（🔒1-10，保留）
  ├── 阶段流水线总览（流程图级别）
  └── 各阶段入口指令："读取 phases/phase-X.md 执行"

phases/
  ├── phase-neg1-bootstrap.md    # 环境安装（当前484-556行）
  ├── phase-0-permission.md      # 权限预检（当前559-1094行，最重的部分）
  ├── phase-1-clarify.md         # 需求澄清（当前1147-1390行）
  ├── phase-2-query.md           # 取数（当前1392-1524行）
  ├── phase-3-analysis.md        # 多维分析（当前1526-1580行）
  ├── phase-35-insight.md        # 洞察报告（当前1583-1718行）
  └── phase-45-delivery.md       # 可视化+报告交付（当前1720-1980行）
```

拆分后每个阶段文件 200-400 行，LLM 只需要在当前阶段加载对应文件即可。

### P1：将 LLM 自律约束下沉为代码防线

**为什么**：prompt 里写"严禁 X"是软约束，代码里 `raise Exception` 是硬约束。`procurement_utils.py` 已经做了一部分（LIMIT 检测、行权限兜底），但还有几个规则没有代码化。

**怎么做**：

在 `procurement_utils.py` 中补齐以下防线：

🔒1 的代码化——`run_query_with_mtdata_bi()` 入口处检测调用来源是否为 `mtdata bi run`，如果检测到 `mtdata query` 或 `mcporter call` 命令模式直接拦截。实际上函数内部已经固定用 `mtdata bi run`，所以这条约束天然被代码保证了，SKILL.md 里可以删掉对应的教学段落。

🔒10 的代码化——全局状态过滤（`po_line_status <> '70' AND po_line_biz_status NOT IN ('10','90')`）当前依赖 AI 主动写入 SQL。可以在 `run_query_with_mtdata_bi()` 里加一个类似行权限注入的"状态过滤兜底注入"：检测 SQL 是否包含事实表B的引用，如果有但缺少状态过滤条件，自动注入。

🔒6 的代码化——`build_html_report()` 的 `main_sql` 参数当前只是文档要求"不得为空"，可以在函数入口加 `assert main_sql and main_sql.strip(), "SQL留痕不得为空"`。

### P2：SSO/双端兼容逻辑封装

**为什么**：SSO Cookie 修复那段代码（约 200 行 Python）和 CatClaw/CatPaw 路径分支逻辑，占据了 SKILL.md 大量篇幅，但这些和"采购分析"的业务逻辑完全无关。它们是基础设施层面的事，不应该出现在业务 Skill 的 prompt 里。

**怎么做**：将 `_handle_cookie_exception()` 和 `_fix_proxy_config()` 完整移入 `procurement_utils.py`，SKILL.md 里只写"若遇到 CookieNotFoundException，调用 `handle_sso_recovery(mis_id)` 自动处理"。双端路径差异也一样，`_resolve_config_path()` 已经做了自动解析，SKILL.md 里不需要再反复解释"CatClaw 用 `~/.openclaw/`，CatPaw 用 `~/.catpaw/`"。

### P3：增加集成测试框架

**为什么**：当前 1390 行的 utils + 870 行的 SQL 模板，没有任何测试。每次改动（比如加个新模板、改个字段映射）都是"改完部署，人肉跑一遍看看对不对"。

**怎么做**：在 `scripts/` 下加一个 `tests/` 目录，优先覆盖三类高风险场景。一是 SQL 模板正确性——用一组固定的品类编码和时间范围，生成 SQL 后检查 WHERE 条件是否包含行权限过滤和全局状态过滤。二是权限注入完整性——模拟 L1/L2/L3 三层混合权限场景，验证 `build_row_filter_clause()` 生成的子句覆盖是否完整。三是 HTML 报告渲染——检查 `build_html_report()` 输出的 HTML 是否包含五个必需区块、ECharts bundle 是否正确内嵌。

### P4：analysis_blueprint 的持久化与版本对比

**为什么**：当前 blueprint 只存最后一次（`last_blueprint`），追问场景下用户可能想对比"加新维度前后"的分析差异。另外"24小时内复用"的规则比较粗糙——用户换了品类范围但在24小时内，是该复用还是重建？

**怎么做**：将 blueprint 改为数组存储（最近5个），每个带 timestamp + hash。复用判断从"时间距今≤24h"改为"核心参数（品类范围+时间范围+分析目的）hash 相同则复用"，参数变了就重建，不管时间间隔。

### P5：预测层彻底清理

**为什么**：当前"半移除"状态（报告中移除，SKILL.md 保留供内部参考）是一个隐患。LLM 的注意力机制不区分"这段是给你内部参考的"和"这段是要你执行的"，残留的预测层指导有概率被 LLM 当作输出指令执行。

**怎么做**：将预测层相关的所有内容从 SKILL.md 和 insight-prompt-template.md 中彻底删除。如果未来要恢复，从 git 历史里找就行。

### P6：管理者身份验证加固

**为什么**：当前管理者判断逻辑是 `os.environ.get('USER')` + SSO 配置文件读取。在沙箱环境中 `$USER` 是可以被篡改的。管理者享有全量查询、跳过权限预检等特权，身份伪造的后果严重。

**怎么做**：管理者身份验证应该强制走 SSO token 校验（从 `sso_config.json` 读取 misId），不再 fallback 到 `$USER` 环境变量。如果 SSO token 过期或不存在，管理者也必须先完成 SSO 认证，不能降级为环境变量判断。

---

### 总结一句话

这个 Skill 的业务设计水平很高（三层分析蓝图、健康度评分、品类交互、排查模式都是精品），优化方向集中在工程化层面——把"教 LLM 做事"的 prompt 约束下沉为"代码替 LLM 做事"的硬防线，把 2003 行的单体文件拆成按阶段加载的模块化结构。核心思想和数仓一样：分层治理、关注点分离、自动化保障。
