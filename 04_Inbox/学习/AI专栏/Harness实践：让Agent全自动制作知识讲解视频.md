# Harness 实践：让 Agent 全自动制作知识讲解视频

## 核心内容概述

本教程讲解了如何用 [[07-Agent|Agent]] + Skill（Harness 工程）把一篇技术文章全自动转化为精美的知识讲解视频。核心思路是：不用视频生成模型，而是让 AI 写出一个可控的网页动画，再配合 TTS 语音合成和录屏，产出画面与口播天然对齐的成品视频。整个过程被封装成一个名为 **web-video-presentation** 的 Skill，体现了 Harness 驾驭工程的六大核心要素。视频分三部分讲解：视频怎么做的、Skill 怎么设计的、手把手完整实战流程。

---

## 一、视频到底咋做的

### 1.1 技术选型：为什么选网页而非视频生成模型

作者明确**不使用** Sora/可灵等视频生成模型，也不用 NotebookLLM，核心原因是**可控性**：

- **网页的优势**：字体、配色、每一步停留时间、某一帧精确数字的展示——改几行代码就搞定，比视频模型"抽卡"式生成稳定得多，成本也更低。
- **NotebookLM 的问题**：做不了动画演示效果，出来的都是静态图。
- **Remotion 框架的问题**：反而限制了模型本身的发挥，有时候还不如直接让模型写 HTML/CSS/JS 来得自由。

![[04_Inbox/学习/AI专栏/assets/harness-video/img_00.png]]

### 1.2 上期视频案例

拿上期视频举例——完全由 AI 生成，输入是一篇 Anthropic 最新发布的文章（[Building [[Claude Code 命令与最佳实践|Claude Code]]: Prompt Caching Is Everything](https://claude.com/blog/lessons-from-building-claude-code-prompt-caching-is-everything)），输出是一个网页：

![[04_Inbox/学习/AI专栏/assets/harness-video/img_01.png]]

- 把原始文章拆成 **13 个章节、100 多个细粒度讲解步骤**
- 每一步都有完整的视觉演示效果
- 画布固定 **16:9**，适配任意大小显示器
- 底部隐藏进度条（鼠标悬浮才出现），支持自由跳转任意章节/步骤
- 画面无页眉、页码、品牌标识——录屏时观众看到的是干净画面，更像视频而非网页

![[04_Inbox/学习/AI专栏/assets/harness-video/img_02.png]]

### 1.3 从文章到视频的四个关键流程

**第一步：文章 → 口播稿**

原始技术文章书面化严重（如"该工具旨在提供高效的解决方案"），需要改写为短句、口语化、第二人称的表达。目标是"像跟人聊天一样"——"这玩意儿是干嘛的呢？"

**第二步：口播稿 → 开发大纲**

每段话对应一个画面步骤，每几步组成一个章节，每章聚焦一个话题。整个开发任务会非常复杂，提前做好大纲是保持后续稳定性的关键。

![[04_Inbox/学习/AI专栏/assets/harness-video/img_03.png]]

**第三步：给每步做视觉演示**

不是把文字糊上去就完事。TCP 三次握手要画时序图，DNS 查询要画节点链路，反垃圾审判要画评分仪表盘。得把抽象概念"演"出来。

![[04_Inbox/学习/AI专栏/assets/harness-video/img_04.png]]

**第四步：步骤和口播对齐**

网页的关键演示步骤必须跟随口播稿的节奏走，这样做出的视频效果才自然。

### 1.4 为什么要做成 Skill

上面这些事情 Agent 都能做，模型并不缺能力。但问题是：

- 怎么让它**每次**都做得出来？
- 怎么让不同的人、不同文章、不同主题都能**稳定产出**？
- 怎么让开发效率**不靠运气**？

直接跟 Agent 说"帮我做个视频网页"可能跑偏：稿子和画面对不上、章节数和音频数不一致、后面章节风格跟前面不搭……

![[04_Inbox/学习/AI专栏/assets/harness-video/img_05.png]]

**模型有能力，但你需要一套系统来驾驭这个过程——划定边界、管理状态、设立检查点、在关键节点拦住错误。这就是 Harness 要负责的事。**

## 二、web-video-presentation Skill 设计

Skill 是 Agent 通用的扩展能力标准——一份"操作手册"，告诉 AI 什么时候该做什么、做到什么标准、哪些红线不能碰。AI 加载 Skill 后按里面的规则干活，不用每次重述规矩。

此 Skill 托管在 [garden-skills](https://github.com/ConardLi/garden-skills) 仓库。

![[img_06.jpg]]

### Harness 六大核心部分

![[04_Inbox/学习/AI专栏/assets/harness-video/img_07.png]]

| 核心部分 | 它解决的问题 | 典型作用 |
|---------|------------|---------|
| [[理论学习_上下文压缩与_Token_管理|上下文管理]] | 模型到底看到了什么 | 组织系统[[提示词]]、项目文档、历史对话、任务状态、外部资料 |
| 工具系统 | 模型到底能做什么 | 搜索、读写文件、调用 API、执行代码、操作浏览器 |
| 执行编排 | 模型下一步该做什么 | 决策、分步执行、工具调用、结果回写 |
| 状态与记忆 | 系统如何跨步骤保持连续性 | 保存任务进度、中间结果、长期偏好 |
| 评估与观测 | 系统怎么知道自己做得对不对 | 结果验收、过程追踪、指标监控、质量评估、错误归因 |
| 约束与恢复 | 出错了怎么办，怎么避免跑偏 | 权限控制、格式约束、重试机制、回滚与校验 |

### 2.1 执行和编排

整个 Skill 把"从文章到视频"切成**四个阶段**，中间卡**两个人工检查点**：

![[img_08.jpg]]

**阶段一：内容编写**
- AI 同时产出两样东西——口播稿（决定信息描述方式和整体节奏）和开发大纲（决定每章做几步、每步屏幕放什么信息）

**检查点 Plan（人工确认）**
- Agent 走到这里被强制停下，人一次性对齐五件事：
  1. 稿子要不要改
  2. 大纲要不要改
  3. 用哪套主题
  4. 素材谁准备
  5. 后续章节是逐章做还是并行做
- 这五件事不拍板，后面的工作全建立在假设上

**阶段二：开发**
- 第一章必须经人验收，确认视觉气质和节奏都对了，它才成为后续章节的基准
- 后续章节可以顺序做，也可以开多个子 Agent 并行写
- 每章写完都要逐项对照检查清单自检，不过关不准说"做完了"

**检查点 Audio（人工确认）**
- 决定要不要合成音频。要→走阶段三；不要→跳到阶段四用人声录

**阶段三：音频合成**
- 从每章口播文本提取合成清单，调 TTS 逐段生成音频文件

**阶段四：录屏**
- 浏览器全屏打开演示页，开自动播放模式，音频和画面天然对齐，录完裁头尾

### 2.2 上下文管理

核心问题：模型到底看到了什么。

链路跨四个 Phase，涉及多份复杂文档。如果启动时全灌进去，模型注意力被严重稀释。

**设计：信息按阶段分层加载**——每份文档只在指定阶段读取：

![[04_Inbox/学习/AI专栏/assets/harness-video/img_09.png]]

| 文档 | 读取时机 | 内容 |
|------|---------|------|
| SCRIPT-STYLE.md | Phase 1 | 文章 → 口播稿规则、平台变体、双层自检（形式/风骨/念出来） |
| OUTLINE-FORMAT.md | Phase 1 | outline.md 字段 spec、命名约定、章节切分、信息池规则 |
| CHAPTER-CRAFT.md | Phase 2 每章 | 十条原则 / 开工5问 / 关系→动作决策树 / 视觉工具箱 / 时长参考 / 反AI味反模式 / 代码硬规则 / 完工自检 / 反馈速查 |
| THEMES.md | 选/造/切主题时 | 完整 token 契约 + 内置主题清单 + 创作新主题流程 |
| AUDIO.md | Phase 3 | MiniMax CLI TTS 合成流程、增量合成、故障排查 |
| RECORDING.md | Phase 4 | 录屏工具推荐、Auto 模式一镜到底、后期合成 |

好处：模型在任何时候面前摆的都是它最需要的信息，不被无关内容干扰。

### 2.3 状态与记忆

核心问题：系统如何跨步骤保持连续性。

![[img_10.jpg]]

最重要的"记忆"是 **outline.md**（开发大纲）。一个项目十几章、上百步，Agent 写到后面章节时，前面定过什么结构、用户在 Checkpoint 改过什么方向，很可能已经不在当前上下文里了。大纲把这些关键决定固定下来，成为开发阶段的持续记忆。

开发阶段 Skill 还强制 Agent 每轮关注三份文件：

- **outline.md（开发大纲）**：管结构边界——Agent 开发任何一章时，大纲就是任务边界
- **script.md（口播稿）**：管叙事节奏——先讲什么后讲什么，画面必须跟着口播走
- **article.md（原文章）**：管信息密度——口播稿更精简，画面里需要的数据案例从原文补

三份文件各有分工，Agent 同时参考它们，就不容易出现"结构对了但画面空"或"信息很多但节奏乱"的问题。

**本质上，这是一种文件化的工作记忆：把关键状态写进文件，需要时再读回来。**

### 2.4 工具系统

核心问题：模型到底能做什么。

这个 Skill 没有用特殊工具，核心就是让模型把 Agent 自带的文件读写工具用好。

关键设计——**多章节并行隔离机制**：

![[04_Inbox/学习/AI专栏/assets/harness-video/img_11.png]]

- **每章独立文件夹**：`src/chapters/01-intro/`、`src/chapters/02-core/`……物理分离
- **每章独立 CSS 前缀**：`.intro-hero`、`.core-card`……不会抢同一个 class name
- **主题 token 兜底视觉统一**：颜色、字体、间距走全局 CSS 变量，即使每个 subagent 独立开发，最终画面在色彩和排版上不会跑偏
- **但风格不强求一致**：每章的动画、节奏、视觉演示方式允许不一样

### 2.5 约束与恢复

核心问题：出错了怎么办。

在这个 Skill 里，"出错"最常见的形式不是代码报错，而是**用户觉得不对**（"这章节奏太快了"、"动画太 AI 味了"、"信息太薄了"）。

Agent 本能反应是重做整章——但重做整章会把已经做对的部分也改掉。用户说"第 3 步节奏太快"，你重做整章，结果第 1 步和第 5 步原来很好的效果也变了。

![[04_Inbox/学习/AI专栏/assets/harness-video/img_12.png]]

**关键原则：反馈修复的最小切片**——先定位问题在哪一层，再只改最小切片：

![[04_Inbox/学习/AI专栏/assets/harness-video/img_13.png]]

| 问题层 | 最小切片 |
|--------|---------|
| 节奏（太快/太慢） | 调整对应 step 的 narration 文本长度，或拆/合 step |
| 视觉（动画不对） | 只改对应 step 的 CSS / 动画逻辑 |
| 内容（信息太薄/太密） | 回 article.md 信息池补充或删减，只改对应 step |
| 代码（编译报错） | 定位具体文件和行号，点修 |

### 2.6 评估与观测

核心问题：系统怎么知道自己做得对不对。

这是作者花心思最多的地方，直接对应 Anthropic 提出的核心问题——**自评失真**。让写代码的 Agent 自己评价"这章写得怎么样"，结果几乎一定是"还不错"。

**设计：硬性自检规则**——每个关键产出完成后必须走"自检 → 修复 → 再汇报"流程：

| 产出 | 自检清单 |
|------|---------|
| script.md（口播稿） | 口语化、B站风、无AI味、信息保留≥60%、念出来不别扭 |
| outline.md（开发计划） | 节奏规划（step数/时长/信息池）、无动画细节、与script时间误差<10% |
| CHAPTER-CRAFT.md（单章实现） | 有动图、不套模板、字大留白、逐个揭示、素材真实、代码隔离 |

**执行方式（优先级从高到低）**：

1. **最优**：用 Agent Teams 开独立 Reviewer Agent——给它产出文件 + 对应清单，让它从零开始逐项核查
2. **次优**：没有 Agent Teams 就用 [[理论学习_SubAgent_子任务分发|SubAgent]]，走同样流程
3. **兜底**：当前 Agent 自己逐项核查——但必须严格逐项，不允许"目测一遍就放行"

### 2.7 回过头看

把这个 Skill 摊开来看，和 OpenAI、Anthropic 做的事本质上没有区别——都是在搭 Harness。只不过他们搭的是工业级的运行系统，我们搭的是一个 Skill 级别的协作协议。

**关键认知：做 Harness 不是非得从零搭建一个 Agent，能用一个 Skill 把一个垂直的开发工作做好，也是在做 Harness。**

### 3.6 Agent Teams 和 tmux（可选）

小 Demo 一个会话够用。但 10 章 100 多步的项目，一个 Agent 从头写到尾太慢。

**Claude Code 两种多 Agent 能力**：

**SubAgent（子进程模式）**：

![[img_29.jpg]]

- 主 Agent 分派任务，子 Agent 在独立上下文完成后交回结果
- 子 Agent 互相不可见、不讨论
- 适合结果导向任务（并行开发多章节、一次性检查代码）
- 省 Token、易调度
- 类比：让三个人分别去三个城市办事，办完回来汇报

**Agent Teams（项目组模式）**：

![[img_30.jpg]]

- 组长 + 组员，每个组员独立会话，可互相发消息、共享任务列表
- 适合需要来回反馈的任务（质检：Reviewer 发现问题→Developer 修改→Reviewer 复查）
- Token 开销更高（每个组员是完整会话）
- 类比：把三个人拉进同一间会议室，边讨论边改方案

![[img_31.jpg]]

开启 Agent Teams：
```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

配合 tmux 可视化（每个组员出现在不同终端面板）：
```bash
brew install tmux
```

## 四、实战：完整流程演示

以《一封邮件发出后的 600 毫秒》这篇文章为例。

### 4.1 启动 Claude Code

```bash

```

### 4.2 先生成脚本和大纲

把原始文章丢给 Claude，告诉它使用 Skill 完成第一步：

```
读取 @article.md ，按照 /web-video-presentation 要求，完成第一步。
注意编写完成后使用 Agent Teams 创建两个独立 Agent 进行质检。
```

![[img_36.jpg]]

流程：
1. 创建 Content Writer Agent 编写口播稿和开发大纲

![[img_37.jpg]]

2. 创建两个 Reviewer Agent 分别质检

![[img_38.jpg]]

3. 主 Agent 根据评审结论校对、输出最终版本

![[img_39.jpg]]

产出两个文件：
- **script.md**：口播脚本——句子更短、节奏更快，听起来像真人聊天而不是念文章

### 4.3 第一次人工确认

Agent 停下来询问几个关键问题：

![[img_42.jpg]]

1. **稿子和大纲要不要改**：AI 味重就调整成自己讲话风格
2. **视觉主题选哪个**：Skill 内置 10 套主题——`paper-press`（编辑纸媒/温暖印刷）、`warm-keynote`（现代演讲）、`midnight-press`（深色编辑）、`blueprint`（技术绘图）、`chalk-garden`（教室黑板）、`terminal-green`（磷光终端）、`bauhaus-bold`（几何宣言）、`sunset-zine`（独立杂志拼贴）、`newsroom`（报纸媒体）、`monochrome-print`（克制字体印刷）
3. **素材怎么准备**：从已有目录挑选 / 用户自己提供 / 全部 placeholder
4. **后续开发模式**：
   - A（默认·逐章确认）：每章做完暂停验收
   - B（首章验收后顺序开发）：第 1 章验收后 2~N 顺序做
   - C（首章验收后并行开发）：第 1 章验收后用 SubAgent 并行做 2~N

## 五、关键结论与核心价值

表面上是"文章生成知识讲解视频"的流程，**本质上是一次 Harness 实践**。

- 模型已经很强：Claude Code 能写代码，MiniMax 提供稳定推理/生成/语音合成能力
- 真正的问题不是"模型会不会做"，而是**怎么把这些能力编排起来让它稳定地做完**
- 这个 Skill 做的事：把复杂内容生产任务拆成**有流程、有状态、有检查点、有自检、有恢复机制**的工程系统
- 不是让 Agent 自由发挥，而是给 Agent 搭了一条**可重复执行的轨道**

**Harness 最核心的价值：把模型能力、工具能力和人的判断，组织成一条稳定可控的生产流程。**

## 六、延伸思考：对数仓开发的启发

这套 Harness 设计思路可以直接迁移到数仓开发场景：

1. **执行编排**：ETL 开发流程同样可以切阶段——需求澄清 → 表设计 → SQL 编写 → DQC 校验 → 人工验收 → 上线调度
2. **上下文管理**：不要一股脑灌所有文档给 Agent，按阶段加载（需求文档只在需求阶段读，血缘关系只在开发阶段读）
3. **状态与记忆**：项目的指标口径定义、表间血缘关系就是"文件化的工作记忆"，跨多表开发时保持一致性
4. **工具系统的隔离**：多表并行开发时，每张表独立 SQL 文件、独立 DQC 规则，物理隔离避免冲突
5. **约束与恢复的最小切片**：数据质量问题定位到具体字段/分区修复，不要推倒重来整张表
6. **评估与观测的自评失真**：写 SQL 的 Agent 自己验收大概率"没问题"，必须开独立 Agent 用 DQC 清单逐项核查

核心方法论：**"发现问题 → 封装 Skill → 可重复执行"的闭环**，与数仓开发"临时取数 → 建模固化 → 接入调度自动跑"同构。
