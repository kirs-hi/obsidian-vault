### 逐维度详评

#### 维度 1：Frontmatter 质量（9/10，加权 7.2）

**优点**：name 符合规范（`procurement-data-analysis`）；description 完整覆盖了"做什么"（采购取数分析）、"给谁用"（erp-catebi 项目组成员）、"怎么用"（9项核心能力枚举）、"触发词"（11个中文关键词）；metadata 信息完整（creator/updater/version/source/skill_id/high_sensitive 全部填写）。

**扣分点**：description 约 380 字符，虽未超限但信息密度偏高，11 个触发词可精简（"采购分析"和"采购数据"语义重叠）。

---

#### 维度 2：工作流清晰度（7/10，加权 10.5）

**优点**：六阶段流水线结构清晰（阶段-1 → 0 → 1 → 2 → 3/3.5 → 4/5），有完整的"执行流程总览"图；每个阶段内部有 Step 编号（如阶段0 的 Step 0-4，阶段1 的 Step A-F）；analysis_blueprint 贯通机制有明确的数据流方向描述。

**扣分点（-3分）**：2003 行单文件中，阶段与阶段之间的边界不够清晰——阶段0 的 SSO Cookie 修复代码（约 200 行 Python）夹在权限预检流程中间，打断了主线叙事。阶段 3.5 末尾的"强制衔接规则"（🔒9 对应执行点）和"HTML 报告用户主动触发规则"（P3-A）逻辑交叉，LLM 执行时容易混淆"自动触发"和"用户主动触发"两条路径的边界。部分 Step 缺少明确的输入/输出声明（如 Step C+D 的输出是 analysis_blueprint 对象，但没有明确标注"本步输出"）。

#### 维度 7：整体架构（5/10，加权 7.5）— 最大短板

**这是全 Skill 得分最低的维度。**

扣分理由（-5分，严重）：

**(1) 单文件 2003 行，认知过载**。SKILL.md 承担了太多角色——它既是设计文档、又是执行 prompt、还是 Python 代码仓库（内嵌了约 400 行 Python 代码块）。LLM 加载全文后，前段的🔒约束在后段的执行中遗忘概率极高。darwin-skill 的评分标准明确要求"不冗余不遗漏"，当前状态是"不遗漏但极度冗余"。

**(2) Prompt 约束与代码防线的职责重叠**。10 条🔒约束用自然语言写在 SKILL.md 顶部，但其中至少 5 条已经在 procurement_utils.py 中有代码实现（LIMIT检测、行权限兜底注入、SQL安全校验、配置文件校验和、取数结果验证）。SKILL.md 花了大量篇幅"教 LLM 做代码已经保证的事"，既增加了文件长度，又没有增加安全性。

**(3) 环境基础设施逻辑与业务分析逻辑未分离**。SSO Cookie 修复（200行）、CatClaw/CatPaw 路径分支、代理白名单自检——这些和"采购数据分析"完全无关的基础设施代码，与品类映射、SQL模板、洞察报告混在一起。

**(4) 预测层"半移除"状态**。文档多处标注"预测层已移除"但又保留了相关指导文本，造成认知噪声。

**(5) 与 CatPaw 生态的一致性**。2003 行远超 CatPaw Skill 的典型体量（通常 200-800 行），文件大小 111.8KB 超过了常规 SKILL.md 的 10 倍以上。

**保留分理由（5分）**：六阶段流水线的整体设计合理，阶段划分边界清晰；analysis_blueprint 贯通机制是优秀的架构抽象；三层分析蓝图（宏观/中观/微观）的决策框架完整。

推演流程：阶段-1 执行 bootstrap.sh → 静默通过 → 读 data-config.json → 老用户跳过阶段0 → 阶段1 快捷通道判断（时间"上个月"明确✓、品类"IT硬件"明确✓、分析目的"采购金额"明确✓）→ 三条件全满足直接跳过澄清 → Step C 推断 analysis_purpose = "采购金额趋势分析"（COST标签）→ Step D 生成蓝图（模板0+模板1+模板2+模板E）→ Step E 品类级联展开（"IT硬件"→查表A模糊匹配→假设唯一匹配→直接使用）→ Step F 输出确认单 → 用户确认 → 阶段2 构建 SQL（含行权限+全局状态过滤）→ 取数 → 阶段3 执行模板0-1-2-E → 阶段3.5 洞察报告 → 自动衔接阶段4→5 → HTML报告 → 大象发送。

**T2："分析一下广告品类的供应商集中度和集采占比"（多意图+品类歧义）— 7/10**

推演流程：阶段1 Step A 识别"广告"→品类映射 → 查表A `LIKE '%广告%'` → 假设返回多条匹配（一级"广告营销"、二级"广告物料"、三级"户外广告"等）→ P1-E 规则触发，按匹配数量分档展示 → 用户选择后 → Step B 无时间歧义（默认近3月）→ Step C 多意图合并：含"供应商集中度"→SUPPLIER + 含"集采占比"→COMPLIANCE → analysis_purpose = "供应商集中度风险评估 + 集采合规率分析（综合分析）" → Step D 蓝图生成（模板0+模板3+模板4+模板E+模板2）→ 确认 → 取数 → 多维分析 → 洞察 → HTML。

潜在问题：多意图合并后的 `meso_modules` 可能包含 5+ 个模板，每个都需要独立取数，总耗时可能 15-30 分钟。SKILL.md 没有对"模板数量上限"做约束。

**T3："外卖业务最近采购了多少钱"（部门探查+时间歧义）— 6/10**

推演流程：阶段1 Step A 识别"外卖业务"→触发部门探查流程 → 执行三层 UNION ALL SQL → 假设美团-1 层匹配到"美团外卖"→只有1条→直接确认 → Step B 检测"最近"为时间歧义 → 反问"默认近3个月，是否确认？" → 用户确认 → Step C 推断 analysis_purpose → 继续。

潜在问题：部门探查 SQL 本身也需要通过 `mtdata bi run` 执行（约1-2分钟），加上后续主查询，用户从输入到看到结果可能需要 5-8 分钟。中间只有"⏳ 正在查询部门信息，请稍候..."一条提示，反馈密度不足。新用户还要先跑完阶段0权限预检（额外3-5分钟），总等待可能超过10分钟。

**T1-T3 综合评分：(8+7+6)/3 ≈ 7/10**

## 四、优化建议（按 darwin-skill 策略库 P0-P3 排序）

### P0：效果问题（实测发现）

**P0-1：多模板取数串行导致长时间等待**

问题：T2 场景（多意图合并）可能产生 5+ 个中观模板，每个都需要独立 `mtdata bi run`（1-3分钟/次），总耗时 10-20 分钟。用户在对话窗口中只看到反复出现的"⏳ 正在查询数据..."。

建议：在 `procurement_utils.py` 中增加 SQL 合并优化——对于同一事实表B、同一时间范围、同一品类范围的多个模板，尝试用 CTE（WITH 子句）合并为一次查询，减少取数次数。或者在阶段2开始前，向用户展示"本次分析包含 N 个模块，预计总耗时约 X 分钟"的全局预估。

**P0-2：calc_health_score() 函数缺失**

问题：SKILL.md 和洞察 Prompt 模板多处引用 `calc_health_score()`，但 procurement_utils.py 中没有实现。LLM 干跑时会卡在"调用不存在的函数"。

建议：在 procurement_utils.py 中实现该函数，按 insight-prompt-template.md 中的 5 维加权规则（集采合规30%+供应商集中度25%+履约完成率20%+品类集中15%+长尾供应商10%）编码。

### P1：结构性问题

**P1-1：SKILL.md 模块化拆分（最高优先级结构改进）**

问题：2003 行单文件是维度7低分的根本原因。

建议拆分方案：
```
SKILL.md（≤400行）
  ├── Frontmatter（保留）
  ├── 强制约束🔒1-10（保留）
  ├── 用户信息呈现规范（保留）
  ├── 中文描述映射表（保留，高频查阅）
  ├── 核心数据表速查（保留）
  └── 各阶段入口："读取 phases/phase-X.md 执行"

phases/（新增目录）
  ├── phase-neg1-bootstrap.md
  ├── phase-0-permission.md
  ├── phase-1-clarify.md
  ├── phase-2-query.md
  ├── phase-3-analysis.md
  ├── phase-35-insight.md
  └── phase-45-delivery.md
```

预期效果：SKILL.md 降至 400 行，每个阶段文件 200-400 行。LLM 按需加载，减少上下文溢出导致的约束遗忘。

**P1-2：SSO/双端兼容逻辑下沉到代码层**

问题：`_handle_cookie_exception()`（200行 Python）和 CatClaw/CatPaw 路径分支与业务逻辑无关，不应出现在 Skill prompt 中。

建议：将全部 SSO 修复代码和双端分支逻辑移入 procurement_utils.py，SKILL.md 中只写"若遇到 CookieNotFoundException，调用 `handle_sso_recovery(mis_id)` → 返回 True 则等用户扫码后重试，返回 False 则阻断"。

### P3：可读性问题
