## 采购取数分析 Skill 设计方案分析

### 一、整体架构概览

这是一个面向美团采购业务人员（品类经理等）的**端到端数据分析自动化 Skill**，由作者  设计，已迭代到 V9 版本。整体流程为六个阶段的流水线：

```
阶段-1（环境安装）→ 阶段0（权限预检）→ 阶段1（需求澄清）→ 阶段2（取数）→ 阶段3/3.5（多维分析+洞察）→ 阶段4/5（可视化+报告交付）
```

核心数据源是一张采购全流程事实表 + 品类维表 + 品类经理维表，通过 `mtdata bi run` CLI 执行 Hive SQL 取数，最终输出单文件自包含的 HTML 可视化报告，并自动通过大象发送给用户。

---

### 二、亮点（设计精妙之处）

**1. 极致的"技术对用户透明"原则**

这是整个 Skill 最核心的设计哲学。它为采购业务人员（不懂技术的品类经理）构建了一层完整的"翻译层"：用户说中文（"广告三级品类有哪些"），系统自动映射到字段名、构建 SQL、执行查询、再翻译回中文。全程严禁暴露英文字段名、表名、SQL 语句、队列名、配置路径。甚至发明了"排查模式"作为有限度的技术信息开放通道，这个分级信息暴露机制设计得很成熟。

**2. 三层行权限过滤的精巧设计**

河图权限系统只返回用户有权看到的品类编码，Skill 将其拆分为 L1（一级通配）、L2（一二级通配）、L3（三级精确）三层，在 SQL 的 WHERE 子句中用 `IN` + `RLIKE` 前缀匹配组合实现层级展开，既保证了安全性（行权限注入不可绕过），又保留了灵活性（不同粒度的权限可混合使用）。`run_query_with_mtdata_bi()` 还做了"兜底注入"，即使 AI 忘记加权限过滤，函数层也会自动补上。

**3. analysis_blueprint 贯通机制**

用一个结构化 JSON 对象（宏观/中观/微观三层）贯穿需求澄清→取数→分析→洞察全链路，保证了"分析什么"和"怎么分析"在各阶段之间的一致性。蓝图一旦构建就只允许追加不允许覆盖，微观层是异常触发后动态追加的，这个设计模式很像数仓里的"执行计划"概念。

**4. 多层防御的安全体系**

安全设计是全 Skill 最厚重的部分，形成了四道防线：配置文件 SHA256 校验和（防篡改）→ SQL 参数消毒（防注入）→ AI 幻觉拦截（`verify_query_result()` 检测模拟数据标记）→ 审计日志（所有取数操作留痕，SQL 只记哈希）。还有"全局状态过滤"（过滤已关闭/待提交/已取消订单），这个业务规则被提到了"强制约束"级别，确保数据口径全局统一。

**5. HTML 报告的单文件自包含方案**

ECharts 5.4.3 和 marked.js 通过 Python 直接内嵌到 HTML 文件中，零 CDN 依赖，内网离线完全可用。报告分五个区块（核心洞察卡→图表→洞察分析→追问建议→SQL留痕），SQL 留痕区用双层折叠默认收起，供产研排查数据问题时展开查看。这解决了美团内网环境下 CDN 不可达的实际痛点。

**6. SSO Cookie 过期的自主修复流程**

面对 CatClaw 沙箱环境的 SSO 问题，设计了 5 步自主修复流程：代理白名单自检→CDP 直连 Chromium 导航→PIL 非白像素验证二维码渲染→S3Plus 上传截图到大象→降级提示。还特别标注了 `3020002` 错误的根因分析（服务端换票 IP 白名单问题），说明作者在实战中被这个问题坑过并深入理解了原因。

**7. 品类多条匹配的优雅交互（P1-E）**

按匹配数量分四档处理：1条直接用、≤5条树形展示让用户选、6~20条按层级分组摘要、>20条提示缩小范围。这是真正面向业务用户的"人话"交互设计，比直接 dump 一大堆编码优雅得多。

**8. 采购健康度评分模型**

5维加权评分（集采合规30%+供应商集中度25%+履约完成率20%+品类集中15%+长尾供应商10%），0-100分 ABCD 四级评级。这给了管理层一个"一眼看懂"的数字，把复杂的多维分析浓缩成了一个直觉指标。

### 三、暗点（设计隐患与可改进之处）

**1. SKILL.md 单文件 2003 行，认知过载严重**

这是最大的结构性问题。所有逻辑（环境安装、权限预检、需求澄清、取数、分析、可视化、报告交付、SSO 修复、管理者命令集……）全塞在一个文件里，LLM 的上下文窗口必须一次性加载全部内容才能正确执行。实际上 LLM 很可能在中后段丢失前面的约束（尤其是否定性约束，如"严禁"），作者自己在 memory 里也记录了"否定性约束最易丢失"。建议按阶段拆分为多个子文件，SKILL.md 只保留总控流程和强制约束。

**2. 过度依赖 LLM 的"自律"来执行程序逻辑**

很多关键逻辑（如 analysis_blueprint 的构建规则、品类字段选择决策树、SQL WHERE 条件组装）是用自然语言描述的，依赖 LLM "理解并遵守"。但 LLM 不是确定性执行引擎，复杂规则越多、越容易出错。比如"三层权限过滤子句拼接"这种事，与其写一大段规则让 AI 自己拼 SQL，不如直接封装在 `procurement_utils.py` 的函数里由代码保证正确性。虽然 `build_row_filter_clause()` 已经做了兜底，但 SKILL.md 里还是花了大量篇幅教 AI "如何自己做"，形成了双轨冗余。

**3. 强制约束（🔒1-10）的执行保障不够硬**

10条强制约束用 emoji + 表格写在文件顶部，本质上是"提示词级别"的约束。在 2003 行的巨大 prompt 中，LLM 能否在每次生成 SQL 时都记住"严禁 LIMIT"（🔒2）、"严禁 mtdata query"（🔒1）？从工程角度，这些约束应该尽量"代码化"——比如 `run_query_with_mtdata_bi()` 已经做了 LIMIT 检测，那 SKILL.md 里就不需要再写一遍让 AI 自律了。每条约束都应有对应的代码防线，而不是仅靠 prompt。

**4. CatClaw/CatPaw 双端兼容增加了大量条件分支**

整个 Skill 需要同时兼容 CatClaw（云端沙箱）和 CatPaw Desk（本地客户端），导致路径解析、浏览器操作、文件上传、SSO 认证等处都有 if/else 分支。SSO Cookie 修复那段代码（Step A-E，约200行）几乎只在 CatClaw 环境下有意义。这种双端兼容的复杂度既增加了 LLM 的理解负担，也增加了维护成本。建议将环境差异完全封装到 `procurement_utils.py` 中，SKILL.md 层面只暴露统一接口。

**5. "管理者特权"设计的安全边界模糊**

管理者（`weijianzhong`）享有跳过全部权限预检、全量查询、直接修改配置等特权。但管理者白名单是硬编码在代码里的（`is_admin_user()` 函数），如果管理者变更需要改代码。更重要的是，管理者的 `admin_query_all()` 跳过了行权限注入，这意味着一旦管理者身份被伪造（比如环境变量 `$USER` 被篡改），就能绕过全部安全防线。

**6. 取数链路是同步阻塞的**

`mtdata bi run` 是同步阻塞调用（SKILL.md 明确写了"无法中途轮询"），对于复杂查询可能需要 5-15 分钟。在这期间 AI 只能等待，用户体验是"死等"。虽然设计了预计时长提示，但没有异步轮询、进度反馈机制。如果查询失败需要重试，又是一次完整的等待周期。

**7. 洞察报告的"数据真实性铁律"与实际执行的张力**

🔒3 约束"严禁大模型推断/估算/虚构"，但阶段 3.5 的洞察报告本质上就是让 LLM 基于数据做"诊断层"分析（因果推断）和"规范层"建议（行动建议）。因果推断和行动建议天然带有 LLM 的推理成分，与"严禁推断"存在概念上的张力。这条约束实际上只对"数字"有效（数字必须来自取数结果），但对"分析判断"的边界并不清晰。

**8. 版本管理简陋**

HTML 报告用"同一 topic + 同一日期最多6个版本，已满则覆盖最旧版本"的策略。没有完整的版本历史、diff 对比、回滚能力。在追问场景下，用户可能想对比"加了新维度前后"的报告差异，当前设计不支持。

**9. 测试/验证体系缺失**

2003 行的 SKILL.md + 1390 行的 utils + 870 行的 SQL 模板，没有看到任何测试代码。`verify_query_result()` 只检测"是否像模拟数据"，没有对 SQL 模板的正确性、权限注入的完整性、报告渲染的准确性做自动化验证。

**10. 预测层的"半移除"状态**

文档多处提到"预测层已从 HTML 报告移除（大模型预测准确度偏低）"，但又说"SKILL.md 中保留预测层分析指导供 AI 内部参考"。这种"留着但不用"的设计容易造成混淆——如果 AI 在某次推理中不小心参考了预测相关的指导并输出了预测性判断，就违反了规则。要么彻底删除，要么明确隔离。

---

### 四、总结评价

这是一个**工程完成度极高、业务理解极深**的 Skill，从 V1 迭代到 V9 可以看出作者在实际使用中不断填坑的痕迹（SSO 修复、品类多匹配交互、排查模式等都是实战经验的沉淀）。它在"让非技术用户无门槛使用数据"这件事上做到了很高的水准。

主要改进方向集中在**工程化**层面：将 prompt 约束代码化（减少对 LLM 自律的依赖）、将大文件模块化拆分（降低认知过载）、补充自动化测试、以及更干净地隔离环境差异。
