## 一、全景概览：数据开发 Skills 到底在解决什么问题

数据开发的日常工作本质上是一条很长的流水线：需求分析 → 找表 → 建模 → 写 ETL → 测试 → 上线 → 运维。这条流水线的痛点不是某个环节难，而是**环节太多、平台太散、重复劳动太多**。万象、起源、探数、数据地图、河图、XT、BI——每个系统都有独立的 Web 界面，一个简单的"查这张表是怎么产出的"就要跨 4 个平台。

AI Skills 的核心价值不是"替代人写 SQL"，而是**把割裂的平台操作串成一条自动化流水线**，让数据工程师从"操作员"变回"设计师"。

### 1.1 技术架构：四层分层

```
用户自然语言请求
        ↓
   AI Agent（理解意图、编排流程）
        ↓
   场景编排层（Skill）—— 如 ETL 测试、ETL 上线、SLA 巡检
        ↓
   工具层（CLI）—— mtdata 命令行，统一封装各平台 API
        ↓
   平台层（API）—— 万象、起源、探数、XT、数据地图...
```

关键设计原则：Skill 负责"知道怎么做"（流程编排），CLI 负责"能做"（工具能力），AI 负责"理解要做什么"（[[01基础_18意图识别与多路由调度策略|意图识别]]）。三者解耦，各自可独立迭代。

## 二、深度解析：mt-data-tools 数据工具包

> 来源：[数据工具包 mt-data-tools](https://km.sankuai.com/collabpage/2752403099)

### 2.1 它解决了什么

一句话：**给 AI 提供数据平台的统一入口**。以前查一张表的血缘要跨 4 个平台，现在一句"帮我看看这张表的上游依赖"就搞定。

### 2.2 能力矩阵（11 个命令域）

| 命令域 | 子命令 | 覆盖平台 |
|--------|--------|----------|
| xt | 任务创建/编辑/测试/上线/配置调度 | 万象 XT |
| origin | 指标/维度/模型/数据集 CRUD | 起源 |
| query | SQL 提交/结果获取/项目组管理 | 探数 |
| table | DDL/字段/分区/权限/搜索 | 数据地图 |
| lineage | 上游/下游/字段级血缘 | 血缘系统 |
| task | ETL 任务状态/依赖/SLA | 万象调度 |
| metric | 指标搜索/详情/维度查询 | 起源 |
| hetu | 表权限申请 | 河图 |
| quality | DQC 规则查询/创建/修改 | DQC |
| bi | BI SQL 查询 | 魔数 BI |
| nau | [[实时数仓]]操作 | NAU |

### 2.3 对我的启发

mt-data-tools 的设计思路和我之前理解的"数仓调度心跳"是同构的——都是**一个统一入口 + 多个原子命令 + 上下文连贯**。关键不是单个命令多强，而是命令之间能串起来：查血缘 → 找到上游 → 看 DDL → 写 SQL → 提交探数 → 验证结果，这个链条在一次对话里就能跑通。

## 三、深度解析：SLA 自动巡检实践

> 来源：[用 AI Skills 告别 SLA 黑盒](https://km.sankuai.com/collabpage/2754540222)（匠心奖）
> 作者：公共数据基建 · 数据开发工程师
> 所用 Skill：data-dev-[[sla]]-guarantee

### 3.1 核心痛点

CIM 项目组上百个核心任务，SLA 保障的核心矛盾是：**问题在破线前很久就埋下了，但没有机制感知**。

三个典型盲区：
1. **Buffer 悄悄缩水**：任务耗时从 2h 慢慢涨到 5.5h，直到某天上游多延迟 40min 就破线
2. **[[数据倾斜实践|数据倾斜]]但不失败**：一个 Task 跑 3 小时，其他 20 分钟就完了，任务不报错只是"慢"
3. **资源严重浪费**：200 个 executor，CPU 利用率 20%，月均浪费数千核时

### 3.2 自动巡检流程（7 步）

```
Step 1: 全量问题盘点 → 自动扫描所有 CIM 任务，1 分钟出清单
Step 2: 获取 Application ID
Step E1: 上游卡点分析（对 [C] 类任务强制执行）
Step 3: 拉取 SparkUI 指标（shuffle_spill、max_task_duration、skew_ratio、cpu_util）
Step 4+5: 诊断分析 + 优化建议（结合 SparkUI 指标 + SQL 代码）
Step 6: 自动提交优化参数 → XT 测试 → 评估效果
Step 7: 自动生成日报 → 推送大象群
```

### 3.3 效果数据

| 指标 | 以前 | AI Skills 后 |
|------|------|--------------|
| 总耗时 | 半天 | 一句话触发，10 分钟 |
| 问题盘点 | 手动翻监控，易遗漏 | 全量自动扫描 |
| 根因定位 | 手动翻 SparkUI | 自动拉取 + 并行诊断 |
| 风险预警 | 破线后才知道 | Buffer 不足提前暴露 |
| 参数优化 | 凭经验试错 | AI 出方案，一键验证 |

### 3.4 踩坑经验（重要）

**铁律：步骤间禁止 AI 判断，所有数据必须 100% 来自脚本输出。** 早期 AI 会"总结" SparkUI 指标，有时把 2.3GB 写成 23GB——这就是幻觉。解法是 AI 只做原样转述，不做任何推算。

**优化参数必须有人工确认节点。** AI 建议缩减 executor，但某任务月底数据量突增 3 倍，缩容后会 OOM。涉及参数修改的步骤必须暂停等人确认。

### 3.5 对我的启发

这篇文章的核心洞察是：**SLA 保障应该从"救火"变成"防火"**。用我之前学到的数仓调度类比——SLA 巡检 Skill 本质上是一个"预防性 DQC"，不是等数据出问题才检查，而是每天主动扫描异常信号。

zscore 算法检测 Buffer 缩水的思路很好，可以迁移到我们自己的任务监控中：不止看"是否破线"，还要看"趋势是否在恶化"。

## 四、深度解析：公共数据切换效率跃迁

> 来源：[将两年工作量压缩至一年](https://km.sankuai.com/collabpage/2753369895)（匠心奖）
> 作者：公共数据基建 · 数据开发工程师
> 所用 Skill：data-etl-infra-data-switch（核心入口）+ 6 个原子 Skill

### 4.1 背景

公共数仓 CIM 要做"总-分"架构切换，梳理出 1333 个待切换模型，按人力配置需要**两年**。单表切换人工流程：资产检索(30min-1h) → SQL 编写(1-2h) → XT 任务开发(2-3h) → 双写比对(3-4h) → 文档归档(1h) = 平均 2PD/表。

### 4.2 Skill 架构设计

**核心设计：1 个入口 Skill + 6 个原子 Skill**

```
data-etl-infra-data-switch（入口 Skill，流程编排）
  ├── data-dev-public-data-retriever（CIM 资产检索）
  ├── data-dev-table-coverage-analyzer（字段级血缘分析 + SQL 生成）
  ├── data-dev-xt-dual-write（XT 任务自动上线）
  ├── dual-write-field-consistency（双写一致性比对）
  ├── Wiki 规范检查
  └── 大象群推送
```

为什么要拆成 6 个原子 Skill？因为**单 Skill 超过 500 行 AI 就容易"偷懒"，选择性跳过某些约束**。这是非常实战的经验。

### 4.3 五步自动化流程

| 步骤 | AI 做了什么 | 人做了什么 | 耗时 |
|------|------------|-----------|------|
| 1. 触发 | 解析表名、识别业务线、创建 Wiki | 一句话指令 | 即时 |
| 2. 资产检索 | 查上游 ODS 血缘 → 查 dw_cim 候选表 → 按覆盖率排序 | 确认选择哪张表 | 5min |
| 3. SQL 生成 | 字段级血缘分析 → 生成复现 SQL | 确认 SQL 正确性 | 2min |
| 4. XT 上线 | 创建任务 → 上传代码 → 静态检查 → 测试 → 标注 → 调度 → 上线 | 审批 | 30-60min |
| 5. 双写比对 | 5 项并行测试 → 全字段 DIFF → 生成 Wiki 报告 → 推送大象群 | 查看结果 | 30-60min |

**总耗时从 2PD 缩减到 30-60 分钟。**

### 4.4 踩坑经验（极其有价值）

1. **长流程必须拆 Skill**：20+ 步骤的切换流程，单 Skill 上下文过长 AI 会发散。拆成细粒度 Skill，每个控制在 500 行内，加状态机和强制代码。

2. **Token 过长导致字段遗漏**：全字段比对 SQL 在字段数超过 100 时，AI 会自主选择部分字段比对。短期方案是让 AI 自主拆分批次，长期要对接 DQC 平台。

3. **业务口径翻译难题**：外卖即时履约业务，00:00-00:20 的退款订单要从昨日 GTV 中剔除，AI 没有适配这个口径，导致双写比对出现差异。**AI 较难翻译理解复杂业务口径，需要人工持续输入。**

4. **历史债务被动发现**：某视图字段名带空格，AI 执行比对时报错，反而暴露了历史债务。

### 4.5 对我的启发

这篇文章最大的启发是**Skill 架构设计**：

- **500 行铁律**：单 Skill 不超过 500 行，否则 AI 偷懒
- **4 个必须确认的节点**：资产选择、SQL 确认、测试用例配置、比对执行
- **8 条生死红线**：双写比对的 D-001 到 D-008，违反任一条结果无效
- **业务口径知识库**：AI 能跑流程但不懂业务口径，需要持续积累

这和我之前记忆里的"否定性约束最易丢失"是一致的——切换规范中的"不能做什么"比"要做什么"更容易被 AI 忽略，必须用强约束（红线机制、状态机）来保障。

## 五、深度解析：数据 API 端到端自动化

> 来源：[履约数据 API 助手](https://km.sankuai.com/collabpage/2757131229)（人气奖）

### 5.1 核心场景

从 Hive 表到线上数据接口的全链路自动化：一句话描述业务场景，实现零代码、零点击、分钟级交付数据 API。

### 5.2 技术架构

**纯 CLI 底层驱动 + LLM-Wiki 知识库架构**：不依赖 [[理论学习_MCP_协议与开放工具生态|MCP]]，用 CLI 封装所有平台操作，LLM 从 Wiki 知识库中获取业务规则和配置模板。

### 5.3 五阶段对话流程

```
阶段 1：需求理解 → 解析用户描述，识别数据源和目标接口
阶段 2：表探查 → 查 DDL、血缘、分区、数据质量
阶段 3：API 配置生成 → 根据知识库模板生成配置
阶段 4：自动部署 → 调用平台 API 创建/发布
阶段 5：验证 → 自动调用接口验证数据正确性
```

### 5.4 踩坑经验

1. **知识库模板比 Prompt 更稳定**：把 API 配置规范写进 Wiki 知识库，AI 从中检索比写进 Prompt 更可靠
2. **验证环节不能省**：自动部署后必须调用一次接口验证，否则配置错误无法及时发现
3. **支持的 API 类型要明确边界**：哪些能自动化、哪些需要人工介入，在 Skill 中要明确声明

## 六、深度解析：需求到上线完整链路工程化

> 来源：[SaaS 业务线离线数据开发提效之路](https://km.sankuai.com/collabpage/2757682792)（人气奖）

### 6.1 核心亮点

不是简单串联已有 Skill，而是构建了三层工程化体系：

### 6.2 三层架构

**6.2.1 业务线专属 Skill**

不同业务线有不同的建模规范、命名规则、口径定义。通用 Skill 无法覆盖这些差异，所以需要在通用 Skill 之上再叠一层业务专属 Skill，把业务线特有的规范、常用表、核心指标等知识注入。

**6.2.2 Hook 体系**

在 Skill 执行的关键节点插入 Hook（钩子），实现：
- **Pre-hook**：执行前检查前置条件（如表是否存在、权限是否就绪）
- **Post-hook**：执行后自动触发后续动作（如测试通过后自动提交上线申请）
- **Error-hook**：异常时自动降级或通知

### 6.2.3 自进化飞轮

```
执行失败 → 记录失败原因 → 反馈给 Skill → 优化流程 → 下次避免
```

每次执行产生的经验（成功模式、失败模式、业务口径）自动沉淀为知识库，Skill 越用越聪明。

### 6.3 三种使用模式

1. **全自动模式**：一句话描述需求，AI 从头跑到尾
2. **分步确认模式**：每个阶段暂停等人确认，适合新需求
3. **单步模式**：只执行某个特定步骤（如只做 ETL 测试）

### 6.4 对我的启发

Hook 体系和自进化飞轮是这篇文章最有价值的部分。Hook 机制本质上就是数仓 DQC 的 [[07-Agent|Agent]] 版本——在关键节点插入检查点。而自进化飞轮和我之前记忆里的"发现问题 → 写 Skill → 定时复用"闭环完全一致。

## 七、综合洞察：对我工作的启发

### 7.1 可以立即落地的

1. **安装 mt-data-tools**：作为底座 Skill，日常查表、查血缘、跑探数都用得上
2. **学习 SLA 巡检 Skill 的设计思路**：zscore 检测 Buffer 缩水、自动拉取 SparkUI 诊断，可以迁移到我们自己的运维场景
3. **500 行铁律**：以后写 Skill 单文件不超过 500 行，拆成原子 Skill 组合

### 7.2 需要团队推动的

1. **业务口径知识库建设**：AI 最弱的环节就是业务口径理解，需要人工持续输入并维护
2. **双写比对红线机制**：如果我们也有表切换场景，可以借鉴 D-001~D-008 红线设计
3. **Hook 体系**：在 Skill 关键节点插入检查点，提升自动化流程的可靠性

### 7.3 方法论沉淀

| 原则 | 说明 | 来源 |
|------|------|------|
| AI 只转述不推算 | 步骤间数据必须 100% 来自脚本输出，禁止 AI 判断 | SLA 巡检 |
| 500 行铁律 | 单 Skill 超过 500 行 AI 会偷懒 | 公共数据切换 |
| 4 必须确认节点 | 资产选择、SQL 确认、测试配置、比对执行 | 公共数据切换 |
| 业务口径需人工输入 | AI 较难翻译理解复杂业务口径 | 公共数据切换 |
| 知识库 > Prompt | 把规范写进知识库比写进 Prompt 更稳定 | API 自动化 |
| Hook 机制 | 在关键节点插入检查点 | 完整链路工程化 |
| 自进化飞轮 | 失败经验自动沉淀为知识库 | 完整链路工程化 |
