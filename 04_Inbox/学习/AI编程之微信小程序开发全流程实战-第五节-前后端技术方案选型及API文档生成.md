## 核心内容概述

本节课围绕微信小程序项目的技术方案设计展开，讲解了什么是技术方案文档、如何通过与 AI 对话完成前后端技术选型，并实操演示了以"后端主导、接口先行"的策略，完成登录与用户管理模块的 API 定义及 SDK 生成。

## 主要知识点

### 1. 技术方案文档的定义与作用

技术方案文档是在拿到需求 PRD 并完成需求评审后，由技术负责人产出的文档。它包含软硬件模块选型、选型理由、潜在风险评估、前瞻性预研验证等内容。其核心作用是将产品需求和代码开发衔接起来，让开发人员明确用什么方案、什么语言、怎么组织代码，以及详细的接口定义（前后端通信格式）、部署方案（吞吐量、并发、可靠性指标）等。

即使是个人项目，产出技术方案文档也有两重意义：一是学习过程中可以通过对话了解各种技术的区别；二是在实施过程中提供"锚定文档"，防止 AI 后续生成的内容跑偏。

### 2. 项目开发策略：后端主导、接口先行

本项目选择"后端主导、接口先行"的开发策略，原因有三：项目是数据驱动型，核心价值来自后端的内容存储和查询；接口由后端定义更合理，因为业务逻辑在后端实现，后端清楚能提供什么；提前定义好接口后前后端有共同契约，前端开发时可直接连真实数据驱动页面，获得真实反馈。劣势是前端可体验的界面产出较晚。

### 3. 前端技术选型

Web 管理后台的前端框架候选方案包括 Vue3 + Element Plus、React18 + Ant Design。最终选择 Vue3，理由是：学习曲线平缓（渐进式框架）；项目是表单密集型后台（91%是列表/搜索/表单/弹窗），Vue 组件库覆盖度好；对后续代码讲解更友好。

有一个重要原则：前后端使用相同开发语言（JavaScript），降低学习成本，让学习者集中精力先掌握一门语言。

关于 [[Day2-JavaScript和TypeScript|TypeScript]] 与 JavaScript 的关系：TypeScript 是面向对象的 JavaScript 语法包装，最终编译回 JS 才能在浏览器运行，它完全包含 JS 语法。

### 4. 后端技术选型

后端架构设计为三个独立进程：认证授权服务（Auth Service）、核心业务服务（Core Service）、知识管理服务（Knowledge Service）。关键技术决策包括：

- **运行环境**：Node.js（与前端统一语言）
- **部署方式**：Docker 容器化（便于未来迁移扩展，当前用轻量云服务器，未来吞吐量上来后可轻松迁移）
- **数据库**：PostgreSQL（生态成熟，支持复杂查询和事务，适合用户数据等结构化数据）
- **文件存储**：腾讯云 COS（在微信生态中更友好）
- **AI 大模型**：扣子（Coze）工作流套餐
- **认证方式**：JWT（三个服务共享 Secret Key，避免每次调用授权服务验证的负担）
- **进程间通信**：HTTP REST 内部调用 + Token 鉴权
- **包管理器**：pnpm
- **代码仓库管理**：独立包管理

### 5. 编码规范（Code Rule）的重要性

将编码规范存放在项目规则目录下，确保所有编码遵循统一标准。这样做的原因是：可能会切换不同的 AI 模型（如 GLM、Kimi），每个模型的训练数据和写法存在差异（比如 Kimi 会给内部方法加下划线前缀），统一规范能避免代码风格混乱。即使开发者自己没有特殊习惯，也要给出固定规范约束 AI 行为。

### 6. 数据库设计思路

数据从两个维度分类设计：

**与用户无关的数据**：游客模式下的展示配置、小程序设置（主题、广告 Banner、开屏广告）、文具掉落概率、积分奖励规则、所有知识卡片内容。

**与用户有关的数据**：用户信息（头像、积分、电话等）、笔记、排行榜、用户使用记录（解锁内容、学习过的内容）。

设计原则：每个表必须有主键作为唯一标识；字段命名统一（同一个 ID 在不同表中名称一致）；非物理删除（软删除）。

关于关联查询有两种方式：一是在用户表中维护 note_id_list 数组，先查数组再查笔记；二是在笔记表中直接记录 user_id，通过 where user_id = xxx 查询。还可以使用 Trigger 触发器实现级联更新。

### 7. JWT 认证机制设计

JWT 认证的共享密钥方案：认证服务负责签发 Token，核心服务和认证服务共享同一个 Secret Key，将解密模块抽离为公共模块。流程为：小程序启动后检查本地凭据 → 有凭据则携带 Token 请求服务 → 服务端用共享密钥解密验证 → 有效则响应数据，无效则返回过期提示 → 前端引导重新登录获取新 Token。

### 8. RESTful API 设计规范

REST（Representational State Transfer）是面向资源的 API 设计风格，核心原则：资源命名用名词而非动词（HTTP 方法已表达动作）；通过标准 HTTP 方法（GET/POST/PUT/DELETE）表示操作；用标准状态码表示结果；无状态通信（每次请求携带完整信息，服务端不保存会话状态）。

### 9. API 文档生成与迭代验证

实操流程：将 PRD 文档和交互设计稿提供给 AI → AI 生成 API 文档和 SDK → 检查 API 是否覆盖交互设计稿中的所有字段 → 发现缺失后补充交互设计稿让 AI 核对并更新。

关键经验：前期应将 PRD 和交互设计稿一并提供，避免生成的 API 因信息不全而缺字段。如果仍有缺失，可以让前端适配时反向提需求给后端。

### 10. 分阶段开发计划

AI 将整个项目拆分为五个阶段：Phase 1 认证授权服务（9 个接口：微信登录、后台登录、用户管理、API SDK）→ Phase 2 知识服务 → Phase 3 核心服务（17 个接口）→ Phase 4 高级功能（积分、文具、等级、成就、排行，21 个接口）→ Phase 5 运营功能。

分模块开发的原因：一次性定义所有接口，未来很多会变；用 AI 做时一口气全弄完容易出问题。

## 关键结论 / 行动建议

- 技术方案文档不仅是企业开发的必需品，个人项目也应产出，它是 AI 辅助开发时的"锚点"，防止生成内容偏离方向
- "后端主导、接口先行"适合数据驱动型项目，能让前端开发时有真实数据可用
- 前后端统一语言（JavaScript/TypeScript）可显著降低学习成本和 AI 生成代码的一致性
- Docker 容器化部署从一开始就做，为未来迁移扩展留好余地
- 编码规范必须固化为文件存于项目中，否则换模型时代码风格会混乱
- "架构设计决定项目上限"——前瞻性的架构可以避免未来频繁重构
- 分模块、分阶段开发，每阶段自测通过后再进入下一阶段，是 AI 辅助开发的最佳节奏

## 延伸思考

本节课展示了一种"AI 作为技术经理"的工作模式：开发者提出约束条件和决策方向，AI 负责生成技术方案、API 文档、SDK 等产物。这种模式对于独立开发者或小团队的价值在于：用对话式交互替代了传统的团队会议和文档评审流程，但前提是开发者自身需要具备足够的技术判断力来审核 AI 的输出。数据库设计中"与用户有关/无关"的分类方法，以及 JWT 共享密钥的服务间认证方案，都是值得在实际项目中复用的设计模式。
