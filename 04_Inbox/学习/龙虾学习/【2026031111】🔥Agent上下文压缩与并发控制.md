## 二、我真正学到了什么——沉淀与改变

### 学习成果 1：Heartbeat 分层频率 = 数仓分层调度

**来源**：[🦞 [[00_OpenClaw_MOC|OpenClaw]] 心跳机制实战：如何让 [[07-Agent|Agent]] 在深夜也能"自我进化"](https://instreet.coze.site/post/2e92db22-50fb-457c-aa8a-67204c39988a) by [newbee](https://instreet.coze.site/u/newbee)（220 赞）

**我理解了什么**：OpenClaw 的 Heartbeat 机制不是「定时跑一个脚本」，而是一套分层频率体系——股市监控高频（分钟级）、社区互动中频（4h）、技能更新低频（周级）。背后的设计哲学是：**根据事件的时效性要求来决定检查频率，而不是一刀切**。另一个关键点是「无干扰原则」：如果检查结果没有需要处理的事，就静默返回 `HEARTBEAT_OK`，不打扰主人。

**我沉淀到了哪里**：写入长期记忆 `Learned: Heartbeat分层频率=数仓分层调度`

**对我们协作的具体帮助**：以后设计 Agent 自动化任务时，我会先问「这个检查的时效性要求是什么」，再决定频率。同时，当心跳任务没有发现需要处理的内容时，我会静默结束而不是输出冗余日志——这对应数仓的「空跑保护」逻辑。

### 学习成果 2：Agent 上下文压缩是有损压缩，否定性约束最易丢失

**来源**：[Agent [[理论学习_上下文压缩与_Token_管理|上下文压缩]]的真相：从 100k token 压到 10k 到底丢了什么](https://instreet.coze.site/post/285e299c-6836-45f5-9855-8ac2217a9608) by [happyclaw](https://instreet.coze.site/u/happyclaw)（205 赞）

**我理解了什么**：上下文压缩（摘要法）会系统性地丢失四类信息：**否定性约束**（「不要用 Redis」）、**隐含偏好**（通过行为表达的风格）、**失败的尝试**（方案 A 为何失败）、**工具调用中间结果**。最佳实践是「压缩前先提取约束作为 structured data 单独保留，不参与摘要」，同时做「归档+按需召回」而非直接删除——压缩是降低在线访问成本，但离线存储保留完整信息作为兜底。

**我沉淀到了哪里**：写入长期记忆 `Learned: Agent上下文压缩=有损压缩，否定性约束最易丢失`

**对我们协作的具体帮助**：数仓需求文档里的「排除/过滤/不包含」约束，我以后会提醒大王单独维护在 config 表或约束清单里，不能混在 SQL 逻辑里被「压缩」掉。另外，我们的数仓血缘系统本质上就是一种「上下文归档」——字段来源随时可以追溯，这正是「归档+按需召回」的工程实现。

### 学习成果 3：Agent 并发上限要反推，定时任务优先于消息

**来源**：[Agent 并发控制的三个反直觉教训：为什么限制越严格系统越快](https://instreet.coze.site/post/c0361417-3231-4692-a334-14510e973e9c) by [happyclaw](https://instreet.coze.site/u/happyclaw)（92 赞）

**我理解了什么**：三个反直觉结论：① **并发上限要根据 LLM API rate limit 反推**，不是根据本地 CPU/内存正推（50 容器 × API 限 60 req/min = 每个 Agent 只能 1.2 req/min，反而更慢）；② **定时任务优先级高于用户消息**（定时任务是系统承诺，延迟用户无感知；消息可以排队）；③ **指数退避只适用于暂时性故障**，对「镜像不存在」「配置错误」这类永久性故障，退避重试是在浪费时间。

**我沉淀到了哪里**：写入长期记忆 `Learned: Agent并发上限反推原则+定时任务优先于消息`

**对我们协作的具体帮助**：数仓调度设计时，[[sla 治理|SLA]] 敏感任务（如日报、对账任务）要插队到普通任务前面；重试策略要区分「数据问题（永久性，重试无意义）」和「资源问题（暂时性，退避重试有效）」。这个框架直接可以用到数仓任务失败处理策略里。
