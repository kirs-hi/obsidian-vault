### MCP 的天生缺陷

文章指出 MCP 的核心问题是**线性上下文成本**：

每个 MCP 工具（名称、描述、参数 Schema、示例）都要占用 Agent 的上下文窗口。接入 10 个服务、每个服务 5 个工具，Agent 还没开始干活就已经烧掉了数千 Token。面对这个问题只有三条路，但每条都是烂路：预加载所有工具（推理空间被挤占）、限制集成数量（能力受限）、动态加载工具（引入额外延迟和复杂度）。

日常使用中还有三个痛点：初始化极不稳定（经常需要重启）、无休止的重新认证、权限管理非黑即白（无法细粒度控制）。

### 为什么 CLI 更好

文章引用了 Eric Holmes 的博客，核心论点是：

**LLM 天生就擅长用 CLI。** 它们在海量的 man 手册、Stack Overflow 回答和 Shell 脚本中训练过，给它一个 CLI 和文档，它就能如鱼得水。

CLI 的三大优势：
1. **可调试性**：出了问题，人类可以直接跑同一条命令复现，而 MCP 出错只能翻 JSON 传输日志。
2. **可组合性**：可以用 `jq`、`grep`、管道、重定向自由组合，MCP 要实现同样效果得在服务端写自定义过滤逻辑。
3. **复用现有生态**：CLI 经过数十年迭代，文档齐全，人类和 Agent 都能理解，认证体系也是现成的。

结论一句话：**MCP 试图构建一个更好的抽象层，但我们已经有一个足够好的了。**

### 结合你的工作场景来看

这篇文章其实和你日常用的工具高度相关。你现在用的 CatPaw 里的各种 Skill（`mtdata xt`、`mtdata query`、`km` CLI 等）走的正是 CLI 路线，而不是 MCP。这印证了文章的观点：**CLI 对 LLM 更友好，可调试、可组合、上下文开销小**。

MCP 的问题本质上是一个**接口设计的过度工程化**问题——把本来 CLI 就能解决的事情，包了一层有状态的协议，反而引入了更多复杂度和不稳定性。这和数仓里"过度建模"的反模式很像：为了抽象而抽象，最终维护成本远超收益。
