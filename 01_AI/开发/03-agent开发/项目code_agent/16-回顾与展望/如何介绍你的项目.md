# 三十秒 介绍

面试官翻到你简历上 MewCode 这一行，抬头看你，说「聊聊这个项目吧」。你有大概三十秒的时间让他决定要不要继续往下问。这三十秒说什么，比后面三十分钟的技术细节都重要 。

先给你一个可以直接用的版本，你根据自己的情况调整措辞就行。

「MewCode 是我独立开发的一个终端 AI 编程助手。用户用自然语言下指令，[[07-Agent|Agent]] 自己读代码、改文件、跑命令，整个过程自主完成，不需要人盯每一步。底层做了 五层架构 ，支持多家模型厂商的 API，有完整的权限控制和[[理论学习_上下文压缩与_Token_管理|上下文压缩]]机制，还支持多个 Agent 并行协作。我做这个项目的出发点是，每天都在用 [[Claude Code 命令与最佳实践|Claude Code]]，但它像个黑盒，想搞清楚 Agent 内部到底怎么决策的，就自己从零造了一个。」

![](如何介绍你的项目-1.jpeg)

这段话不长，但每句都有设计意图。

开头说「终端 AI 编程助手」加上「自主完成」，面试官马上能想象这东西长什么样、干什么用。如果你上来就说「 我用某某语言和某某框架做了一个终端应用 」，面试官脑子里画不出画面，注意力就散了。先让他看到产品，再聊技术。

中间提「五层架构」「多家模型厂商」「权限控制」「上下文压缩」「多 Agent 并行」，这些都是钩子。你不展开，但面试官听到任何一个感兴趣的都会追问，而你每个都有准备。如果你在这里就 把五层分别是什么全说了 ，面试官反而没东西可问了，对话就死了。

结尾说动机。「每天在用 Claude Code 但它像个黑盒」这句话特别重要，因为它回答了面试官心里一定会有的疑问：为什么要做这个？如果你的动机是「 想练手某个技术 」或者「课程作业要求的」，面试官对项目的兴趣会直接降一个档。但「想搞懂黑盒内部的决策机制」是一个技术驱动力很强的动机，说明你不满足于当用户，想理解底层。

几个容易犯的错误。

不要以「我参加了一个课程，然后...」开头，面试官不关心你怎么接触到这个项目的，他关心这个项目本身。

不要上来就列技术栈，一串语言名、框架名、协议名堆在一起不构成项目介绍。

绝对不要说 MewCode 是 Claude Code 的「克隆」或者「复制品」，这会让面试官觉得你只是在抄作业。

# 面试官听到这个项目的第一反应

说完电梯演讲，面试官的反应通常就那么几种。提前准备好怎么接，比临场发挥强十倍。

![](如何介绍你的项目-2.jpeg)

最常见的一种：「所以就是照着 Claude Code 做了一个一样的？」

这个问题几乎每个面试都会遇到，怎么接决定了整场面试的基调。慌了或者含糊过去，面试官就会给你贴上「做了个复制品」的标签，后面再怎么讲技术细节都很难翻盘。

你可以这样说：「功能形态是对标 Claude Code 的，但代码是参照它的架构理念独立实现的。Claude Code 本身闭源（ **目前是有泄漏一个开源快照，但是本质上不是开源的，这个版本迟早会过时** ），我的每一层都是从零写的。而且有些地方做了 Claude Code 没有的设计，比如我的 LLM Client 同时支持 Anthropic 和 OpenAI 两套协议，Claude Code 只接 Anthropic 自家的 API。」

这段话的核心逻辑是：承认参考对象，但立刻给出 差异化的事实。独立实现、多协议支持，这些都能证明 这不是简单的照搬。

第二种常见反应：「这个项目是你一个人做的吗？」

诚实回答：独立开发。但不要止步于此。你可以补一句：「开发过程中大量使用了 Claude Code 本身做辅助，等于我在用 AI 编程 Agent 来造 AI 编程 Agent。这个经历挺有意思的，让我同时从使用者和构建者两个角度理解了 Agent 系统。」

这样既诚实地说明了开发方式，又把 AI 辅助开发变成了一个独特的叙事优势。

第三种：「这个项目有实际用户吗？有在生产环境跑吗？」

不要心虚，也不要编。「MewCode 是一个学习型项目，目标是理解 Agent 架构，不是做一个生产级产品。它和 Claude Code 的差距不在架构层面，而在细节打磨上。Claude Code 有几千个边界场景的处理、有 LSP 集成、有 OAuth 鉴权流程、有官方 [[理论学习_MCP_协议与开放工具生态|MCP]] 注册中心，这些都是把一个架构原型变成可以卖钱的产品需要的东西。我能说清楚这个差距在哪，恰恰说明我理解从原型到产品之间的距离。」

面试官问这个问题不是要否定你的项目，是想看你有没有工程成熟度。能清晰地说出自己项目的边界，比号称「什么都有」但说不出细节要强得多。

# 项目介绍的叙事结构

电梯演讲之后，面试官通常会说「展开讲讲」。这时候你需要一个大概三分钟的叙事弧线，既能让面试官看到全貌，又能自然地把他引向你最想被追问的方向。

很多人在这一步会犯一个错误：把 五层架构 像目录一样列一遍。「引擎层负责 [[理论学习_ReAct_范式与_Agent_Loop|Agent Loop]]，工具层负责工具注册，安全层负责权限...」面试官听完会觉得你在背书，不知道该从哪里接话。

更好的方式是把架构讲成一个成长故事。MewCode 不是一次性设计出来的，它是一步步长出来的，每加一层都是因为上一层暴露了新的问题。

你可以这样讲。

「最开始我只做了一件事：接上 LLM API，让 AI 在终端里能说话。这一步很简单，但做完之后发现它跟一个聊天机器人没区别，问什么答什么，完全不能干活。」

「所以我给它加了六个工具：读文件、写文件、改文件、跑命令、找文件、搜内容。有了工具之后，它能做事了，但每次只能做一步，做完一步就停下来等我。」

「于是我做了 Agent Loop。一个循环，让模型自己判断要不要继续调工具，不需要调了就自动停。从这一步开始，MewCode 从聊天机器人变成了 Agent，你给它一个任务，它自己跑完。」

「但光有循环还不够。同样的模型同样的工具，加不同的 [[理论学习_System_Prompt_如何设计_|System Prompt]] 表现天差地别。所以我设计了一套系统[[提示词]]组装管线，让 Agent 知道自己是谁、该遵守什么规则、用什么语气跟用户交互。这一步决定了 Agent 不只是能干活，而是能干得像样。」

「但是一个能自己跑 Bash 命令的 Agent 是很危险的。我做测试的时候它真的尝试过 rm -rf，幸好在沙箱里跑的。所以我做了五层权限系统：危险命令检测、路径沙箱、规则引擎、权限模式、人机确认，任何一层拒绝就短路，不管模型怎么想都执行不了。」

「权限解决了安全问题，但跑长了又有新问题：上下文窗口撑不住。一个任务可能要调几十轮 LLM，每轮都带完整对话历史， token 涨得很快 。 所以我做了两层上下文压缩。第一层是大结果落盘，工具输出太长就写到磁盘只保留引用；第二层是自动摘要，对话历史超过阈值就让 LLM 压缩一遍。用户还可以手动触发压缩来精确控制成本 。 」

「最后，有些任务单个 Agent 搞不定，或者太慢。我做了多 Agent 协作：用 Git Worktree 给每个子 Agent 一个独立的工作目录避免文件冲突，用文件邮箱做异步通信，Coordinator Agent 专门负责拆任务和调度，自己不碰代码。」

这个叙事弧线有几个好处。它不是在列功能，是在讲问题和解法的因果链，面试官能跟上你的思路。每一层的出现都是因为上一层暴露了痛点，这说明你不是照搬别人的设计，而是真正理解每一层存在的理由。

而且你在讲的过程中已经丢出了好几个钩子：权限系统五层是哪五层？ 上下文压缩两层怎么分 ？Worktree 隔离具体怎么做？面试官会自然地挑一个追问，而你每个都有深度准备。

讲完架构全景之后，你可以主动收一下。「这大概就是 MewCode 的整体架构。我可以深入聊任何一层，技术上最有意思的大概是流式协议归一化、上下文压缩策略、还有多 Agent 的文件隔离机制。」

把选择权交给面试官，但你指定了菜单。

# 把 AI 辅助开发变成双重优势

有些同学听到「你是用 AI 写的代码吗」这个问题会紧张。不用。AI 辅助开发在 2025 年已经是行业标配，紧张反而显得你对这个趋势不适应。

你可以坦然地说：「我用 Claude Code 来开发 MewCode，等于是用 AI 编程 Agent 来造 AI 编程 Agent。」这句话本身就是一个很有记忆点的表述。

更重要的是，这个经历给了你一个别人没有的双重视角。你不光是 MewCode 的构建者，你还是 Claude Code 的深度用户。这两个角色的经验是互相印证的。

![](如何介绍你的项目-4.jpeg)

比如你在使用 Claude Code 的过程中，跑长了会发现输出质量明显下降，后面的回复开始忘记前面讲过的东西。这是用户侧的感知。但因为你同时在造 MewCode，你知道这背后是上下文窗口逼近上限，模型注意力被稀释了。

这个体感直接指导了你设计压缩系统时阈值的选择，你知道在什么节点用户会开始感觉到退化，就在那之前触发压缩。

再比如你在用 Claude Code 的时候，学会了一套高效的协作方法：先让它读完代码出方案，确认了再动手；每个子任务独立一轮；改完立即提交，方便回滚。这些「怎么给 AI 拆任务」的经验，其实就是你做 MewCode 的 Plan Mode 和 HITL 机制时想要支持的工作流。

如果面试官直接问「那代码到底是你写的还是 AI 写的」，你可以说：「具体到每一行代码，确实有很大比例是 AI 生成的。但架构决策全是我自己做的， 五层怎么分 、权限系统为什么五层、多 Agent 用邮箱不用 RPC，这些 AI 帮不了你选。AI 帮我加速了实现，但系统怎么设计、踩了坑之后怎么调整，这些判断力是在过程中积累的。」

# 简历上怎么写才能引出好问题

面试其实从面试官打开你简历那一刻就开始了。简历上每一行都是一个信号，它决定了面试官会问你什么。好的简历条目是「面试诱饵」，每一条都引向一个你有深度准备的话题。

先看一个反面例子：「使用某某语言开发了一个 AI 编程助手」。面试官看到这行，唯一能问的就是「 这个语言哪些特性你用到了 」，这是一个很无聊的问题，聊不出深度。

再看一个好的写法：「独立设计并实现 AI [[理论学习_什么是_Coding_Agent_|Coding Agent]]， 五层架构 覆盖 Agent Loop、工具注册、权限沙箱、上下文压缩和多 Agent 协作」。这一行里至少有五个追问点，面试官随便挑一个你都能展开聊三分钟。

给你几条推荐的简历 bullet，每一条对应一个你有深度准备的技术亮点。

「 设计两层上下文压缩策略（大结果落盘 + 自动摘要），支持 50+ 轮长会话稳定运行，压缩边界对齐 [[01基础_12理解函数调用Function Call|Function Calling]] 协议约束 」。这条对应上下文管理的亮点，面试官很可能会问「什么叫压缩边界对齐协议约束」，而你对这个问题有完整的答案。

「实现 五层权限拦截模型 （危险命令检测/路径沙箱/规则引擎/权限模式/HITL），支持三级规则优先级和 symlink 安全解析」。这条对应安全模型的亮点。

「基于 Git Worktree 实现多 Agent 文件隔离，配合文件邮箱异步通信和 Coordinator 调度模式，支持并行任务执行」。这条对应多 Agent 协作。

「抽象 Anthropic/OpenAI 两套流式协议为统一事件接口，新增 LLM Provider 只需实现一个 Parser」。这条对应协议归一化。

有一点要特别注意：简历上不要写「课程项目」。MewCode 是你独立设计和实现的，写「个人项目」就好。课程是你学习的途径，不是项目的属性。

# 常见翻车场景

最后说几个我见过最多的翻车场景，帮你提前避坑。

![](如何介绍你的项目-5.jpeg)

第一种是技术清单模式。有的同学介绍项目的时候就像在念购物清单：「我用了某某语言、某某框架、MCP 协议、JSONL 存储、Git Worktree...」。面试官听完不知道你在这些技术上做了什么决策，也不知道你遇到过什么问题。

每提到一个技术，都应该连接到一个决策。不是「我用了 JSONL」，而是「会话持久化我选了 JSONL 而不是 SQLite，因为 Agent 的写入模式是纯追加，JSONL 追加写不需要加锁，崩溃恢复也更简单，而且日志文件可以直接用 grep 排查问题」。

第二种是过度声称原创。面试官稍微搜一下就能知道 MewCode 的架构参考了 Claude Code。如果你表现得好像一切都是自己首创的，被戳穿的瞬间信用就崩了。正确的做法是大方承认参考对象，然后说清楚你在哪些地方做了不同的选择、为什么。诚实本身就是一种面试竞争力。

第三种是回答不了「为什么」。你说「我做了五层权限」，面试官问「为什么五层不是三层」，你愣住了。每一个你提到的设计决策，都要准备好「为什么这样做」和「还考虑过什么别的方案」。如果说不清楚取舍，面试官会怀疑这个设计不是你做的。

第四种是否认项目局限。面试官问「如果让你重做，有什么想改的」，你说「挺好的，没什么大问题」。这在面试官眼里是一个危险信号，说明你要么没认真想过，要么没发现问题。

准备两三个真实的改进方向：比如目前工具输出是等执行完才一次性返回，应该支持流式产出让 UI 能实时看到进度；比如多 Agent 场景缺少一个可视化的状态面板，调试时看不到各个 Agent 的进展；比如前期花了太多时间做架构设计，后来发现不如直接动手写、遇到问题再调整来得快。

第五种是对 AI 辅助开发感到心虚。面试官问「这些代码都是你自己写的吗」，你开始支支吾吾。不要这样。AI 辅助开发就是一种工具选择，跟你用 IDE 自动补全、用 Stack Overflow 查答案没有本质区别。坦然说出来，重点放在你做了哪些 AI 替代不了的判断，比如架构选型、安全模型设计、跨层问题的排查。
