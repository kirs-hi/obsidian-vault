### 简历里这个 Mewcode Coding Agent 项目，先用一两句话讲清楚它是什么？

考点：基本认知，能否清晰说明项目定位。

**参考回答** ：

Mewcode [[理论学习_什么是_Coding_Agent_|Coding Agent]] 是一个轻量级的终端 AI 编程助手。用户用自然语言下指令，Agent 自己完成多步任务，比如读文件、改代码、跑命令、修 bug，整个过程不需要人盯着每一步。功能上覆盖了主流 Coding Agent 的核心能力：常驻终端、[[理论学习_ReAct_范式与_Agent_Loop|Agent Loop]] 自主循环、调用工具完成任务、支持权限控制和会话记忆。我最初想做这个，是因为光用现成的 Coding Agent 总觉得它像个黑盒，不知道里面每一步到底怎么决策的，自己造一遍才能真正搞懂。

**追问** ：那既然 [[Claude Code 命令与最佳实践|Claude Code]] 已经有了，为什么还要再造一个？价值在哪？

答：就像刚才说的，从工程角度光看输入输出搞不清内部怎么跑的，自己造一遍对每一层的取舍才有发言权。再就是产品角度，Claude Code 闭源（ **目前是有泄漏一个开源快照，但是本质上不是开源的，这个版本迟早会过时** ），自研版本可以审计每一步行为，落地到企业内部更可控。

### 你觉得 Agent 有哪些必要元素？

考点：对 Agent 本质的理解，区分 Agent 和普通 LLM 应用。

**参考回答** ：

我做 MewCode 下来，觉得 Agent 的本质就四个东西。

首先得有个大脑，就是 LLM。它负责理解任务、做判断、决定下一步干什么。没有它 Agent 就是一堆写死的 if-else，碰到没见过的情况就傻了。

然后得有手脚，就是工具。光有大脑不行，LLM 只能想不能做。加了工具它才能读文件、改代码、跑命令，真正跟外部世界交互。而且是 Agent 自己决定该用哪个工具，不是我预先编排好流程让它走。

第三个是循环。拿最直观的对比来说，早期的 ChatGPT 网页版就是一问一答， 发一句它回一句就结束了 。Agent 是做完一步看看结果，再决定下一步，自己反复跑，直到它觉得任务搞定了才停。其实现在 ChatGPT 也加了 Agent Mode，能自主浏览网页、跑代码、多步推理，本质上也是往 Agent 方向演进了。

第四个是环境反馈。Agent 每执行一步都能拿到结果，跑了命令能看到输出，改了代码能跑测试看报错。这些反馈驱动着它下一轮的决策。没有反馈它就是闭着眼干活，不知道自己做的对不对。

这四个缺哪个都不行。没大脑不会想，没工具不会动，没循环干一步就停了，没反馈全程盲跑。

**追问** ：那安全机制和记忆不算必要元素吗？

答：严格来说不算。一个没有权限拦截和记忆的 Agent 也能跑起来完成任务。但从工程角度，缺了安全机制的 Agent 不敢放出去用，缺了记忆的 Agent 每次都要重新交代背景，体验很差。四要素解决的是 Agent 能不能跑的问题，安全和记忆解决的是敢不敢用、好不好用的问题。我做 MewCode 的时候就在四要素基础上加了五层权限拦截和跨会话记忆，这些是工程层面的增强。

### Agent 有什么工作模式？

考点：对 Agent 执行范式的理解，不只是背概念，而是知道什么场景用什么模式。

**参考回答** ：

主流的 Agent 工作模式有四种，MewCode 里面用到了三种。

最基础的是 ReAct，就是边想边做。Agent 每一步先推理当前情况决定做什么，然后调工具执行，拿到结果后再推理下一步，如此循环。MewCode 的 Agent Loop 底层就是 ReAct，模型看到用户需求后自己决定先读哪个文件、改哪段代码、跑什么测试，每一步都是根据上一步的结果来决策的。这个模式的好处是灵活，碰到意外能随时调整，但 token 消耗大，因为每一步都要完整推理一次。

第二种是 Plan-and-Execute，先想好再做。跟 ReAct 的区别是它把规划和执行分开了：先让模型把完整计划想清楚，然后按计划逐步执行，不用每步都重新思考全局。MewCode 的 Plan Mode 就是这个思路，进入 Plan Mode 后 Agent 只读代码不动手，出一个完整的实施方案，确认后再切到执行模式逐步完成。比 ReAct 省 token，因为规划只做一次，但不够灵活，执行到一半发现情况变了就得重新规划。

第三种是 Reflection，做完再检查。Agent 完成任务后不急着交付，先自我审查一遍，发现问题就修改再审查，直到满意。也可以用两个 Agent 来做，一个生成一个评审，来回迭代。MewCode 目前没有独立的 Reflection 模块，但实际上每次改完代码跑测试、看到报错再改，本质上就是一种隐式的反思循环。

第四种是 Multi-Agent，团队协作。任务太大一个 Agent 搞不定，就拆给多个 Agent 各自负责一块。MewCode 的多 Agent 协作就是这个模式，Coordinator 负责拆任务，Worker 各自在独立的 Worktree 里并行干活，最后汇总结果。

实际项目里这四种模式不是互斥的，MewCode 就是组合使用：每个 Agent 内部跑的是 ReAct，复杂任务前先用 Plan Mode 规划，多个 Agent 并行协作处理大规模改动。

**追问** ：你在 MewCode 里为什么没有单独做 Reflection 模块？

答：主要是投入产出比。独立的 Reflection 意味着每次任务完成后额外调一轮 LLM 来审查，token 成本翻倍。对 Coding Agent 来说，测试框架本身就是最好的审查机制，编译通过、测试绿了就是最硬的反馈，比让模型自己审自己靠谱。如果未来要加，我会考虑在特定场景触发，比如涉及安全相关的代码改动时自动加一轮审查，而不是每次都审。

### 你说Agent能自主完成多步任务，那Agent怎么知道该停了？万一它停不下来了呢？

考点：对AgentLoop终止条件和工程兜底的理解。

**参考回答** ：

这其实就是 [[01基础_12理解函数调用Function Call|Function Calling]] 协议决定的：模型每一轮返回里如果还带着工具调用，说明它觉得活没干完，循环继续工作；如果只回了纯文本、不调任何工具，就表示它觉得搞定了，循环停止。模型用「调工具」表达「我还要干活」，用「只说话不动手」表达「我做完了」。

但光靠模型自觉肯定不够，程序上必须有兜底。最基本的是给循环轮次加一个硬上限，到了上限不管模型想不想继续都强制退出。

再就是 token 用量监控，如果连续几轮 token 消耗越来越少，说明模型在原地打转没有实质进展，也会主动停掉。用户随时可以中断也是兜底的一部分。我做 MewCode 的时候遇到过模型陷在改代码、跑测试的循环里出不来，每轮都在改但就是修不对，靠轮次上限兜住的，从那之后我就明白终止条件不能只指望模型自己判断，程序上该拦的得拦住。

### 你在简历里说独立完成整体架构设计，能简单讲一下分了哪几层、为什么这么分？

考点：架构设计能力。

**参考回答** ：

整体分了五层。最核心的是引擎层，Agent Loop、对话管理、LLM Client 都在这里，相当于大脑。引擎往外伸出两只手就是工具层，内置工具加 [[理论学习_MCP_协议与开放工具生态|MCP]] 工具都挂在这。再往外是交互层负责 CLI、配置、TUI、命令解析和 Skill 系统，安全层负责危险命令检测、路径沙箱和 HITL 审批，记忆层管会话持久化和[[理论学习_上下文压缩与_Token_管理|上下文压缩]]。

这么分的好处是每层都能独立替换。比如 LLM Client 从 Anthropic 切到 OpenAI 不影响其他层，交互层从 TUI 换成 Web UI 只要重写一层引擎照跑，工具从 6 个内置扩展到任意 MCP 工具不需要改 Agent 代码。

**追问** ：分层之间最容易出问题的边界是哪里？

答：我踩过最疼的两个。一个是引擎层和记忆层之间，记忆注入时机和压缩时机要对齐，否则刚注入的长期记忆下一步就被压缩掉了，等于白注入。另一个是协作层和引擎层之间，多 Agent 邮箱消息消费时机必须在 Loop 开头，如果插在工具执行中间会破坏对话历史一致性，调试起来很痛苦。

### 你简历写熟悉 Function Calling 协议，能用大白话讲一下这个协议大概是什么形状？

考点：对协议本身的理解。

**参考回答** ：

Function Calling 是模型告诉调用方「我想调一个工具」的标准格式。请求时把工具列表和对话历史一起发给模型，每个工具有 name、description、input\_schema。

模型返回里如果包含 tool\_use 块，里面就有工具名、调用 ID、入参 JSON。调用方拿到后真正去执行工具，把结果以 tool\_result 块塞回对话历史，包含同一个调用 ID，再发下一轮请求。

协议本质是把「模型想做什么」和「实际做了什么」明确分开。模型不会真的调任何外部接口，它只是输出一个想调的请求，执行权完全在调用方。

**追问** ：如果模型一次返回了三个 tool\_use，你怎么处理？是串行还是并行？

答：要看工具是否并发安全。读类工具可以并发跑，写类工具必须串行。我的做法是按相邻同性质分组：连续的并发安全工具合并到一个并发批次，遇到非并发安全工具就开一个新串行批次。

三个 tool\_use 可能被切成几个批次按顺序执行，所有结果统一打包成 tool\_results 回传给模型，作为下一轮的输入。

### ReAct 范式能用大白话讲一下吗？它和普通 LLM 调用有啥区别？

考点：范式理解。

**参考回答** ：

普通 LLM 调用是单次问答，问一句答一句。ReAct 是 Reasoning 加 Acting，把模型放进一个循环：模型先思考要做什么，然后调一个工具去做，拿到结果后再思考下一步，再调下一个工具，直到模型觉得任务完成不再调工具为止。

关键区别是状态延续。普通调用模型不记得自己上一次说了什么、做了什么，每次都要在 prompt 里塞历史。ReAct 把工具调用、工具结果都作为对话历史的一部分滚动进下一轮，模型能看到自己之前所有行为和结果，基于事实决策，不需要凭空想。

**追问** ：那如果模型思考是错的，比如选错了工具，怎么办？

答：靠工具结果反馈纠正。选错工具通常会得到错误输出或不符合预期的输出，下一轮模型看到这个结果会重新思考。

我的工程层会把工具错误明确标记 is\_error=True 回传给模型，让模型清楚知道刚才那一步失败了，又或者模型可以看结果发现自己是否上一步思考错误。如果模型连续多次选错，那是 prompt 或工具描述的问题，要回去优化[[提示词]]。

### 你的 Agent Loop 一轮里大概会做哪些事？按顺序讲一遍。

考点：核心流程理解。

**参考回答** ：

每一轮进来先检查上下文长度，超阈值就触发压缩，保证不会因为历史太长让下一次 API 调用爆掉。然后把 system prompt、工具列表、对话历史、环境信息、项目指令、长期记忆打包发给 LLM，开始流式接收，边收边把事件推给 UI 渲染。流结束后判断模型是不是要调工具：不调就把回复落进历史，Loop 结束；要调就把模型回复连同工具调用一起落历史，按并发安全性分批，每个工具过权限检查、执行、结果落历史，然后进入下一轮。中间穿插着各种 hook，轮次开始、发送前、调用前后、轮次结束都有，方便外部扩展。Multi-Agent 场景下还会在轮次开头检查邮箱，把队友发来的消息拼进历史。

**追问** ：哪一步最容易出 bug？

答：单 Agent 场景下最容易栽在对话历史一致性上。Function Calling 协议要求每个工具调用必须有对应的工具结果，缺一个下一轮请求直接报错。工具执行抛异常没正确返回、或用户在工具执行到一半 ESC 中断，历史里就会出现孤儿调用。我专门做了一个对话链校验逻辑，从持久化日志恢复会话时也会校验，发现孤儿就截断到最近一个完整边界。

Multi-Agent 场景下更难调。我遇到过两个典型的：一个是 Worker 看到别的 Worker 发来的进度同步消息，误以为是新指令照做了一遍，重复劳动甚至覆盖别人的成果；另一个是 Worker A 等 Worker B 的输出，B 又在等 A 的输入，邮箱里没人推进，整个团队悄无声息地停滞。这类 bug 单元测试覆盖不到，要靠真实的多 Agent 跑长任务才暴露，复现还很难。

### 工具并行执行你怎么设计的？怎么判断哪些工具可以并行？

考点：并发安全性设计。

**参考回答** ：

每个工具自己声明能不能并发。我们想对于设置并发的标准是：

-   只读类不修改任何外部状态：可并发，比如 ReadFile、Glob、Grep
-   写类会修改文件、执行命令、影响外部状态：不可并发，比如 WriteFile、EditFile、Bash
-   spawn 子 Agent 这类工具：不可并发，因为 spawn 本身有副作用，要创建 worktree、注册任务、加入 team 邮箱等共享状态，几个 spawn 同时跑会竞争；但 spawn 完之后子 Agent 自己在后台并行跑，主 Agent 不阻塞，所以并发能力是通过 background 模式实现的，不是工具级并发

做的时候我试过全部串行，发现读[[08-文件操作|文件操作]]多的时候明显慢，才加的分批。分批算法是相邻同性质合并：遍历模型返回的工具调用列表，连续的并发安全工具合并到同一个并发批次，碰到非并发安全工具就开一个新的串行批次。这样既保留了模型给出的执行顺序，又最大化了并发。

**追问** ：两个 ReadFile 读同一个文件，并发会有问题吗？

答：单次 ReadFile 是只读 IO，并发没问题。

如果是模型在一轮里既 ReadFile 又 WriteFile 同一个文件，分批后 ReadFile 在并发批次跑、WriteFile 在串行批次跑，顺序由模型给出的顺序保证，所以读后写不会冲突。

如果模型给出的顺序是读写读，分成三个批次顺序执行，结果跟串行一致，因为并发只在批次内部生效。

追问 2：并发跑的话 工具之间会不会看见对方的中间状态？比如 A 读到 B 还没写完的文件？

答：不会，因为并发批次内部全是只读类工具，按定义不修改外部状态。读类工具之间互相隔离。

读和写之间靠分批保证顺序：不管模型返回多复杂的读写序列，分批算法都是按顺序遇到写就切一刀，保证写前的读先跑完、写后的读能看到写的结果。逻辑顺序由模型给出，工程层只做并发优化不改顺序。

**追问** ：并发批次里如果有一个工具特别慢，会不会拖垮整批？

答：批次内并发等所有任务完成，最慢的工具决定批次耗时。这不算 bug，因为下一批必须等前一批全部结果。

优化空间在两处。一是给每个工具配独立超时，比如 ReadFile 10 秒、Bash 5 分钟，超时直接返回错误结果避免卡住整批。二是让慢工具支持流式产出中间状态，UI 上能看到进度不至于以为死了，但批次还是要等完成。

更激进的方案想过但没做，工程复杂度不值。

### Plan Mode 是什么？为什么要专门搞这个模式？

考点：用户控制权设计。

**参考回答** ：

Plan Mode 是只读规划模式：Agent 可以读文件、grep、glob 收集信息，Bash 也能用但限制为只读命令比如 ls、find、git log，不能写文件、不能执行会产生副作用的命令。模型最终输出一份结构化执行计划，等用户确认后再切换到执行模式。

为什么需要它：自动化执行的边界要交给用户控制。用户给的需求可能本身有歧义，或者想先看 Agent 准备怎么做再决定要不要让它真改。如果直接放它跑，万一方向跑偏，改完再回滚成本高。我做 MewCode 的时候就碰到过，让 Agent 重构一个模块，它理解偏了把接口签名全改了，下游调用全炸，回滚花了不少时间。从那之后我就觉得大改动一定要先出 Plan 让人看一眼。

**追问** ：用户看完 Plan 同意后，怎么切换出去？怎么保证 Agent 按 Plan 执行？

答：Agent 规划完成后会调用 ExitPlanMode 工具，用户在 UI 上确认同意，系统自动恢复到进入 Plan Mode 之前的权限模式，权限层立即生效。

工程层做了两件事来帮助按 Plan 执行。一是 Plan 会自动持久化写到磁盘文件，不怕上下文压缩丢掉。二是退出 Plan Mode 的时候，完整的 Plan 文本会通过工具返回值回灌到模型的上下文里，模型下一轮就是基于这份 Plan 继续工作的。再加上上下文连贯性带来的自然约束，实际效果已经很好了。当然如果用户切完模式后给个新指令，Agent 会优先响应新指令，不能百分百保证严格按 Plan 走。

追问 2：Plan Mode 输出的计划质量 怎么保证？有时候模型说要做 5 步，实际开干发现拆得太粗或漏步骤。

答：质量本质靠 prompt 约束。Plan Mode 的 system prompt 要求模型按固定结构输出：上下文分析、推荐方案、涉及的文件路径、可复用的现有代码引用、验证方式。每个步骤要写到改哪个文件的哪个函数这个粒度，不能是重构 auth 模块这种粗糙描述。

工程层没做计划质量自动评估，因为评估本身又要调 LLM 而且结果未必比 Plan 准。我的态度是把质量决策交给用户，看不顺眼就改。

经验上 Plan 出错最常见原因是模型没读够代码，prompt 里我强制要求 Plan Mode 至少调多次工具读相关文件。

**追问** ：用户同意 Plan 切到执行模式后，模型中途想到更好的方案偏离原 Plan，应该让它偏离吗？

答：允许偏离，但要标记。模型在执行中如果发现 Plan 的步骤 3 走不通，应该明确说出来：我发现 Plan 的某步不成立，原因是什么，现在调整成什么，然后继续。这是工程上没法强约束的，靠 system prompt 里要求偏离 Plan 时必须显式说明。

工程层能做的是审计：所有偏离行为通过对话历史可以回放，事后用户能看到哪一步背离了原计划。如果用户希望百分百严格执行不允许偏离，那 Plan Mode 应该改成自动化脚本生成而不是 Agent 模式。现在模型能力上来了，除非长程任务或方案本身太粗，基本不会偏离， 偏离了也往往会停下来确认 。

### 你的工具系统怎么注册和分发？新加一个工具要改多少地方？

考点：扩展性设计。

**参考回答** ：

工具系统核心就是个字典：key 是工具名字，value 是工具实例，全局共享。每个工具自己写好几个标签：叫什么、是干嘛的、入参长什么样、属于读还是写还是命令、能不能并发跑。

新加一个工具就两步：写代码继承基类，再把它扔进字典里。剩下都是自动的：给模型看的工具描述从入参定义里直接推出来，权限根据读写命令的设置来定，并发分批根据并发设置来定。Agent Loop 不用动、权限层不用动、UI 不用动。

MCP 工具更省事，连第二步都不用：启动时连上 MCP Server，拉到的每个远程工具用一个统一包装包一层就扔进字典，等于零代码扩展。

**追问** ：那如果一个工具需要访问 LLM Client 怎么办？比如内置一个 Agent 工具用来 spawn 子 Agent。

答：构造的时候直接传进去。基类不强制构造函数的样子，子类按自己需要扩展，比如 AgentTool 写好自己的构造函数，把要的依赖列在参数里，Agent 加载器、任务管理器、父 Agent、模型配置这些都行；在建字典时按需塞给它。

### 工具执行结果太大，比如一个 grep 跑出来 50MB 输出，怎么避免撑爆上下文？

考点：输出预算管理。

**参考回答** ：

我第一次遇到这个问题是让 Agent 跑 find，结果几万行直接把上下文撑爆了，那次之后我才认真做了输出预算管理。

做法是工具执行完之后马上检查结果大小，单个结果超过阈值就把原始内容写到会话目录下一个专门的工具结果子目录里，对话历史里只留头部预览加一个文件路径，模型如果真需要完整内容可以自己调 ReadFile 去读那个文件。但光管单条还不够，因为模型有时候一次调好几个工具，每个结果中等大小，单条不超标但加起来超了。所以在每轮 Loop 开始时我还会遍历历史，把同一条消息里的工具结果按大小排序，逐个落盘直到总量降到阈值以下。

这样模型平时只看摘要做决策，真正需要原始数据时按需去取，上下文预算保住了，信息也没丢。

**追问** ：模型自己去 ReadFile 那个 .txt 文件不会很笨吗？

答：会多一轮调用，但概率不高。头部预览已经能覆盖大多数场景，比如错误日志基本前几行就有关键信息。比起撑爆上下文整个对话挂掉，偶尔多调一轮成本完全可以接受。

追问 2：落盘的文件之后 怎么清理？长会话会不会堆几百个落盘文件？

答：每个会话有自己的工具结果目录，会话结束时整个目录删掉，过期会话清理时也跟着一起删。实际堆积量不大，长会话也就几十个文件。

**追问** ：头部预览的截断点怎么选？硬截可能截在一行中间或 JSON 半边。

答：文本输出按行截断，找最后一个换行符；结构化输出遇到不完整 JSON 就加省略标记。极端情况模型确实可能被截断误导，但有读文件兜底。

### 你简历说五层权限拦截，能从内到外讲一遍吗？

考点：多层防御理解。

**参考回答** ：

最开始其实只有 HITL 一层，就是每次工具调用都弹框问用户。结果用户吐槽做个任务要点几十次 allow，完全受不了，才逼着我在前面加了自动判断的层。最终做成了五层，从硬编码拦截到用户确认，越靠内越严格，任何一层 deny 就短路退出，不再往后走。

设计取舍最多的是中间两层。 **危险命令检测** 只针对 Bash 类工具，用正则匹配 rm -rf /、mkfs、fork bomb 这类绝对不能执行的模式。这一层只能 deny 或者 pass，永远不能 allow，保证最严格的拦截不被后续任何层覆盖。这也是整条链路的设计洞察：deny 是单调的，任何一层的 deny 不能被别的层 allow 掉，否则安全模型就崩了。 **规则引擎** 是另一个花心思的地方，用户级、项目级、本地级三层规则文件，按 Tool(pattern) 匹配。我吃过一个亏：最早优先级搞反了，本地规则应该覆盖项目规则，结果我让项目规则优先，用户在本地配的 deny 被项目级 allow 盖掉了，一个 Bash 命令莫名放行，排查半天才发现。改完之后加了单测专门覆盖这个 case。

剩下三层相对简单。路径沙箱只管文件类工具，把路径解析成绝对路径看是不是落在项目根目录或 /tmp 内。权限模式就是 default、acceptEdits、plan 几种档位，控制读写和命令各自放不放行。最外面才是 HITL，前面四层都没做出决定的才弹框问用户。

**追问** ：为什么搞这么复杂，不能就一层用户确认吗？

答：Agent 做一个任务可能调几十次工具，每次都问体验太差了。前面四层把明显能放、明显不能放的自动判断掉，HITL 只处理灰色地带。同时也是多层防御，万一某一层配错了还有别的层兜底。

追问 2：这五层每一次 工具调用都跑一遍，性能开销大吗？

答：完全可以忽略，每层都是纯本地计算，正则、路径前缀匹配、字典查找，前四层加起来 1ms 以内。真正的瓶颈是 HITL 等用户点按钮，但那是交互问题不是性能问题。

### MCP 是什么？为什么不直接把工具写在 Agent 进程里？

考点：MCP 价值理解。

**参考回答** ：

MCP 全称 Model Context Protocol，是 Anthropic 提出的一套工具服务发现和调用协议。简单说就是把工具放到独立进程里跑，Agent 通过 MCP Client 连上去，拿到工具列表，代理调用。

我做 MewCode 的时候，一开始确实是把所有工具直接写在 Agent 进程里的，读文件、写文件、跑命令，全是内置函数。后来想加一个 GitHub 相关的工具，比如创建 PR、查 issue，我发现问题来了：这些工具逻辑和 Agent 核心循环完全没关系， 而且如果别的 Agent 也想用这套 GitHub 工具，我得把代码复制一份过去，后续维护两边同步很痛苦。MCP 的做法就是让工具以独立 Server 的形式存在，一个 GitHub MCP Server 写好之后，Claude Code 能用，MewCode 能用，Cursor 也能用，社区共建共享。

另一个让我很有感触的点是配置即生效。用户想加一个新工具，不需要改 Agent 的任何代码，编辑一下配置文件指定 MCP Server 的启动命令就行。这对终端工具的可扩展性太重要了，用户不可能为了加个工具 去重新编译 Agent 。而且 MCP Server 可以跑在不同的账号甚至不同的容器里，天然形成权限隔离，比如数据库查询工具跑在只读账号下，Agent 进程本身不需要持有数据库凭证。

**追问** ：那 MCP 引入了进程间通信，性能损耗大吗？

答：单次调用毫秒级开销，Coding Agent 场景下 LLM 响应动辄好几秒，这点 IPC 开销完全可以忽略。性能瓶颈从来不在 MCP 协议本身，在工具实现内部， 比如工具要读一个大文件 、跑一个慢查询，那才是真正的耗时。

### MCP 都支持哪些通信方式？

考点：对 MCP 协议传输层的理解。

**参考回答** ：

MCP 协议定义了三种传输方式，每种对应不同的部署场景。

第一种是 stdio，也是最常用的。Agent 启动一个子进程跑 MCP Server，两边通过标准输入输出交换 JSON-RPC 消息。零网络配置，Server 生命周期跟着 Agent 走，Agent 退出子进程也跟着结束。本地工具场景基本都用这个，比如文件操作、代码索引、git 操作。我做 MewCode 的时候一开始只实现了 stdio，因为终端工具天然就是本地跑的，这种方式最简单也最稳定。

第二种是 SSE，全称 Server-Sent Events。本质是 HTTP 长连接，客户端发请求用普通 POST，服务端推消息用 SSE 流。适合连远端已经部署好的服务，比如公司内部的 RAG、知识库这些。好处是一个 Server 可以同时服务多个客户端，适合团队共享工具。

第三种是 Streamable HTTP，这是 MCP 协议后来新增的，设计上是要替代 SSE 的。和 SSE 的区别是它不要求服务端维持长连接，每次请求都是独立的 HTTP 调用，服务端可以选择返回普通 JSON 或者开一个 SSE 流。对无状态部署更友好，比如 Serverless 环境。不过目前生态里大部分 Server 还是用 stdio 或 SSE，Streamable HTTP 还在逐步推开。

在 Client 这边，我做了一层统一的会话抽象来屏蔽传输差异。业务代码只关心列工具和调工具两个动作，底层走哪种传输它不需要知道，连接方法里根据配置选择实现，上层完全透明。

**追问** ：你实际用下来，stdio 和远端传输最大的体感差异是什么？

答：延迟。stdio 走的是进程间通信，微秒级响应，工具调用几乎感觉不到延迟。远端传输走网络，快的也要几十毫秒，工具多了用户能明显感觉到每一轮变慢了。另外 stdio 不用操心认证和网络抖动，远端的得处理超时、重连这些。所以我的原则是能本地跑的工具一律走 stdio，只有确实跑不了本地的才走远端。

### MCP Server 启动失败，你的 Agent 怎么处理？让 Agent 启动失败还是降级？

考点：故障隔离设计。

**参考回答** ：

降级，不让 Agent 启动失败。

MCP 管理器在注册阶段对每个 Server 做异常隔离：成功就连上、拉工具、注册进去；失败就记日志告警、把错误信息加进错误列表，继续处理下一个 Server。所有 Server 处理完，错误列表返回给上层，UI 在启动后展示哪些 MCP 失败了。

理由：MCP 是扩展能力，不是核心能力。一个 GitHub MCP 挂了不影响用户用 Agent 改代码。如果让一个 MCP 失败就拖死整个 Agent，对用户体验是灾难。

**追问** ：那 Server 跑到一半挂了呢，比如网络断了？

答：管理器在取客户端时会检查存活状态，发现挂了就尝试重连一次。重连失败就抛错，工具调用层捕获后返回错误结果，对话历史里告诉模型这次工具调用失败。

模型可以选择重试、换工具、或者放弃。整个降级链路对用户透明。

### 你简历写 MCP 工具延迟加载，Token 占用减少 85%， 是如何实现的？

考点：延迟加载的设计动机与量化依据。

**参考回答** ：

MCP 工具数量是不可控的，我自己用 Claude Code 常配的 MCP Server 就有 5 个，加起来大概五六十个工具。每个工具的完整 schema 包含名字、描述、参数定义，这些工具全塞进上下文就消耗了两万左右的 Token。对于只有 200k 上下文窗口的模型来说，还没开始干正事就吃掉了 10%，多聊几轮触发上下文压缩的时机会明显提前。

做法是构建 API 请求的时候，tools 数组里不放 MCP 工具的完整 schema。系统提示词里告诉模型「有些工具还没加载，需要先通过 ToolSearch 查询才能调用」，然后每轮在一条合成的 user message 里列出所有延迟工具的名字。模型判断需要某个工具时先调 ToolSearch 拉取完整定义。客户端这边维护一个「已激活工具」的集合，ToolSearch 命中就把工具加进去，下一轮请求的 tools 数组自然就会带上它的完整 schema。实测 58 个模拟 MCP 工具，全会话维度 Token 减少约 85%。

### 上下文压缩具体怎么做的？分别什么时机触发？

考点：压缩策略分层。

**参考回答** ：

按代价从小到大、触发时机 由早到晚分两层 。

最轻的是落盘。每次工具结果产出时立即检查，单结果超 50K 字符或一轮工具结果总和超 200K，就把大结果落到磁盘文件，对话历史里只留 2K 预览加文件路径。不调 LLM、不丢信息，原始数据还在磁盘上，代价最小。

落盘顶不住的时候就靠自动压缩接着扛。每轮 Loop 开始前检查上轮的输入 token 是否超过阈值，超了就调 LLM 把较早的消息生成摘要，同时保留最近大约 1 万 token 或至少 5 条消息的原文不动。压缩后的对话历史就是一条摘要 user 消息（包含摘要内容、恢复的文件快照、会话记录路径）加上保留的近期原文。阈值不是拍脑袋定的，Claude Code 的做法是从上下文窗口里先扣掉 20K 的压缩输出预留（这是根据实际 p99.99 数据算的安全边际），再扣 13K 给当前轮次留余量，比如 200K 窗口实际触发点大概在 167K，也就是 83.5% 左右。我做 MewCode 的时候参照了这个思路，也是预留一块 buffer 再算阈值，不是等快满了才压。代价中等，要调一次 LLM，早期对话细节会丢，但近期原文和主线都保住了。

还有一种是用户主动触发的 compact，逻辑跟自动压缩一样但安全余量更小，不算独立的一层，只是触发方式不同。典型场景是自动压缩还没触发，但用户感觉对话已经太啰嗦了想手动清理一下。

**追问** ：自动压缩之后，模型会不会忘了用户最初的需求？

答：两道防线。第一道是近期原文保留，最近几轮对话一字不改地留着，模型能直接看到最近的指令和反馈。第二道是摘要 prompt 里专门列了 9 个必须保留的部分，第 6 项要求尽量保留用户消息原文。prompt 里写的是强约束语气，但本质还是 prompt 层面的引导，模型不一定每次都严格遵守。再加上摘要消息里附带了完整会话记录的文件路径，模型真需要早期细节可以自己去读。

这样最初的用户指令、后续的纠正、风格偏好都不会轻易丢掉。摘要后还会重新注入最近读过的文件快照、技能定义和工具列表，让模型恢复工作上下文。

追问 2：如果用户在长会话里频繁触发压缩 ，比如几小时内压缩 5 次，最后的对话历史会不会越来越失真？

答：会，摘要的摘要必然有损累积。不过工程层有个缓解：压缩只替换内存里的对话历史，磁盘上的会话日志是追加写的不会被改，原始消息一直在，需要时可以回日志里全文搜索。多次压缩之后失真到一定程度，用户会感知到需要新建会话，这个生命周期由用户自己决定。

### 摘要本身也要调 LLM，万一摘要这次调用又超了上下文呢？

考点：边界情况处理。

**参考回答** ：

首先是预防。算阈值的时候我把摘要本身的输出预算扣掉，再扣一个安全余量，这样压缩触发时，剩下的 token 余量足够生成摘要不超限。我一开始没扣这个预算，结果压缩的时候摘要请求本身就炸了，才意识到阈值得算得更保守。

但如果预防没顶住，真超了的话，摘要请求失败时我会检测错误信息里是不是上下文超长，是的话按轮分组把对话历史的最早 1/5 直接丢掉，再试，最多重试 3 次。

如果 3 次后还失败，触发压缩熔断器，这个会话后续不再自动压缩了，但用户手动 /compact 还能用。Claude Code 的源码里这个上限也是 3 次，他们发现线上有一千多个会话出现过连续 50 次以上的压缩失败，最极端的一个会话连续失败了 3000 多次，所以加了这个硬限制。

**追问** ：丢掉最早 1/5 不会丢关键信息吗？

答：会，但是必要妥协。早期对话通常是项目背景、需求澄清，正常情况下这些通过看后面的对话历史也能知道大概。

真要丢的话也是成组丢，按轮分组保证工具调用和工具结果配对，不会出现孤儿工具调用让历史不合法。这是最后的安全网，正常路径阈值算得保守就基本不会触发。

### 记忆系统怎么设计的？为什么要分层？

考点：记忆分层设计。

**参考回答** ：

两层：

-   用户级：跟用户走，跨项目。放在用户 home 下，存放用户偏好和纠正反馈，比如编码风格、命名规范、语言偏好
-   项目级：跟项目走，团队共享。放在项目 .mewcode 目录里，存放项目知识和参考资料，比如技术栈、目录结构、外部文档链接

为什么分：不同性质的记忆作用域不同。我喜欢 4 空格缩进应该跨项目生效，这个项目用 PostgreSQL 15 这种记忆只在当前项目有意义。混在一起，要么用户偏好污染所有项目，要么项目知识丢失重新进入项目又得重交代。

实现上，记忆提取出来后按章节标题自动归类：用户偏好和纠正反馈写到用户级文件，项目知识和参考资料写到项目级文件。模型不需要知道这个分层，自己按 4 个固定章节输出，工程层面分发。

**追问** ：用户级如果记错了，比如把临时偏好当永久偏好记下来，怎么办？

答：用户能直接编辑记忆文件，所见即所得。我提供一个 memory 命令展示当前所有记忆，用户可以自己改、删行，也可以触发清空。

这是 markdown 不是数据库，编辑零成本。坏处是模型可能下次又把同样的内容提取出来，所以记错通常需要从对话源头纠正，让模型知道这条不要记。

### 记忆什么时候被注入对话？怎么注入？

考点：注入时机与位置。

**参考回答** ：

注入时机：会话开始或第一次注入调用时，整个会话只注入一次，用一个注入标志位防止重复。压缩之后注入标志重置，下一轮重新注入到摘要后面。

注入位置在对话历史开头，环境信息之后。先放环境信息（cwd、OS、当前时间），再放项目指令（CLAUDE.md 或 AGENTS.md 内容），接着是自动记忆文件内容，最后补一条假装的助手消息说已经了解项目背景和记忆。最后那句假装的助手回复是为了让对话历史保持 user/assistant 交替，符合 LLM 的对话格式期望，这是个常见的工程 trick。

为什么放在对话历史里而不是 system prompt：记忆是用户产物，跟用户消息放在一起更自然。system prompt 定义的是 Agent 的角色和行为规则，记忆和项目指令属于用户侧的上下文信息，放在对话历史开头作为背景交代，语义上更清晰。

**追问** ：那要是用户的记忆有 100KB，全注入会不会本身就吃太多上下文？

答：记忆设计上是 markdown 短文本，正常几百行内，几 KB 量级。如果用户暴力堆 100KB 记忆，那是用户用法问题。

工程层不会自动截断，因为不知道哪条该砍。我的处理是 memory 命令展示总长度，让用户自己感知。极端情况下记忆吃太多上下文，自动压缩会触发，把摘要加记忆重新组织。系统有自我调节能力，但不主动限制用户。

理论上可以再加一层记忆摘要的摘要，但目前这个量级够用。

### 为什么需要多 Agent？单 Agent 不够用吗？

考点：多 Agent 的动机与边界。

**参考回答** ：

单 Agent 的瓶颈有两个。上下文窗口是最先碰到的，一个 Agent 改 30 个文件，所有文件内容、工具调用结果、中间推理过程都挤在同一段对话历史里，很快就逼近上下文上限。压缩能续命但会丢细节，改到后面模型可能忘了前面改过什么。另一个是串行效率，单 Agent 一次只能做一件事：读完文件才能改，改完才能测，测完才能改下一个。如果有多组互不依赖的文件，串行跑完全浪费了并行的机会。

多 Agent 同时解决这两个问题。每个 Agent 有自己独立的上下文窗口，改前端的 Agent 不需要看后端的代码，上下文压力天然分散。多个 Agent 并行跑互不依赖的任务，总耗时取决于最慢的那个，而不是所有任务加起来。

当然多 Agent 不是万能的。每个 Agent 都要重新理解项目背景，token 总消耗翻倍甚至更多。Agent 之间的协调本身也是开销，消息传递、任务拆分、结果合并都有出错的可能。所以小任务单 Agent 直接跑，只有任务可拆分且单任务耗时明显的场景，多 Agent 才值得开。

**追问** ：你的多 Agent 协作有哪几种模式？

答：两种。Agent Team 模式是多个 Agent 平等协作，没有中心节点，通过共享任务清单和邮箱通信，每个 Agent 在独立 Worktree 里跑。Coordinator 模式是一个 Coordinator 统筹，spawn 多个 worker，自己不写代码只做拆分、分发和验证，worker 跑完异步通知结果 。 Coordinator 的 system prompt 明确告诉它你是协调者，不要自己实现。

追问 2：这两种模式怎么选 ？

答：Coordinator 模式类似于技术负责人带专业团队：负责人理解全局需求，拆成具体任务分给专人做，做完验收合并。适合需要多阶段协调的复杂任务，因为有人站在全局视角把控方向、分配依赖、处理冲突。不然会忙不过来导致出错。

Team 模式更像一线组长带游击队：每个人看任务清单自己认领，干完汇报，彼此之间靠邮箱松散协调。适合几个 Agent 长期驻守各管一块的场景，比如一个管前端一个管后端一个管测试，各自独立推进不需要频繁协调。

实际项目里 Coordinator 用得更多，因为大多数复杂任务天然有依赖关系，需要有人决定先做什么后做什么、谁的产出给谁用。没有统筹角色的话，多个 Agent 容易各干各的最后合不到一起。

**追问** ：Coordinator 自己也是个 Agent，它的上下文不会爆吗？毕竟要处理所有 worker 的报告。

答：会有压力，所以 Coordinator 有几个设计：worker 用结构化 XML 格式返回任务完成通知，比自然语言更紧凑；worker 的中间过程不回传 Coordinator，只回最终摘要；Coordinator 自己也走标准的上下文压缩流程。

但确实这是 Coordinator 模式的天然限制，深度长任务对 Coordinator 上下文压力大。这种场景可以再嵌套一层 Coordinator，或者拆分成多个独立会话。

### 多 Agent 之间用邮箱通信，为什么不用直接调用？

考点：异步消息设计。

**参考回答** ：

直接调用是同步阻塞：Agent A 调 Agent B 的方法，A 必须等 B 返回。我最早试过直接函数调用，结果 A 等 B 的时候整个 Loop 卡住了。而且 B 在另一个进程或另一个 worktree，跨边界调用本身就复杂，一个 Agent 同时被多个 Agent 调用还要做同步控制，任意一个 Agent 挂了会拖垮整条调用链。

邮箱模型就不一样，A 给 B 发消息立即返回不阻塞，B 在自己的 Loop 里主动消费邮箱。消息存储用文件系统，每个消息一个 JSON 文件，天然解耦，B 没在线消息也能落盘等 B 上线再消费。

实现上每个 Agent 一个目录，消息按时间戳命名落盘，消费时按时间序读取并删除。简洁可靠，崩溃安全，消息已写盘就不会丢。

**追问** ：那如果 A 给 B 发消息要等 B 回复呢？

答：邮箱没有 RPC 语义，只有消息传递。A 发完消息继续干自己的活，B 处理完结果再发一条消息回给 A。A 在自己的下一轮 Loop 开始时消费邮箱会读到 B 的回复。

这种异步回调模式比 RPC 复杂，但更容错。如果业务上需要 A 等 B 的结果再继续，用 Coordinator 模式更合适。Coordinator spawn worker 后不阻塞，系统每隔几秒轮询后台任务状态，worker 完成后通知自动注入到 Coordinator 的对话历史里，Coordinator 在下一轮看到结果再做下一步决策。

### 你做这个 Agent，最大的工程挑战是什么？

考点：项目反思深度。

**参考回答** ：

最难的是多 Agent 协作。单 Agent 的工程问题大多边界清晰，比如对话历史要保持工具调用和结果配对，知道原则之后机械执行就行。多 Agent 开起来之后，问题变成了分布式状态同步：Worker 误读进度消息当成新指令重复劳动，Coordinator 误判 Worker 已完成，两个 Worker 基于旧版代码各自改完合并后语义互相破坏。这类 bug 单元测试覆盖不到，要靠真实的多 Agent 跑长任务才暴露，复现还很难因为 LLM 本身有随机性。我后期大部分时间花在调这一块。

![](MewCode_项目面试题-9.jpeg)

紧跟着的是上下文预算管理。Agent 一个任务可能跑几十轮，每轮的工具调用和结果都往对话历史里堆，不主动管理很快撑到上下文窗口上限。我做了三层防御：大结果即时落盘只留预览、接近上限时调 LLM 生成摘要替换历史、摘要本身超限时丢最早的轮次重试加熔断。每层都有边界情况，比如落盘截断点怎么选、摘要质量怎么保证、多次压缩的累积失真怎么缓解。

还有一个始终绕不开的是模型的不确定性。同样的 prompt 同样的工具，模型今天可能调一次 ReadFile 解决，明天可能调五次 Glob 找半天。工程层没法约束模型的思路，只能优化工具描述、system prompt、工具数量来引导。Agent 工程一半是工程一半是 prompt 工程，两者调试方法论完全不同。

**追问** ：这三个挑战里哪个是你一开始没想到的？

答：模型的不确定性。一开始我以为把架构搭好、工具做好，模型自然就能稳定跑。实际上模型的行为漂移才是最大的不可控因素，同一个 prompt 换个模型版本表现可能完全不同。这逼着我从纯工程思维转向工程加 prompt 工程的混合思维。

追问 2：多 Agent 协作里有没有 哪一次踩坑让你印象特别深？

答：印象最深的是 Coordinator 误判 Worker 完成。Worker 跑到一半触发了上下文压缩，压缩后摘要里把任务进度写成了已完成的语气，Worker 继续跑完后通知 Coordinator。Coordinator 看到通知里的摘要以为任务做完了，实际上 Worker 只做了一半，压缩把后半段的待办丢了。

这个 bug 排查了很久，因为从日志上看每一步都是正常的：压缩成功、Worker 正常结束、通知正常发送。问题出在摘要的语义失真，工程层面完全检测不到。

后来的缓解方式是在 Coordinator 的 prompt 里强调：不要只看 Worker 说完成就信，要看具体产出，比如有没有 commit、测试过没过。但这本质上是 prompt 层面的软约束，没法完全杜绝。

**追问** ：这个项目最让你有成就感的是什么？

答：多 Agent 协作真的跑起来那一刻。3 个 Worktree、3 个 Agent 同时改不同模块，Coordinator 在另一个会话看任务清单和邮件，30 分钟完成了正常需要 2 小时的重构。Loop、工具、权限、上下文、记忆、隔离每一层累积到最后合在一起才发挥威力，任何一层不扎实多 Agent 都跑不起来。

### 你的 Skill 系统是怎么设计的？Skill、MCP、Function Calling 三者是什么关系？

考点：扩展机制的层次理解。

**参考回答** ：

Skill 是行为层面的扩展。每个 Skill 是一个 markdown 文件，frontmatter 声明名字、描述、允许用的工具列表、执行模式，正文是一段 prompt 模板，支持占位符接收参数。

三个来源按优先级：项目级放 .mewcode/skills 目录，用户级放 home 目录，内置的打包在代码里。同名 Skill 项目级优先覆盖。

两种执行模式：inline 模式把 Skill 的 prompt 注入当前会话，Agent 带着额外指令继续跑；fork 模式开一个独立子 Agent，有自己的对话历史和过滤过的工具集，跑完把结果返回主 Agent。支持热加载，每次调用时重新读源文件，改完不用重启。

我做 MewCode 的时候，最初 Skill 和普通 prompt 混在一起管理，后来发现不同任务的指令互相干扰，才把 Skill 独立出来做模块化。这个设计让我在调代码审查和重构这两类任务时省了很多事，各自的工具集和约束互不打架。

三者的关系：Function Calling 是底层协议，模型输出工具调用请求，Agent 执行后把结果回传，所有工具调用都走这条路。MCP 扩展的是工具集，解决 Agent 能做什么，MCP Server 提供远程工具，拉过来包装成普通工具注册进字典。Skill 扩展的是行为模式，解决 Agent 该怎么做，本质是 prompt 配置加工具过滤，不提供新的工具能力，而是约束和引导 Agent 在特定任务上的思路和可用工具范围。

打个比方：Function Calling 是嘴和手的连接方式，MCP 是往工具箱里加工具，Skill 是给 Agent 一份作业指导书告诉它这次任务的目标、约束和做法。

**追问** ：Skill 跟直接写 system prompt 有什么区别？为什么不直接改 system prompt？

答：system prompt 是全局固定的，对所有任务生效。Skill 是按需激活的，用户触发某个 Skill 时才加载进来，任务结束可以卸掉。

区别在于粒度和生命周期。system prompt 定义 Agent 的基础人格和通用规则，Skill 定义特定任务的指令和工具约束。两者叠加：system prompt 是底层，Skill 是上层覆盖。

如果把所有 Skill 的内容都塞进 system prompt，一是太长浪费 token，二是不同任务的指令互相干扰，三是用户没法按需组合。Skill 的价值就在于把行为配置模块化了。

### 为什么 Agent 需要工具？没有工具的 LLM 不也能回答问题吗？

考点：Agent 与普通 LLM 的本质区别。

**参考回答** ：

LLM 本身只能生成文本。它能回答问题、写代码片段、做分析，但这些全都停留在文字层面。它没法读一个真实文件看里面写了什么，没法跑一条命令看结果是什么，没法改一行代码让项目真的变化。

工具让 LLM 从聊天变成做事。有了 ReadFile 它能看到项目的真实状态，有了 Bash 它能跑测试验证自己的判断，有了 EditFile 它能把想法落地成代码变更。工具把模型的推理能力和外部世界连接起来。

这也是 Agent 和普通对话机器人的分界线：普通对话只有输入输出两个端点，Agent 在中间有一个持续的感知、决策、行动循环。每一轮工具调用都是一次真实的环境交互，模型基于真实反馈做下一步决策，而不是凭训练数据猜。

**追问** ：那工具越多越好吗？

答：不是。工具数量直接影响模型的选择质量。工具列表作为请求的一部分发给模型，工具越多模型要理解的选项越多，选错工具的概率也越高。

我的经验是内置工具控制在个位数，覆盖读、写、搜索、执行这几个核心动作就够。更专业的能力通过 MCP 按需接入，用的时候才加载，不用的时候不占工具列表。

另一个问题是工具描述的质量比数量重要。两个工具描述含糊模型分不清该用哪个，比缺一个工具更糟糕，因为模型会反复选错浪费轮次。

追问 2：如果没有工具，纯靠 LLM 生成代码，能做到什么程度？

答：能做到单次问答级别的代码生成。用户贴一段代码问怎么改，模型给出修改后的版本，用户自己复制粘贴。这就是最早期的 AI 编程助手形态。

局限在于模型看不到项目全貌。它不知道这个函数被谁调用，不知道改完之后测试过不过，不知道项目用的是哪个版本的依赖。所有这些上下文都要用户手动喂进去。

工具解决的核心问题就是让模型自己获取上下文。与其让用户一段一段贴代码，不如让模型自己读文件、搜代码、跑测试，基于真实项目状态做决策。这是从辅助工具到自主 Agent 的关键跨越。

### 为什么 Agent 需要记忆？没有记忆不也能完成任务吗？

考点：[[理论学习_跨会话记忆与会话持久化|记忆系统]]的必要性理解。

**参考回答** ：

单次会话内确实不需要记忆，对话历史本身就是短期记忆，模型能看到这轮会话里发生过的一切。

问题出在会话之间。用户今天告诉 Agent 自己喜欢 4 空格缩进、项目用 PostgreSQL、测试必须跑 pytest，明天开新会话这些全忘了，用户得重新交代一遍。项目越复杂、协作越深入，这种重复交代的成本越高。

记忆解决的是跨会话的知识延续。把用户偏好、纠正反馈、项目知识持久化到文件里，每次新会话开始时注入回对话历史，模型从第一句话就带着这些背景在工作。

分两层是因为不同知识的作用域不同。编码风格这种个人偏好跨项目通用，放用户级；项目的技术栈和目录约定只在当前项目有意义，放项目级。混在一起要么偏好污染其他项目，要么换项目就丢知识。

**追问** ：那 CLAUDE.md 这种项目指令文件跟记忆有什么区别？都是持久化的背景信息。

答：区别在于谁写的、怎么维护的。

CLAUDE.md 是人写的项目指令，跟着 git 走，版本可控，团队所有人看到的是同一份。内容是项目约定和工程规范，比如目录结构、命名约定、协作流程。

记忆是 Agent 自动提取的，从对话中抓取用户偏好和反馈，写到记忆文件里。内容是个人化的、隐式的，比如用户习惯性的纠正、风格偏好，这些不适合写进项目文档。

两者互补：CLAUDE.md 提供项目的显式规范，记忆捕捉用户的隐式偏好。Agent 启动时两者都注入，一个是团队共识一个是个人经验。

追问 2：记忆会不会越积越多 最后变成噪声？

答：会有这个趋势。时间长了过时的记忆和当前需求混在一起，模型注入这些内容反而可能被误导。

缓解方式有两个。一是提取 prompt 要求模型在已有记忆基础上更新，把过时的替换掉，保持记忆文件精简。二是用户可以直接编辑记忆文件，看到哪条不对就删。

但工程层没有做自动过期或衰减。记忆是 markdown 文本，没有时间戳，没法自动判断哪条老了该删。这个问题目前靠用户自己管理，是一个已知的设计局限。但是也没什么大碍，因为其实这不是Coding Agent的重点，目前这样够用，否则ROI会较低。

### 你的 MewCode 项目和 Claude Code 相比，核心差异是什么？有什么比它做得好，什么不如它？

考点：对自研项目的客观定位，能否诚实评价差距和亮点。

**参考回答** ：

核心差异在定位和体量。Claude Code 是 Anthropic 官方商业产品，将近 2000 个 [[Day2-JavaScript和TypeScript|TypeScript]] 源文件、背后有专职团队迭代。Mewcode Coding Agent 是学习项目，一个人从零搭的，目标是把 Coding Agent 的每一层架构走通，不是做到产品级。

差距最明显的是生态和精细度。Claude Code 做了 LSP 集成、OAuth 认证、MCP 官方注册表、语音输入、IDE Bridge 这些产品级功能，MewCode 都没有，学习项目里投入产出比不划算。上下文管理上 Claude Code 有五六层压缩策略针对不同粒度分别优化，MewCode 做了三层核心机制都有但精细度差一个量级。工具数量上 Claude Code 50 多个覆盖各种垂直场景，MewCode 内置 8 个核心工具加 MCP 扩展。

但 MewCode 有自己的取舍优势。架构透明度上，每一层都是从零写的没有黑盒依赖，权限五层怎么短路、压缩三层怎么分级、多 Agent 邮箱怎么投递，每一行都能解释为什么这么写。多模型支持上，LLM Client 层从一开始就抽象了 Anthropic 和 OpenAI 两套协议，通过配置切换，实际用 GLM 的兼容端点跑过。Claude Code 硬绑 Anthropic 协议，这方面反而受限。

**追问** ：如果让你把 MewCode 做到产品级，你觉得最大的 gap 在哪？

答：不在架构，在细节打磨和边界覆盖。产品级意味着每一个异常路径都要有优雅降级，每一个用户交互都要有合理反馈。比如 Claude Code 的 StreamingToolExecutor 里有 sibling abort 机制，一个 Bash 工具报错能立刻终止同批次其他子进程；有 file history 做文件级 checkpoint，改坏了能 undo。这些都是真实用户踩坑之后才补上的防御层，不是架构设计阶段能预见的。MewCode 的架构骨架是对的，但从骨架到产品之间差的是几千个这样的细节。

### 自己做 Agent 的时候，踩过最大的坑是什么？

考点：真实踩坑经验，区分「看过博客」和「真做过」。

**参考回答** ：

最大的坑是对话历史的一致性问题。

Function Calling 协议有一个隐含的硬约束：每个 tool\_use 块必须有且仅有一个对应的 tool\_result 块，缺了或多了，下一轮 API 调用直接报错，整个会话崩掉。

这个约束看起来简单，但实际开发中到处都是破坏它的场景。

最先遇到的是用户中断。模型返回了 3 个工具调用，第 1 个执行完了，第 2 个正在跑，用户按 ESC。这时候对话历史里已经有 3 个 tool\_use，但只有 1 个 tool\_result。如果不补上剩余 2 个的结果（哪怕是错误结果），下一轮直接炸。我最初没处理这个，调试了很久才发现是中断导致的孤儿 tool\_use。

后来恢复会话的时候又踩了一次。用户关掉终端再打开，从 JSONL 日志恢复对话历史，如果上次进程是被 kill 的，日志最后可能只写了 assistant 消息没写 tool\_result。恢复时不做校验就会加载一个不合法的历史，下一轮调 API 立刻失败。我后来专门加了对话链校验逻辑，恢复时检测孤儿 tool\_use，发现就截断到最近一个完整轮次。

做压缩的时候又碰到了。压缩会把对话历史替换成摘要，如果替换的边界恰好切在 tool\_use 和 tool\_result 之间，历史又不合法了。我的压缩逻辑按轮分组，保证切点一定在完整轮次之间，不会切在中间。

这个坑的教训是：Agent 的对话历史不是普通的消息列表，它是一个有严格语法约束的协议结构。任何修改历史的操作，包括中断、恢复、压缩、截断，都必须保证协议合法性。

**追问** ：除了对话历史一致性，还有哪些坑是你事先没想到的？

答：流式解析里工具调用参数的累积。文本增量边来边拼就行，但工具调用的入参是 JSON，模型按 token 推送，推到一半是不完整 JSON 没法解析。我最初试过每收到一个 delta 就尝试解析，结果每次都失败。后来改成等 ToolCallComplete 事件才解析，中间只做字符串拼接。

更坑的是不同协议的流式事件结构完全不同。Anthropic 的是层次化的块结构，一个块走完才开始下一个；OpenAI 兼容协议是平铺事件流，多个工具调用的 delta 是交错的，必须按 tool call index 路由到对应的累积器。我最初只写了 Anthropic 的 parser，接 OpenAI 兼容端点时被交错事件搞得一头雾水，调了两天才理清。

### 从开发者的角度，做 Agent 最难的部分是什么？

考点：对 Agent 工程本质难点的理解，能否跳出代码层面看问题。

**参考回答** ：

最难的是确定性和不确定性混在一起。

Agent 工程里有一半是传统的确定性问题，对话历史的协议合法性、工具分批的并发控制、权限五层的短路求值，写对了就稳定，跟普通后端没区别。但另一半是模型的不确定性，同样的 prompt 同样的工具，模型今天可能调一次 ReadFile 直接搞定，明天先 Glob 再 Grep 再 ReadFile 绕一大圈。工具描述差一个词，模型可能从稳定选对变成反复选错。

这两半混在一起，调试变得非常痛苦。一个 bug 表面上是工程问题，比如工具返回了错误结果，但根因可能是 prompt 问题，模型给了错误的参数。也可能反过来，表面上是模型乱来，但根因是工程问题，比如上下文压缩丢了关键信息导致模型做了错误决策。 没法像传统 debug 那样隔离变量 ，唯一可靠的方法是回放完整的对话历史，逐轮看模型的推理和工具调用，人肉判断哪一步走偏了。

我印象很深的一次是模型在重构任务里反复读同一个文件，日志上看工具调用完全正常，ReadFile 每次都成功返回了内容。排查了半天才发现是上一轮压缩把文件的修改计划摘要丢了，模型失去了上下文不知道自己已经读过这个文件，所以一直重复。工程层面看是死循环，prompt 层面看是摘要质量问题，两边都有责任，这种交叉归因就是 Agent 工程最折磨人的地方。

**追问** ：这种不确定性问题，你有什么实际的应对方法？

答：工具描述当 prompt 写不当注释写，每改一次跑典型任务观察模型行为有没有退化。对话历史全量写进 JSONL 日志，出了问题回放日志逐轮排查，这是 Agent 工程里唯一可靠的调试手段。把确定性的部分做到极致，对话历史校验、入参校验、压缩熔断这些工程层越稳固，不确定性问题的排查范围就越小。

### 在使用 Claude Code 时候的心得和技巧？

考点：实际使用深度，区分「装了没用过」和「重度使用者」。

**参考回答** ：

我用 Claude Code 做了整个 MewCode 项目，积累了不少经验，说几个我觉得最重要的。

最有用的一点是 CLAUDE.md 一定要写好，但不要写长。我最开始写了 200 多行的 CLAUDE.md，什么代码风格、命名规范、提交格式都写进去了，结果 Claude 反而开始忽略某些指令。后来精简到 150 行以内，每条规则自问「删掉这行会导致 Claude 犯错吗？」不会就删。遵守率明显提高。最关键的是要写清楚项目的测试命令和 lint 命令，让 Claude 改完代码能自己验证，这一条顶十条风格规范。

在这个基础上，我越用越觉得上下文是最稀缺的资源。我现在的习惯是一个功能点一个会话，做完就 /clear。之前吃过亏，一个会话里又改功能又修 bug 又加测试，到后面 Claude 的输出质量明显下降，注意力被之前的对话内容分散了。复杂任务用 Subagent 做调研，比如让子 Agent 去探索某个模块的实现，只把结论带回来，主会话保持干净。

还有就是复杂任务一定要先 Plan 再 Code。我做 MewCode 的多 Agent 协调那块的时候，直接让 Claude 上手写，结果方向完全跑偏了，删掉重来。后来学乖了，先进 Plan Mode 让它把相关代码都读一遍，出个方案我看完确认了再动手，一次就对了。

**追问** ：还有没有其他小技巧？

答：几个比较实用的：/btw 快速问个不相关的问题不会污染上下文；修了两次还没修对就 /clear 重新来，写更好的初始 prompt 比在烂上下文里挣扎高效得多； 用 /effort 切换思考强度（low/medium/high/xhigh/max），让模型在复杂问题上花更多时间推理 。

**追问** ：你觉得 Claude Code 最大的缺点是什么？

答：终端交互看 diff 不方便。改了十几个文件，在终端里一个个看 diff 很痛苦，不如 Cursor 那种可视化界面直观。我现在的做法是让 Claude 改完后自己 git diff 输出给我看，或者我切到编辑器里用 git 插件看。另一个是网络依赖，模型调用全走云端，网络差的时候延迟很高。

### 如何用 AI coding 工具从需求到实现全流程？

考点：对 AI 辅助开发完整工作流的掌握。

**参考回答** ：

我做 MewCode 的时候总结出一套流程，核心就三个环节。

首先是把需求聊清楚。不要拿到需求就让 AI 写代码，这是最常犯的错。我会先开一个会话，把需求描述给 Claude，让它反过来问我问题：「这个边界情况你怎么处理？」「这个接口的错误码怎么定义？」「和现有模块怎么交互？」。聊两三轮下来，需求里很多没想清楚的细节就暴露出来了。聊完以后让 Claude 整理成一个 spec 文件，确认无误再动手。

需求确认了就进 Plan Mode 出方案。让 Claude 读完相关代码后出个实施方案，包含改哪些文件、大概什么思路、有哪些风险点。我看完确认了再退出 Plan Mode 开始写代码。这一步看似浪费时间，实际上省了大量的返工。我做压缩模块的时候跳过了这步，Claude 写了一版方案方向完全不对，改来改去两个小时白费。

方案定了就进实现。如果需要改大量文件，就用 Spec 模式，先让 Claude 生成需求规格和任务拆分，按 SDD（Spec Driven Development）的方式一步步推进，改完一步 commit 一步。每步完成后让 Claude 自己跑测试、跑 lint，通过了再继续。关键路径我会再开一个全新会话让 Claude 审查，全新上下文没有偏见更容易发现问题，同时频繁 commit 保证随时能回滚。

**追问** ：这个流程和不用 AI 的开发流程区别在哪？

答：最大的区别是反馈循环更快了。传统流程里需求分析、设计、编码、测试是比较线性的。用 AI 以后，我在第一步就能让 AI 帮我做方案推演，相当于写代码之前就「试跑」了一遍。而且迭代速度快，一个小改动从写到测试可能就几分钟，传统可能要半小时。但 代价是要花更多精力在审查和引导上 ，不是省心了而是换了一种劳动方式。

### 你怎么用 AI 做项目的，全部交给 AI 生成还是每次只提一部分的需求？

考点：对 AI 辅助开发粒度控制的实际经验。

**参考回答** ：

肯定是分着给的，一次全交不行。

我做 MewCode 的时候试过一次「大包大揽」，让 Claude 一口气把整个工具系统设计加实现全做了。结果它生成了一大堆代码，接口定义和我预期的不一样，内部状态管理的方式也不对，改起来比从头写还痛苦。从那以后我就不这么干了。

我现在的做法是按任务复杂度分。小活（改个变量名、修个拼写错误、调个格式）直接让 AI 做，不需要前戏。中等任务（加个功能、修个 bug）给明确的范围，比如「在 tool\_registry.go 里加一个热插拔接口，参考现有的 Register 方法，加完跑一下测试」。大活（架构调整、新模块设计）一定先聊方案再写代码，先 Plan Mode 让它读完代码出方案，确认了再实现，每个子任务单独一轮。

关键是一个原则：每次给的任务，小到 AI 在当前上下文里能完全理解，小到能看完它写的所有代码。如果一次改动超过 十个文件都审不过来了 ，那粒度就太大了。

做 MewCode 整个项目下来，我的体感是：AI 最大的价值不是替人写代码，是加速迭代。人定方向、拆任务、审结果，AI 负责把想法变成代码。 相当于从「打字员」变成了 「技术总监」，但技术判断力一点不能少。

**追问** ：你觉得什么时候应该完全不用 AI？

答：涉及到关键的架构决策时。比如 MewCode 选五层架构、权限系统设计五层权限拦截、多 Agent 通信选邮箱模式，这些决策我都是自己想的，不是让 AI 帮我选的。AI 可以帮忙做方案推演、列出利弊，但最终拍板必须是人。因为 AI 没有对项目全局的理解 ，它不知道未来三个月要往哪个方向走。

### 你怎么保证 AI 写的代码是正确的？

考点：对 AI 生成代码质量保障体系的理解。

**参考回答** ：

说实话，保证「正确」这个词有点绝对，更准确的说法是怎么尽早发现问题。前面聊约束稳定性的时候讲了 Hooks 和测试这些自动化手段，这里我更想说的是人怎么去发现 AI 留下的坑。

自动化能拦住的问题前面已经聊过了，测试和 lint 通过 Hooks 自动跑，确实能挡住大部分明显错误。但 AI 最容易犯的错不是语法和格式这些，而是「happy path 写得很对，异常路径漏了」。所以每次让 Claude 改完我都会过一遍 diff，重点就看两个地方：边界条件有没有处理，和现有代码的交互是不是对的。

真正关键的改动我会开第二个会话让 Claude 审查。写代码的那个 Claude 已经有了「我觉得这样是对的」的偏见，换一个全新上下文的视角更容易发现问题。再加上频繁 commit，每完成一个小功能就提交，出问题能快速回滚。AI 改代码的速度比人快很多，可能几分钟就改了十个文件，没有存档点的话回滚起来很痛苦。

**追问** ：有没有 AI 的错误是你所有这些手段都没拦住的？

答：有。最难拦的是「逻辑正确但设计不对」的问题。比如 AI 实现了一个功能，代码逻辑没问题，测试也过了，但它选的实现方案和后续扩展的方向冲突了。这种问题测试拦不住，lint 拦不住， 只有自己对整个系统有全局理解 才能发现。所以架构层面的决策不能依赖 AI，必须自己把控。

### Claude Code 有哪些常用命令？

考点：实际使用熟练度。

**参考回答** ：

我每天用得最多的就四个。

/clear 最频繁，每个独立任务前我都会清一次上下文。之前在一个会话里做了太多事，后面的输出质量明显下降，养成了这个习惯。

/compact 长任务必备。做 MewCode 的时候经常一个功能模块要跑几十轮对话，上下文涨得很快。我一般看到 70% 左右就手动 compact 一下，还可以带参数告诉它压缩时重点保留什么。

/plan 复杂任务前必用。进 Plan Mode 后 Claude 只读不写，先让它把相关代码都看一遍，出个方案我确认了再执行。避免了上来就改然后方向跑偏的问题。

/rewind 是救命的。Claude 改完发现不对，直接回退到上一个检查点，比手动 git checkout 方便。

其他像 /init、/review、/btw、/mcp 这些也会用到，但不是每天都碰。快捷键的话 Esc 中断和 Shift+Tab 切换模式这两个用得很多。

**追问** ：还有哪些不太常用但值得了解的？

答：/btw 挺值得知道的。 在一个任务中途突然想问 个不相关的问题，用 /btw 问完就过去了，不会进入对话历史，不会影响后续任务的上下文。CLI 参数的话，claude -c 能继续上一个会话，claude -p "task" 非交互模式可以用在脚本里。

### 有没有遇到过 AI 应用或者工具无法解决的场景？

考点：对 AI 工具局限性的诚实认知。

**参考回答** ：

遇到过好几次。

最印象深刻的一次是做 MewCode 流式解析的时候。Anthropic 和 OpenAI 的流式事件结构完全不同，Anthropic 是层次化的块结构，一个块走完才开始下一个；OpenAI 兼容协议是平铺的事件流，多个工具调用的 delta 是交错推送的。我让 Claude 帮我写 OpenAI 的流式 parser，它写出来的代码对单个工具调用没问题，但一旦遇到多个工具调用的交错 delta，就乱了。我反复描述问题，给了示例数据，Claude 改了三四版都不对。最后还是我自己画了个状态机的图，理清了每种 delta 应该路由到哪个累积器，手写了核心逻辑，让 Claude 帮我补边界处理和测试。

类似的还有一次调试压缩后的死循环 bug，根因是压缩丢了一段关键的 tool\_result，但症状表现为模型反复调同一个工具，因果链拉得很长，AI 只会在症状位置打补丁，最后还是我自己回放 JSONL 日志逐轮对比才找到。

这两个经历给我的感受是，涉及到跨多个层面的全局推理，或者需要隐性知识才能判断的问题，AI 目前确实搞不太动。碰到这种情况修了两三次还不对，我就果断自己上了，别跟 AI 死磕。

**追问** ：那你觉得这些局限未来能解决吗？

答：部分能。随着模型上下文窗口变大、推理能力变强，「因果链长的 bug」这类问题会逐步改善。但「隐性知识」的问题本质上不是 AI 的问题，是信息没有被显式化。如果团队把决策背景都写进文档、写进 CLAUDE.md，AI 就能用上这些信息。所以 Harness Engineering 强调知识库建设，某种意义上就是在把隐性知识变成 AI 能读的显性知识。

## Claude Code 源码八股相关

### 介绍一下 Claude Code 主循环 Query 的流程？

考点：对 Coding Agent 核心运行机制的理解深度。

**参考回答** ：

Claude Code 的主循环，我读源码印象最深的就是它其实特别朴素。入口是 QueryEngine.submitMessage()，用户消息进来后先把上下文准备好，系统提示词、CLAUDE.md 内容、git status 这些环境信息并行获取，拼到一起。然后就进入核心的 queryLoop()。

这个循环就是一个 while(true)，配一个可变的 state 对象在每轮之间传递状态。每轮做的事情概括起来就三件：先做上下文预处理，主要是各种压缩策略防止上下文爆掉，比如裁剪历史、微压缩、自动摘要这些，具体哪种触发取决于当前 token 量；然后调模型，拿到[[01基础_20SSE协议与流式响应|流式响应]]；最后看响应里有没有 tool\_use，没有就结束循环，有的话执行完工具、把结果拼回对话，continue 进入下一轮。

这里有个我觉得很巧妙的设计，就是 Streaming Tool Execution。模型还在流式输出的过程中，前面已经输出完的 tool\_use block 就已经开始并行执行了，不用等整个响应结束。我做 MewCode 的时候也参照了这个思路，用 Go 写了一个 StreamingExecutor，工具调用一解析完就扔进 goroutine 跑，最后 CollectResults() 统一收集。实测下来对多工具场景的感知延迟改善非常明显。

循环终止条件很简单：模型回复里没有任何 tool\_use，只产出文字，就 break 出去了。另外还有 maxTurns 限制作为兜底，防止死循环。

**追问** ：为什么用 while(true) 循环而不是递归调用？

答：递归每轮会在调用栈上叠一层帧，长任务跑几十上百轮就有栈溢出风险。循环只在一层帧里跑，状态通过对象显式传递，内存恒定，也方便做断点恢复。

### Claude Code 的记忆架构是什么？

考点：对 Agent 持久化记忆系统的设计理解。

**参考回答** ：

Claude Code 的记忆其实就两套机制，一个是静态注入，一个是动态检索。

静态注入靠的是 CLAUDE.md 层级加载。它会从多个位置去找配置文件，比如项目根目录的 CLAUDE.md、.claude/rules/ 下的规则文件、用户主目录的 ~/.claude/CLAUDE.md，还有企业级的全局配置。加载的时候从当前目录往上一层层遍历，每层都检查有没有这些文件，最后全部拼在一起注入到上下文里。它还支持 @include 引入其他文件，支持 paths: frontmatter 做条件注入，比如只有操作到某个目录下的文件时才加载对应的规则。这个设计让团队可以把编码规范、架构约定都写进 CLAUDE.md，跟代码一起版本管理。

动态检索靠的是 Auto Memory 系统。记忆文件存在 ~/.claude/projects/<slug>/memory/ 下面，有一个 MEMORY.md 做索引，指向各个主题文件。每轮用户输入进来的时候，系统会异步启动一个 prefetch，拿用户消息的文本去跟记忆文件的 frontmatter 做相关性匹配。这个匹配不是关键词搜索，而是用 Sonnet 模型做判断，选出最相关的几条，在工具执行后作为 attachment 注入。因为是异步的，模型流式输出的那段时间刚好把记忆检索做完了，不阻塞主循环。

我做 MewCode 的时候也实现了类似的记忆系统，不过简化了很多。MewCode 的记忆存在 .mewcode/memory/auto\_memory.json 里，每隔几轮对话用 LLM 提取关键事实存下来，下次对话直接全量注入。没有做 Claude Code 那种按相关性检索的机制，因为记忆条数少的时候全量注入反而更简单可靠。

**追问** ：为什么记忆检索用 LLM 做相关性判断，而不是向量搜索？

答：用户说「别用 mock」和记忆里写「integration tests must hit a real database」之间没有关键词重叠，关键词匹配搞不定。向量搜索要维护嵌入索引，记忆文件经常增删改，同步是额外负担。记忆文件通常就几十个，用 Sonnet 直接读 frontmatter 判断，成本低延迟可控，够用了。

### Claude Code 是如何去检索代码之间的相关性的？比如给一个中文描述的问题，它怎么找到对应的代码？

考点：对 Coding Agent 代码检索策略的理解。

**参考回答** ：

Claude Code 没有用[[01基础_09向量数据库的原理与选型|向量数据库]]，也没有提前建索引，核心思路就是让模型自己推理该搜什么，然后用 ripgrep 之类的传统工具去执行。工具层面就是 GrepTool、GlobTool、FileReadTool 这几个，分别负责正则搜索、文件名匹配、读文件内容。

关键在于搜索策略是模型自主决定的。比如用户用中文问「登录逻辑在哪里」，模型不会去做翻译或者关键词提取，它是直接根据自己对代码的理解去推理：先用 GlobTool 找 */auth 、* \*/login\* 这类文件名，用 GrepTool 搜 login、authenticate、session 这些英文关键词，读到一些结果后发现线索不够，再调整策略搜第二轮、第三轮。这个迭代过程跟人类开发者拿 grep 搜代码的思路其实是一样的，只不过模型的「直觉」来自训练数据里见过的大量代码库。

我读源码还注意到一个细节，系统提示词里明确写了「ALWAYS use Grep for search tasks, NEVER invoke grep or rg as a Bash command」。这不是随便写的限制，因为如果模型直接在 Bash 里跑 rg，原始输出可能几千行，直接塞进上下文就爆了。走工具层的话，结果会经过格式化和截断控制。

我做 MewCode 的时候也实现了 GrepTool，用 Go 的 filepath.Walk 加标准库正则来做。功能上比 ripgrep 简单不少，但核心思路是一致的：把搜索能力给到模型，让模型自己决定搜什么、怎么迭代。实际体验下来，模型对搜索策略的把控比我预期的要好很多，大部分场景两三轮就能定位到目标代码。

**追问** ：不用向量搜索，面对几百万行代码的大仓库效率够吗？

答：ripgrep 本身百万行级别全文搜索也就几百毫秒，搜索速度不是瓶颈。真正的瓶颈在模型的搜索策略，如果关键词选得不对就得多跑几轮，消耗的是 LLM 调用时间。但这个方案的好处是零预处理，clone 下来就能用，对任何语言任何项目都通用。

## Harness 相关

### Harness Engineering 是什么？

考点：对 AI Agent 工程方法论前沿的了解。

**参考回答** ：

简单说，裸模型就是一个推理引擎，它没有记忆、没有工具、没有执行环境，光靠它什么实际的事都干不了。Harness 就是给它补上这些能力的工程层。

核心公式是 Agent = Model + Harness。我喜欢用一个计算机类比来解释：Model 就是 CPU，只管算；Context Window 是 RAM，临时存点东西但断电就没了；Harness 是操作系统，管调度、管权限、管 I/O；Agent 是跑在上面的应用程序。 没人会拿一颗裸的 CPU 去跑业务，同理也不该拿 一个裸模型去做 Agent。

这个概念最早是 HashiCorp 的创始人 Mitchell Hashimoto 在 2026 年 2 月提出的。他的核心思想特别实用： 每当发现 Agent 犯了一个错误 ，就花时间工程化一个解决方案，让这个错误结构上不可能再发生。不是靠 prompt 叮嘱它别犯错，而是从工程上让它犯不了。六天后 OpenAI 发了一篇文章正式用了这个术语，Anthropic 和一些技术大佬也都跟进了深度分析。

Harness 具体包含什么呢？知识库（CLAUDE.md 这些项目指令文件）、工具集成（MCP、CLI）、权限约束（allow/deny 规则、沙箱）、 自动验证（CI、lint、测试） 、反馈回路（错误信号返回 Agent 让它自我修正）、状态管理（跨会话的持久化）。

说一个很能说明问题的数据：LangChain 团队在 Terminal Bench 2.0 上，底层模型一个参数没动，光优化 Harness，编码 Agent 得分就从 52.8% 飙到了 66.5%，排名从第 30 跳到第 5。这说明在 Agent 工程里，更好的 Harness 往往比更好的 Model 更重要。

**追问** ：Harness Engineering 和传统软件工程里的框架、中间件有什么本质区别？

答：最大的区别是 Harness 面对的「执行者」是不确定的。传统框架里，代码写了什么就执行什么，完全确定性。但 LLM 的行为是概率性的，在 prompt 里说「不要删除文件」，它 99% 的时候会听，但总有 1% 会忘。所以 Harness 必须用确定性的工程机制来兜底概率性的行为：权限系统硬拦危险操作，lint 在生成后立即检查， CI 检查挡住不合规代码 。本质上就是给一个「不完全可控的执行者」套一套「完全可控的约束系统」。

### 具体描述一下你对 Harness 工程的理解，它与 Context 工程/Prompt 工程的区别。

考点：对三代 AI 工程方法论演进的深度理解。

**参考回答** ：

这三个概念其实是包含关系，不是并列的。Prompt Engineering 管的是「怎么跟模型说话」，把指令写清楚、写结构化，模型回答就更准，但它只作用于单轮对话。Context Engineering 往外扩了一圈，管的是「给模型什么信息」，通过 RAG、检索、token 排序把正确的知识塞进上下文窗口，但它有个盲区：信息给够了，模型可能拿着信息干危险的事，Context 层没有任何机制拦它。Harness Engineering 再往外扩一圈，管的是「整个系统怎么运转」，在模型外围建确定性的 约束、反馈和验证 ，让 Agent 跨多个 context window 都能稳定工作。所以是 Prompt 在 Context 里面，Context 在 Harness 里面。

有个类比我觉得很到位：Prompt 是油门指令，Context 是燃油，Harness 是整辆车，方向盘、刹车、车道线、仪表盘全算。光踩油门加燃油，没有方向盘和刹车，车是要出事的。

我做 MewCode 的时候体感很明显：花时间调 prompt 的措辞，效果改善是有的但很有限。真正让 Agent 可靠性产生质变的，是加权限拦截、加错误信息回传、加工具输入校验这些 Harness 层的东西。Mitchell Hashimoto 说过一句话我印象很深：「每次 Agent 犯错就工程化一个永久修复」，我觉得这抓住了本质。换模型提升天花板，搭 Harness 提升落地能力。后来也有人把 Harness 细化成 约束、反馈、验证三层 ，思路是一样的，就是用确定性的工程机制把模型的不确定性兜住。

![](MewCode_项目面试题-11.jpeg)

**追问** ：这三者是替代关系还是叠加关系？

答：叠加的，不是说有了 Harness 就不需要 Prompt 和 Context 了。区别在于 ROI 变了。早期模型能力弱，打磨 prompt 回报最高；现在模型够聪明了，瓶颈不在理解意图，在行为可控性，所以 Harness 的投入产出比最高。

### 你这个项目有没有用到 Harness 的思想？

考点：能否将方法论和自己的工程实践对应起来。

**参考回答** ：

有的。我理解 Harness 就是除了模型本身之外，所有让 Agent 能稳定交付的东西。OpenAI 那篇博客里讲得很清楚，他们团队靠 Agent 写了一百万行代码，靠的不是模型多聪明，而是围绕模型搭了一整套约束、反馈、工具集成和状态管理的基础设施。换模型提升的是天花板，搭 Harness 提升的是落地能力。

我做 MewCode 的时候还没接触过这个概念，后来了解到才发现，我做的那些东西其实就是在给大模型套缰绳。虽然做得还比较粗糙，但核心方向都有覆盖。

最能体现 Harness 思想的是权限系统。MewCode 做了五层权限约束，任何一层 deny 就短路，后面的层不会翻转 。不管模型怎么想，rm -rf / 到了权限层就被硬拦住了 。 我不会指望在 prompt 里写 「别删文件」模型就百分百听话，必须用工程机制硬拦。

另一个花了不少精力的是反馈回路。工具执行失败后，错误信息不是简单地说「失败了」，而是结构化地告诉模型哪一行出了什么问题、应该怎么改。权限拒绝也一样，不是把 Agent 杀掉，而是作为 tool\_result 返回让模型调整策略。Harness 要解决的不光是「别跑偏」，还有「出了错能自己爬起来」。

再就是 MEWCODE.md 和记忆系统。Mitchell Hashimoto 最早提 Harness 这个概念，说的其实就是这个东西： Agent 犯了错，就往规则文件里加一条 ，下次它就不犯了，就像打疫苗一样，他管这个叫「复利效应」。MewCode 里的 MEWCODE.md 就是这个角色，项目规范、架构约定、编码习惯全写在里面，Agent 启动时自动加载。另外还有一层自动记忆，Agent 自己从对话中提取用户偏好和反馈存下来，下次会话也会带上。一个是人写的显式规范，一个是 Agent 自动积累的隐式经验，两者配合让 Agent 越用越懂这个项目。这块我觉得投入产出比最高，成本很低但效果很明显。

除了这三块重点做的，MewCode 其他几个方向也都有覆盖。上下文管理做了两层压缩，大结果存磁盘只留预览，上下文到大概 80% 多就自动触发 LLM 摘要，参照 Claude Code 的做法预留了 20K 的压缩输出空间和 13K 的当前轮次 buffer，实在压不动就丢最老的消息腾空间。[[理论学习_System_Prompt_如何设计_|System Prompt]] 是结构化组装的，七个来源分三个通道注入，稳定内容走 system 通道利用 Prompt Cache，动态信息走 system-reminder 不破坏缓存。工具系统每个工具都有元信息标记是否只读、是否可并发，注册中心统一管理，分批执行时根据这些标记决定并行还是串行。Agent Loop 有四种停止条件兜底，包括迭代上限和连续异常检测。Hook 系统在工具执行的关键节点挂了 15 个生命周期事件，可以做自动化的 lint、格式化这些。MCP 做了 transport 抽象，stdio 和 HTTP 两种传输对上层透明。会话持久化用 JSONL 追加写入，恢复时做消息链完整性校验。多 Agent 协调用 Git Worktree 做文件隔离，邮箱模式做异步通信。

**追问** ：这里面你觉得哪块做得最不够？

答：验证机制。 Agent 改完代码说「搞定了」，但测试没跑过、lint 有报错，目前没有强制挡住。如果重做，我会在任务完成前加自动验证，过不了就打回去继续改。另一个是学习闭环，Agent 执行完任务后没有自动提取经验生成可复用的 [[skill]]，每次都从零开始，这块后续想补。
