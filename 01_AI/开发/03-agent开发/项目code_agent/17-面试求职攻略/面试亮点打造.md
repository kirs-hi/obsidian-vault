# 面试官到底在听什么

![](面试亮点打造-1.jpeg)

很多同学介绍项目的时候，习惯从技术栈开始讲。「我用了某某语言、用了某某框架、用了 MCP 协议、用了 JSONL 做持久化...」这种介绍方式有一个致命的问题：它只告诉面试官你「用」了什么，没有告诉他你「想」了什么。

面试官听候选人讲项目的时候，核心在听三件事。

第一是设计决策能力。你能不能在有约束的情况下做出选择，并且讲清楚为什么选这个不选那个。「我用了 JSONL」是事实陈述，「我选了 JSONL 而不是 SQLite，因为 Agent 的写入模式是纯追加，JSONL 不需要加锁，崩溃恢复也更简单」是设计决策。前者谁都能说，后者证明你真的想过。

第二是真实的踩坑经历。教程和课程能教你正确的做法，但踩坑经历是抄不来的。你说「我做了上下文压缩」，面试官可能觉得你照着文档写的。你说「我做压缩的时候切错了边界，把一个 tool\_ use 和它对应的 tool\_ result 拆到了两段里，下一轮 API 调用直接报协议错误，排查了半天才发现是压缩的截断位置不对」，面试官就知道你真的做过这个事情。

第三是对边界的认知。你知道自己做了什么，也知道自己没做什么。你能说清楚 MewCode 和 Claude Code 的差距在哪， 比如 LSP 集成、OAuth 鉴权、官方 MCP 注册中心这些 MewCode 没有做，为什么没做 。这比号称「什么都做了」可信得多。

MewCode 涉及系统设计、协议解析、并发处理、安全防御和分布式协作，每个维度都有具体的设计决策可以聊。问题是很多同学会把它讲浅了，变成「我做了一个 Agent」。

所以你需要一套结构化的方法来提炼亮点。每个亮点的叙事公式是这样的：我遇到了什么约束，考虑了哪些方案，选了哪个、为什么，后来发现了什么预期之外的问题。这个结构天然就包含了面试官想听的三件事：决策、踩坑、边界。

# 七个高价值技术亮点

![](面试亮点打造-2.jpeg)

下面七个亮点是从 MewCode 项目里提炼出来的，每一个都按「场景、决策、结果」展开。你不需要在一场面试里全部用上，根据面试的方向挑两三个重点讲就够了。

## 流式协议归一化

MewCode 同时支持 Anthropic 和 OpenAI 兼容协议，两家的流式事件结构完全不同。Anthropic 的流式响应是层次化的块结构，一个 tool\_ use 块从开始到结束是连续的，走完一个才开始下一个。OpenAI 兼容协议是平铺的事件流，多个工具调用的增量是交错推送的，必须按索引把每个工具的增量分流到对应的累积器里。

我的做法是在 LLM Client 层吃掉这个差异。对外只暴露一套抽象事件：文字增量、工具调用开始、工具入参增量、工具调用结束、流结束。上层的 Agent Loop 和 TUI 只认抽象事件， 根本不用管底层走的是哪家协议 。这样未来加一个新的模型提供方，只需要写一个新的 Parser 把它的事件流归一化成抽象事件，Agent 代码一行不用动。

这个亮点之所以有面试价值，是因为它展示了接口抽象不是教科书概念，而是用来解决一个真实且棘手的协议差异问题。面试官如果在做多模型接入的公司，会觉得这跟他们的痛点直接相关。

## 上下文预算的渐进式管理

Agent 的一个任务可能要调几十轮 LLM，每一轮都带着完整的对话历史发请求。如果不主动管理，token 涨得很快，逼近上下文窗口上限之后 API 直接拒绝请求，整个会话就崩了。

我做了两层压缩，设计哲学是能轻则轻。第一层是大结果存磁盘，工具返回超过阈值的部分写到磁盘只留摘要，几乎零成本零损失。第二层是全量摘要，token 数逼近窗口上限时调 LLM 生成结构化摘要替换历史消息。早期我也试过更多层，比如中间加折叠旧轮次、渐进式压缩，但实测下来收益不大，最后简化成这两层就够了。

这里面最让我头疼的一个细节是压缩的截断边界。Function Calling 协议有一个硬约束：每个 tool\_ use 必须有且只有一个 tool\_ result 与之配对。如果压缩的时候不小心从中间切了一刀，把 tool\_ use 留在历史里但 tool\_ result 被压缩掉了，下一轮 API 调用会直接报协议错误。所以我的压缩逻辑必须保证只在完整的对话轮次边界上切割，任何时候历史里的 tool\_ use 和 tool\_ result 都是成对的。

这个约束看起来简单，但压缩摘要请求本身也会消耗 token，如果剩余空间不够做摘要请求，还需要一个降级方案直接丢弃最老的消息腾空间。

面试讲这个的时候，成本意识和生产级思维是得分点。每个用 LLM API 的公司都关心 token 成本，能讲清楚怎么管理上下文预算的候选人很加分。

## 对话历史一致性保障

刚才提到的 tool\_use 和 tool\_result 必须配对，这个协议约束不只在压缩时会被破坏。用户按 ESC 中断 的时候，如果正好在工具执行阶段，已经进入历史的 tool\_use 就会变成孤儿，没有对应的 tool\_result。进程意外崩溃恢复会话的时候也可能出现不完整的对话轮次。

我做了一个对话链校验逻辑。在三个关键时机运行：从 JSONL 日志恢复会话时、上下文压缩之后、用户中断之后。它会遍历整个对话历史，检查每个 tool\_ use 是否有匹配的 tool\_ result，发现孤儿就截断到最近一个完整的对话轮次边界。

这个问题的面试价值在于它是一个「你只有真的造过才知道」的问题。看教程不会告诉你 ESC 中断会破坏对话历史一致性，只有当你真的在终端里跑长任务、按了几次 ESC 之后发现 Agent 莫名其妙报错，排查半天才定位到历史里的孤儿 tool\_ use，你才会理解为什么需要这个校验。

面试里能讲出这种排查过程，比任何理论知识都有说服力。

## 五层权限拦截模型

一个能自己跑 Bash 命令和写文件的 Agent 是天然危险的。如果只靠一层防御，万一那一层出了 bug 或者配错了，就完全没有兜底。

我做了五层，从内到外依次是：危险命令正则检测、路径沙箱、三级规则引擎、四种权限模式、HITL 人在回路确认。设计原则是任何一层的 deny 不能被其他层翻转，只要有一层说不行，后面的就不用看了。

这里面踩过一个真实的坑。规则引擎有三层优先级：用户级、项目级、本地级。最早我把优先级搞反了，让项目级规则覆盖本地级。结果用户在本地配了一条 deny 规则禁止某个 Bash 命令，但项目级的 allow 规则把它盖掉了，命令被放行了。排查了半天才发现是优先级方向错了。改完之后专门加了单测覆盖这个 case。

路径沙箱也有一个容易忽略的点：symlink 解析。如果只检查路径字符串是否在项目目录下，攻击者可以创建一个 symlink 指向项目外的文件，路径看起来合法但实际指向了不该访问的位置。所以路径校验必须先把 symlink 解析成真实路径再做判断。

安全设计在面试中是稀缺项。大多数个人项目不涉及安全层面的考量，你能讲出多层防御、优先级冲突、symlink 绕过这些具体场景，会让面试官觉得你的工程成熟度很高。

## 多 Agent 协作与 Worktree 隔离

有些任务单个 Agent 搞不定，或者串行执行太慢。比如一个大型重构，涉及三个独立的模块，串行一个一个改可能要两个小时，但三个模块之间没有依赖关系，完全可以并行。

问题是多个 Agent 同时改同一个代码库会冲突。我用 Git Worktree 解决文件隔离：每个子 Agent 分配一个独立的 Worktree，有自己的工作目录，文件操作互不干扰。

Agent 之间用文件邮箱做异步通信，每个 Agent 有自己的收件箱目录，要发消息就往对方的目录里写一个文件，收消息就在 Loop 开头检查自己的目录。Coordinator Agent 是一个特殊角色，它放弃了所有代码操作工具，只保留调度能力，专门负责拆任务、分发、监控进度。

这个设计里最难调的 bug 是 Worker 把进度广播当成新指令。Coordinator 会给所有 Worker 广播其他人的进展，比如「Worker A 已经完成了模块 1 的重构」。Worker B 收到这条消息后，把它理解成了一个新任务，开始去检查模块 1 的重构结果，偏离了自己原本的工作。这个 bug 推动了消息类型的结构化设计：每条消息带上明确的类型标记（status\_update、task\_assignment、shutdown\_request），Worker 只响应分配给自己的任务类型，忽略纯状态更新。

多 Agent 协作是 Agent 工程里比较深的话题，面试官如果感兴趣，可以从文件隔离聊到通信机制再聊到协调策略，层次很丰富。

## Skill 热加载与模式隔离

固定的 Slash 命令只能覆盖预定义的功能。但不同类型的任务需要不同的工具集和提示词。做代码审查的时候不需要写文件的工具，做重构的时候不需要搜索引擎的工具。

我做了一个 Skill 系统。每个 Skill 是一个 YAML 元数据加 Markdown 指令体的文件。元数据声明名字、描述、适用场景和标签，指令体就是注入给模型的提示词。加载优先级分三层：项目级覆盖用户级，用户级覆盖内置。

关键的设计决策有两个。第一个是两阶段加载，也就是渐进式披露。平时模型只看到 Skill 的名称和简短描述，不占多少 token。当模型判断用户意图匹配某个 Skill 时，调用 LoadSkill 按需加载完整的提示词和专属工具。这样做的好处是 Skill 数量多了也不会撑爆上下文，同时收窄了模型的注意力范围，工具调用准确率更高。第二个是执行模式。inline 模式把 Skill 指令直接注入当前会话，适合轻量级的行为调整；fork 模式拉起独立子 Agent 执行，上下文隔离不污染主会话。

热加载的意思是不需要重启 Agent。Skill 目录下加一个文件，下一次调用的时候加载器重新扫描就生效了。这让你可以在一个会话中间临时加一个 Skill，不用丢掉已有的对话历史。

## MCP 工具按需加载

MewCode 通过 MCP 协议接入外部工具生态。用户可能配了好几个 MCP 服务器，加起来几十甚至上百个工具。每个工具的完整 schema 包含名字、描述、参数定义，少则几百 token，多的能到上千。如果全塞进上下文，光工具定义就吃掉好几万 token，还没开始干活上下文就占了一大块。

Claude Code 解决这个问题靠的是 Deferred Loading：平时只在系统提示词里列工具名字，不传完整 schema。模型需要用某个工具时先调一个 ToolSearchTool 搜索，搜到后返回一个 tool\_ reference 内容块，Anthropic 的 API 服务端识别到这个块后在下一轮请求里把完整 schema 展开注入给模型。这样几百个工具在上下文里只占几行文本，用到哪个才按需展开。

我在 MewCode 里参照了这个思路，但碰到了一个实际问题：tool\_ reference 是 Anthropic API 的专属协议能力，DeepSeek 这些兼容端点不支持。我实际测过，给 DeepSeek 发 tool\_ reference，模型收到后不知道工具的参数定义，只能靠名字猜，把 city 猜成了 location。

所以我改成了客户端自管理的方案：维护一个 discoveredTools 集合，ToolSearch 搜到工具后把它加进去，下一轮请求时客户端自己把这个工具的完整 schema 带上。效果和 Claude Code 一样，只是「展开 schema」这步从服务端搬到了客户端。

这个亮点的面试价值在三个层面。一是对 MCP 生态的深入理解，不只是「接了 MCP」，而是知道工具数量膨胀后的上下文管理问题怎么解。二是跨协议兼容的工程能力，同一个功能在 Anthropic 原生和第三方兼容端点上的实现路径不同，你能讲清楚差异和替代方案。

三是「发现服务端能力缺失后自己补位」的解决问题思路，这种从踩坑到绕过的经历面试官很爱听。

# 踩坑经历怎么变成加分项

![](面试亮点打造-3.jpeg)

面试里有一个反直觉的规律：「我搞砸了 X 然后这样修的」比「一切都很顺利」更加分。面试官不期待你一路完美，他期待你能从错误中学到东西。

MewCode 开发过程中有几个真实的踩坑经历，每一个都可以在面试里用。

第一个是前面提到的 Coordinator 误判。上下文压缩把 Worker 的进度报告做了摘要，摘要用了过去时态的表述，Coordinator 把摘要当成任务完成的信号。这个 bug 特别难排查，因为每一条日志看起来都是对的，只有把压缩前后的对话历史逐轮对比，才能发现语义信息在压缩过程中丢失了。

最终的修复方案是在 Coordinator 的提示词里加上具体的验证标准，不能只看文字描述，要检查实际的工具执行结果。

第二个是 Worker 把进度同步消息当成新指令。多 Agent 场景下，Coordinator 会给所有 Worker 广播其他人的进展。有一次 Worker A 收到「Worker B 已经完成了数据库迁移脚本」这条消息，它把这句话理解成了一个新指令，自己也去写了一遍迁移脚本，覆盖了 Worker B 的成果。 这个问题的根源是消息没有类型区分，进度通知和任务指令走的是同一个通道。正是这个 bug 推动了后来结构化消息类型的设计：每条消息带上 status\_update、task\_assignment 等类型标记，Worker 只响应任务类型，忽略状态更新。

第三个是流式断连导致的不完整 JSON。模型推送工具调用参数的时候是按 token 推的，中途任何时刻参数都是不完整的 JSON，无法解析。如果这时候网络断了，你手里就只有半个 JSON。我最初试过做精细的断点续传，维护每个工具调用的累积状态，但状态一致性的风险比重发的成本还高。最终的方案很简单：所有流式断连都视为整轮失败，丢弃这一轮的所有输出，重发上一轮的完整请求。

讲踩坑经历的时候有一个技巧：不要只讲「遇到了什么、怎么修的」，要讲「我一开始以为是什么原因、为什么那个判断是错的、真正的原因是什么」。排查过程比修复方案更能展示你的工程能力。

另外准备一个「如果重来一次会怎么做」的答案也很有价值。比如：工具执行应该支持流式输出，现在是执行完一次性返回，长命令跑的时候 UI 上看起来像卡死了。多 Agent 场景应该有一个可视化的状态面板，能看到每个 Agent 在干什么、进展到哪一步，现在排查问题只能翻日志。

整体开发策略上，前期不应该花太多时间做完整架构设计，应该先动手写、遇到问题再调整，细节驱动的迭代比前期规划更高效。

# 技术深度怎么展示不翻车

![](面试亮点打造-4.jpeg)

讲亮点的时候有一个度需要把握。太浅了面试官觉得你只是跟着教程走了一遍，太深了他跟不上你的思路或者觉得你在背课文。最好的深度是「比面试官预期的多讲一层」。

举个例子。面试官问你上下文压缩怎么做的，大多数候选人的回答是「用 LLM 生成摘要替换旧的对话历史」。你要多讲一层：「压缩必须对齐 tool\_ use 和 tool\_ result 的配对边界，否则下一轮 API 调用会报协议错误。而且摘要请求本身也会消耗 token，如果剩余上下文空间不够做摘要请求，需要一个降级方案直接丢弃最老的消息腾空间。」这一层信息证明你不是在复述概念，而是真的实现过、遇到过边界问题。

为每个亮点准备两个版本会很有帮助。一个 30 秒的短版，只说场景和结论，用来快速引出话题。一个 2 分钟的长版，包含决策过程和踩坑细节，用来应对深入追问。根据面试官的反应切换版本：如果他追问了，切长版；如果他点头想换话题，就停在短版。

同样重要的是诚实地标定自己项目的边界。MewCode 和 Claude Code 的差距不在架构思路上，而在细节覆盖上。Claude Code 有 LSP 集成能做代码智能补全，有 OAuth 鉴权流程，有官方 MCP 注册中心和工具市场，有文件修改历史的检查点方便回滚。这些 MewCode 都没有做，因为它们不是架构验证所必需的。

能说清楚这个差距在哪、为什么不做，比假装差距不存在强得多。

还有一条铁律：不要提到你答不上追问的东西。如果你提了「自适应上下文压缩」，面试官问「四个阈值分别是多少、为什么选这些数字」，你答不上来，效果比不提更差。每个你主动提到的技术点，都要能承受至少两层追问。

# 追问应对策略

![](面试亮点打造-5.jpeg)

面试的真正考验不在你主动讲的部分，而在面试官的追问。你讲的内容是准备过的，追问才能暴露你的理解是不是真的深。好消息是，追问也是可以提前准备的，因为每个亮点引出的追问方向是可预测的。

拿「多 Provider LLM 客户端」这个亮点来说。你说了之后，面试官第一个追问大概率是「两套协议具体差在哪」。你回答完之后，第二个追问可能是「如果一个 Provider 支持 Extended Thinking 另一个不支持，你怎么处理」。再往下可能会问「如果模型返回的工具调用格式两家不一样，你怎么归一化」。这三层追问对应的答案在你的开发经历里都有，提前想一遍就不会被打断节奏。

五层权限系统也是一样。你一说完，面试官可能会问「正则检测一定有漏网之鱼，你怎么看」。答案是正则只是第一层防线，它是尽力而为的，后面还有路径沙箱和 HITL 兜底。接下来可能问「LLM 有没有试过绕过权限」，这个你可以说模型确实会生成在权限边界上的命令，比如用管道拼接规避关键词匹配，所以单靠正则不够，路径沙箱和 HITL 是硬兜底。

多 Agent 的追问链通常是「为什么用 Worktree 不用 Docker」，接着「Worktree 改完怎么合并回去」，再接着「合并冲突怎么处理」。这些问题都有具体的技术答案，但如果你在面试现场才第一次想，回答的质量会差很多。

当追问超出你的准备范围时，不要硬编。一个有效的技巧是「桥接回来」：承认你对这个具体问题没有深入研究，但它跟你做过的另一个问题有关，然后把话题引到你有准备的领域。比如面试官追问到 Docker 容器的网络隔离细节，你可以说「容器方案我确实没有深入对比过，我选 Worktree 主要考虑的是文件系统级别的隔离足够了，而且 Git 原生支持合并流程。在多 Agent 通信层面，我花了更多精力在邮箱消息的时序问题上...」这样既诚实又不冷场。
