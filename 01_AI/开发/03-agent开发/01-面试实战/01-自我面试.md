### 1、你简历写熟悉 Function Calling 协议，能用大白话讲一下这个协议大概是什么形状？
考点：对协议本身的理解。
**参考回答** ：
**Function Calling 可以理解成，模型不会真的去调用工具，它只会“提需求”。**
比如我先把有哪些工具告诉它，比如读文件、改文件、跑命令，每个工具的说明叫什么、能干嘛、参数长什么样。
然后模型如果觉得需要用工具，它不会自己去执行，它会吐出一个结构化请求，意思是：**“我想调用这个工具，参数是这些。”**
这时候真正执行工具的是我们自己的程序，不是模型。  
我们程序拿到这个请求后，去本地执行，比如真去读文件、真去跑命令。执行完以后，再把结果塞回对话里告诉模型：**“刚刚你要的工具我已经帮你执行了，结果是这个。”**
模型看到结果之后，再决定下一步是继续调工具，还是直接回答用户。
所以这个协议本质上就是把两件事拆开：
- **模型负责决定要做什么**
- **系统负责真正去执行**
- **执行结果再反馈给模型**
这样做的好处是边界很清楚，也更安全。因为模型本身没有权限直接碰你的文件系统、终端或者外部接口，它只是提出“调用请求”，最后拍板执行的还是宿主程序。

**追问** ：如果模型一次返回了三个 tool\_use，你怎么处理？是串行还是并行？

答：要看工具是否并发安全。读类工具可以并发跑，写类工具必须串行。我的做法是按相邻同性质分组：连续的并发安全工具合并到一个并发批次，遇到非并发安全工具就开一个新串行批次。
三个 tool\_use 可能被切成几个批次按顺序执行，所有结果统一打包成 tool\_results 回传给模型，作为下一轮的输入

### ReAct 范式能用大白话讲一下吗？它和普通 LLM 调用有啥区别？

考点：范式理解。

**参考回答** ：

普通 LLM 调用是单次问答，问一句答一句。ReAct 是 Reasoning 加 Acting，把模型放进一个循环：模型先思考要做什么，然后调一个工具去做，拿到结果后再思考下一步，再调下一个工具，直到模型觉得任务完成不再调工具为止。

关键区别是状态延续。普通调用模型不记得自己上一次说了什么、做了什么，每次都要在 prompt 里塞历史。ReAct 把工具调用、工具结果都作为对话历史的一部分滚动进下一轮，模型能看到自己之前所有行为和结果，基于事实决策，不需要凭空想。

**追问** ：那如果模型思考是错的，比如选错了工具，怎么办？

答：靠工具结果反馈纠正。选错工具通常会得到错误输出或不符合预期的输出，下一轮模型看到这个结果会重新思考。

我的工程层会把工具错误明确标记 is\_error=True 回传给模型，让模型清楚知道刚才那一步失败了，又或者模型可以看结果发现自己是否上一步思考错误。如果模型连续多次选错，那是 prompt 或工具描述的问题，要回去优化提示词。

### Plan Mode 是什么？为什么要专门搞这个模式？
考点：用户控制权设计。
**参考回答** ：

Plan Mode 是**只读规划模式**：Agent 可以读文件、grep、glob 收集信息，Bash 也能用但限制为只读命令比如 ls、find、git log，不能写文件、不能执行会产生副作用的命令。模型最终输出一份结构化执行计划，等用户确认后再切换到执行模式。

为什么需要它：自动化执行的边界要交给用户控制。用户给的需求可能本身有歧义，或者想先看 Agent 准备怎么做再决定要不要让它真改。如果直接放它跑，万一方向跑偏，改完再回滚成本高。我做 MewCode 的时候就碰到过，让 Agent 重构一个模块，它理解偏了把接口签名全改了，下游调用全炸，回滚花了不少时间。从那之后我就觉得大改动一定要先出 Plan 让人看一眼。

**追问** ：用户看完 Plan 同意后，怎么切换出去？怎么保证 Agent 按 Plan 执行？

答：Agent 规划完成后会调用 ExitPlanMode 工具，用户在 UI 上确认同意，系统自动恢复到进入 Plan Mode 之前的权限模式，权限层立即生效。

工程层做了两件事来帮助按 Plan 执行。一是 Plan 会自动持久化写到磁盘文件，不怕上下文压缩丢掉。二是退出 Plan Mode 的时候，完整的 Plan 文本会通过工具返回值回灌到模型的上下文里，模型下一轮就是基于这份 Plan 继续工作的。再加上上下文连贯性带来的自然约束，实际效果已经很好了。当然如果用户切完模式后给个新指令，Agent 会优先响应新指令，不能百分百保证严格按 Plan 走。

追问 2：Plan Mode 输出的计划质量 怎么保证？有时候模型说要做 5 步，实际开干发现拆得太粗或漏步骤。

答：质量本质靠 prompt 约束。Plan Mode 的 system prompt 要求模型按固定结构输出：上下文分析、推荐方案、涉及的文件路径、可复用的现有代码引用、验证方式。每个步骤要写到改哪个文件的哪个函数这个粒度，不能是重构 auth 模块这种粗糙描述。

工程层没做计划质量自动评估，因为评估本身又要调 LLM 而且结果未必比 Plan 准。我的态度是把质量决策交给用户，看不顺眼就改。

经验上 Plan 出错最常见原因是模型没读够代码，prompt 里我强制要求 Plan Mode 至少调多次工具读相关文件。

**追问** ：用户同意 Plan 切到执行模式后，模型中途想到更好的方案偏离原 Plan，应该让它偏离吗？

答：允许偏离，但要标记。模型在执行中如果发现 Plan 的步骤 3 走不通，应该明确说出来：我发现 Plan 的某步不成立，原因是什么，现在调整成什么，然后继续。这是工程上没法强约束的，靠 system prompt 里要求偏离 Plan 时必须显式说明。

工程层能做的是审计：所有偏离行为通过对话历史可以回放，事后用户能看到哪一步背离了原计划。如果用户希望百分百严格执行不允许偏离，那 Plan Mode 应该改成自动化脚本生成而不是 Agent 模式。现在模型能力上来了，除非长程任务或方案本身太粗，基本不会偏离， 偏离了也往往会停下来确认 



### 你用 ReAct 还是 Plan-and-Execute？为什么？
先明确两者的核心机制。**ReAct**（Reasoning + Acting）是推理与行动交替的循环——模型先 Thought（思考当前该做什么）、再 Action（调用工具）、拿到 Observation（工具返回），根据新信息再 Thought，循环直到任务完成。它的特点是**每一步决策都基于上一步的真实反馈**，天然适应不确定性。**Plan-and-Execute** 则是规划与执行分离——Planner 先一次性生成完整步骤序列，Executor 按序执行，互不干扰。它的特点是**全局最优、可预测、可审计**，但对中间变化的适应性弱。

我们按**任务不确定性**来选。流程固定的内部工具（如日报生成）用 Plan-and-Execute，省成本。面对用户的开放任务（如旅行规划）必须用 ReAct，因为用户下一秒就可能改需求。

核心技巧是在 Plan 的每个步骤里嵌入“ReAct 式检查点”——比如调用天气 API 后，自动检查返回字段是否完整，不完整立刻触发修复，而不是让整个计划崩掉。


### Q：了解过 Agent 的设计范式吗？



好，我给你一个最精炼的理解框架，记住 Agent 设计范式只需要抓住一条主线和三个层次。

**一条主线：Agent = LLM + 循环**

所有 Agent 范式的本质区别，就在于"这个循环怎么转"。普通 LLM 是单次输入→输出，Agent 是把 LLM 放进一个循环里，让它反复地想、做、看、再想，直到任务完成。不同范式，就是不同的循环结构。

**三个层次，由简到复杂：**

**第一层：单步循环——ReAct**

最基础的范式。思考→行动→观察，一步一步来。就像一个人边想边干，干一步看一步。优点是简单可控，缺点是没有全局视野，容易走弯路。你记住"走一步看一步"这五个字就够了。

**第二层：先想后干——Plan-and-Execute**

先花时间列一个完整计划，然后按计划执行。执行中发现计划不对，回去改计划。就像你接到一个大项目，先写个方案再动手。比 ReAct 多了一个"抬头看路"的环节。记住"先列提纲，再写文章"。

**第三层：团队作战——Multi-Agent**

一个人忙不过来，拆成一个团队。每个 Agent 有自己的专长和职责，通过一个"经理"协调，或者彼此直接沟通。就像一个项目组：有人写代码、有人做测试、有人做 review。记住"分工协作"。
   典型的框架思路包括 supervisor 模式（一个主 Agent 调度其他子 Agent）和 peer-to-peer 模式（Agent 之间平等协作）。

**然后横切两个关键能力，贯穿所有层次：**

**工具调用**——Agent 的"手"。光会想不会动手没用，LLM 需要能调 API、查数据库、执行代码。这是所有范式的基础设施。

**记忆系统**——Agent 的"脑"。工作记忆是当前桌面，短期记忆是便签本，长期记忆是档案柜。没有记忆，Agent 永远是个新人。

**一句话总结**：Agent 范式的演进就是循环结构从简单到复杂的过程——从"边想边干"（ReAct），到"先想再干"（Plan-and-Execute），到"分头干"（Multi-Agent），再叠加工具和记忆两项基础能力。

你把这个"一条主线、三个层次、两个横切"的框架记住，遇到任何具体的 Agent 框架（LangGraph、CrewAI、AutoGen 等），都能快速定位它在这个图谱里的位置。

### Q：你的 Agent 项目用了什么框架？为什么选它？
没用 LangChain，主要三个原因：
1. **透明度**：Agent 的核心循环（推理→工具调用→结果整合）代码量不大，自己写更可控
2. **调试**：自己实现的代码，每一步的 Prompt 和响应都能直接打日志，不用穿透框架抽象
3. **灵活性**：我们的工具调用逻辑需要自定义的重试策略和错误恢复，框架的默认行为反而是限制

## 六道防线，从硬到软

CCB 防止多 Agent 踢皮球和死循环的设计不是靠某一个机制，而是六道防线叠加，任何一道被触发都会终止问题行为。

### 第一道：硬性轮次上限（maxTurns）

这是最暴力也最可靠的兜底。每个 Agent 在 `query()` 循环的每次迭代末尾检查 `turnCount`：

```typescript
if (maxTurns && nextTurnCount > maxTurns) {
  return { reason: 'max_turns', turnCount: nextTurnCount }
}
```

不同 Agent 类型有不同的上限：Fork Agent 默认 200 轮，自定义 Agent 可在定义中配置 `maxTurns`。不管 Agent 在做什么——即使它认为自己还需要继续——到了轮次上限就强制终止。这保证了即使所有其他防线都失效，系统也不会无限运行。

### 第二道：递归禁止 + 深度控制

踢皮球的本质是 Agent A 把任务交给 Agent B，B 又交回给 A，或者 B 再创建 C、C 创建 D……CCB 用两种机制斩断递归链。

**Fork 的双重递归检测**：主防线检查 `querySource`（抗 autocompact，不会因为上下文压缩丢失标记），后备防线扫描消息历史中的 `<fork-boilerplate>` 标记。两道检测任一命中就抛异常：

```typescript
if (toolUseContext.options.querySource === `agent:builtin:${FORK_AGENT.agentType}` 
    || isInForkChild(toolUseContext.messages)) {
  throw new Error('Fork is not available inside a forked worker. Complete your task directly.')
}
```

**工具黑名单切断派遣能力**：Verification Agent 的 `disallowedTools` 包含 `AgentTool`——它不能再创建子 Agent。Worker Agent 的黑名单包含 `TeamCreate`、`SendMessage`——它不能编排其他 Worker。Teammate 被明确禁止"spawn other teammates"（团队名册是扁平的）。这在架构层面保证了调用链是单向的、有深度限制的。

| Agent 类型 | 被禁止的关键工具 | 效果 |
|---|---|---|
| Verification | AgentTool, FileEdit, FileWrite | 不能派 Agent，不能改代码 |
| Explore | AgentTool, FileEdit, FileWrite | 不能派 Agent，只读 |
| Worker | TeamCreate, TeamDelete, SendMessage | 不能编排其他 Worker |
| Teammate | AgentTool（name 参数被禁） | 不能再 spawn teammate |

### 第三道：预算熔断（Cost + Token）

**成本上限**：通过 `--max-budget-usd` 设置整个会话的美元预算，每轮 turn 外循环检查，累计 API 成本达到阈值立即终止整个会话——不管有多少子 Agent 在运行。

**Token 递减收益检测**：当因 `max_tokens` 截断而自动续接时，如果连续 3 次续接每次新增不足 500 tokens，判定为递减收益，停止续接。这防止了 Agent 在一个方向上"空转"——不断生成但不产出有效内容。

**max_output_tokens 恢复上限**：最多 3 次恢复尝试（从 8K 升到 64K 再多轮恢复），超过直接终止，不会无限升级。

### 第四道：Prompt 层的反模式约束

这是"软"防线，但非常有效，因为它直接约束 LLM 的决策倾向。

**Coordinator 系统提示**中明确列举了"不要做什么"，并附带反面示例：

- 不要懒惰委派（Bad: "Based on your findings, fix the auth bug" → 应该自己做能做的事）
- 不要用一个 Worker 检查另一个（"Do not use one worker to check on another. Workers will notify you when they are done."）
- 不要过度并行

**Fork 子进程**的指令更加直接："Your system prompt says 'default to forking.' IGNORE IT — that's for the parent. You ARE the fork. Do NOT spawn sub-agents; execute directly."

**压缩后的运行中任务提示**："Do NOT spawn a duplicate. You will be notified when it completes." 防止 autocompact 丢失上下文后，Agent 以为任务没启动而重复创建。

### 第五道：状态机的 10 种终止路径

`query()` 的 `while(true)` 循环不是只有"正常完成"一个出口，而是有 10 种终止原因：

```typescript
type Terminal =
  | { reason: 'completed' }           // 正常完成
  | { reason: 'max_turns' }           // 轮次上限
  | { reason: 'blocking_limit' }      // 阻塞限制
  | { reason: 'prompt_too_long' }     // 上下文溢出
  | { reason: 'model_error' }         // 模型错误
  | { reason: 'aborted_streaming' }   // 流式中断
  | { reason: 'aborted_tools' }       // 工具中断
  | { reason: 'image_error' }         // 图片错误
  | { reason: 'stop_hook_prevented' } // Hook 阻止
  | { reason: 'hook_stopped' }        // Hook 终止
```

每一种都是强制退出点。这种穷举式的终止类型设计（TypeScript 联合类型，编译器会检查是否处理了所有分支）确保了不会遗漏退出条件。

同样，"继续"也有 7 种明确的原因（`Continue` 类型），不是随意的 `continue`，每次继续都必须声明理由——这使得循环的每次迭代都是有意识的决策，而非默认行为。

### 第六道：通信层的死锁防护

Team/Swarm 模式中，多个 Agent 通过 mailbox 异步通信，存在"所有人都在等消息，没人发消息"的死锁风险。CCB 有三重防护：

**Idle 检测与回调**：每个 Teammate 变为 idle 时通过 `onIdleCallbacks` 通知 Leader。Leader 可以通过 `waitForTeammatesToBecomeIdle()` 等待所有 Teammate 完成，不会无限等待。

**权限请求自动拒绝**：当 Pipe 逻辑断开时，如果 Teammate 发来权限请求（需要人类审批），系统自动 deny 而不是挂起等待：

```typescript
if (msg.type === 'permission_request') {
  client.send({ type: 'permission_response', behavior: 'deny',
    feedback: 'Permission auto-denied: pipe is logically disconnected.' })
}
```

**流式空闲看门狗**：90 秒无任何 chunk 到达，主动 abort 当前 stream，防止网络问题导致的无限挂起。

## 还有一道隐藏防线：autocompact 熔断器

`autocompact`（上下文压缩）本身也可能死循环——如果压缩总是失败，Agent 就会不断重试压缩而不干正事。CCB 设了 circuit breaker：连续 3 次压缩失败后停止重试。同时 compact agent 自身被豁免 blocking limit 检查，避免"压缩 agent 被 block → 无法压缩 → 主 agent 也被 block"的连锁死锁。

## 面试总结话术

> 防止多 Agent 踢皮球和死循环，核心思路是**分层防御，任一层触发即终止**。最外层是硬性约束——轮次上限、成本预算、重试次数，不管 Agent 在做什么，到了就停。中间层是架构约束——通过工具黑名单切断递归派遣能力（Verification Agent 根本没有 AgentTool，Worker 没有 TeamCreate），使调用链在结构上不可能形成环。内层是行为约束——系统提示中明确禁止懒惰委派、重复派遣、递归 fork，并附带反面示例教 LLM "什么不该做"。最底层还有通信协议的死锁防护——idle 回调、权限自动拒绝、流式看门狗。这六层中任何一层被触发，循环就会终止，它们互为备份、互不依赖。‘




工程化拆坑
## Q：了解 Kubernetes 吗？在 Agent 项目里有没有实际用到？

> 来源：视频面经汇总

**新手答**：“了解一些，用过 Docker 部署。”

**高手答**：

K8s 在 Agent 项目中主要解决三个问题：

**1. 弹性伸缩**：Agent 请求是典型的“长连接 + 高 token 消耗”——一个复杂任务可能执行 30s+。用 HPA 基于自定义指标（如当前活跃 Agent 会话数）做 Pod 扩缩容，而不是简单的 CPU/内存。

**2. 服务编排**：一个完整 Agent 系统拆成多个服务——Gateway、Planner、Tool Executor、RAG Service、Memory Service。K8s 的 Service + Deployment 天然适合管理这种微服务拓扑，配合 ConfigMap 做各环境配置隔离。

**3. GPU 调度**：如果自部署 Embedding 或 Rerank 模型，K8s 的 `nvidia.com/gpu` 资源类型 + Node Affinity 可以精确调度 GPU Pod，避免抢占。

**实际使用中的踩坑**：

- Agent 长任务 Pod 被 HPA 缩容杀掉 → 设置 `terminationGracePeriodSeconds` + 任务 checkpoint，被 kill 前保存状态
- SSE 长连接通过 Ingress 时被 nginx 超时断开 → 调整 `proxy_read_timeout` 到 300s+
- 向量数据库（Milvus）的 StatefulSet 扩容后需要 rebalance segment → 用 CronJob 定期触发

**差距在哪**：面试官不是要考你 K8s 八股，而是看你有没有在 AI 应用场景下用过。能说出 Agent 长任务的优雅终止、SSE 超时配置、GPU 调度，说明你真正在生产环境部署过 Agent 系统。
