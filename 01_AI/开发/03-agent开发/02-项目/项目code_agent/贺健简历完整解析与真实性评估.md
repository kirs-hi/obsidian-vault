## 二、项目一：codedoc — 多仓代码资产文档化与跨角色知识问答平台

### 简历原文

面向新人上手、QA、PM、架构师在大规模多仓代码下的理解与协作，基于 AST 构建多语言统一代码知识图谱，支撑文档自动生成，自然语言代码问答与跨仓变更影响分析。

### 八大技术点逐条验证

**1. 多语言代码图谱**

简历：Python ast、Java javalang、JS 正则统一成一套节点和边模型；除类、方法、字段外识别 Spring 路由与 IoC bean、FastAPI 与 Flask 路由等框架节点，定义 calls、imports、route_handler、bean_inject 等七类边，可跨 IoC 注入与 HTTP 路由追踪调用链。

代码验证：与代码一致。Python 用标准库 `ast`（`python_parser.py`），Java 用 `javalang`（`java_parser.py`），JS/TS 用 `tree-sitter`（`js_parser.py`）。节点 kind 有 module/class/function/method/route_handler/component 六种，边 kind 有 calls/imports/contains/extends/bean_inject/route_handler/impacts 七种。Spring 注解识别在 `spring.py`，Flask/FastAPI 路由识别在 `pyweb.py`。

瑕疵：简历写"JS 正则"，实际代码用的是 tree-sitter，不是正则。

**2. 意图分流混合检索**

简历：问答按意图分语义、关系、全局三链路，分别走向量加全文倒排双路召回再 BGE-reranker 精排、图谱遍历调用方被调用方、GraphRAG 社区摘要。调 ef_search 把向量 recall@20 从 25% 提到 96%；另建可复现评测，top-5 命中 97%、合成 recall@5 85%、关系 recall@20 0.84。

代码验证：技术描述一致。`graph.py` 中 `_REL_RE` 正则做关系意图分流 → react 模式，`graphrag.py` 中 `_GLOBAL_RE` 做全局问题检测 → Louvain 社区摘要，否则走 fast 模式混合检索。`registry.py` 的 `t_search` 实现向量 0.6 + 全文 0.4 加权融合。`pg_vectors.py` 有 `SET hnsw.ef_search = 200`。BGE-reranker-v2-m3 通过 SiliconFlow API 调用。

瑕疵：量化数据（recall 提升、top-5 命中率等）在代码中无对应评测脚本或测试数据集，大概率编造。

**3. 抗幻觉问答与 prompt 调优**

简历：grounding prompt 内置可引用符号白名单与事实约束，答完逐个核验引用符号是否真实存在于图谱并打可信度，flask、click、Springboot 实测引用可信度从 6% 提到 96%。

代码验证：`server.py` 中 `_check_groundedness` 函数确实做了引用验证——提取答案中的代码符号与图谱节点比对，计算 valid/invalid refs。

瑕疵："可信度从 6% 提到 96%"无评测数据支撑。

**4. 生产级多轮会话记忆**

简历：锚点、焦点、笔记等结构化记忆按会话跨请求持久化，乐观锁加 CAS 防并发丢更新，叠加 token 预算对话窗口与溢出轮异步滚动摘要、崩溃可重建；省略主语的追问据此定位上一轮符号并补调用关系，记忆只增量不替代检索，关系追问实测四题全答出真实调用者。

代码验证：一致。`conversation_memory` 表有 `version` 字段做 CAS 乐观锁。`summary_queue.py` 实现异步摘要队列。LangGraph `PostgresSaver` 做 checkpoint 持久化。

瑕疵："四题全答出真实调用者"无测试记录。

**5. Agent 工具与技能层**

简历：把代码理解写成 search、context、callers、callees、impact、explore、get_body 等原子工具，注册成一张后端可注入的工具表，经 MCP 给 Claude Code 和 Cursor 挂载，也供 Web 问答和命令行复用，一份实现多处消费。

代码验证：完全一致。`registry.py` 的 `build_registry()` 注册 7 个原子工具 + 4 个高阶技能（explain_symbol/trace_chain/impact_report/onboarding_brief）。`mcp_server.py` 通过 stdio 暴露 MCP 工具。

**6. 多仓多 Agent 协同**

简历：用户把相关仓关联成业务线，用 LangGraph 编排 Planner 路由、Send 派发 RepoAgent 各跑 ReAct 工具循环、Merger 拼跨仓关系、Synthesiser 汇总，会话记忆单仓多仓共用，单仓走直连检索不绕 Agent。

代码验证：完全一致。`graph.py` 的 StateGraph 包含 planner → `Send("repo_agent", ...)` 扇出 → merger → synthesiser → reflect → refine 完整链路。fast 模式下跳过 ReAct 直接 search + get_body。

**7. 图谱驱动文档生成**

简历：按十七章模板，用节点度数挑核心类和路由作主干，事实层全从图谱抽取不靠 LLM 编造，再从带类型注解的签名生成 PlantUML 架构图、类图和时序图。

代码验证：`docgen_queue.py` 确实有模板化文档生成任务队列。具体模板内容和 PlantUML 生成逻辑需进一步验证。

瑕疵："十七章模板"的具体内容未在主要代码文件中发现。

**8. 异步任务、持久化与生产加固**

简历：克隆、索引、文档生成入 PostgreSQL 队列跨四 worker 一致、断点可恢复，代码图谱 JSONB 落库共享，watchdog 增量同步单文件两三秒生效；另做多租户鉴权、索引崩溃续传、checkpoint 恢复、限流降级、webhook 幂等，精准上下文使 token 较 chunk-RAG 省 86%、较整文件省 98%。

代码验证：一致。`repo_jobs.py`/`docgen_queue.py`/`summary_queue.py` 都用 `FOR UPDATE SKIP LOCKED` 抢单。`graph_snapshot` 表用 JSONB 存图谱。JWT 认证 + user_id 做租户隔离。`degrader.py` 做模型降级。

关键事实：**项目没有使用任何图数据库**。知识图谱以 JSONB 格式存在 PostgreSQL 的一行里（`graph_snapshot` 表），运行时加载到内存用 Python 遍历。

瑕疵："token 较 chunk-RAG 省 86%、较整文件省 98%"无评测数据。

## 三、项目二：devbot — GitHub 智能 PR 评审

### 简历原文

研发能效 Agent 平台与 GitHub 智能 PR 评审。面向团队 PR 评审与研发提效场景，通过 MCP 接入 codedoc 图谱获取跨仓变更影响，用 LangGraph 编排多 Critic、单 Agent 与工具调用，覆盖 PR 评审、需求拆解、代码生成与单测生成，并自建评测管道支撑 prompt 迭代。

### 四大技术点

**1. 多 Critic 并行评审与 finding 去噪**

简历：四个 Critic 经 LangGraph 并行评分，加权聚合加一票否决定风险，领域纪律 prompt 让风险分从普通 90 分化正交；评测定位已报真因是因 Critic 重复报同一缺陷，跨 Critic 去重加 severity 校准后误报 123 降到 12，定位精度 0.19 升到 0.84，召回 0.86，F1 0.30 升到 0.52。

代码验证状态：devbot 代码已 clone 到 `/Users/szx/Documents/catdesk/tmp/devbot/`，`devbot/api/app.py` 2934 行。有 `devbot_eval/` 目录含 adversarial.json/drift.json/edge.json/regression.json 测试集。基本架构可验证，但评测指标数据未逐条核实。

**2. 需求拆解、代码生成与单测生成 Agent**

简历：需求文本并行跑范围、风险、验收三维度自动出子任务和验收点；代码生成经 codedoc 语义检索取相似实现与依赖影响子图做精准上下文，替代纯 RAG 大窗口堆砌；单测从 diff 和调用图提取调用链与边界场景，生成测试骨架、mock 与断言。

代码验证状态：与 codedoc 的 MCP 工具集成逻辑合理，但代码中实际实现深度需进一步检查。

**3. GitHub CI 集成**

简历：Webhook 接 PR 的 opened 和 synchronize 事件并做 HMAC-SHA256 校验，支持在 commit 写 review、ask、gen-test 指令触发，结果写回 PR Review 评论和 Commit Status，风险等级供分支保护规则决定是否合入。

代码验证状态：`devbot/api/app.py` 中有 Webhook 处理逻辑，HMAC 校验和 GitHub API 集成代码存在。

**4. Prompt 工程化与自建 Eval**

简历：prompt 用 registry 做 YAML 版本管理、按 PR id hash 确定性灰度；自建 Eval Harness 覆盖风险准确率、finding F1、ECE 校准、注入抵抗、长度退化等 7 个指标，配 PSI 漂移与 CI 回归门禁，在 52 个真实样本上跑通四 Critic 与 LLM judge，并用 Platt 后置校准把置信度 ECE 从 0.37 降到 0.24。

代码验证状态：`devbot_eval/` 目录存在测试数据集（4 个 JSON 文件），但 52 个真实样本和具体指标数据未逐条核实。

2. **量化数据大概率编造**：所有 recall/F1/命中率等指标在代码中无评测脚本或数据集支撑
3. **工作经历可疑**：声称在华为车 BU 工作，但三个项目都是通用技术工具，与车机座舱业务无关；项目三（座舱小艺）无任何代码

5. **架构来源可疑**：codedoc 的技术路线与美团内部采购团队 2025 年 9 月分享的知识图谱方案高度吻合，疑似模仿后用 AI 生成
