# codedoc 选仓与混合检索解析（codedoc_select.py）

## 1. 为什么要选仓

系统里有几十个代码仓库，用户问一个问题，不能每个仓都查：
- 贵：每个仓向量检索+rerank，几十个仓成本爆炸
- 慢：串行查几十个仓要几秒
- 没必要：大部分仓跟问题无关
- 有害：无关仓的弱证据会毒化答案

所以要先轻量挑出最相关的几个仓（3-5个），再交多 Agent 深挖。

**选仓的命门**：宁可多选不可少选。**漏仓=下游缺证据=瞎答**；多选只是多查一下不影响。所以 Recall 要尽量高，项目追求 Recall=1.0。

## 2. 核心概念

### 向量检索（Vector Search）
文本变向量，用余弦相似度算语义接近。项目用 BGE-M3 变向量，pgvector 存储，HNSW 索引加速。

HNSW：近似最近邻索引，建分层图结构快速定位近邻。关键参数 ef_search：搜索时邻居队列大小，越大越准越慢。

### Reranker（重排器）
向量检索是**双塔模型**——问题和文档各自变向量算余弦，快但精度有限（独立编码没互相看过）。
Reranker 是**交叉编码器**——问题+文档拼一起送进模型，精度高但慢（每个文档都要过模型）。

策略：先用向量粗召回 top-40（快），再用 reranker 精排（慢但准，只对小批量做）。

### Recall / Precision / F1
- Recall = 该选的仓里选对了多少（漏仓=Recall掉）
- Precision = 选的仓里该选的有多少（多选无关仓=Precision掉）
- F1 = 两者调和平均
- 选仓取舍：Recall 优先（宁可多选不漏）

## 3. 整体流程

```
用户问题（中文）
  ↓ 先翻译成英文（代码是英文的，跨语言检索更准）
  ↓ 全库向量检索，取 top-40 候选节点（不按仓，全局搜）
  ↓ 对 40 个候选用 BGE-reranker 交叉编码精排
  ↓ 取 top-15 节点
  ↓ 按仓聚合：每个仓权重 = 它名下节点的 rerank 分之和
  ↓ 留权重 ≥ 40%×最高仓 的那些仓
  ↓ 返回选中的仓列表
```

大白话：全局搜出最相关的 40 个代码符号，rerank 精排后取最好的 15 个，看这些符号分布在哪些仓，按"rerank分加权和"给仓打分，留分数够高的几个仓。

## 4. 逐段讲代码

### 4.1 中文翻译成英文（_to_english）

为什么翻译？代码是英文的，中文跨语言检索效果差，翻成英文同语言更准。实测 F1 0.89→0.96。

设计细节：
- lru_cache(512)：LRU 缓存避免重复翻译
- 先判断有没有中文，没中文免翻
- 翻译失败降级回原文

### 4.2 核心函数 select_by_nodes

步骤1：翻译
步骤2：问题变向量 + 全库检索 top-40
- _embed_query_cached：问题向量走缓存
- SET hnsw.ef_search = 200：关键参数，默认40偏低召回不够，设200召回25%→96%
- embedding <=> vector：pgvector 余弦距离算符
- LIMIT 40：全库取 top-40，不按仓查

为什么不按仓查？按仓查每个仓单独搜一次慢。全局搜一次拿 top-40，靠 repo_name 字段聚合到仓。

步骤3：reranker 精排
- 对 40 个候选用 BGE-reranker 精排
- 喂的不是原始 document，是 _code_repr 处理过的
- 返回 [(repo_name, score)] 按分降序

步骤4：按仓加权聚合
- 取 rerank 后 top-15 节点
- 按仓聚合：仓权重 = 名下节点 rerank 分之和
- 为什么用"和"不用"均值"？和能体现"多个节点都相关=仓更相关"。均值让"1个高分"和"10个高分"一样，区分不出。

步骤5：相对阈值过滤
- mx = 最高仓权重
- 留权重 ≥ ratio × mx（ratio=0.40）的仓
- 兜底：至少返回1个仓
- 为什么相对阈值不用绝对？rerank分绝对值没固定意义，不同问题分布不同。相对阈值跟问题难度无关更稳健。
- 为什么 0.40？13题测评调出的最优值。

### 4.3 _code_repr——喂给 reranker 的文本处理

问题：document 里 docstring 排在源码前面，直接截前N字会大半是注释把源码截掉。而注释是人写的可能错。

解决：找到 def/class/async def 开头行，把签名（前3行）+ 真实源码提前，跳过中间散文注释块，截前400字。

设计哲学：**代码不会撒谎，注释会**。rerank 优先喂源码，压低 docstring 权重。

### 4.4 select_scored——带置信度的版本

和 select_by_nodes 一样，但额外返回置信度 = 最高节点 rerank 分。这个置信度后面反思门控要用。

### 4.5 _reformulate——问题重表述

让 LLM 把问题换个说法，变成更利于代码检索的英文关键词。
为什么需要？同意思不同说法检索效果不同。"登录流程"vs"login authentication flow"，后者更匹配代码函数名。

### 4.6 select_with_reflect——置信门控反思选仓

流程：
1. 先选一遍，拿到 repos 和置信度 conf
2. conf >= conf_threshold（有把握）→ 直接用，不反思（省成本）
3. conf < conf_threshold（没把握）→ 反思：
   - LLM 把问题换说法（_reformulate）
   - 用新说法再选，拿到 repos2, conf2
   - 取置信度更高的一版

关键设计：
- 便宜门控：有把握不反思。反思要调 LLM（贵），只在没把握时才反思
- grounded：反思依据是真实 rerank 置信，不是 LLM 凭空说"我觉得不对"
- 取更高置信：不是无脑采纳第二版，而是比谁更有把握

### 4.7 route_question——问题路由

把选仓和 GraphRAG 统一：
1. 看问题是不是全局问题（is_global_question）
2. 跑选仓反思（select_with_reflect）
3. 不自信（选仓没把握=笼统/非代码问题）或命中全局关键词 → 走 global（GraphRAG）
4. 否则 → 走 repos（选中的仓 → 多 Agent 深挖）

为什么低置信也走 global？选仓没把握说明问题笼统（"这项目是干嘛的"），不是针对具体代码，GraphRAG 社区摘要更合适。

## 5. 量化指标（必背）

23题真索引测评：top=15 ratio=0.40 → **recall 1.00 / precision 0.93 / F1 0.96 / ~620ms**

select_eval.py 的 13题微服务多仓测评，对比 4 种策略。

Recall=1.0 是命门：漏仓=瞎答；多选不毒化（Merger 跨仓推理时弱证据贡献小）。

## 6. 关键设计决策

### 为什么全局搜不按仓搜
按仓搜每个仓单独查一次慢。全局搜一次拿 top-40，靠 repo_name 聚合到仓。快且统一。

### 为什么用"分之和"不用"分之均值"
和能体现"多个节点都相关=仓更相关"。均值让"1个高分"和"10个高分"一样，区分不出。

### 为什么相对阈值不用绝对
rerank 分绝对值没固定意义，不同问题分布不同。相对阈值（40%×最高）跟问题难度无关更稳健。

### 为什么 rerank 喂源码不喂注释
代码不会撒谎，注释会。优先喂源码，压低 docstring 权重。

### 为什么 ef_search 设 200
默认 40 太低召回不够（25%）。设 200 召回到 96%。召回和延迟的平衡点。

### 为什么有把握不反思
反思要调 LLM（贵）。有把握（rerank分高）说明选对了，不用反思。只在没把握时反思，成本控制。

## 7. 面试深问应答卡

### Q1：选仓怎么做的
A：全局向量检索 top-40 候选 → BGE-reranker 精排 → 取 top-15 → 按仓聚合（仓权重=名下节点rerank分之和）→ 相对阈值过滤（≥40%×最高）。中文先翻英文（F1 0.89→0.96）。23题测评 Recall 1.0 / Precision 0.93 / F1 0.96 / 620ms。

### Q2：Recall=1.0 为什么是命门
A：漏仓=下游缺证据=瞎答；多选近邻仓只多查一下，弱证据不毒化（Merger 跨仓推理时弱证据贡献小）。所以选仓宁可多不可少，Recall 优先。

### Q3：为什么用 reranker？放哪一步
A：向量检索是双塔模型，快但精度有限。reranker 是交叉编码器，精度高但慢。先用向量粗召回 top-40（HNSW快），再用 reranker 精排（只对小批量做）。不能对全库做 reranker，太慢。

### Q4：仓权重为什么用"和"不用"均值"
A：和能体现"多个节点都相关=仓更相关"。均值让"1个高分"和"10个高分"一样，区分不出。

### Q5：为什么相对阈值不用绝对
A：rerank 分绝对值没固定意义，不同问题分布不同。相对阈值跟问题难度无关更稳健。0.40 是测评调出的最优值。

### Q6：为什么中文先翻英文
A：代码是英文的，中文跨语言检索效果差。翻成英文同语言更准。实测 F1 0.89→0.96。LRU缓存避免重复翻，失败降级回原文。

### Q7：反思选仓怎么做
A：置信门控反思。选完看最高 rerank 分，够高直接用（省反思）；低于阈值=没把握，LLM 换说法再选，取置信更高的一版。反思依据是真实 rerank 置信，不是 LLM 凭空说。有把握不反思是便宜门控。

### Q8：ef_search 是什么为什么设 200
A：HNSW 搜索时邻居队列大小，越大越准越慢。默认 40 太低召回不够（25%），设 200 召回到 96%。召回和延迟的平衡点。

### Q9：rerank 喂的文本做了什么处理
A：把签名和真实源码提前，跳过散文注释。因为代码不会撒谎注释会。直接截前N字会大半是注释把源码截掉。

### Q10：问题路由怎么统一选仓和 GraphRAG
A：route_question 合并两个信号——选仓置信度和全局关键词。低置信或命中全局关键词走 GraphRAG；否则走选仓多 Agent。两个信号合并决策。
