## s01 Agent Loop — 怎么让 AI 动起来

**1. Agent 和聊天机器人有啥区别？**
聊天机器人是"一问一答"，Agent 是"自己干到完"。区别就在于**循环**：模型说完话后，程序检查它要不要调工具，要调就调，调完把结果喂回去，再问模型下一步——如此循环，直到模型说"我做完了"。

**2. 这个循环长什么样？**
就四步，绕圈跑：① 调模型 → ② 模型说要调工具？→ ③ 调工具，把结果塞回去 → ④ 模型说"完了"？没完就回到①。

**3. 循环什么时候停？**
看模型的 `stop_reason`。如果它想调工具，就是"没干完，继续"；如果它不调工具了，说明它认为任务做完了，循环结束。

---

## s02 Tool Use — 怎么让 AI 用工具

**1. 模型怎么"调用"工具？它又不能直接跑代码。**
模型只是输出一段"我想调用某个工具，参数是这些"的标记。真正的执行靠你的程序：解析这段标记 → 找到对应函数 → 执行 → 把结果包装好塞回对话。模型是"指挥官"，程序是"执行者"。

**2. 工具怎么告诉模型？**
用一个 JSONL 列表，每个工具写清楚名字、说明、参数 schema。模型靠"读说明"来决定该用哪个、传什么参数。说明写得好，模型就用得准。

**3. 一次能调几个工具？**
能调多个。模型一轮里可以输出多个 `tool_use` block，你的程序依次执行它们，把所有结果打包成一个 user 消息喂回去。

---

## s03 Permission — 怎么防止 AI 闯祸

**1. 为什么需要权限检查？**
Agent 能跑 bash、能写文件，一句 `rm -rf /` 就完蛋了。它没恶意，只是"不知道轻重"。所以执行工具前，得有人拦一下。

**2. 三道闸门是啥？**
- **第一道（黑名单）**：绝对的禁令，比如 `sudo`、`rm -rf /`，直接拦
- **第二道（规则匹配）**：看情况，比如"往工作目录外写文件"就可疑
- **第三道（问用户）**：拿不准就弹窗问人

**3. 权限检查插在哪？**
插在"调工具前"。循环里就是一行：`if not check_permission(block): continue`——权限不过，这个工具就不执行，直接返回"被拒了"。

---

## s04 Hooks — 怎么加功能不改循环

**1. 不用 hook 会怎样？**
每加一个检查（日志、git add、通知……）就改一次 `agent_loop` 函数体。改十次后循环变成一锅粥，没人看得懂，也不敢动了。

**2. Hook 是什么？**
一个**挂钩板**（字典），钉了几个挂钩（事件点）：开工前、干活前、干完后、下班前。想加功能就写个函数挂上去，循环只是"经过挂钩时把挂的卡片都执行一遍"。**循环一个字不改，功能随便加。**

==**3. Hook 怎么"叫停"？**==
==回调返回 `None` = 没意见，继续；返回非 `None` = "停！"。比如权限 hook 检测到危险命令，返回个字符串，循环就知道"这个工具别执行了"==。

### 阶段 2：做复杂任务

---

## s05 TodoWrite — 怎么让 AI 不跑偏

**1. Agent 为什么会跑偏？**
任务一长，工具输出把"最初的目标"挤出上下文了。做着做着就忘了最初要干啥，被中间的细节带跑。

**2. 怎么解决？给它一个便签本。**
干活前先写清单（todo_write）：1.干啥 2.干啥 3.干啥。干完一个划掉一个。这样它随时能看到"我还剩啥没做"。这个工具**不能干活**，只能规划——增加的是规划能力，不是执行能力。

**3. 忘了更新便签怎么办？**
加个催促：连续 3 轮没动过便签，就塞条提醒"更新你的待办"。

---

## s06 Subagent — 大任务怎么拆

**1. 为什么要拆？**
一个大任务（"重构整个认证模块"）本身是几十个小任务。全塞一个对话里，上下文被中间过程撑爆，Agent 越来越"健忘"，记不住最初要干啥。

**2. 怎么拆？开子 Agent。**
主 Agent 调 `task` 工具 → 派一个子 Agent，给它**全新的消息列表**（干净上下文）→ 子 Agent 专心干这一件事 → 干完**只把结论摘要**回传给主 Agent，中间过程全丢弃。

**3. 子 Agent 能随便开吗？**
不能。三个限制：① 子 Agent **没有 task 工具**，不能再套娃；② 子 Agent 的工具调用**仍然走权限 hook**，上下文隔离不代表权限放水；③ 有**轮数上限**（30 轮），防止跑飞。

---

## s08 Context Compact — 上下文满了怎么腾地方

**1. 为什么需要四层压缩？**
Agent 干活会产生大量中间过程（读文件、跑命令）。上下文窗口有限，满了就 `prompt_too_long` 报错。需要一套"便宜的先跑，贵的后跑"的管线。

**2. 四层分别干啥？**
- **L1 snip**：消息超 50 条，裁掉中间（留头 3 条+尾 47 条）
- **L2 micro**：旧的 tool_result 只留最近 3 条完整，更旧的换成一行占位符
- **L3 budget**：单条消息里 tool_result 总和超 200KB，把大的落盘到文件，上下文里只留预览
- **L4 compact**：前三层全跑完还不够，调 LLM 把整段对话总结成一条摘要

**3. 顺序为什么不能换？**
L3（budget）必须在 L2（micro）前面。因为 micro 会把旧的大 tool_result 替换成一行占位符，budget 必须在那之前把完整内容存盘，否则内容就丢了。

### 阶段 3：记住和恢复

---

## s09 Memory — 压缩会丢，要一层不丢的

**1. 压缩之后记忆为什么丢了？**
s08 的压缩是把对话总结成摘要，细节会丢。而且新开会话，连摘要都没了。LLM 没有持久状态，需要一层不参与压缩、跨会话保留的存储。

**2. 记忆怎么存？**
`.memory/` 目录下每个记忆一个 `.md` 文件（带 YAML frontmatter 记录类型）。`MEMORY.md` 是索引，一行一个链接，注入 system prompt。四类：user（你是谁）、feedback（怎么做事）、project（发生什么）、reference（东西在哪）。

**3. 记忆怎么自动写入？**
每轮对话结束时，提取器扫最近 10 条对话，发现"用户表达了偏好/约束/项目事实"就写成记忆文件。积累到 10 个文件时，整理器去重合并。

---

## s10 System Prompt — 运行时组装，不写死

**1. 为什么不能硬编码 system prompt？**
从 s01 到 s09，prompt 越加越长。换项目要重写整个 prompt，改一处可能影响全局，每次都带全部内容浪费 token。

**2. 怎么组装？**
把一大段字符串拆成独立段落（identity、tools、workspace、memory）。**始终加载**的每轮都带，**按需加载**的看真实状态（记忆文件存不存在、工具有没有注册）决定是否拼接。

**3. 为什么要缓存？**
同一轮对话里多次调 LLM，context 没变就不需要重新拼接字符串。用序列化 context 做 key，命中缓存直接返回。这不是 API 的 prompt cache，只是避免重复拼字符串。

---

## s11 Error Recovery — 错误不是终点

**1. 不处理错误会怎样？**
API 报 529（过载）、429（限流）、`prompt_too_long`、输出截断——这些不是 bug 是常态。不处理就是"一碰就熄火的车"，直接崩溃。

**2. 三条恢复路径各管什么？**
- **输出截断**（max_tokens）：先升级 8K→64K，还不够就让它"接着刚才说"（续写提示，最多 3 次）
- **上下文超限**（prompt_too_long）：触发 reactive compact（更激进的压缩），压缩后重试
- **临时故障**（429/529）：指数退避（0.5s→1s→2s…），连续 3 次 529 切换备用模型

**3. 恢复后怎么继续？**
每条路径恢复后都 `continue` 回到循环开头，重新调 LLM。不是重启，是带着修复后的状态继续。

### 阶段 4：让任务长期运行

---

## s12 Task System — 大目标拆成持久化任务图

**1. 和 s05 的 TodoWrite 有啥区别？**
TodoWrite 是"当前会话的便签本"，存在内存里，关了就没了。Task System 是"磁盘上的任务图"，每个任务一个 JSON 文件，有依赖关系，跨会话能恢复。

**2. 依赖怎么管？**
每个任务有 `blockedBy` 字段（依赖哪些任务）。`can_start` 检查所有 blockedBy 是否都 completed，没全完成就不能 claim。完成任务时自动扫描下游，找出刚被解锁的任务。

**3. 状态怎么流转？**
`pending →（claim）→ in_progress →（complete）→ completed`。claim 设置 owner（谁在做），complete 解锁下游。教学版没有 release 回退路径。

---

## s13 Background Tasks — 慢操作丢后台

**1. 为什么要后台？**
`npm install` 要 10 分钟，Agent 干等就是浪费（LLM 按 token 计费）。像洗衣机一样，扔进去按启动，去干别的，好了提醒你。

**2. 怎么判断该后台？**
模型可以显式说 `run_in_background: true`。如果没说，用启发式兜底——命令里有 install/build/test/deploy 等关键词就丢后台。

**3. 后台完成了怎么通知？**
先回一个占位 tool_result（"后台任务 bg_0001 已启动"）。后台完成后，结果包装成 `<task_notification>`，在下一轮和正常 tool_result 一起注入 messages。不复用原来的 tool_use_id（因为已经回复过了）。

---

## s14 Cron Scheduler — 给 Agent 装闹钟

**1. 为什么需要 cron？**
s13 的后台任务是"你手动触发的慢操作"。但"每天 9 点跑测试"这种定时任务，不该需要人每次来推。闹钟不需要你盯着才会响。

**2. 四层模型是什么？**
- **Scheduler**：daemon 线程每秒检查"时间到了没"
- **Queue**：到了就把任务塞进 `cron_queue`
- **Queue Processor**：发现队列有任务且 Agent 空闲，拉起一轮执行
- **Consumer**：agent_loop 从队列拿已触发任务，注入 messages

四层解耦：生产者（查时间）、交付者（找空闲）、消费者（执行）各管各的。

**3. durable 是什么意思？**
任务定义写进 `.scheduled_tasks.json`，Agent 重启后能恢复。但前提是 Agent 进程得在跑——进程关了调度也停。durable 只保证任务定义不丢。

### 阶段 5：让多个 Agent 协作

---

## s15 Agent Teams — 一个搞不定，组队
队友+邮箱通信

**1. 和 s06 子 Agent 有啥区别？**
子 Agent 是"临时工"，干完一件事就走，只回传结论。队友是"同事"，能多轮通信，通过收件箱随时交流，能协作。

**2. 怎么通信？**
文件收件箱——每个 Agent 有个 `.jsonl` 邮箱。发消息 = 往对方文件里 append 一行 JSON。读消息 = 读文件 + 删除（消费式）。

**3. 队友怎么工作？**
Lead 调 `spawn_teammate` 启动队友线程。队友有自己的 system prompt、自己的 messages、自己的简化工具集（bash/read/write/send_message）。做完后自动发 summary 给 Lead。Lead 每轮检查收件箱，把队友消息注入 history。

---

## s16 Team Protocols — 队友之间要约定 收发格式

**1. 为什么不能直接杀线程关机？**
Lead 想让 Alice 关机，直接杀线程，Alice 写了一半的文件留在磁盘上。需要握手：Lead 发请求，Alice 确认收尾后关机。

**2. 协议怎么工作？**
四方步：① 发请求（带 request_id）→ ② 队友收到并 dispatch → ③ 队友回复（带同一个 request_id）→ ④ Lead 收响应，通过 request_id 匹配，更新状态。`request_id` 是贯穿全链路的关联键。

**3. 队友做完任务不退出了？**
s15 的队友跑完 10 轮就退。s16 加了 idle 阶段——做完任务不退，每秒轮询收件箱，收到 shutdown_request 才回复合退出，收到新消息就继续干。

---

## s17 Autonomous Agents — 自己看板自己认领

**1. 为什么需要自治？**
s16 里队友等 Lead 分配任务。看板上有 10 个任务，Lead 得手动 assign 10 次。队友应该自己看板，发现没人做的就认领，做完再找下一个。

**2. 怎么自动认领？**
队友空闲时每 5 秒扫描任务看板，找"pending + 无 owner + 依赖已完成"的任务。找到就 claim，不用 Lead 操心。`claim_task` 检查 owner，避免重复认领。

**3. 队友什么时候退出？**
60 秒空闲没新任务就超时退出。或者收到 shutdown_request 立即退出。三阶段循环：WORK → IDLE → SHUTDOWN。

---

## s18 Worktree Isolation — 各干各的目录

**1. 为什么要隔离目录？**
s17 里 Alice 和 Bob 在同一个目录工作，都改 `config.py`，互相覆盖。分不清谁的改动是谁的，也无法干净回滚。

**2. 怎么隔离？**
用 git worktree——同一仓库创建多个独立工作目录，每个有自己的分支。Alice 在 `.worktrees/auth-refactor/`，Bob 在 `.worktrees/ui-login/`，互不干扰。

**3. 任务和 worktree 怎么绑定？**
创建 worktree 时可选传 `task_id`，绑定写到 task 的 `worktree` 字段。队友认领带 worktree 的任务时，bash/read/write 自动切换到那个目录执行。绑定不改任务状态（还是 pending，等 claim 才推进）。

### 阶段 6：接外部能力合体

## s07 Skill Loading — 知识用到才加载

**1. 为什么不能把所有文档塞进 system prompt？**
React 规范 2000 行、SQL 指南 1500 行、API 文档 3000 行……全塞进去，改 CSS 颜色时也带着 SQL 指南，99% 白花 token。

**2. 两级加载是什么？**
- **第一级（目录）**：启动时扫 `skills/` 目录，把每个技能的"名字+一句话描述"塞进 system prompt（每条 ~100 token，每轮都带）。Agent 知道"我有哪些技能"。
- **第二级（内容）**：Agent 需要完整规范时，调 `load_skill("sql-style")`，完整内容通过 tool_result 注入（~2000 token，按需）。

**3. 技能内容和 system prompt 有什么区别？**
技能内容是一次 tool_result，进了对话历史就跟着走，不是每轮都重复带的。system prompt 每轮都全量发。

---

## s19 MCP Plugin — 外接工具，标准协议

**1. 为什么需要标准协议？**
s01-s18 的工具都是手写的。现在要接 Jira、部署系统、Notion……不想为每个服务重写一套代码。外部服务只要实现标准协议，Agent 就能直接调用，不管服务用什么语言写的。

**2. 怎么接入？**
调 `connect_mcp("docs")` 连接外部 server。server 通过 `tools/list` 暴露自己的工具，Agent 组装进工具池。工具名加前缀 `mcp__server__tool` 避免不同 server 的工具名冲突。

**3. 工具池变了怎么办？**
s10-s18 有 prompt cache 避免重复拼字符串。s19 去掉了——因为 `connect_mcp` 后工具池变了（新增了 `mcp__docs__search` 等），缓存的旧工具列表会导致模型调不到新工具。每次重新组装。

## s20 Comprehensive — 机制很多，循环一个

**1. 这章讲了什么新东西？**
没有新机制。把前 19 章的所有机制（工具、权限、hooks、todo、任务图、记忆、压缩、错误恢复、后台、cron、团队、协议、自治、worktree、MCP）挂回**同一个循环**上。

**2. 所有机制挂在循环哪里？**
- 用户输入前后：UserPromptSubmit hooks
- LLM 前：cron 通知 + 后台通知 + 压缩管线 + 组装 system prompt
- LLM 调用：错误恢复（重试/升级/压缩/换模型）
- 工具执行前：PreToolUse hooks + 权限检查
- 工具执行时：慢操作丢后台
- 工具执行后：PostToolUse hooks
- 没有工具调用时：Stop hooks

**3. 核心始终没变的是什么？**
不管外面挂了多少机制，循环本身永远是四步：

```python
while True:
    response = LLM(messages, tools)
    if not has_tool_use(response.content):
        return
    results = execute_tools(response.content)
    messages.append(tool_results)
```

**复杂性在 harness（支架），不在"另一个 agent 大脑"。** 模型负责判断和行动选择；harness 负责把环境、工具、权限、记忆、团队组织好。
