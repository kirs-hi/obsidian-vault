## 概述

Friday 平台预置了 12 个评测指标，按链路分为三层：

| 层级 | 指标 | 数量 | 特点 |
|------|------|------|------|
| 召回层 | 知识有召回率、知识命中率、上下文召回率、上下文精确率、召回知识冲突率 | 5 | 衡量"检索"阶段的质量 |
| 知识库层 | 知识覆盖率 | 1 | 衡量"知识库本身"的完善度，与检索无关 |
| 端到端层 | 正确拒答率、错误拒答率、答案相似度、答案准确率_相似度、答案准确率_大模型、答案缺漏率、答案错误率、答案无幻觉率、答案忠实度、正确知识转化率_相似度 | 10 | 衡量"最终答案"的质量 |

> 指标定义以起源为准：[FRIDAY 应用工厂指标体系](https://origin.sankuai.com/kpi?busiLineId=338)

---

## 一、召回层（5 个指标）

### 1. 知识有召回率

**一句话**：召回了没有？到底有没有返回任何结果。

**算法**：检查 `recallResponse` 中 `docRecallItems` 或 `faqRecallItems` 不为空，有就是 1，没有就是 0。无 `recallResponse` 则直接为 0。

**公式**：

```
知识有召回率 = 有召回结果的 traceId 数 / 全部测评集 traceId 数
```

**依赖**：不依赖 Ground truth，不调 LLM。

**解读**：最基础的底线指标。这是"有没有"的问题，不是"对不对"的问题。如果这个指标很低，说明系统就没在召回，需要先排查工程链路。

**对比 Ragent**：Ragent 没有直接统计"有召回的比例"，但 `requires_rag` 字段 + 误拒率（该召回但没召回的占比）本质上是反向视角。Friday 这个是正向的"覆盖率"。

---

### 2. 知识命中率

**一句话**：有召回的样本里，召回内容到底有没有包含正确答案？

**算法**：

- **docQA**：ground truth 和每个 doc 切片逐个算相似度，取最大值，≥ 阈值 k1 则记为命中
- **faqQA**：ground truth 和每个 answer 逐个算相似度，取最大值，≥ 阈值 k2 则记为命中
- 以上两个条件取"或"：任意一个命中就算命中
- 无 `recallResponse` 直接取 0

**公式**：

```
知识命中率 = 有知识命中的 traceId 数 / 有召回上下文的 traceId 数
```

**依赖**：依赖 Ground truth，不调 LLM（纯相似度计算）。

**解读**：衡量召回阶段的准确度。如果指标低，需要补充相似问法或提 tt 优化召回。

**对比 Ragent**：这是 Ragent 自建 `Hit@K` 的 Friday 对应版。区别在于：
- Ragent：在文档 ID 集合上做集合交集（确定性的，二元结果）
- Friday：在文本内容上算相似度（连续值 + 阈值，能处理"意思差不多"的情况）
- Friday 把 docQA 和 faqQA 分开处理

---

### 3. 上下文召回率（Context Recall）

**一句话**：标准答案里提到的每一个点，召回上下文覆盖了多少？

**算法**：

1. LLM 把 ground truth 拆成多条子句
2. LLM 逐条判断每条子句能否从召回块中推断出来
3. 覆盖比例 = 能在召回块中找到的子句数 / total 子句数

**公式**：

```
上下文召回率 = 覆盖子句数 / ground truth 子句总数
```

逐样本计算后，筛选"有召回"的样本取均值。

**依赖**：依赖 Ground truth + 依赖 LLM。

**解读**：衡量检索到的上下文与标准答案的一致程度。数值越高说明召回上下文涵盖正确答案越完整。如果指标低，说明知识库不完善。

**对比 Ragent**：完全对应 RAGAS 的 `context_recall`。Friday 的算法描述跟 RAGAS 的做法一模一样——拆 statement → LLM 判断能否推导 → 算覆盖比例。与 Ragent 自建 `Recall@K` 的区别：

| 维度 | 自建 Recall@K | Friday/RAGAS Context Recall |
|------|---------------|---------------------------|
| 粒度 | 文档级 ID | 语义子句级 |
| 算法 | 集合运算 | LLM-as-judge |
| 速度 | 秒级 | 慢 |
| 能发现 | 文档找到没 | chunk 切得对不对 |

---

### 4. 上下文精确率（Context Precision）

**一句话**：相关的召回片段有没有排在前面？

**算法**：

1. LLM 判断每个召回块是否包含 ground truth 相关内容
2. 对 k=1 到 K，依次计算前 k 个召回块中包含 ground truth 内容的比例（precision@k）
3. 各 precision@k 加权求和，除以 K 个召回块中包含 ground truth 的块数
4. docQA 和 faqQA 分开计算，取较大的值

**公式**：

```
Context Precision = Σ(k=1..K) precision@k / (包含 GT 的块数)
```

**依赖**：依赖 Ground truth + 依赖 LLM。

**解读**：对标业界 NDCG。关注排序质量——因为 LLM 有 "lost in the middle" 现象，排在后面的内容容易被忽略。指标越高说明检索排序越合理。

**对比 Ragent**：Ragent 使用 `MRR@10` 来衡量排序（只看第一个命中文档的排名倒数），算法更简单。Friday 的 Context Precision 更精细——整体评估前 K 个里相关内容的排名分布，类似 NDCG 的加权思路。

---

### 5. 召回知识冲突率

**一句话**：召回的内容里，有没有两篇文档说了互相矛盾的话？

**算法**：

- docQA：暂无法计算，默认无冲突
- faqQA：两两比较 FAQ 项的 question 和 answer
  - 若 question 相似度 > k1（问题很像）
  - 且 answer 相似度 < k2（答案却不一样）
  - 则认为存在知识冲突

**公式**：

```
召回知识冲突率 = 有冲突的 traceId 数 / 有召回结果的 traceId 数
```

**依赖**：不依赖 Ground truth（不需要标注标准答案），但需要 LLM 判断相似度。

**解读**：衡量知识库本身的质量问题。例如 FAQ 中一条说"退货期限 7 天"，另一条说"退货期限 15 天"——两个问题问的都是退货期限，答案却不一致，这就是知识冲突。

**对比 Ragent**：这是 Friday 独有的指标，Ragent 评测体系中没有。Ragent 的检索指标只关心"有没有命中正确的"，不关心"命中的内容之间是否矛盾"。这个指标在生产环境中非常有价值——知识库维护时间长了，不同人写的内容容易产生矛盾。

## 二、知识库层（1 个指标）

### 6. 知识覆盖率

**一句话**：标准答案需要的那部分知识，知识库里到底存不存在？

**算法**：

拿 ground truth 去绑定知识库中全局检索，通过相似度判断：
- docQA：ground truth 与 doc 切片算相似度，≥ 0.6 → 覆盖
- faqQA：ground truth 与 answer 算相似度，≥ 0.65 → 覆盖
- 任意匹配到一个就算覆盖

**公式**：

```
知识覆盖率 = 知识库覆盖的 traceId 数 / 全部测评集 traceId 数
```

**依赖**：不依赖 Ground truth（实际上用到了 ground truth 但作为 query 去检索），不调 LLM。

**解读**：这是衡量知识库完善程度的第一道检查——跟检索系统无关，单纯看知识库本身有没有这个内容。如果知识覆盖率就低，后面的检索指标再好也没用，因为"巧妇难为无米之炊"。

**对比 Ragent**：Ragent 评测体系中用 `init` 阶段确保知识库灌好文档后再评测，但没有单独的"知识覆盖率"指标。Friday 把它作为前置指标，先判断知识库是否完善，再判断检索是否有效——这个分层设计的思路更好。

## 三、端到端层（8 个核心指标 + 2 个衍生指标）

### 7. 正确拒答率

**一句话**：不该回答的问题，系统有没有正确拒绝？

**触发条件**（两个条件取"且"）：

1. 大模型输出内容是应用的兜底文案（LLM 判断）
2. 知识覆盖率 = 0（知识库里没有这个知识）

**公式**：

```
正确拒答率 = 正确拒答的 traceId 数 / 全部测评集 traceId 数
```

**依赖**：依赖 Ground truth（间接，因为需要知识覆盖率），需要 LLM 判断是否为兜底文案。

**解读**：衡量系统的"边界意识"——知识库没有的就别说。如果指标低，说明系统在不该回答的时候乱编了。

**对比 Ragent**：Ragent 没有单独的"正确拒答率"，但有 `requires_rag` 字段分流。Friday 把这个拆得更细——正确拒答 + 错误拒答两个指标配合看。

---

### 8. 错误拒答率

**一句话**：知识库里有答案，系统却说"不知道"？

**触发条件**（两个条件取"且"）：

1. 大模型输出内容是应用的兜底文案（LLM 判断）
2. 知识覆盖率 = 1（知识库里明明有这个知识）

**公式**：

```
错误拒答率 = 错误拒答的 traceId 数 / 全部测评集 traceId 数
```

**依赖**：不依赖 Ground truth，但需要 LLM 判断是否为兜底文案。

**解读**：衡量系统的"过度保守"——该答的不答。如果指标高，需要调整 prompt 或模型参数。

**对比 Ragent**：对应 Ragent 的"误拒率"——`requires_rag=true` 但系统没有给出 RAG 答案的比例。本质上是同一个概念。

---

### 9. 答案相似度

**一句话**：系统输出的答案和标准答案长得像不像？

**算法**：用 Rouge-2 Recall 计算输出答案与 ground truth 的文本相似度。

**公式**：

```
答案相似度 = Σ Rouge2Recall(answer_i, ground_truth_i) / 样本数
```

**依赖**：依赖 Ground truth，不调 LLM（纯文本相似度计算）。

**解读**：最直观的指标。但注意：相似度高不代表语义正确，相似度低也不代表错（可能只是改写了一下）。需要配合其他指标一起看。

**对比 Ragent**：Ragent 自建指标里没有这个维度的，因为它只评检索不评内容。Friday 用 Rouge-2 做文本级别的快速判断，再配合"答案准确率_大模型"做语义级别的精确判断。

---

### 10. 答案准确率_相似度

**一句话**：在模型不编造的前提下，输出答案和标准答案够不够像？

**前置条件**：

1. 大模型输出非兜底文案
2. 答案忠实度 = 100%（没有编造内容）
3. Rouge-2 Recall ≥ 阈值 k

三个条件取"且"，全部满足才算准确。

**公式**：

```
答案准确率_相似度 = 判定准确的 traceId 数 / 全部测评集 traceId 数
```

**依赖**：依赖 Ground truth，不调 LLM（纯相似度 + 忠实度阈值）。

**解读**：在排除了"编造"和"兜底"之后，再用文本相似度判断答案是否正确。比单纯看相似度更可靠——因为先确保了模型没有瞎编。

---

### 11. 答案准确率_大模型

**一句话**：用另一个大模型当裁判，判断输出答案和标准答案在含义上是否完全一致（无冗余、无缺漏）。

**触发条件**（三个条件取"且"）：

1. 输出不是兜底文案
2. 答案忠实度 ≥ 阈值 K%
3. 通过 LLM 将输出与 ground truth 对比 Q 次（默认 6 次），每次生成模糊逻辑分数，平均值 > 0.55 才算通过

**公式**：

```
答案准确率_大模型 = 判定准确的 traceId 数 / 全部测评集 traceId 数
```

**依赖**：依赖 Ground truth + 依赖 LLM。

**解读**：这是 Friday 评测体系的"终极裁判"。比相似度更准确——能判断"换了一种说法但意思一样"的情况。多次取均值的设计是为了减少 LLM-as-judge 的单次方差。

**对比 Ragent**：对应 RAGAS 的 `answer_correctness`。Friday 的做法更工程化——Q 次取平均、模糊逻辑打分、阈值卡控，三保险。

---

### 12. 答案缺漏率

**一句话**：输出答案跟标准答案比，正确但不全——少了关键信息。

**触发条件**：

1. 输出不是兜底文案
2. LLM 判断输出内容是 ground truth 的真子集（输出内容包含在 GT 中，但 GT 有更多内容）

两个条件取"且"。

**公式**：

```
答案缺漏率 = 答案缺漏的 traceId 数 / 全部测评集 traceId 数
```

**依赖**：依赖 Ground truth + 依赖 LLM。

**解读**：衡量答案完整度。需要结合其他指标一起诊断：
- 缺漏率高 + Context Recall 高 → 召回没问题，是 LLM 选择和输出有问题，需要优化 prompt
- 缺漏率高 + Context Recall 低 + Faithfulness 高 → 召回阶段就没召回全，需要优化检索

**对比 Ragent**：Ragent 和 RAGAS 都没有单独评估"缺漏"，RAGAS 的 `answer_correctness` 的 Recall 部分可以间接反映，但不是独立指标。Friday 把它单拆出来，定位问题更准。

---

### 13. 答案错误率

**一句话**：模型选了错误的知识片段，生成了错误答案。

**触发条件**（三个条件取"且"）：

1. 输出不是兜底文案
2. 答案忠实度 ≥ 阈值 K%（说明不是编造的，确实是基于召回的）
3. LLM 判断输出信息超出 ground truth（多了不该说的）

**公式**：

```
答案错误率 = 答案错误的 traceId 数 / 全部测评集 traceId 数
```

**依赖**：依赖 Ground truth + 依赖 LLM。

**解读**：衡量模型"选错了参考片段"的问题。同样需要结合诊断：
- 错误率高 + Context Recall 高 → 召回没问题，LLM 选错了片段
- 错误率高 + Context Recall 低 → 召回本身就没找全

---

### 14. 答案无幻觉率

**一句话**：不考虑缺漏的前提下，系统有没有给出正确的或有边界的回答？

**算法**：

```
答案无幻觉率 = 答案准确率_大模型 + 正确拒答率 + 错误拒答率
```

**依赖**：依赖 Ground truth + 依赖 LLM。

**解读**：把三个正面情况加起来——答对了、该拒答时拒了、不该拒答但拒了也说明没有胡编。这是一个"没有更坏"的复合指标。

---

### 15. 答案忠实度（Faithfulness）

**一句话**：大模型输出的内容里，有多大比例能回溯到召回上下文中找到依据？

**算法**：LLM 将输出答案与 ground truth 对比 Q 次（默认 6 次），生成模糊逻辑分数，计算平均值。结果是"每个要点在召回上下文中的匹配比例"。

**公式**：

```
答案忠实度 = (可回溯到上下文的内容 / 输出总内容) 的百分比
```

筛选有召回的样本取均值。

**依赖**：不依赖 Ground truth！只依赖 LLM。

**解读**：这是整个体系里最关键的生成质量指标之一。**不需要人工标注就能自动算**——LLM 只要判断"这句话有没有在召回片段的依据"。在 2B 问答场景，一般要求忠实度尽量接近 100%（不期望 LLM 额外发挥）。

**对比 Ragent**：完全对应 RAGAS 的 `faithfulness`。Friday 还特别指出：

> 答案准确率、答案缺漏率、答案错误率 都是建立在答案忠实度基础上的。

也就是说，Friday 评测体系的三层校验逻辑：
1. 先看忠实度——有没有编造
2. 确认没有编造后，再看相似度——对不对
3. 再用 LLM judge——语义上准不准

---

### 16. 正确知识转化率_相似度

**一句话**：召回了正确的知识，且最终输出也正确——整条链路有没有断？

**算法**：

```
正确知识转化率_相似度 = 答案准确率_相似度 / 知识命中率
```

**依赖**：依赖 Ground truth，不调 LLM。

**解读**：这是一个衍生指标，衡量"检索到生成"的转化效率。如果知识命中率高但正确知识转化率低，说明检索找到了对的文档，但生成环节出了问题——prompt 设计、模型选择、上下文利用等需要排查。

## 四、三层架构全景图

```
测评数据集（带 Ground truth）
        │
        ▼
┌─ 知识库层 ──────────────────────────────────────┐
│ 知识覆盖率    知识库本身有没有这个知识？           │
│ → 没有就补知识                                     │
└──────────────────────────────────────────────────┘
        │（知识已覆盖）
        ▼
┌─ 召回层 ────────────────────────────────────────┐
│ 知识有召回率    有没有返回任何结果？               │
│ 知识命中率      结果里有没有对的？                 │
│ 上下文召回率    对的覆盖全了没？                   │
│ 上下文精确率    对的排前面了没？                   │
│ 知识冲突率      "对的结果"之间矛盾了没？           │
│ → 召回不行就优化检索策略                           │
└──────────────────────────────────────────────────┘
        │（检索结果到位）
        ▼
┌─ 端到端层 ──────────────────────────────────────┐
│ 答案忠实度      有没有编造？         ← 不依赖GT    │
│ 正确拒答率      不该答时拒了没？                   │
│ 错误拒答率      该答时拒了没？                     │
│ 答案相似度      文本像不像？                       │
│ 答案准确率      语义准不准？                       │
│ 答案缺漏率      输出全不全？                       │
│ 答案错误率      选没选错片段？                     │
│ 答案无幻觉率    整体幻觉程度                       │
│ 知识转化率      从检索到生成的转化效率             │
│ → 生成不行就优化 Prompt / 换模型                  │
└──────────────────────────────────────────────────┘
```

## 五、与 Ragent 评测体系对照

| 维度 | Ragent（ragenteval） | Friday 平台 |
|------|---------------------|-------------|
| 评估集规模 | 20 条主力集 + 150 条全量集 | 按业务自定义 |
| 意图评估 | `intent_top1`（自建，秒级） | 元（无独立指标） |
| 检索底线 | 误拒率 | 知识有召回率 |
| 检索命中 | `Hit@K`（文档 ID 集合运算） | 知识命中率（文本相似度） |
| 检索覆盖 | `Recall@K`（文档 ID） | 上下文召回率（LLM 语义） |
| 排序质量 | `MRR@10` | 上下文精确率（类似 NDCG） |
| 知识冲突 | 无 | 召回知识冲突率 |
| 知识库完整性 | 无独立指标 | 知识覆盖率 |
| 回复忠实度 | `faithfulness`（RAGAS） | 答案忠实度 |
| 回复正确性 | `answer_correctness`（RAGAS） | 答案准确率_大模型 |
| 回复相似度 | 无 | 答案相似度（Rouge-2） |
| 拒答分析 | 误拒率 + 过召回率 | 正确拒答率 + 错误拒答率 |
| 缺漏分析 | 无独立指标 | 答案缺漏率 |
| 错误分析 | 无独立指标 | 答案错误率 |
| 性能 | `ttft_p50` / `ttft_mean` | 无 |

---

## 六、关键差异总结

1. **Friday 评测体系更完整**：12 个指标覆盖了知识库、召回、生成三个层面，且有明确的层间依赖关系（知识覆盖率 → 召回指标 → 端到端指标）

2. **Friday 多了一个"知识冲突"维度**：Ragent 没有这个指标。在实际业务中，知识冲突是常见问题——不同版本的政策、不同部门写的 FAQ 产生矛盾。

3. **Friday 使用文本相似度做检索评估**：Ragent 用文档 ID 精确匹配，更确定但更严格。Friday 用相似度，更宽松但能覆盖"差不多对"的情况。

4. **Friday 有独立的"缺漏"和"错误"分析**：RAGAS 的 `answer_correctness` 是一个综合分数（F1），Friday 把它拆成三个子指标（相似度、缺漏、错误），排障更精准。

5. **Friday 引入了模糊逻辑打分 + 多次取均值**：相比 RAGAS 的标准 LLM-as-judge，Friday 做了额外的工程加固——防方差、防单次误判。

6. **Friday 不评意图和性能**：这可能是它跟 Ragent 评测体系最大的互补点——Friday 侧重内容质量，Ragent 自建指标侧重意图路由和性能延迟。
