# 智能 OnCall Agent 面经题分类与参考答案

> 本文根据面经分享记录中的真实面试题，结合项目资料整理出分类答案，供面试前复习使用。

---

## 一、Agent 架构与设计

### Q1: 你的对话上下文是有几轮？阈值是怎么设置的？

**参考答案：**

我设计了分层记忆管理机制来处理对话上下文。第一层是滑动窗口记忆，保留最近 5 轮对话的完整内容直接放入 prompt，保证对话连贯性。第二层是摘要记忆，当对话轮数超过 5 轮，会对之前的历史对话做总结压缩，把 10 轮对话压缩成 2-3 句关键信息。

阈值设置方面，我参考了一些实践文章的经验，当上下文 token 达到模型窗口的 70% 时触发压缩。之所以选 70% 而不是更高，是为了给当前轮对话和工具返回留足空间，避免动态超限。如果压缩后仍然超限（极端情况下模型可能一直在调用工具），就转人工处理，因为这时大概率是推理出了问题。

---

### Q2: 你的对话 Agent prompt 是如何调优的？

**参考答案：**
明确指令 和角色
减少 

Prompt 调优我主要遵循以下几个原则：
首先是**明确指令和角色**，在 system prompt 中明确 Agent 的身份是"团队内部的智能 OnCall 助手"，并严格约束"只基于文档内容回答，不知道就说不知道"。

其次是**减少幻觉的具体方法**：引入假设约束（限定输入输出格式）、提供背景信息（将召回的文档片段标记为"参考文档"）、要求分步推理（ReAct 模式自带）、限制输出范围（只回答与 OnCall 相关的问题）。

然后是**反向提示**，主动排除不该回答的内容，比如"如果用户的提问与你的专业知识无关，则回复不知道"。

最后通过测评集验证效果，构造几十个真实问题的标准答案，跑一遍对比准确率，不断迭代优化 prompt 的表述。

---

### Q3: 用户乱问问题怎么处理？

**参考答案：**

主要从 prompt 工程层面做限制。在 system prompt 中明确写了约束："如果用户的提问与你的专业知识无关，则回复'抱歉，这个问题超出了我的能力范围'"。

同时配合 RAG 的相似度阈值过滤（设置为 0.8），如果召回的文档相似度全部低于阈值，说明知识库中没有相关信息，系统会直接返回"暂无相关信息"而不是让大模型猜测。这样从两个维度避免了大模型对无关问题胡说八道。

---

### Q4: 调用工具异常/失败怎么处理？

**参考答案：**

工具调用失败本质上就是函数执行 error，我从两个层面处理：

第一是**重试机制**，调用失败后会进行最多 3 次重试（for i < 3 retry），处理网络抖动等临时性问题。

第二是**ReAct/Plan 模式本身的容错能力**。因为 ReAct 是"思考-行动-观察"循环，工具返回失败信息后，大模型在下一轮"思考"时可以看到这个失败结果，自主决定是换个参数重试，还是调用其他工具绕过，或者直接告知用户该信息暂时无法获取。

在 Plan-Execute-Replan 模式中更明确：Replanner 评估执行结果时如果发现失败，会调整计划，比如换一种排查路径。

---

### Q5: Agent 执行异常/失败怎么处理？

**参考答案：**

Agent 执行异常最典型的情况是**无限调用 Tool 陷入循环**。解决方案是设置一个最大 Step 限制，比如最多执行 15 步，超过就强制中断并返回当前已收集到的信息。

第二个层面是事后排查，检查 prompt 是否有歧义导致模型反复调用同一个工具，或者模型能力是否不足以理解任务终止条件。这种情况下需要优化 prompt 表述或者更换更强的模型。

---

### Q6: 多 Agent 的时候，如果某个 Agent 挂掉了，怎么保证稳定性，怎么及时发现？

**参考答案：**

这本质上是可观测性的问题。

**及时发现**方面：我在每个 Agent 的关键节点都做了埋点，包括工具调用耗时、大模型响应耗时、执行步骤数等。配合监控告警（比如 Prometheus + AlertManager），如果某个 Agent 的响应时间超过阈值或者错误率飙升，就会触发告警通知到值班群。

**稳定性保障**方面：对话 Agent 和运维 Agent 是独立运行的，互不影响。单个 Agent 内部通过最大 Step 限制、超时控制、错误重试等机制保证不会无限挂起。最坏情况下某次请求失败，会返回友好的错误提示并建议转人工。

---

### Q7: 项目中多个 Agent 怎么交互的？

**参考答案：**

在我的项目中，对话 Agent 和运维 Agent 是**独立的**，它们不直接交互。用户通过不同入口触发不同 Agent：对话框输入问题触发对话 Agent，告警触发运维 Agent。

它们底层**共享同一套工具集和知识库**。比如 `query_internal_docs` 这个知识库召回工具，对话 Agent 用它来回答业务咨询，运维 Agent 也用它来检索告警处理手册。但工作模式和目标不同：对话 Agent 用 ReAct 灵活应答，运维 Agent 用 Plan-Execute-Replan 结构化排查。

知识库本身不是 Agent，而是一个基础设施层，为上层 Agent 提供向量检索能力。

---

## 二、工作流与编排

### Q8: 工作流的具体创建流程是什么？

**参考答案：**

工作流本质上就是把多个节点用边连起来，形成一个有向图。每个节点代表一个原子能力，比如大模型节点、Retriever 召回节点、Prompt 构建节点等。边代表数据流向和执行顺序。

具体创建流程：首先定义需要哪些节点（根据场景需求），然后用代码把它们串联起来。比如对话 Agent 的工作流是：用户输入 → RAG 召回 → Prompt 拼装（把召回片段 + 用户问题 + 历史对话拼在一起）→ 大模型推理 → 工具调用（如果需要）→ 返回结果。

可以通过代码实现（像我项目中用框架的 API 编排），也可以通过低代码可视化平台拖拽实现。

---

### Q9: 为什么使用图编排，和其他工作流有什么区别？

**参考答案：**

我选择图编排主要是因为**直观和可配置**。图编排的好处是整个 Agent 的执行过程一目了然，每个节点做什么、数据怎么流转都清晰可见，调试和维护方便。

和"其他工作流"相比，其实本质区别不大。区别在于图编排后面接的是 ReAct 循环（对话 Agent）或 Plan-Execute-Replan 循环（运维 Agent），而不是简单的线性链。图编排更像是定义了 Agent 的骨架（哪些能力节点、什么顺序），具体执行逻辑由 Agent 的设计模式决定。

选择框架提供的图编排方式，主要是工程效率的考虑——可视化编排很快，比纯代码手写状态管理要轻松。

---

## 三、RAG 与知识库

### Q10: 你的向量数据库存了多少记录？

**参考答案：**

要结合项目立意来回答——这是内部值班场景使用的工具，知识库里存的是告警处理手册、服务接口文档、错误码文档这些内容。

按照我们团队维护的服务数量（大约十几个），每个服务有对应的告警处理文档和接口文档，经过分片后大概有几百条向量记录。数据量不大，但质量高、针对性强。这也和项目定位一致——不是面向全公司的通用知识库，而是聚焦团队内部值班场景。

---

### Q11: 你是多路召回还是单路，有没有做关键词检索？

**参考答案：**

目前是**单路召回**，只用了余弦相似度的向量检索。

原因是我们的文档数据量不大（几百条），且文档经过了认真的清洗和规范化（要求大家写文档时同一个问题集中在一个段落），单路向量召回在我们场景下效果已经很好了。

如果后续数据量增大或者场景变复杂，可以考虑加入关键词检索（BM25）做多路召回，然后融合排序。但在当前场景下，单路 + 重排的方案已经够用，没必要过度工程。

---

### Q12: 有了 Skill 之后，项目哪里可以改进？

**参考答案：**

最大的优化点是**用 Skill 替换部分 RAG**。

我们原来用 RAG 的目的是把告警处理 SOP 从文档中检索出来给大模型参考。现在有了 Skill 机制后，可以把不同告警类型的处理流程封装成独立的 Skill。比如"服务下线处理 Skill"、"接口超时处理 Skill"，每个 Skill 里面直接写清楚处理步骤和决策逻辑。

这样做的好处是：Skill 是按需加载的（只加载当前告警类型对应的 Skill），比 RAG 召回更精准，不存在"召回了不相关片段"的问题。而且 Skill 天然支持结构化的操作指引，比从文档中提取信息更可靠。

---

## 四、大模型能力与安全

### Q13: 大模型怎么处理幻觉？

**参考答案：**

我从四个层面控制幻觉：

**第一，Prompt 强约束**：system prompt 中明确要求"严格按照文档内容回答，不允许使用文档外的信息""不知道就说不知道，不要编造"。

**第二，RAG 增强**：通过检索给大模型提供可靠的事实依据，让它基于文档回答而不是凭空生成。

**第三，相似度阈值过滤**：设置 0.8 的高阈值，如果召回文档相似度都低于此值，直接告诉用户"暂无相关信息"，不让模型硬答。

**第四，人工反馈闭环**：加入 👎 功能，用户反馈不满意时记录 log id，人工 trace 整个流程排查原因。通常是文档缺失导致的，补充文档后问题就解决了。

另外还可以考虑：降低 temperature（减少创造性）、多模型交叉验证、后处理规则校验等方案。

---

### Q14: 大模型输出安全问题怎么处理？

**参考答案：**

输出安全主要关注两个方面：

**敏感词过滤**：可以在大模型输出后加一层过滤器，对照敏感词库进行检测和替换。也可以用另一个小模型专门做安全审核。

**Prompt 注入防护**：在 prompt 中明确角色边界，避免用户通过特殊输入绕过限制。比如用户试图说"忽略之前的指令"时，系统不会被劫持。

不过在我们的场景下（内部团队使用），安全问题相对可控，主要还是聚焦在防止幻觉和确保回答准确性上。

---

## 五、项目立意与思考

### Q15: 这个需求是你自己提的，方案设计和上线是怎么思考的？

**参考答案：**

这个项目源于真实的值班痛点——上游同事反复问相同问题、翻日志看监控重复劳动、告警一多就手忙脚乱。

方案设计经历了几个阶段的演进：

**第一版：粗暴塞文档**。直接把整个告警处理手册丢给大模型，发现受上下文窗口限制，成本高、效果差。

**第二版：引入 RAG**。把文档分片向量化，先检索再生成，效果好了很多，但还有幻觉问题。

**第三版：ReAct + Plan-Execute-Replan**。让 Agent 具备工具调用能力，不仅能检索文档，还能查日志、查监控，结合多源信息给出准确答案。同时通过结构化的规划执行模式，让复杂排查任务变得可控。

上线方面，项目部署在办公环境开发机上，用 Docker 容器化启动（向量数据库、前端、后端各一个容器），办公网直接可用，不需要打通线上网络。

---

### Q16: 怎么分析问题、解决问题，你当时是怎么想的？

**参考答案：**

核心思路是**从痛点出发，用技术解决问题**。

值班时我发现几个明确痛点：80% 时间在当"人肉客服"回答重复问题、翻日志切系统效率低、新人值班不知道标准流程。这些都是高频、标准化、可自动化的工作。

然后思考解决方案时，AI 这两年发展很快，大模型 + RAG + Agent 的技术组合恰好能覆盖这些场景：重复咨询用对话 Agent + RAG 解决，告警排查用运维 Agent + 工具调用解决。

这体现了我的主观能动性——不是被动等分配任务，而是主动发现问题、分析问题、用技术手段解决问题，最终产出实际价值。

---

### Q17: 除了人工反馈，你打算用什么指标来评价 OnCall Agent 的好坏？

**参考答案：**

核心指标是**进线率/拦截率**——即 Agent 能自动处理掉多少比例的咨询和告警，不需要转人工。

具体可以从几个维度衡量：

- **自动解决率**：Agent 回答后用户没有追问或没有点 👎 的比例
- **响应时间**：从用户提问到 Agent 给出答案的平均时间
- **准确率**：通过定期抽样人工评审，或构造测评集自动化评测
- **召回命中率**：RAG 检索到的文档是否包含正确答案

这些指标结合起来，就能比较全面地衡量系统效果。

---

### Q18: 能不能补充项目实际会出现的问题？

**参考答案：**

这个问题需要讲故事，展示项目迭代的过程：

**问题一：初期直接把文档塞给大模型**，发现上下文窗口不够，回答质量差，而且 token 成本很高。→ 引入 RAG，先检索相关片段再生成。

**问题二：RAG 召回效果不稳定**，有时候召回了不相关的文档，导致大模型被错误信息误导产生幻觉。→ 加入相似度阈值过滤（0.8）+ 重排模型精筛 + prompt 强约束。

**问题三：用户问题需要实时数据支撑**（比如"最近 5 分钟的错误日志是什么"），纯文档检索做不到。→ 引入 ReAct 模式 + 工具调用（日志查询、告警查询等工具）。

**问题四：复杂告警排查步骤多且需要动态调整**。→ 运维 Agent 采用 Plan-Execute-Replan 模式，结构化拆解 + 动态重规划。

每个"难点"本质上都是一次技术迭代，已经解决了所以可以从容讲述。

## 六、性能与系统设计

### Q19: 你的项目 QPS 多少？

**参考答案：**

从项目立意来说，这是内部值班场景使用的工具，QPS 不会太高。日常使用的并发大概就 1-2，因为是团队内部几个人用，使用频率本来就不频繁。

如果 QPS 说有 100 反而很奇怪——这是给内部十几个人用的值班工具，不是面向 C 端用户的产品。项目的价值不在于高并发，而在于解决实际痛点、提升值班效率。

---

### Q20: 如果项目是 IO 密集型，CPU 打不满，QPS 增加了怎么优化单机性能？

**参考答案：**

这个项目确实是 IO 密集型——主要时间花在等待大模型返回结果。CPU 打不满是正常的，因为瓶颈不在我们的服务端，而在大模型的推理速度。

如果 QPS 真的增加了，压力也是大模型侧的压力增加，对我们的服务本身几乎没有影响（Go/Java 天然支持高并发，处理请求转发是很轻量的操作）。

如果面试官追问优化方案：可以考虑请求队列 + 限流（控制并发请求大模型的数量）、缓存热点问题的回答（相同问题直接返回缓存）、异步处理（SSE 流式返回，不阻塞主线程）。

---

### Q21: 有没有遇到过性能问题？

**参考答案：**

没有遇到过。因为项目是内部使用，数据量不大（向量数据库几百条记录），并发也不高（1-2 QPS），整个系统的瓶颈在大模型的响应速度（通常 2-5 秒），而不是我们的服务本身。

如果未来要扩展到更大规模，可能需要关注向量数据库的检索性能（数据量到百万级时需要优化索引策略）和大模型调用的并发限制（可能需要做请求排队或多模型负载均衡）。

## 七、运维与基础设施

### Q22: 公司内部的服务，为什么会通过腾讯云收集日志？

**参考答案：**

日志采集和存储有两种主流方案：一是使用云服务商的日志服务（如腾讯云 CLS、阿里云 SLS），二是自建 ELK 集群。

现在大部分公司选择托管到云服务商，因为自建 ELK 也需要存储成本和运维成本（要维护 Elasticsearch 集群、处理扩容、保证高可用等）。用云厂商的服务开箱即用，按量付费，还能利用他们的检索能力（比如 CLS 的 MCP 支持自然语言查询日志）。

---

### Q23: 日志是怎么上传到云服务的？

**参考答案：**

在服务器上安装一个日志采集器（腾讯云 CLS 提供的 LogListener），配置好采集路径（指定日志文件的目录），采集器就会自动将日志上传到云端。

整个过程非常简单：一行命令安装采集器 → 在控制台配置采集规则（路径、解析格式等）→ 日志自动上传。实际操作成本很低。

## 八、ReAct vs Plan-Execute-Replan 对比

### Q24: ReAct 和 Plan-Execute-Replan 有什么区别？为什么分别用在对话和运维场景？

**参考答案：**

| 维度 | ReAct | Plan-Execute-Replan |
|------|-------|---------------------|
| 核心思路 | 边想边做，实时决策 | 先拆步骤，按计划执行，动态调整 |
| 步骤特点 | 无预设步骤，动态生成下一步 | 提前规划步骤序列，可动态修改 |
| 适用场景 | 灵活探索类（问答、咨询） | 复杂流程类（告警排查、多步任务） |

**对话场景用 ReAct**：因为用户问题灵活多变，可能问完 API 怎么用又问错误码含义，需要实时决策。执行链条不会预设很长，灵活性优先。

**运维场景用 Plan-Execute-Replan**：告警排查是一个明确目标下的多步骤复杂任务（查日志 → 查监控 → 检索文档 → 分析根因），需要结构化拆解确保不遗漏。而且排查过程中可能遇到意外（比如日志没异常），需要 Replan 动态调整方向。

两者底层共享同一套工具集和知识库，但工作模式和适用场景不同。

## 九、SSE 与前后端交互

### Q25: 为什么用 SSE 不用 WebSocket？

**参考答案：**

SSE（Server-Sent Events）是基于 HTTP 的单向推送协议，WebSocket 是全双工双向通信协议。

选择 SSE 的原因是**简单且够用**：我们的场景是服务器单向推送 AI 生成的文本流给前端（用户体验像看实时打字），不需要前端在流式过程中反向发消息。SSE 基于标准 HTTP，不需要处理新协议，后端实现简单，调试方便。

如果是需要双向实时交互的场景（如协同编辑、在线聊天），才需要 WebSocket。对于"AI 逐字输出"这种典型的服务器推流场景，SSE 是最轻量的选择。

实现上就是把 HTTP 响应头的 `Content-Type` 设为 `text/event-stream`，然后按 SSE 格式（`data: xxx\n\n`）持续发送数据即可。

## 十二、RAG 核心知识面试题


### Q27: RAG 和微调怎么选？什么时候用 RAG，什么时候用微调？

**参考答案：**

先区分问题出在哪里。如果是**知识缺失**（企业内部文档、产品手册、FAQ、接口说明、政策制度等外部知识），优先用 RAG，因为这些内容变化快、可追溯性要求高，放进知识库可以随时更新，也方便引用来源。如果为了这些动态知识去微调模型，成本高、更新慢，而且模型还可能记混或产生幻觉。

如果是**确定性的业务规则**（权限判断、价格计算、流程状态流转、字段校验），不该靠模型也不该靠 RAG，而是用规则、代码、工具调用来增强。

真正适合微调的场景是：模型的行为模式本身不符合业务需求。比如模型长期不会按某种专业风格回答、不能稳定遵守某种输出格式、对领域术语理解差、分类/抽取/意图识别在大量样本上都表现不好，且有大量高质量标注数据可以让模型学到稳定模式。微调改变的是模型的行为分布，而不是临时塞几段知识。

---

### Q28: RAG 评估指标有哪些？RAGAS 框架是什么？

**参考答案：**

RAG 场景下最著名的评估框架是 **RAGAS**，它的优点是不需要参考答案。主要从三个维度评判：

**忠诚度（Faithfulness）**：判断 LLM 的输出是否基于召回文本生成的。计算方式是让 LLM 从回答中生成多个陈述，然后判断每个陈述是否能从召回文本推断得出，"可推断的陈述数 / 总陈述数"就是忠诚度得分。

**答案相关性（Answer Relevance）**：衡量生成的答案是否和用户问题相关。方法是根据 LLM 的回答生成若干假问题，然后计算假问题和真实问题的相似度，取平均值作为得分。

**上下文相关性（Context Relevance）**：判断召回文本是否和用户问题相关。让 LLM 挑选出召回文本中能够回答问题的子集，用子集占召回文本的比例作为得分。

在项目中实际使用：每次 query 时后台记录召回的文档和最终答案，定期跑 RAGAS 指标监控系统质量变化。

---

### Q29: 你的 RAG 系统用了什么切块策略？为什么？

**参考答案：**

切块策略需要根据文档类型区分：

**强结构化数据（如技师列表、FAQ、错误码表）**：不切块，每个实体完整保存为一个 Document。原因是这些是不可分割的语义实体，强行切分会导致"张冠李戴"。

**长文本非结构化数据（如技术文档、用户协议、SOP）**：使用递归字符分块（RecursiveCharacterTextSplitter），chunk_size 设为 400-500 tokens，overlap 设为 50 tokens（约 10%-20%）。优先按段落（`\n\n`）切，其次按句子切，最后才按长度兜底。这样既保留语义完整性，又适配 embedding 模型的输入窗口。

选择依据：embedding 模型（如 BGE）最佳输入区间在 100-400 token；chunk 不宜过长导致语义稀释，也不宜过短导致信息密度不够；overlap 防止关键信息落在边界被切断。

---

### Q30: 如何确定切块的大小？

**参考答案：**

chunk size 的确定是一个有方法论的工程决策，从五个维度考虑：

**第一，确认计量单位**：chunk_size 是字符还是 token。中文 1000 字符约 1500-2000 token，英文 1000 字符才 200-250 token，脱离单位谈数字没意义。

**第二，看 embedding 模型的最佳输入区间**：不是不超过 max_token 就行。比如 text-embedding-ada-002 上限 8191 token，但最佳区间在 100-800 token；BGE 类模型上限 512 token，最佳 100-400 token。

**第三，看文档的自然语义粒度**：FAQ 一条 200-300 字，SOP 一个步骤 400-600 字，技术文档一段 500-800 字。理想状态是一个 chunk 刚好包含一个完整语义单元。

**第四，算 LLM 的窗口预算**：召回 top-k 个 chunk 拼进 prompt 不能撑满窗口。top-10 时单个 chunk 控制在 500-1000 token 比较安全。

**第五，overlap 设 10%-20%**：太小边界信息丢失，太大冗余浪费。

**最终定值**：以上步骤给出经验区间，然后用 golden test set（20-50 条真实 query + 标注正确来源）跑 3-4 个候选值，对比 hit_rate 和 MRR，选最优。

---

### Q31: 跨块语义丢失怎么办？对于同一个问题，相关内容可能跨块，怎么优化？

**参考答案：**

三种互补策略：

**第一，切块方式本身的优化**：递归字符分块器按语义切割保留完整性；块之间设置 overlap 重叠区（10%-20%），让边界信息在相邻 chunk 中都有保留。

**第二，召回内容上下文扩充**：召回的是短文本（子块），但输入给 LLM 的是它的父文本（长文本）。LangChain 提供 `ParentDocumentRetriever`，同时维护 parent_splitter 和 child_splitter，以及父子映射关系。这样既精准（基于短文本匹配），又完整（基于长文本输入）。

**第三，什么时候需要上下文扩充**：在向量化模型对长文本编码信息损失较大时，从向量数据库召回的通常是短文本，但 LLM 需要连贯且包含更多上下文的长文本才能总结出好答案。上下文扩充打破了"召回内容 = 输入 LLM 内容"的约束，解决了向量化模型需要短文本而 LLM 需要长文本的矛盾。

---

### Q32: 如何提高 RAG 的召回准确性？

**参考答案：**

可以从 RAG 全链路各个环节优化：

**切块策略优化**：选择适合文档类型的切分方式，保证语义完整。

**召回优化策略**：
- 短文本全局信息增强：为切块后的短文补充所属长文本的标题、前三句话、关键词
- 召回内容上下文扩充：召回短文本但输入 LLM 长文本（ParentDocumentRetriever）
- 文本多向量表示：对同一段文本生成子集向量、摘要向量、假设性问题向量等补充表示
- 内容查询优化：用 LLM 改写和扩充用户口语化的模糊 query（如 MultiQueryRetriever）
- 结合元数据召回：利用标签信息（日期、分类、来源）先过滤再检索，减少无关结果

**精排/重排**：粗排后用 Cross Encoder（如 BERT）对候选文档做深层语义排序，提高精度。

**多路召回**：稠密向量检索（语义匹配）+ 稀疏向量检索 BM25（关键词匹配），互补后融合排序。

---

### Q33: 稠密向量和稀疏向量有什么区别？什么是对称/非对称检索？

**参考答案：**

**稠密向量 vs 稀疏向量**：
- 稠密向量检索模型（基于 BERT/GPT 等深度学习模型）：更擅长提取文本的语义信息，捕捉深层含义和同义表达
- 稀疏向量检索模型（如 TF-IDF、BM25）：更擅长提取关键信息，对精确的关键词匹配效果好

两者互补：稠密向量理解"意思相近"，稀疏向量抓住"关键词相同"。RAG 系统可以双路召回后融合排序。

**对称检索 vs 非对称检索**：
- 对称检索：找相似的句子，query 和 document 形式相同（如两个句子比较相似度），与向量相似度计算天然匹配
- 非对称检索：根据问题召回答案，query 和 document 形式不同（问题 vs 答案文本），要求模型能将问题和答案映射到同一空间

并非所有模型都支持非对称检索，只有经过大量问答数据训练的模型才能支持。RAG 场景大多是非对称检索（用户提问 → 召回文档片段）。

### Q34: 常见的向量数据库有哪些？怎么选？

**参考答案：**

常见的有 Milvus、Weaviate、Chroma、Qdrant：

**Milvus**：支持数十亿甚至万亿级向量，毫秒级检索延迟，支持数据分片、持久化、流式数据摄取、混合搜索，高可用高性能易扩展，支持 K8s 部署，有数据复制和故障恢复。适合大规模生产环境。

**Weaviate**：鲁棒性好，云原生，百万级向量毫秒级检索，方便调用 OpenAI/Cohere/HuggingFace 的向量化接口。缺点是目前只支持 HNSW 一种检索算法。

**Chroma**：AI 原生，轻量化，针对 RAG 场景设计，目前只实现了单机方案，使用 FAISS 索引。适合快速原型验证和数据量不大的场景。

**选择依据**：数据量大（百万级以上）选 Milvus；需要快速验证、数据量小选 Chroma；需要云原生特性选 Weaviate。内部场景数据量不大时，甚至 SQLite + FAISS 也够用。

### Q35: 你用的什么 Embedding 模型？怎么选的？

**参考答案：**

Embedding 模型选择策略：

**第一，看检索问题类型**：普通知识库问答选面向"问题找文档"训练的检索模型（如 BGE、E5、GTE）；代码/法律/金融/医疗选领域专用模型；中英混合选跨语言能力强的模型。

**第二，看数据特点**：中文多还是英文多？长文档还是短 FAQ？有没有大量产品名、缩写、错误码、内部黑话？纯语义模型对精确匹配（订单号、版本号）不一定比关键词检索好。

**第三，用业务样本试**：不看榜单看实效。拿几十到几百个真实问题，标出应命中的文档片段，比较不同模型的 hit_rate。重点观察：该召回的有没有召回，不该召回的有没有混进来。

**第四，工程约束**：向量维度、索引大小、查询延迟、建库速度、调用成本、是否能私有化部署。商业 API 接入快但有费用和合规问题；开源模型可控但需自己维护。

**第五，和切分/重排一起调**：Embedding 不是孤立选择的，chunk 切太大语义稀释，切太小上下文不够；专有名词需结合关键词检索；召回质量要求高时加 rerank。

---

### Q36: Cross Encoder 重排是怎么做的？和双塔模型有什么区别？

**参考答案：**

**Cross Encoder**：把 query 和每个候选文档拼在一起输入模型（如 BERT），让模型直接判断二者相关性，输出一个相关性分数。优点是语义匹配能力强（因为 query 和 doc 之间有充分的注意力交互），精度高。缺点是每个候选都要单独过模型，计算量大、延迟高，不适合大规模召回，只适合对小规模候选集做精排。

**双塔模型（Bi-Encoder）**：query 和 document 分别编码成向量，通过向量距离计算相似度。优点是可以预先计算所有文档的向量，在线只需计算 query 向量然后做 ANN 检索，速度极快。缺点是 query 和 doc 之间没有交互，语义匹配精度不如 Cross Encoder。

**实际 RAG 系统的做法**：先用双塔模型（稠密向量检索）做粗排/召回 top-k，再用 Cross Encoder 对这 k 个候选做精排重排序，兼顾速度和精度。

---

### Q37: 召回时如何处理时间/元数据敏感的问题？（如"今天天气怎么样"要召回正确日期的内容）

**参考答案：**

结合两个策略：

**内容查询优化**：当用户提问过于模糊时，先对查询做处理和扩充。比如用户问"今天天气怎么样"，系统先把当前日期补充进去，改写为"6月20日的天气怎么样"，再去检索。

**结合元数据召回**：在存储文本向量时，同时记录元数据标签（如日期、来源、分类）。召回时先根据元数据过滤出符合条件的向量子集（如只看今天的数据），再在子集中做语义检索。LangChain 提供了 `Create_metadata_tagger` 接口，可以利用 LLM 自动生成元数据。

两者结合：先通过查询改写明确意图，再通过元数据过滤缩小范围，最后语义匹配精确召回。

---

### Q38: RAG 中如果召回的内容有恶意注入的错误信息，怎么避免影响生成？

**参考答案：**

从多个层面防护：

**第一，数据源控制**：限制知识库数据的来源，只接入可靠的数据源，建立数据写入审核机制。

**第二，安全性检验**：对召回的数据进行安全性检测，检查是否包含恶意指令（如 prompt injection）或明显错误信息。可以用规则匹配或专门的安全分类模型。

**第三，多路召回交叉验证**：如果多个来源的文档观点一致才采纳，对单一来源的极端信息降权。

**第四，Prompt 约束**：在 system prompt 中要求模型"如果召回的信息相互矛盾或不可信，请明确指出"。

**第五，忠诚度监控**：通过 RAGAS 的忠诚度指标持续监控输出是否偏离召回文档，异常时触发告警。

### Q39: 你的 RAG 系统端到端延迟是多少？怎么优化？

**参考答案：**

RAG 系统的端到端延迟主要由三部分组成：Query Embedding 时间 + 向量检索时间 + LLM 生成时间。

通常 Query Embedding 几十毫秒，向量检索（数据量不大时）几十毫秒，瓶颈在 LLM 生成（2-5 秒）。

**优化方向**：
- 向量检索：使用 HNSW 等 ANN 索引加速；数据量大时做分片和预过滤
- 缓存：对热点问题的回答做缓存，相同 query 直接返回
- 流式输出：使用 SSE 流式返回，用户无需等全部生成完再看到结果
- 并行化：Query Embedding 和历史记录加载可以并行
- 减少 LLM 输入 token：精排后只取 top-3 而非 top-10，减少 context 长度

---

### Q40: 什么是 RAG 的"Lost in the Middle"问题？怎么解决？

**参考答案：**

"Lost in the Middle"是指当向 LLM 输入多个检索到的文档片段时，模型倾向于更关注开头和结尾的内容，而中间位置的关键信息可能被忽略。

**解决方案**：
- 精排控制数量：不贪多，只传最相关的 3-5 个 chunk，减少总量
- 调整排列顺序：把最相关的内容放在开头和结尾位置（如最相关 → 次相关 → ... → 第二相关）
- Prompt 明确指引：在 prompt 中强调"请综合所有参考文档回答"
- Map-Reduce 策略：对每个 chunk 单独让 LLM 提取关键信息，然后汇总
- 控制 chunk 大小：避免单次输入过长的上下文

---

### Q41: 多轮对话场景下，RAG 的查询改写怎么做？

**参考答案：**

多轮对话中，用户的当前提问往往依赖上下文（如指代消解、省略主语），直接用当前 query 去检索效果差。

**查询改写方法**：
- 利用 LLM 将当前问题 + 历史对话 condensed 成一个独立、完整的 query（LangChain 的 `ConversationalRetrievalChain` 就有这个功能）
- 例如：用户先问"Milvus 怎么安装"，再问"它支持 K8s 吗"。改写后变成"Milvus 是否支持 Kubernetes 部署"
- 改写后的 query 再去向量库检索，效果远好于直接用"它支持 K8s 吗"检索

这属于 **内容查询优化** 的一种，核心思想是在检索前让 query 变得完整和明确。

---

### Q42: Embedding 模型的向量是怎么拿到的？常见的 Embedding 方法有哪些？

**参考答案：**

常见 Embedding 方法按类型分：

**通用语言理解模型（BERT/RoBERTa）**：通用文本理解底座，不是专门为向量检索训练的，直接用句向量做相似度搜索效果不稳定。在 RAG 中更常作为精排模型（Cross Encoder）使用。

**句向量模型（SBERT/Sentence-Transformers）**：在 BERT 基础上改造，目标是让句子向量适合相似度计算。开源、好部署、生态成熟，适合语义搜索和 RAG 召回。

**检索专用开源模型（E5/BGE/GTE）**：按"问题找文档"的目标训练，用大量问答对和搜索点击数据训练。召回能力强，适合知识库问答，可私有化部署。

**商业 API（OpenAI/Cohere/Voyage）**：开箱即用，效果稳定，多语言能力强。缺点是成本、延迟、数据合规问题。

**领域专用模型**：针对代码/法律/金融/医疗等特定领域优化，处理专业术语更准确。

### Q43: LangChain 中如何实现流式响应？用了什么协议？

**参考答案：**

在 LangChain 中，所有核心组件都被封装为 Runnable 抽象类，定义了 `invoke()`（一次性输出）和 `stream()`/`astream()`（流式输出）接口。流式方法通过 Python 的异步迭代器，让开发者用 `for chunk in runnable.stream(...)` 逐块消费生成内容。

**底层实现分两层**：

后端内部：LangChain 迭代器从底层模型获取 token 片段（前提是模型支持 stream=True）。

后端对外传输（三种协议）：
- **SSE（Server-Sent Events）**：基于 HTTP 长连接，单向推送，浏览器用 EventSource 接收。最常用于 AI 流式输出场景。
- **WebSocket**：全双工双向实时，适合需要交互（如停止生成）的场景。
- **HTTP Chunked**：普通 HTTP 分块传输，前端用 ReadableStream 读取。

AI 应用场景最常用 SSE，因为简单够用——服务器单向推流给前端即可。

### Q44: 精排（Rerank）有哪些方法？各有什么优缺点？

**参考答案：**

精排是位于召回之后的排序环节，对候选文档做更细粒度的相关性判断。主要四种方法：

**Cross Encoder 精排**：把 query 和候选文档拼在一起输入 BERT 类模型判断相关性。优点是语义匹配能力强、分数稳定；缺点是每个候选都要单独过模型，延迟高。

**LLM 精排**：让大模型结合问题、候选文档和业务要求做综合判断。优点是解释性强，能处理复杂意图和多维标准；缺点是成本高、延迟高、输出稳定性需额外约束。

**规则精排**：根据业务目标打分或过滤（字段完整性、标题匹配、时间、来源可靠性、关键实体）。优点是可解释、成本低、延迟低；缺点是语义泛化能力弱。

**混合精排**：模型相关性 + 规则约束结合。效果最均衡但工程复杂度最高，需要调权重和持续评估。

实际项目中通常采用"粗排（向量召回 top-k）→ Cross Encoder 精排 → 规则过滤"的流水线。
