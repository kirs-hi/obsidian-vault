## 背景

团队 7 名产品成员，从 2025 年 3 月开始在「飞象——下一代大象」项目中实践 PDE 工作模式。3 个月内累计完成 40+ 功能闭环（需求→开发→上线），提交 102 个 PR、380+ commit。需求复杂度从简单的「换图标」「改布局」逐步升级到前后端联动的复杂功能，并真实上线产生高频使用。

## 踩坑经历

早期误区：认为「对话式编程」就是「随便描述一下，AI 就实现了」。简单需求可以搞定，但复杂度上来后问题频发：

| 踩坑               | 具体说明                                         | 真实案例                                                                 |
| ------------------ | ------------------------------------------------ | ------------------------------------------------------------------------ |
| 一句话需求         | AI 自己脑补产品目标，方向偏离                    | —                                                                        |
| AI 越改越多        | 改动范围失控，引入无关变更                       | 让 AI 修改一个字段取值逻辑，AI 改了 3 处共 8 处代码                      |
| 重要文件误改       | 潜在全局风险                                     | 改动 app.js（学城骨架文件），差点引入全局风险                            |
| 堆上下文           | 信噪比低，AI 被无关约束牵走                      | —                                                                        |
| 跳过设计状态       | 只描述 happy path，漏掉 loading/empty/error 状态 | —                                                                        |
| 没有测试命令       | 无法验证的代码只是草稿                           | 多维表格项目 70%+ 时间花在测试上                                         |
| 表面正确整体风险   | AI 局部改对，改坏其他逻辑                        | 会议室改造中多次「看起来修好了，实际改坏了其他逻辑」                     |
| CR 成本高于开发    | 研发不熟悉 AI 生成代码的改动逻辑                 | 后端改造开发 1pd，CR 耗时 1.5-2pd                                        |
| 让 agent 直接发布  | 影响用户的动作缺少人类审批                       | —                                                                        |
| 不复盘失败         | 同类错误反复发生                                 | —                                                                        |

## 六阶段协作模型

> [!tip] 核心理念
> 无论项目复杂度如何，每个成功交付的项目都遵循这六个阶段。

### Phase 1：理解现状

**核心动作**：让 AI 先帮你看懂系统，不要急着改。

操作要点：
- 先和 AI 梳理主流程，理解入口、用户路径、实现逻辑和影响范围
- 明确「从哪里改」，避免一上来就动手导致改错位置
- 对存量系统重点梳理接口影响面、调用链路、兼容风险
- 把上下文当作「给 agent 的接口」来设计

**提示词示例**：
```
先不改代码。帮我理解这个功能现在怎么工作的。
用产品能听懂的话说明入口、用户路径、功能逻辑、可能影响范围。
```

**产出物**：现状分析、影响面评估、可改范围

---

### Phase 2：对齐方案

**核心动作**：让 AI 输出计划，人确认后再动手。

操作要点：
- 复杂任务必须先要求 AI 输出实现计划（修改哪些文件、为什么改、如何验证）
- 遵循「先探索→再计划→再执行」
- 让 AI 先列出边界和禁区
- 任务规格包含：用户是谁、解决什么问题、功能范围、不能破坏什么、怎么判断完成

**提示词示例**：
```
先不要改代码。请基于现有代码和需求，给出实现计划：
涉及文件、每个文件的改动意图、关键风险、验证方式。
计划通过后再实现。
```

**产出物**：实现计划、文件清单、风险点、验证方式

---

### Phase 3：最小实现

**核心动作**：一次只做一个可验收的闭环目标。

操作要点：
- MVP 策略：先保证能用，后续保证好用
- 一轮只做一个可验收的小目标，不顺手加太多需求
- 每个改动说明它服务于哪条产品规则
- 最适合 agent 的任务：1 小时级、几百行以内、边界清楚、可测试

**提示词示例**：
```
实现 XXXX，但只做这一个目标，
不要顺手改无关逻辑、样式、文案、结构等。
```

**产出物**：最小可验证的代码变更

---

### Phase 4：验证闭环

**核心动作**：跑测试、列清单、确认没有破坏已有功能。

操作要点：
- 在提示词里写「完成前必须运行哪些测试或检查」
- 没有证据的完成只是未经验证的草稿——必须提供 diff、测试结果、截图
- 问题分类处理：MVP 核心遗漏→立即修复；非核心→记录后续；代码 bug→分析原因后修复

**提示词示例**：
```
实现后请运行相关测试。最终回复必须包含：
改了什么、运行了什么命令、哪些用例通过、
哪些没跑以及原因、还需要我人工验收什么。
```

**产出物**：测试结果、验证清单、未覆盖项说明

---

### Phase 5：提交合入

**核心动作**：拉主干、解冲突、Code Review、安全合入。

操作要点：
- 拉取远端最新代码，合并到开发分支
- 冲突涉及功能取舍必须联系人确认
- 用 AI 做 Code Review，逐条分析建议后再动手
- PR 是 agent 的交付界面，结果落在分支/diff/PR/CI 中

**提示词示例**：
```
帮我 Review 当前分支到 develop 分支最新提交的更改。
```

**产出物**：无冲突代码分支、CR 通过、PR 已创建

---

### Phase 6：复盘沉淀

**核心动作**：总结改动、记录风险、沉淀可复用经验。

操作要点：
- 让 AI 说明：改了什么、为什么改、验证了什么、没验证什么
- 把反复出现的要求提炼为长期规则写进 AGENTS.md
- 用可观察指标衡量效果：一次通过率、人工修改量、测试失败率、返工量、交付周期

**提示词示例**：
```
请根据这次任务中反复出现的要求，
提炼 3-5 条适合写进 AGENTS.md 的长期规则。
每条必须具体、可检查、不过度泛化。
```

**产出物**：改动说明、风险清单、经验文档、AGENTS.md 更新

## 13 条实践技巧（庞经纬）

| #  | 技巧                                         | 核心要点                                                         |
| -- | -------------------------------------------- | ---------------------------------------------------------------- |
| 1  | 用「目标、上下文、约束、完成标准」替代一句话需求 | 把业务目标、相关文件、交互约束、验收结果分开写                   |
| 2  | 先让 AI 复述需求和计划，再让它动手           | 复杂任务先要求输出计划，检查是否覆盖边界条件                     |
| 3  | 把产品规则写成状态、触发、结果和例外         | 用「当用户做 X 时，系统进入 Y 状态，并展示 Z」表达               |
| 4  | 主动声明「保留什么」和「不要做什么」         | 明确旧主流程必须保留，指定不能改的体验/变量/接口                 |
| 5  | 让验收标准进入任务                           | 在提示词里写「完成前必须运行哪些测试或检查」                     |
| 6  | 让 AI 帮你恢复上下文和工作现场               | 跨天/跨线程时，先恢复需求、文件、分支、验证、风险                |
| 7  | 把团队习惯沉淀到 AGENTS.md                   | 稳定、重复、团队共识类规则写进长期配置                           |
| 8  | 前端体验先做可看的原型，再进工程实现         | 先做静态 HTML 原型对齐体验，再进代码仓库                         |
| 9  | 批量数据任务必须要求进度、失败清单和可重跑   | 保留原始输入、输出进度表、失败表和汇总报告                       |
| 10 | 视觉/素材生成要写「生产规格」和「QA 指标」   | 写清尺寸、网格、透明度、帧数、禁止元素                           |
| 11 | 收尾时要求 AI 做风险复盘和差异说明           | 改了什么、为什么改、验证了什么、没验证什么                       |
| 12 | 多线程并行时隔离任务和文件范围               | 并行任务按领域拆开，每个线程写清可读/可写范围                    |
| 13 | 人负责边界，Agent 负责路径探索               | 先定义不能牺牲的东西，再让 AI 在边界内探索方案                   |

## 组织协作模式变化

**传统模式**：产品→设计→研发→QA，串行流水线。

**新模式**：PDE（Product-Design-Engineering）融合角色。
- PDE 初阶：具备跨职能中的部分职能，一部分精通，一部分初阶
- PDE 中高阶：具备大部分跨职能能力，大部分精通

核心变化：岗位职能加速融合，协作从「串行流水线」变为「PDE 与 PDE 之间的并行协作」。产品也能干代码实现，工程师也能干需求对接。

## 产品经理生存指南

### 三件必须新学的事

**1. 学会驾驭 AI，而不是会写代码**

核心能力是把用户问题描述到 AI 可以动手的程度——学会结构化表达需求、一次只给一个闭环目标、要求 AI 解释它做了什么。

**2. 学会工程协作语言**

| 概念           | 需要理解到什么程度                               |
| -------------- | ------------------------------------------------ |
| 分支 Branch    | 知道改动在哪个分支，不和别人混在一起             |
| PR             | 知道这是代码进入主干前的「审批单」               |
| Code Review    | 知道这是研发帮你把关质量的环节                   |
| 测试           | 「跑通了」不等于「没问题」，要覆盖多种场景       |
| 回滚           | 出问题可以退回去，前提是改动足够小和独立         |

**3. 学会验证和质量判断**

追问清单：改了哪些文件？跑了什么测试？老用户会不会被影响？还有什么风险没覆盖？

### 分阶段建议

**1-3 年 PM**：最大优势是「没有包袱」。All in 实践、用实际交付建立信任、尽早进入真实代码仓库跑通全流程。

**3-5 年 PM**：最大挑战是「重新定义自己的价值」。价值在于定义正确的问题、主动承担「边界定义者」角色、学习工程语言提升协作效率。

**5 年+ / 管理者**：最大机会是「重新设计协作模式」。设计新的协作模式、亲自完成至少一个 PR、建设团队级基础设施、用数据衡量转型效果。

### 七条生存建议

1. **把 Coding 当基本功** — 像学 Axure 一样自然，不是变成工程师，而是拥有「从想法到验证」的最短路径
2. **不要丢掉核心竞争力** — 洞察需求、定义价值、做出取舍，这些不会因学会 Coding 自动获得
3. **找合适的需求成为 PDE** — 不是包揽所有需求，挑选边界清楚、风险可控的需求作为起点
4. **敬畏工程流程** — AI 写的代码也要 CR、测试、走发布流程，「稳妥地快」才是目标
5. **接受最后 30% 仍需工程师** — 后端深水区、系统架构、线上稳定性仍需专业工程判断
6. **拥抱不适，让可能性发生** — 转型伴随迷茫和阵痛，关键是继续往前走一步
7. **把每次实践变成可复用资产** — 经验文档、提示词模板、踩坑记录就是新时代的「作品集」

### 一句话总结

> 传统产品经理的生存之道，不是变成工程师，而是成为**能用专业判断力驾驭 AI 的人**——把「想到」和「做到」之间的距离压缩到最短，同时对最终交付质量负责。
