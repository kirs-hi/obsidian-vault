# Python类与对象——从零理解面向对象

> 学习日期：2025-06

> 学习方式：对话式逐步拆解

## 费曼总结（用三句话给别人讲明白）

类就是一个模板，规定了"有什么数据"和"能做什么事"；对象是按模板造出来的具体实例，每个对象数据独立；当你发现多个函数反复传同一批参数时，就该用类把数据和操作打包在一起了。

---

## QAR 1：类和对象到底是什么关系？

**Q：** 什么是类？什么是对象？两者什么关系？

**A（我的猜测）：** 类可能就是一种数据类型的定义？对象是变量？

**R（验证后的结论）：** 类是模具，对象是用模具做出来的产品。类定义"有什么属性"和"能做什么事"，对象是类的一个具体实例，每个对象属性值可以不同但共享同一套方法。

```python
class Dog:
    def __init__(self, name, age):
        self.name = name
        self.age = age

dog1 = Dog("旺财", 3)  # 对象1
dog2 = Dog("小白", 1)  # 对象2，数据独立
```

**一句话费曼：** 类是饼干模具，对象是压出来的饼干——模具一个，饼干无数个，每块饼干上的花纹（数据）可以不一样。

## QAR 2：self 是固定写法吗？必须加吗？

**Q：** `self` 是 Python 关键字吗？能不能换个名字？方法定义里必须写 self 吗？

**A（我的猜测）：** 感觉 self 是固定语法，像 Java 的 this 一样是关键字，不写可能也行？

**R（验证后的结论）：** self 不是关键字，叫 `this`、`x` 都能跑，Python 只认"第一个参数"这个位置。但 PEP 8 约定所有人都写 self，实践中当作强制。**位置必须留**，不留就报错：

```python
def __init__(name, age):  # 少了 self
    pass
Dog("旺财", 3)
# TypeError: takes 2 able but 3 were given
# 原因：Python 自动塞了对象进来，2个位置装3个值
```

**一句话费曼：** self 就是"我自己"的占位符，Python 自动把对象塞进第一个位置，你不留位置它就放不下，会报错。

---

## QAR 3：`self.name = name` 是在干什么？

**Q：** `self.name = name` 这句话属于什么概念？为什么要 `self.` 前缀？

**A（我的猜测）：** 可能就是赋值？把参数存到某个地方？

**R（验证后的结论）：** 这是在创建**实例属性**——把数据绑到具体对象身上。右边 `name` 是传进来的参数（临时的），左边 `self.name` 是绑到对象上的属性（永久的）。不加 `self.` 的变量是局部变量，方法执行完就消失。

```python
self.name = name   # 实例属性：对象在它就在
temp = "临时"       # 局部变量：方法结束就没了
```

实例属性 vs 类属性：实例属性每个对象各一份（like 分区数据），类属性所有对象共享（like 表 schema）。

**一句话费曼：** `self.xxx = yyy` 就是给当前对象贴一个标签，标签跟着对象走，方法结束了标签还在。

---

## QAR 4：`__init__` 是什么？为什么前后有下划线？

**Q：** `__init__` 是什么方法？我需要手动调用它吗？

**A（我的猜测）：** 看着像初始化？双下划线可能是 Python 的某种特殊标记？

**R（验证后的结论）：** `__init__` 是初始化方法（构造方法），对象创建时 Python **自动调用**，不需要手动调。双下划线 `__xxx__` 是 Python 魔术方法标记（dunder method），不同的名字在不同时机自动触发（`__str__` 在 print 时，`__len__` 在 len() 时）。日常最常用就是 `__init__`。

```python
class Dog:
    def __init__(self, name):
        print("自动执行了！")  # 你没调我，但我自己跑了
        self.name = name

Dog("旺财")  # 输出：自动执行了！
```

**一句话费曼：** `__init__` 是对象的出厂设置程序——产品一下线，这段程序自动跑一遍，把初始数据装配好。

---

## QAR 5：`Dog("旺财", 3)` 这句话到底做了什么？Dog 哪来的？

**Q：** 为什么写 `dog1 = Dog("旺财", 3)`？Dog 是哪来的？参数怎么对应的？

**A（我的猜测）：** Dog 应该是上面定义的类名？但参数有3个（self, name, age），我只传了2个？

**R（验证后的结论）：** Dog 来自 `class Dog` 定义（先定义后使用，和函数一样）。self 不用你传，Python 自动塞对象进去，你只传后面的参数：

```python
def __init__(self, name, age):
#            自动   第1个  第2个
Dog("旺财", 3)
#    第1个  第2个
```

背后四步：造空对象 → 自动调 `__init__` → 绑属性 → 赋值给变量。

**一句话费曼：** `Dog("旺财", 3)` 就是跟 Python 说"按 Dog 模具造一个，名字旺财，3岁"，Python 自动处理剩下的。

## QAR 6：方法和函数有什么区别？

**Q：** 类里面的 `def` 为什么叫"方法"而不叫函数？有什么本质不同？

**A（我的猜测）：** 可能只是换了个名字？写在类里就叫方法？

**R（验证后的结论）：** 本质区别是方法**绑定了对象**，能通过 self 访问对象的数据；普通函数是无状态的，数据全靠参数传入。

```python
# 函数："给我数据我处理"
def bark(name):
    print(f"{name}：汪汪！")

# 方法："我自己有数据，直接干活"
def bark(self):
    print(f"{self.name}：汪汪！")
```

**一句话费曼：** 函数是外卖员（你得告诉他地址），方法是住家保姆（自己知道家在哪）。

## QAR 7：什么时候该用类，什么时候函数就够了？

**Q：** 我平时写取数脚本，到底需不需要定义类？

**A（我的猜测）：** 简单脚本应该不需要？类是写大项目才用的？

**R（验证后的结论）：** 判断标准很简单——**好几个函数反复传同一批参数时，就该用类了。**

```python
# 函数版：db, table 重复传了3次
get_partition("dw", "fact_order", "2025-06-01")
check_partition("dw", "fact_order", "2025-06-01")
get_row_count("dw", "fact_order", "2025-06-01")

# 类版：创建一次，后面干净利落
t = HiveTable("dw", "fact_order", "2025-06-01")
t.get_partition()
t.check_partition()
```

口诀：临时取数 → 函数够了；封装通用工具（建表/查分区/DQC）→ 用类。

**一句话费曼：** 当你发现自己在复制粘贴同样的参数列表时，就是类登场的信号。

## QAR 8：对象之间怎么交互？

**Q：** 一个对象怎么使用另一个对象？比如"主人遛狗"怎么写？

**A（我的猜测）：** 可能把另一个对象当参数传进去？

**R（验证后的结论）：** 完全正确！把对象当参数传入，或者存为属性，然后通过 `.` 链式访问。

```python
class Owner:
    def __init__(self, name, dog):
        self.dog = dog  # 把狗对象存为属性

    def walk_dog(self):
        self.dog.play()  # 通过 . 调用狗的方法
```

数仓版：ETLTask 组合 source 表和 target 表两个对象，各自管各自的全名拼接。

**一句话费曼：** 对象交互就是积木拼装——把小零件（对象）插到大零件（另一个对象）上，通过 `.` 连接访问。

## QAR 9：继承解决什么问题？

**Q：** "哈士奇是狗的一种"这种关系在代码里怎么表达？

**A（我的猜测）：** 复制一份 Dog 类的代码改改？

**R（验证后的结论）：** 不用复制！用继承——子类括号里写父类名，自动拥有父类的一切，只需要写"差异部分"。

```python
class Husky(Dog):      # 括号里写 Dog = 哈士奇继承自狗
    def bark(self):     # 重写：覆盖父类的叫法
        print("嗷呜~~~")
    def demolish(self): # 新增：父类没有的独有技能
        print("沙发没了")
```

规则：自动拥有父类一切 → 想改就重写 → 想加就直接加。数仓场景：BaseTask 写公共逻辑（log、初始化），HiveTask/DorisTask 各自只写 run 方法。

**一句话费曼：** 继承就是"抄作业但改名字"——公共答案抄父类的，不一样的题自己写。

## 知识地图

```
类和对象
├── 类 = 模具（定义属性 + 方法）
├── 对象 = 产品（类的具体实例）
├── self = "我自己"（第一个参数位，必须留）
├── __init__ = 出厂设置（自动调用的魔术方法）
├── 实例属性 = self.xxx = yyy（每个对象私有）
├── 类属性 = 写在 class 下（所有对象共享）
├── 方法 vs 函数 = 自带数据 vs 需传数据
├── 对象交互 = 组合（对象当属性/参数）
└── 继承 = 公共提父类，子类写差异
```


# python 类继承 和工厂模式区别
这两个东西解决的问题层面不同，但经常被放在一起比较是因为它们都涉及"创建对象"的场景。

**类继承**解决的是"代码复用和多态"问题。你有一组对象共享部分行为，把共性抽到父类里，子类各自实现差异部分。核心关系是 is-a：Dog is an Animal。

```python
class Animal:
    def speak(self):
        raise NotImplementedError

class Dog(Animal):
    def speak(self):
        return "汪"

class Cat(Animal):
    def speak(self):
        return "喵"
```

调用方直接 `Dog()` 或 `Cat()`，它知道自己要什么类型。

**工厂模式**解决的是"把创建逻辑和使用逻辑解耦"的问题。调用方不关心具体是哪个子类，只描述需求，由工厂来决定实例化哪个。

```python
# 定义了一个普通函数，名字叫 animal_factory（动物工厂）。
#`animal_type: str` 表示这个函数接收一个参数，类型是字符串（str）。`-> Animal` 表示它返回的东西是一个 Animal 类型的对象。
def animal_factory(animal_type: str) -> Animal:
# `Dog()` 加了括号就是"根据 Dog 这张图纸，造一个实际的对象出来"。类是图纸，`Dog()` 是按图纸生产出来的实物。
    if animal_type == "dog":
        return Dog()
    elif animal_type == "cat":
        return Cat()
    else:
        raise ValueError(f"未知类型: {animal_type}")
        

# 第一行：调用工厂，传入 `"dog"`，工厂内部执行 `return Dog()`，所以 `animal` 这个变量里装的是一只 Dog 对象。
animal = animal_factory("dog")
第二行：调用这只 Dog 的 `speak()` 方法，返回 `"汪"`。
关键点在于——写这两行代码的人**完全不需要知道 Dog 这个类的存在**。他只知道"我有一个工厂，给它一个字符串，它会还我一个能 speak 的动物"。至于这个动物具体是怎么造出来的、有多少种动物可选、内部实现长什么样，他都不用关心。
animal.speak()
```

---

两者的关系是：继承提供了"多个类可以被统一对待"的基础（多态），工厂利用这个基础，把"选择具体用哪个类"的决策集中到一个地方。没有继承（或接口/协议），工厂返回的东西调用方都不知道怎么用；没有工厂，调用方就得自己写一堆 if-else 来选类型，创建逻辑散落各处。

用数仓场景举个例子：你有多种数据源（MySQL、Hive、Kafka），每种的连接和读取方式不同，但对上层暴露统一的 `read_data()` 接口。继承解决的是让它们共享接口定义，工厂解决的是"传入一个数据源配置，自动返回对应的连接器实例"，上层代码完全不需要关心底层具体是哪种数据源。

一句话总结区别：继承是"我是什么"（类型体系），工厂是"谁来决定创建什么"（创建责任的归属）。继承是结构层面的设计，工厂是控制流层面的设计。两者互补而非互斥。
