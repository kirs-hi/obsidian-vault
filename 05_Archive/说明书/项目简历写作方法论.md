# 项目简历写作方法论

## 核心原则

### 原则一：量化一切

"优化了查询性能"是废话，"TP95 响应从 2500ms 降至 1300ms"才是信息。量化维度包括：

- 性能指标：延迟、吞吐、响应时间、QPS
- 规模指标：任务数、数据量、表数量、服务的产品数
- 效率指标：开发周期缩短 X%、故障率下降 X%
- 业务指标：覆盖 X 条产品线、支撑 X 万用户

### 原则二：写决策，不写执行

错误示范："用 Flink 开发了实时 ETL 任务"（执行层面，谁都能做）

正确示范："针对业务生命周期长导致的 State 膨胀问题，对比 Regular Join / Interval Join / Window Join 三种方案后，选择 Regular Join + JoinKeyContainsUniqueKey 优化"（决策层面，体现判断力）

面试官想看到的是：面对问题时你的思考过程，为什么选 A 不选 B，权衡了什么。

### 原则三：先讲问题，再讲方案

没有问题背景的方案毫无意义。要让面试官理解这件事的价值和难度。

错误示范："搭建了双链路容灾体系"

正确示范："实时数据直接参与供应商结算，一次故障就是真金白银的资损，因此搭建了跨机房双链路容灾"

### 原则四：体现技术深度而非广度

堆砌技术名词是最常见的错误。面试官不关心你用过多少工具，关心的是你对某个技术理解到什么深度。

错误示范："熟悉 Flink 开发"

正确示范："深入理解 Flink State 管理机制，解决过 RocksDB 堆外内存不受 JVM 管控导致的 Container OOM 问题"

### 原则五：一个项目讲一个核心故事

每个项目应该有一个主线。"从零搭建实时数仓"讲架构设计能力，"稳定性治理"讲工程治理能力。主线清晰，面试官才记得住。

## 推荐结构：背景-职责-亮点-成果四段式

### 第一段：项目背景（2-3 句话）

回答两个问题：这个项目解决什么业务问题？技术挑战是什么？

要求：不写公司内部术语，让任何技术面试官都能看懂。点明业务规模和技术难度。

### 第二段：核心职责（1 句话）

明确角色和负责范围。是主导架构设计、负责模块开发、还是做性能优化？不要模糊。

### 第三段：技术亮点（3-5 个 bullet point）

每个 bullet point 的结构：**做了什么 + 为什么这么做 + 效果如何**

按技术含量从高到低排列，最有深度的放第一条。每条控制在 2-3 行。

示例格式：
> **设计并落地三引擎混合架构（Flink + Doris + Blade）**，根据时效性、数据量级、状态大小三个维度制定引擎选型决策树，纯实时链路覆盖率从 20% 提升至 80%，核心指标延迟从 10 分钟级降至秒级。

### 第四段：项目成果（3-4 个数据点）

纯量化结果，分为技术成果和业务成果。

技术成果示例：TP95 响应降低 48%、故障率下降 70%、任务告警量下降 60%

业务成果示例：支撑 10+ 核心数据产品、助力履约效率提升 X%

### 最后一行：技术栈

只列实际深度使用的技术，不要把只碰过的也写上去。

## 面试深挖准备

每个项目写完后，额外准备 3-5 个"深挖问题"的回答：

- 为什么选这个技术方案？对比过哪些方案？
- 遇到过最难的问题是什么？怎么排查和解决的？
- 如果重新做，你会怎么改进？
- 这个项目的技术债是什么？
- 你在项目中最大的成长是什么？

## 常见错误

- 写成工作日志（"负责了 XX、完成了 XX"）而不是能力证明
- 所有 bullet point 都是同一个层次，没有主次
- 只写技术方案不写业务价值
- 数据造假或过度夸大（面试一追问就露馅）
- 项目描述太长，超过半页 A4 纸
