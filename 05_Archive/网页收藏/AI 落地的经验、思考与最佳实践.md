到目前为止，我已经解决了不同场景、不同业务的 40~50 个 AI 问题、应用以及解决方案。  
再结合最近在落地的 6 个 AI 项目，里边的技术涉及到 Java、Python、[[Day2-JavaScript和TypeScript|TypeScript]]，不同的场景和框架都用了很多，所以这里面有很多是我自己的体悟和感悟，还有一些真正的坑和相关的难点，以及现有场景的一些现状。  
接下来和你谈一谈 AI 在技术中的最佳实践。  
希望这些内容对你也有收获，也希望有一些借鉴的经验能帮助到你。

## 一、落地现状

### 应用现状

我特别想跟你说，现在都是 AI 时代，所有东西都是 AI native 的，大家只通过 CLI 和 [[07-Agent|Agent]]，就构建一个一个应用。  
但是在实际咨询和做项目的过程中，并非如此。尤其是传统企业，他们更多还是习惯于操作之前的页面。所以即使有 AI 化的融入，也是 AI 方面的一个改造，比如说在某些方面加上 AI 能力，再以传统的表格的方式把它做好。  
这是我看到的一个现状。并非所有企业都在做 AI native，有很多企业是基于原有系统在进行 AI 化或者说加 AI。  
这是一块很大的市场，也是我看到的一个非常现实的要求和落地的现状。

### 技术栈现状

关于技术现状，我非常想跟你说，现在大家都在使用 Python 和 TS，但是我们不得不承认，在真实的项目中会有很多 Java 的项目，尤其是 Java Spring 项目。  
5 年前或者 8 年前非常火的一段时间是 Spring Boot，当时我也在基于 Spring Boot 这种框架给企业做架构。我也深知 Spring Boot 这种非常抽象的架构可以带来稳定，但同时也会带来很多复杂的研发成本和冗余代码。  
后来，我做了很多关于 Python 的应用，因为 Python 改变了 Spring Boot 这种冗余的写法，尤其是 Java 这种复杂的写法。  
使用了 Python 之后，我觉得 Python 非常好用，它一定会替代 Java 的。但是目前的现状是，依然有很多项目在使用 Spring Boot，而且这个东西暂时不会消失。我接到企业更多的事情是基于 Spring Boot 怎么去做 AI Coding 开发。给大家指导了很长时间，发现如果去做 Spring Boot 的 AI Coding 开发，微服务再去结合 AI 自动化 Coding 的时候，需要补充大量的上下文，这是一个挑战。  
但是想跟大家讲的是，技术现状并不都是 Python 和 TypeScript 的，更多的是 Java 架构。所以也有更多的企业希望我们不把 Java 的整个架构推翻，而是在 Java 的基础上加 AI。这也是很多企业中层或者上层领导不希望看到整个架构变化的原因，因为这就意味着人员的变动、技术栈的变动、系统不稳定因素的增加。所以大家在考虑，能不能在现有架构里边去加一些东西，也就是说在现有架构上加 AI——加 AI Coding 也好，加 AI 功能也好，让它看起来 AI 化了。

## 二、技术选型

### 图谱重要吗？

我们之前做过很多关于图谱的事情，比如说构建图谱，将图谱工业化，将图谱作为基础的逻辑、基础的数据源提供给上层。但是我们渐渐地发现，图谱抽取的一个很好的方式就是使用模型去抽取。  
我们当时使用了很多相关的框架和架构，但是图谱底层抽取的时候，依然有置信度的问题。之前我给大家讲过关于置信度逐渐改良的问题，但在改良的过程中，仍然发现如果需要图谱去稳定，需要 100% 审核。  
如果非 100% 审核，我们把它融入到产品的时候，就发现一个问题：即使图谱的准确度已经做到 85% 以上，只要有一个图谱的点在其中有问题，比如说不准确，用户就会对整个产品有一个不好的体验。因为你所有的图谱数据其实是基于原始数据进行的再加工。所以如果在给用户提供绝对事实的情况下，图谱的这种方式是被高估的。原因是你在做图谱的时候，用了大规模的技术手段去做抽取、[[提示词]]优化、流程优化、人工审核，花了大量时间。  
但你会发现，你想构建的一个精准的东西，居然依赖于一个概率型的模型。所以最后又产生了一个信任问题，这个东西是被高估的。  
所以大家如果准备做一个真正的图谱项目，一定要先想好。如果只是给用户提供事实源的追溯，其实更应该使用传统的 [[06-RAG|RAG]] 或者 Agentic RAG 的方式，把溯源情况给到他，效果会更好。所以图谱一定要慎用，图谱的很多场景是被高估的。  
（当然有的场景是图谱足够好就行，不需要精准的图谱准确性。这种场景的话，就是可以的。比如说社交型的分析图谱、趋势走向的图谱、新闻聚焦的图谱，这种就是可以的。但对于绝对事实可信度的这种图谱数据，一定要谨慎。）

### RAG 通用知识库？

结合数据这一点，你再去看 RAG。很多人认为 RAG 已经被淘汰了，但在企业实际使用的时候，RAG 还是在被大量应用。  
大量应用的原因是企业真的有很多数据，如果不使用 RAG，全量交给模型的话，第一会遇到模型成本的问题，第二会遇到模型在大海捞针时准确度丢失的问题。  
为什么还在使用 RAG？因为 RAG 一下子可以把企业的很多数据聚拢起来。  
如果 RAG 不会消失，那它使用的最佳实践是什么？我用起来的感受，RAG 没有一个唯一通用的策略，都是结合企业不同的场景、不同的人员在使用数据库时候的方式，来定制 RAG 的用法。  
比如你在做法律相关的，你会发现法律上面有很多条文。我们使用的方式是：把这个条文先进行粗略的清洗，清洗之后再进行一条一条条文的梳理，梳理之后进入 RAG 的库。这个方式之前在知识星球的其他文章中我也给大家介绍过。  
非常关键的一点是，RAG 必须提供数据源的追溯支持，这个非常重要。比如我们以前做的时候，可能一个问答系统就结束了；  
现在我们做的时候，会把 RAG 的溯源信息——来源于哪条信息——都列出来。第一个，你要根据场景去切你的 RAG 策略，没有一种完全通用的 RAG 策略。你可以想象一下，如果 RAG 真的有一种通用的方案出来，肯定会有一个通用库的解决方案。你发现 GitHub 上没有，商业社区也没有。从这个也能判断出，像这种非常宽的赛道，没有统一的切入点的时候，这个东西就没有统一解。  
所以 RAG 的第一步是把 RAG 按照功能、按照需求、按照场景切分好，这个非常重要。第二步是你要给它完整的溯源信息，要知道这些数据来源于哪。  
第三步也是非常重要的，我们在做 RAG 的时候经常使用到一种方式：用户可以接受你多花一些时间，但还是希望你能把这个东西做准。当然，这是在要求准度比较高的情况下，也就是我们会引入 Agentic RAG，让 Agentic RAG 基于循环的方式不断去找信息缺漏，从信息缺漏里边归纳出问题，再去 RAG 里边找。它是一个不断循环的过程，虽然耗时，但它给 RAG 提供了一个非常好的方向。

#### Java

Spring Al Alibaba ：[https://java2ai.com/](https://java2ai.com/)  
java - semantic-kernel : [https://github.com/microsoft/semantic-kernel](https://github.com/microsoft/semantic-kernel)

你知道，如果是 Java 项目去做 AI，很少有企业会从 0 开始搭建一个 Java 项目，更多时候是在原有项目中引入一个 AI 框架、AI 功能，将系统进行 AI 化改造。不管是因为上级领导的关注，还是希望功能落地，或者因为其他原因，都是希望 AI 可以融入到系统中，所以很多时候是把这些框架嵌入进来。

基于 Java 的一个前置原因——对稳定性的追求，你会发现 Java 所有的框架，抽象做得都很好。比如 interface、继承、接口、设计模式，其实做得都很好。我们以前在研究 GoF 23 设计模式的时候，都是以 Java 为基础去研究，就是因为它的数据结构等各方面都很好。Java 对稳定性的要求很高，这也是为什么它很抽象的原因。

基于这些原因，我们去看 Java，就得给它选一个非常稳定的框架。在两个框架之间，一个是 Spring AI，阿里巴巴自己出的框架，另一个是 Java 的 Semantic Kernel，微软出的框架。当时考虑这两个框架的原因是阿里巴巴和微软提供的框架基本上都会很稳定。就想它们可能谁会持续下来并且能把框架持续更新好，后来我们选的是微软。原因是微软离 OpenAI 最近，它是大股东，所以我们想它提供的框架肯定对 AI 很友好，当时就选择了这个框架。

选择这个框架之后，大家最关注的是它能否很好地融入到 Java 这个抽象体系，并且提供一个稳定的 AI 内核出来。并不是说其他框架不好，这个是想跟大家聊的。所以大家去思考的时候，你会发现 Java 体系里边对框架稳定度的要求是非常高的，因为 Java 去换一整个框架成本比较高，写法也比较难，所以大家在做 Java 体系的时候，一定要把这件事情调研好。核心思考逻辑，也是当时我们跟团队探讨的逻辑：在整个发展过程中它是不是会持续更新这个框架，这对未来发展来讲非常重要。

所以为什么跟大家说没有一个特别唯一的框架选择，其实它还是一个平衡的选择，平衡地去选择这些框架。

（有很多伙伴跟我说，这个框架效果非常好，值得试一试。但实际应用中，我没有去使用过。如果大家感兴趣的话，可以去研究一下: Spring Al Alibaba)

#### Python

python - langchain : [https://github.com/langchain-ai/langchain](https://github.com/langchain-ai/langchain)

Python 框架我们肯定就使用 Langchain、LangGraph 这套体系去做了。选择 Python 的原因，更多是因为这些企业是传统的计算型企业，比如做计算算法、科学算法的，他们更喜欢选 Python。所以当时我们选的框架大多数都是 Python 相关的框架。使用这些框架的时候，大家希望无缝融入起来，语言和框架就选择的是 Python，底层我们就使用 Langchain。

Langchain 前期使用最大的问题就是版本变动太多了。有一段时间我们弃用了 Langchain，直接通过 API 的方式去调用，自己写了一个简单的小框架，因为当时 Langchain 太不成熟了。用了一段时间之后发现一个很大的问题：当 Agent 变得复杂的时候，自己做那个框架也会变得麻烦复杂，这其实是一个大问题。

后边在 Langchain 稳定之后，我们又踩了一个大坑，把整个 AI 框架又换成了 Langchain。为什么呢？因为我们发现如果做 AI 应用，一定要去做监控。做监控的话，就要连 LangSmith 或者 LangFuse 这种平台。如果你自己再去做一个 OTel 平台，问题其实是比较大的，包括后边很多集成的问题。所以当 Langchain 出了 middleware，也就是 Langchain 1.0 之后，我们发现很多东西就稳定了，又切回了 Langchain 框架。最近使用 Langchain 的话稳定了很多，而且他们的联合创始人 Harrison，总在结合很多不同的技术栈追求最新的一些工程实践，我读他们的博客有很多启发。  
他们这些博客都是基于 Langchain 框架怎么去改造、怎么去升级的，有自己独特的一些见解，我觉得特别有帮助，这也是我们最后又继续选择 Langchain 的一个原因。

langchain 博客： [https://www.langchain.com/blog](https://www.langchain.com/blog)

#### TypeScript

typescript - pi-mono : [https://github.com/earendil-works/pi](https://github.com/earendil-works/pi)

你需要知道这个框架是 [[00_OpenClaw_MOC|OpenClaw]] 的核心框架。它的整个框架和生命周期我们都看过了，觉得逻辑灵活度非常好。我们把它做了一个非常有意思的实验——在一个 TS 应用中使用它。

你知道 TS 应用为什么现在比较火吗？并不是说它的能力有多么强，最主要的原因是它的交付成本最低。比如你使用 AI 工具，像 Lovable，之前跟大家介绍过的，或者 Bolt、V0，这些东西都使用 TS 的架构去做的，不管是单页面的纯 React 方式，还是 Next JS 的方式，它的交付成本都非常低，也就是说很快能交付。所以很有必要为它选一个框架，这样我们就选了 Pi Mono。

Vercel AI SDK： [https://ai-sdk.dev/](https://ai-sdk.dev/)  
mastra： [https://mastra.ai/](https://mastra.ai/)

并不是说我们选择了 Pi Mono 这个框架就是最好的，而是我们当时有了一些思考。对于 TS 应用，如果想这么快交付的话，一定是希望实验型的东西经过一些小型实验，就可以无缝迁移到整个项目体系中。我们看到 Pi 这个框架可以通过扩展的方式注入一些命令和工具——工具给模型用的，命令是给人用的。这些步骤可以做一些 SOP 或者上下文注入相关的内容。再结合它生命周期的扩展，我们发现它做起来的灵活度非常高。如果切过去的时候，我们只需要把 SDK 调起来，把核心文件的地址给它，就可以无缝切过去了。这是我们当时选择它的一个原因。

为什么说它也不是最好的？因为它太灵活了，前期做起来很快，后期却需要做很多工程化的东西。比如要给它做一个监控，把 AI 的生命周期都监控起来，我们就发现需要自己写大量的扩展插件。包括后边要给它做一些其他的工程化方式，都要围绕这个灵活的框架去开发很多东西，到中后期它的能量其实是有些不足的，这也是我们发现很大的一个问题。

当然选择它的原因，也是因为它可以很快通过扩展的方式迁移过去。但是大家如果不存在 "测试完了以后直接迁移到正式项目" 这种场景的话，其实就可以去选其他的框架，比如 Vercel AI SDK、Mastra，这些框架都可以。

## 三、落地实践

### AI 自动化 Coding 的避坑指南

AI 自动化 Coding 这个方向，我相信大家已经在各个地方应用起来了。大家会发现，之前我也给大家讲过，不管是 Superpowers 也好，[[【GSD】AI编程最佳工作流 GSD 详细指南|GSD]] 也好，它们都在完成不同的 AI Coding 方向和要解决的问题。我们的自动化 Coding 是为了真的解决企业自动化开发的一些问题。

这里边一个关键点，也是在我们实践里边反复跟大家聊的。我给企业也在做内训，做内训的时候给他们讲 AI 自动化 Coding。我就会发现，包括我们社群的伙伴们也是，有的人或者企业把这个东西用得很好，有的就用得不太好。原因其实还是在于对自动化 Coding 的 Harness 那块做得不够。我们知道 Harness 比如说 Agents.md，非常关键的一点是，它是给要处理你问题、帮你做 task 的模型来准备的一个手册。这个手册肯定是因人而异的，所以这是一方面不稳定的因素。

为什么说要把 Harness 工程跟大家再强调一下？你把 Harness 工程做好了以后，未来做很多东西的时候就有了相关的手册，包括我们的复利工程其实也是为了完善 Harness 的手册。大家叫 Harness，你看 OpenAI 给它定义的是 Codebase map，有的企业给它定义的是约束规范，但实际上，Harness 工程的核心就是让模型往你想要的方向、正确的方向去走。所以 Harness 工程这两点都包括：如果你有约束，可以放在 Harness 工程里边；如果你有 Codebase map，也可以放在 Harness 工程里边。

如果真的想把 AI 自动化 Coding 做起来，一定要把文档手册写好。我也给大家一个方式，就是使用 GSD 的 map Codebase 先把 Harness 工程跑起来。这是一个非常重要的点。

第二个点，除了这个以外，你要把 Plan 计划做好。有的企业为了快速跑起来，Plan 计划做得非常差，基本上不怎么审，觉得自己已经描述清楚了，都听 AI 的，下一步、下一步、下一步，AI 生成了一个 PRD.json，一跑的时候发现跑了 8 个小时、10 个小时，不是自己想要的，就觉得这个东西不好用。

所以我想给大家讲一个事情：你的 Plan 一定要跟你对齐。对齐的方式，你使用我之前跟大家讲的 AI 杠铃审核策略也好，使用其他自我问答、自我采访的方式也好，哪种方式都好。但是你的目的是什么？你的目的是把 AI 自动化 Coding 这件事的前置工作，也就是需求和 AI 与你对齐这件事，先搞清楚，这个非常重要。

所以 AI 自动化 Coding 是一个非常棒的利器。不要先着急追求让 AI 全自动化开发，要先追求 Harness 工程，先追求把问题和 AI 对齐，这些方面都是最重要的。它其实比你创建一个并行任务去开发，或者快速让 AI 把工程跑起来更重要。因为只有让 AI 往你想要的方向去做，它才是对的，这其实是关键。

### 最佳实践的思考

接下来终于要跟大家讲最佳实践了。前面的技术层次和相关思考我已经跟大家讲完了，后边就跟大家讲落地的最佳实践。上面那几个语言框架其实也在跟大家说明，没有一个正确与错误的绝对标准，它们都是基于场景的实用性来考虑的。

首先要接受一个现实，不是说有一家企业提出来说我要做一个传统的表单级项目、审核级项目、报销型项目或者 OA 项目，他们就不适合做 AI 化或者纯 AI 化改造。其实并不是的，有很多企业我们要承认这个现实，他们不太希望大批改造自己企业内部的系统，有可能是使用习惯上的原因，也有可能是稳定性的考虑。所以如果他们真的提出这个需求，那就是真的有这种需求，这也是 AI 化改造中的一部分。

所以一个点就是，整个 AI 化的系统需求其实什么样的需求都有，这是我们需要接受的一个现实。

### 老项目、微服务与 AI

下一个想跟大家说的是 Java 的技术栈问题，短时间内 Java 技术栈也不会消失。比如说很早之前，大概是 2011 年或 2012 年，有朋友大家都在学 Python，说 Python 以后会怎么样。但当时我没有接触到 Python，因为那阵 Java 正好从 90 年代刚出来，然后慢慢火起来，到 2010 年左右到达了一个挺巅峰的状态。2013 年、2014 年那阵，还到处都是关于 Java 架构方面的事情。后边 Python 火起来，也就是近几年，比如从 2023 年借 AI 的机会才火起来。你会发现一个技术的周期演进，至少需要 10 年。

所以在 Java 现在还有这么多体系没有消失、Python 刚起来的情况下，还会有大量的程序在沿用 Java，因为那些技术架构、技术栈不可能一下全都拆散、全都换掉。所以最关键的一个点是，这些技术栈还会存在，我们就要结合这些技术栈来思考，接到这些项目的时候怎么去开发。

我碰到一个最棘手的情况，就是现在有很多遗留的微服务架构，这些微服务架构跟当年的初衷已经完全不一样了。当年的初衷是希望一个团队去维护一个微服务，让这个架构越来越专业化。但是你会发现，这几年由于各个平台入口的分发，这样的一个微服务已经没有以前那么高的并发或复杂度去承接这样的项目了，除非是老项目继续沿用。这样就导致很多开发人员被削减或者已经离开原本的企业机构，变成了一个人去维护多个微服务的架构。一个人去维护多个微服务架构，本身就已经远离了之前 DDD 领域驱动开发、微服务拆分的逻辑了。

但是我们接手这些东西的时候，还是要耐心地去做 Harness，做一些 plan 计划，AI 自动化 Coding 依然还是可以用起来的。原因大概率是训练 Java 的语料足够多，所以 Java 现在的自动化 Coding 能力是非常强的。这部分我们只需要把 AI 自动化 Coding 做好，把它以前的架构、目录结构这些 Codebase map 都做好，基于这些让它自动化 Coding 开发起来。相当于我们沿用了之前的架构，用新的 AI 自动化 Coding 继续把它推进起来。

### 新项目与最佳实践

后边对于一些新项目的逻辑和思考，跟大家分享一下。新项目我们使用过很多方法，比如使用 Next JS、HTML 的方式去做。但最后我们发现，如果一个项目要启动起来，最好的方式不是没有页面，最好的方式应该是先展示页面。

以前我们觉得，现在什么都能做了，你把功能给我说清楚，我就拿回去做吧。其实这个是错误的。第一个方面，我们通过录音的方式，先跟客户把需求谈清楚。这个也是之前我在知识星球跟大家讲过的录音谈需求的方式，通过录音谈需求之后，我们把整个逻辑让 AI 给我们梳理起来，确认这是我们最终的需求。结合这个需求形成一个需求文档，形成之后我们就给到 Lovable，让 Lovable 先给我们生成一个初步的页面。因为我们现在测了很多 skills，这些 skills 的生成效果在实际测试下都不如 Lovable。所以你先拿 Lovable 去生成一个原型，先去跟用户对齐用户的需求。这个非常重要，是一个非常关键的点——先用 Lovable 把用户需求对齐。并不是说你不需要页面，而是你现在需要一个非常关键的前期跟用户对齐的工作——产品需求对齐。因为只有这样，你后边的 AI 自动化 Coding 也好，AI 的各种设计也好，这些才有意义。

（毕竟 AI 自动化 coding 不能对齐前端页面， 这部分需要 copilot 和人的确认， 确切的说是 HILP）

跟用户对齐的过程，其实也是你了解用户需求的过程。所以 Lovable 生成之后的页面，不要直接转给用户，一定要对着你录音里边整理好的需求文档和 Lovable 产生的页面，真正地去看一遍。看一遍之后，把每一个功能都去跟业务之前聊过的东西对齐，把这些东西先确定下来。用户肯定会给你反馈几遍，我们很清晰地能看到，基本上反馈 3~5 遍，整个页面才有一个确定性的结果。

这里边非常关键的一点：你一定一定一定不要直接做功能，一定要先去对齐。我们做过太多的项目，用户说完了之后，有一些团队就比较着急，抓紧就开发，说你要不要这样？你要不要那样？用户说你这要给我改一点，那要给我改一点。回来改的时候发现他们用的其实是 Next JS 架构，用 Next JS 架构就会把前后端全都改一遍。当然更恐怖的是，他们在给用户做原型的时候就使用了 React 加 Python 的方式，或者使用 Python 加 HTML 的方式去给用户做演示。这些都是把过度的工程化提前到了用户的前期确认阶段，都是不对的。

所以你会发现，传统软件的东西是通过 AI 加速了，它并没有省去哪些步骤，只不过是把可以加速的部分加速了，放大了整个流程的效率。所以第一步你是不能去掉的，也不要因为 AI 工具的加速能力，而过早地过度工程化。第一步就是跟用户的需求对齐，把页面好好去做对齐。

有了第一步之后，我们才会结合第一步产生的原型，做数据对齐。整个用户的需求对齐之后，让客户给我们提供这些数据，看这些数据是否能跟每一个页面或者底层的设计流程对上。你会发现有很多项目，用户说是这样，但这个数据没有、那个数据没有，最后你不得不在工程化之后再去改前端的页面，返工也是非常严重的。所以最高效的方式就是减少返工，最好的方式就是跟用户先聊清楚他这些数据能不能支持产品。

（哈哈， 这个道理怎么看都是对的： less is more）

有了这些东西之后，我们可以接下来基于 Lovable 继续调。因为它的样式开发很好，跟用户已经确定完数据、要求等之后，在 Lovable 里边会调很多相关的细节，使用两三次的对话语音继续去调。在 Lovable 里边调这些细节的时候，有一个点需要注意：当你调到五六十条对话信息之后，或者说 100 条信息之后，你每一次跟 AI 的对话费用就非常高了。这个费用基本上你很难接受，因为一次的话十几刀美金，还是非常高的。

所以最佳实践，我们是这么来做的：使用 Lovable 把初版做好之后，把它 Git 同步下来，然后用本地的 Codex 工具或者其他 IDE 工具，基于 Lovable 给的框架在上面进行 AI 的页面 Coding。因为它属于 SPA，属于纯 React 加 Vite 的方式，所以我们用本地的去 Coding 就行。本地 Coding 之后，可以给它同步回 Lovable，让 Lovable 进行演示。因为 Lovable 还可以把你在本地改的代码同步回来，这样费用就减少很多，要不然 Lovable 自己本身的费用其实是非常高的。

而且当你同步回的代码，你可以利用 Lovable 的远程部署功能方便地给客户进行演示。连域名部署的问题都省了。 （lovable 自带了发布 publish 的功能）

我们做了很多项目，跟大家说 Lovable 的原因，其实是 Lovable 跟后边很多东西是连接起来的。我们之前做一个项目使用 Next JS 去做，就发现没有特别好的 Next JS 生成 IDE 工具，就在本地去做。我们发现 Next JS 在改一些页面的时候时间比较长，可能跟它的编译加载机制有关系，所以我们不太想用 Next JS。

之前拿 Next JS 做了一些应用，我跟大家讲里边的问题在哪：拿 Next JS 做一版应用，做完了之后，开始觉得它前后端都能写，都是 TS 的，写起来非常好。它确实写起来比较好，也可以跟用户去对接、聊需求。但是后边就会发现一件事情：任何项目做到中后期的时候，都会有管理后台，你不可能有一个项目没有管理后台（RBAC 这套）。怎么办呢？就需要再给它做一个管理后台，就会发现 Next JS 的很多功能都已经集成在项目内部了。而它本身是一个快速开发产品的框架，管理后台这块非常弱，你要为它单独做一个管理后台就很麻烦。后边我们不得不从外部给它套一个管理后台给 Next JS 去用，再结合现有的项目把它开发出来，发现其实绕弯了。

后来我们放弃了 NextJS。我们说，Lovable 开发得这么好的话，那我们直接去用 Lovable。Lovable 它自己的后端数据库是使用 [[Day5-Supabase和实战项目|Supabase]] 的，Supabase 里边会有边缘函数，那我们就觉得直接使用这一套东西吧。然后我们在 Lovable 里边做了很多方式的更新。更新完了以后，想给它无缝接回到国内的平台上，但是我们发现，国内平台的 Supabase，比如阿里平台，其实是非常不完善的，里边是没有边缘函数功能的。所以我们就放弃了这个想法，应该是在阿里平台是不能直接迁移过来的。那 Supabase 在国内又不支持，怎么办呢？我们就要把它所有的边缘函数和 Supabase 的权限管理，迁移到一个可以承载这些功能的框架上。 （这里说的阿里的 supabse 不是 ADB 那套， 而是 supbase 的关系数据库 AI 功能那套。）

后来我们怎么去做？后来我们发现，管理后台这么麻烦，为什么不使用一个现成的？之前我们又做过若依相关的应用开发，我们就想，若依平台的 Python 版本，Python 加 Vue 加 FastAPI 那个版本，拿过来直接用就好了，AI 现在改起来也很轻松。所以若依平台的管理后台，它自带的 RBAC，就成为了我们非常强大的候选平台。

ruoyi: [https://github.com/insistence/RuoYi-Vue3-FastAPI](https://github.com/insistence/RuoYi-Vue3-FastAPI)

所以整个后边的 AI 部分，我们就舍弃了 Next JS，因为我们发现它到中后期去做强大的 RBAC 系统管理的时候，会花费很多时间。

我们的最佳实践就变成：前端我们使用 React 这种单页面的方式，SPA 的方式，使用 Lovable 去把整个项目开发出来。  
开发出来、对齐之后，先使用 FastAPI 加 Python 最简单的方式把整个应用先搭建起来。  
搭建起来之后，如果项目发展到中后期，我们就把之前 FastAPI 写好的 Python 程序，用 AI 无缝整合到若依的平台里边，把权限这部分切换掉。  
这样整个若依平台的后端管理平台就和我们无缝对接上来了。  
整个项目从原型到最终生产就都平滑的发展过来了。
