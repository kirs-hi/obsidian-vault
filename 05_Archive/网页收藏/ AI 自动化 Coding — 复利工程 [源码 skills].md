写给刚接触 AI 自动化 Coding 的朋友。这篇文章讲清楚一件事：**当 AI 帮你写完一批代码之后，怎么让它下次写得更好？** 这就是 "复利工程" 要解决的问题。

* * *

## 零、先搞清楚几个名词

在进入正文前，先把几个后面会反复出现的词解释清楚，不然看起来会很累。

*   **AI 自动化 Coding**：让 AI（比如 Claude、Cursor、各种 coding agent）根据你的需求，自动写代码、改代码、提交代码的整个流程。你下指令，它干活。
    

*   **Harness（马具 / 缰绳）**：字面意思是 "给马套上的缰绳"。在 AI Coding 语境里，它指**一整套约束 AI 行为的规则与上下文**，让 AI 按照你想要的方式干活，不要乱跑。本文里，Harness 先简化理解为一个规则文件：`agents.md`。
    

*   **`agents.md`**：一个 Markdown 文件，写着项目的规则、约定、目录结构、注意事项。AI 每次干活前先读它，就知道该怎么做。可以理解成 **AI 的 "员工手册"**。
    

*   **Story**：一个开发任务单元，通常是 "实现某个功能" 的小任务。一批 story = 一批要做的事。
    

*   **Commit**：Git 的一次代码提交。每完成一件事就 commit 一次，相当于**存档**。
    

*   **分支（Branch）**：Git 的概念，可以理解成一条**独立的工作线**。AI 在一条分支上干活，不影响主线。
    

**复利工程是什么？**  
类比 "复利"——利滚利。**每做完一批任务，经验和结构都沉淀到规则文件里，下一批任务就能在更好的基础上开始，越做越顺、越做越强。** 这就是复利。

* * *

## 一、背景：AI 自动化 Coding 缺失的两块拼图

现在大家都在做 AI 自动化 Coding，但整套流程其实**还缺两块关键拼图**。没有这两块，AI 干得越多，项目可能反而越乱。

### 拼图一：如何构建 Harness（规则文件）

也就是：**`agents.md` 这份 "员工手册" 该怎么写？**

写得好，AI 能准确理解项目、稳定产出；写得烂，AI 乱写一通，你越改越累。

本文不深入讲这一块。固定的编程范式、记忆机制等更复杂的话题也先不展开。如何设计 `agents.md` 的细节，之前的《AI 自动化 Coding 2.0 多子项目最佳实践》那篇文章已经讲过结构了， 后续我做好 skills 再分享给大家

### 拼图二：如何把完成的经验回流到 Harness

这是本文的重点。场景是这样的：

*   3.
    
    干活过程中，它生成一个 `progress.txt` 文件记录进度
    

**问题来了：**  
这一轮跑完，项目结构其实发生了变化 —— 新增了文件、改了目录、甚至调整了架构。调试过程中 AI 也积累了不少经验（哪些坑踩过、哪些写法更合适）。

**但这些新结构和新经验，没有回流到主项目的 `agents.md` 里（以及相关的索引文件中）。**

下次 AI 再来干活时，读到的还是**旧的员工手册**，自然不知道项目已经变样了，也不知道上次踩过什么坑。

这就是拼图二要解决的问题：**怎么把这一轮的成果，沉淀回 `agents.md`？**

* * *

## 二、核心问题：结构没有更新

用一句话说透：

**干完活了，但 "员工手册" 没更新。**

如果 `agents.md` 不更新，下一批 story 开工时，AI 拿到的还是过期信息。它既不知道新的目录长什么样，也学不到上次的教训。

**没有沉淀，就没有复利。** 每一轮都在同一起跑线上，甚至可能因为项目已经变化、手册却没变，反而越跑越差。

* * *

## 三、两个设计原则

要让沉淀这件事真正靠谱，必须守住两条原则。

### 原则一：以 Commit 为事实数据源

AI 在干活过程中会产生很多东西：临时文件、`progress.txt`、中间产物、没用的草稿…… 这些都**不可靠**，可能只是半成品。

**只有 commit 进 Git 的代码，才是真正 "落地" 的事实。**

所以更新 `agents.md` 的时候，依据必须是 **Git commit 的实际提交数据**，而不是那些临时文件。

**为什么？**

*   Commit 代表 AI 真正认可并提交了这个改动
    

*   Commit 有完整的 diff（改了什么）、message（为什么改）
    

一句话：**commit 就是事实。 我们要基于事实来做修改和完善 agents.md。**

### 原则二：顺着 `agents.md` 的关联文件逐层读取

更新 `agents.md` 之前，你得先搞清楚**现在的结构长什么样**。

怎么搞清楚？**顺着 `agents.md` 自己往下摸。**

`agents.md` 不是一个孤立的文件，它里面会引用、关联很多其他文件（子项目的说明、各模块的规则等）。这些关联文件又会继续关联下一层。整个结构是**一棵树**。

**关键点**：不要自己猜 `agents.md` 关联了什么，而要沿着它已经写明的关联文件，一层一层往下读，直到读完整棵树。

这样做的好处：

**然后**，把 "读到的现有结构" 和 "commit 提交的新改动" 放在一起对比，去更新 `agents.md` 本身以及相关的规则文件。

* * *

## 四、完整流程闭环

把前面两个原则串起来，整个复利工程的流程是这样的：

textjavascripttypescriptcsshtmlbashjsonmarkdownpythonjavaccpprubygorustphpsqlyaml Copy

```
1. 启动分支
   ↓
2. AI 在分支上完成所有 story（每完成一个 → commit 一次）
   ↓
3. 所有 story 完成后，分析这条分支的全部 commit
   （看它提交了哪些改动、新增了什么、调整了什么）
   ↓
4. 顺着 agents.md 往下，逐层读取现有的关联文件
   （搞清楚当前项目结构和规则）
   ↓
5. 基于"commit 事实 + 现有结构"，更新 agents.md 及其关联文件
   （把新结构、新经验写进去）
   ↓
   ✅ 员工手册升级完成，下一轮开工自动用上新手册



```

每跑完一轮分支，目录结构和经验就沉淀一次。**跑得越多，手册越完善，AI 干得越好 —— 这就是复利。**

* * *

## 五、落地成果：已经封装成 skill

基于上面这一整套思路，我已经把它封装成了一个 **skill**。

实际用下来，**效果非常好**，强烈推荐大家也用起来。

**使用建议：**

*   如果觉得默认流程跟你的项目不完全匹配，**随时按自己的需求改** —— 这个 skill 设计得很灵活
    

*   不要只停留在 "觉得这个思路不错" 的层面，**真正把它跑起来**，你才能体会到复利带来的加速感
    

**最后想说的：**

AI 自动化 Coding 不是 "让 AI 帮你写一次代码" 那么简单，而是**让 AI 每次都比上一次写得更好**。

而实现这一点的关键，就是今天聊的这件事 —— **复利工程**。

希望大家都能真正用起来。
