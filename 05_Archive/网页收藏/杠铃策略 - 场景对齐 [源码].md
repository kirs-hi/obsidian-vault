在 AI 自动化 coding 流程里，最难的不是让 AI 写代码，而是让 AI 写对东西。

一个成熟的自动化流程里，Harness、Agents、固定 Workflow、Builder / Validator 反馈回路、复利工程都可以逐渐沉淀下来。它们决定的是 “AI 如何做”：目录怎么建、代码怎么写、测试怎么跑、实现规范怎么遵守、项目经验怎么回写到 `AGENTS.md` 和 codebase 文档里。

（可以看之前的视频和源码：[https://articles.zsxq.com/id_xjpq4lwk8zee.html](https://articles.zsxq.com/id_xjpq4lwk8zee.html) ）

但在所有自动化开始之前，还有一个更关键的问题：

**这次到底要做什么？用户真实的场景是什么？哪些行为必须被系统支持？哪些结果需要被保存下来？**

这就是杠铃策略要审核的第一端：场景对齐。

`scenario-alignment` 这个 Skill，就是杠铃策略里的 “场景对齐” 部分。它负责在技术方案和自动化 coding 之前，把用户的大段口述、需求文档、bug 描述和页面调整，压缩成一份可审核的业务 PRD 对齐稿。

## 什么是杠铃策略

杠铃策略的核心，是不要把人类审核时间浪费在所有细节上。

以前做 plan 审核时，AI 往往会一次性输出很长的方案：用户场景、页面调整、接口设计、数据库表、组件拆分、实现步骤、测试计划全都混在一起。文档看起来完整，但问题是人类很难快速审完。一个不复杂的需求，也可能需要 40 分钟到 1 小时才能看清楚。

杠铃策略把审核重点拆成两端：

*   第一端：场景对齐。用户旅程是什么？角色是谁？入口在哪里？用户要完成哪些动作？权限、可见性、历史记录、生成、审核、保存、下载规则是什么？
    

*   第二端：技术对齐。用户行为最终如何持久化？表、Redis、OSS、外部接口、权限系统、状态机和兼容策略怎么承接？
    

中间大量 “怎么写代码” 的细节，尽量交给已经沉淀好的 Harness 和自动化 Workflow。

这样人类不需要审核所有实现细节，只需要握住杠铃两端：先审业务场景，再审关键技术合同。两端清楚后，自动化 coding 才有稳定输入。

当前这篇文章讲的是第一端：`scenario-alignment` 如何完成场景对齐。

技术对齐部分还没有完全写好，后续会由另一个 Skill 或流程承接。

## 它解决什么问题

`scenario-alignment` 解决的是 AI 自动化前最容易失真的问题：AI 是否真的理解用户要解决的场景。

用户给出的需求经常不是标准 PRD，而是一段口述、一堆页面链接、一些抱怨、几个 bug、几个按钮改名、再加上临时想到的权限和流程规则。比如用户可能会同时说：

如果直接让 AI 开始做，它会把这些内容拆成开发任务，但很可能没有先确认业务口径。结果就是代码写了很多，最后发现最上游理解错了。

`scenario-alignment` 会先做几件事：

*   查当前代码和 harness，确认现有系统已经有什么，目标行为和当前行为差在哪里。
    

*   把生成、审核、保存、下载、权限、可见性、历史记录这些规则写进业务场景。
    

*   只提出少量真正会改变产品方向的问题，并给出推荐答案。
    

它的产物不是技术方案，而是一份可以让人快速拍板的业务 PRD 对齐稿。

## 为什么它是自动化 coding 的前置条件

AI 自动化 coding 的后半段可以越来越自动化。Builder 负责实现，Validator 负责反馈，Harness 负责规范，复利工程负责把经验重新沉淀回项目。

但 “要做什么” 不能完全交给自动化自己猜。

`scenario-alignment` 的价值，是把这类问题在 coding 前拦住。它把 “我觉得 AI 大概懂了” 变成一份可审核文档，让用户和 AI 在同一张业务地图上。

## 它如何工作

这个 Skill 的工作方式很克制。它不会一上来写长篇技术方案，也不会把能查到的东西都丢给用户问。

它会按这个顺序处理：

*   1.
    
    **读取项目约束**：先看根级 `AGENTS.md`，再按项目规则读取架构文档、接口边界和相关子项目 codebase 文档。
    

*   2.
    
    **抽取业务意图**：不复述用户原文，而是提炼用户真正想达成的结果，以及涉及的角色、入口、对象和状态。
    

*   3.
    
    **查证代码事实**：只查和当前需求有关的页面、接口、服务、状态和数据结构。能从代码里判断的，不再问用户。
    

*   4.
    
    **沿用户旅程整理场景**：先整理上游决策，再整理成功路径、失败路径、权限、可见性、历史数据和例外状态。
    

*   5.
    
    **提出关键问题**：如果仍有会改变验收口径的问题，最多问 1-3 个，并给出推荐答案。
    

*   6.
    
    **输出业务 PRD 对齐稿**：文档只写业务口径，不写数据库字段、接口设计、组件拆分和任务顺序。
    

这个过程的重点不是 “多问问题”，而是 “减少没必要的问题”。能通过代码、文档和用户材料判断的，就直接查证；只有需要用户拍板的地方，才提出问题。

## 输出是什么

`scenario-alignment` 的标准输出是一份业务 PRD 对齐稿，结构通常是：

```
# [功能名] 业务 PRD 对齐稿

## 目标

- 一句话说明用户要达成的业务结果

## 范围

- 包含：本次明确要做的内容
- 不包含：本阶段不处理的内容

## 业务场景

场景 1：场景标题

- 角色 / 入口 / 触发动作
- 目标行为、当前问题或要调整的内容
- 权限、可见性、历史记录、生成 / 保存 / 下载规则

## 验收口径

- 用户可观察到的业务结果

## 需要对齐的问题

1. 会改变产品方向的问题

推荐：推荐答案



```

这里有一个重要原则：它不写技术实现。

它不会写数据库怎么建、接口字段是什么、组件怎么拆、任务怎么排。因为这些属于杠铃策略的另一端，应该在场景确认后进入技术对齐。

## 怎么使用

最简单的用法，是用 `align:` 开头，把需求文档路径或口述需求交给 agent。

```
align: docs/需求文档/skills市场的一些问题和调整.md



```

也可以直接粘贴需求：

```
align:

家长登录后要选择自己是家长还是某个学生。

如果选择学生，只能看到这个学生自己的课程包、错题本和学习报告。



```

如果需求涉及页面，最好一起给出入口：

textjavascripttypescriptcsshtmlbashjsonmarkdownpythonjavaccpprubygorustphpsqlyaml Copy

```
align: docs/需求文档/xxx.md

相关页面：

- /skills/create?subdomain=teacher
- /biz/skill
- /vibe
- /document-upload?student=AIBA-2026-013



```

输出后，如果 `需要对齐的问题` 里还有问题，用户只需要确认这些关键问题即可。确认后，再让 agent 把推荐口径合并进文档并保存。
